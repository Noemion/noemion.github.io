项目/常见问题

常见问题

先回答能否使用、解决什么问题和怎样开始，再解释目标制品、安全边界与未来实现。

当前可用性开发者入口安全边界

开始之前

现在可以安装或试用吗？

目前只能阅读和审查规范草案、实验格式、威胁模型、架构决策与验证方案。项目还没有正式版本、有效安装命令、在线运行服务或可执行组件；任何第三方同名软件都不能代表 Noemion 当前实现。

为什么人工智能时代需要 Noemion？

Agent 已能根据自然语言委托工具、转交任务并改变外部系统，但人的目标仍常被留在临时对话、提示词或某次模型解释中。Noemion 研究怎样把自然语言中的期望终态形成具有明确来源、稳定身份、判据和未决内容的工程制品，让后续模型、协议与运行可以引用同一个目标。

目标制品只固定“要达到什么”，不替主体获得动作权限。使用前仍要核对行动者、对象、动作、目的、范围与截止点；运行后再依据有范围证据判断是否满足，并由具名权威作出最终决定。

Noemion 是自然语言写代码工具吗？

不是。代码生成关注产生哪段代码，Noemion 关注目标怎样跨模型、协议、工具与运行保持同一、可检查且不可被另一份内容冒充。代码、计划和工具调用都只是实现目标的候选手段，不能替代目标、证据或最终决定。当前也没有编译器或代码生成组件。

第一次接触项目应该先读什么？

先读入门指南的依赖升级案例，分清目标、外部任务状态、运行权限、证据和最终决定。遇到具体工程问题时，再用规范与查询定位权威条款；不必先记住全部项目术语。

项目与目标制品

Noemion 与 Endem 有什么区别？

Noemion 是项目与研究方向的总名；Endem 是项目定义的目标制品，表示一个可以独立判断的最小期望终态单元。名称由 end 与表示最小区别单位的构词成分 -eme 组合而来。它是否适合作为正式发行名称，还要经过目标语言读音、口头区分和专业商标检索。

Endem 是传统目标文件的新名字吗？

不是。传统目标文件主要承载机器代码、符号和链接信息；Endem 描述目标的自然语言来源、已确认意义、期望事态、目标方向、满足判据和待确认事项。项目借鉴目标文件的确定性、显式引用、独立读取与裁剪发布纪律，但不继承机器指令本体。

Endem 包含什么？

来源保留的形成版分开六项职责：来源表达、已确认意义、一个根可能事态、目标方向、满足判据和待确认事项。现行设计阶段字段是 source_expression、meaning_projection、situation、goal_direction、satisfaction_criteria 与 unresolved_meaning；这些普通英语字段已经按词首、职责和关键字语料接受。

最终发布版只保留经过确认的目标结构与所需来源绑定，不把提示词或整段原始自然语言直接包装进发布制品。

为什么最终发布版会移除原始自然语言？

原始自然语言用于形成、复核和解释，就像调试资料帮助理解制品怎样产生；运行消费者只需要经过确认的目标结构。形成版因此保留原文与形成关系，发布时再按独立 Profile 移除原文和可逆重建材料，并为裁剪后的内容形成新身份。

这项裁剪只说明发布制品不再携带原文，不证明原始资料、备份或伴随记录已经删除。来源保留、访问权限、披露和删除仍由各自的数据路径与政策负责。

什么时候需要 closure、contract 和 evidence？

多个 Endem 需要共同解析依赖、激活条件或发布范围时使用 closure；一次受限运行需要固定对象、政策、环境、能力和预算时建立 contract；需要说明哪些观察支持哪项结论时形成 evidence。三者分别处理组合、会话和证据，不能互相替代。

安全、模型与决定

供人工智能系统安全使用的目标制品，是否意味着文件本身已经获准执行？

不意味着。发布制品用稳定内容身份和确定结构保护目标不被静默替换，但它不会给自己签发工具调用、修改、部署或跨会话权限。控制平面仍要核对请求主体、实际行动者、被代表主体、对象、动作、目的、范围和截止点；本次能力只取授权决定、contract、环境与预算的交集。

运行完成后还要分开证据有效性、对判据的覆盖度、满足结果与最终决定。签名、模型置信度、工具成功或外部 completed 状态都只能支持自己的声明范围。

模型能直接生成规范 Endem 吗？

模型可以提出带来源与待确认状态的候选，但不能自行形成规范内容。未来 producer 只消费已经具备精确语义授权绑定的输入，并按固定 Profile 确定性写入；它不替具名权威选择意义，也不授予动作权限。当前模型接入和 producer 都还没有组件实现。

模型能调用工具或宣告完成吗？

模型可以提出能力请求、工具参数和候选结果；实际调用由控制平面依据授权、contract、协议状态和当前预算决定。handoff 改变对话控制权，作为工具调用的 Agent 则保持在管理者的受限能力内，两种机制都不能隐藏实际行动者或扩大权限。

NIST AI Agent Standards Initiative把身份、安全和互操作列为 Agent 标准化重点；OpenAI Agent 编排说明也把 handoff 与 agents-as-tools 分开。这些外部机制只提供运行事实，目标是否满足仍由 satisfaction_criteria 与适用证据判断。

自然语言候选由谁确认？

具名语义权威或版本化确定性规则只能在精确政策、来源、对象与范围内确认候选意义。模型置信度、结构合法、签名、登录或单次点击不能替代这项关系。

意义确认也不是动作授权。工具调用还要单独形成 allowed / denied / pending，并核对委托、同意、多人规则、撤销、时效与重放。文件中的权威名称只是待核对的选择器，不能证明该权威已经针对当前内容作出有效决定。

确定性路径会调用模型吗？

不会。未来的形成、解码、绑定、布局、摘要范围和规则型检查必须能在无模型环境复现。模型参与的生成、解释或评测留在候选与有范围记录中，不进入确定性结果路径。当前还没有这些实现。

开放性、服务与实现

开放权重、开源模型和自由软件是一回事吗？

不是。开放权重只说明参数可以取得；OSI Open Source AI Definition 1.0还要求修改系统所需的数据说明、训练与运行代码及参数。GNU 自由软件定义则关注用户运行、研究、修改与再分发软件的自由。两者都不能由公开仓库、模型卡或一个许可证标签自动证明。

Noemion 当前没有模型权重、训练数据或模型服务。未来若发布相关材料，必须逐项说明代码、参数、数据说明、数据、文档、许可、限制与复现范围。完整对象边界见模型开放性与软件自由研究。

托管模型、设备内模型和自主管理服务怎样区分？

按实际执行者、控制平面和数据路径区分。设备内执行只说明某个组件在当前设备运行；自主管理服务要求用户或组织能控制软件负载、模型、配置、密钥、数据路径和停启；第三方托管执行则把实际计算交给服务运营者。

GNU 对他人服务替代用户计算的分析说明，服务器软件自由并不会让服务用户控制正在运行的实例。Noemion 因而把软件权利、实际服务控制、数据外发、保留、导出与退出分别记录；源码公开、AGPL、数据驻留或 API 兼容都不能单独概括整条服务路径。进一步案例见托管人工智能服务与用户控制研究。

为什么当前使用一个命令命名空间？

形成、检查、组合、查看和运行属于同一开发者工作流，当前用 endem 组织这些动作名称。它只让公开词汇保持一致，不确定最终需要几个 CLI、可执行文件、进程或发行包。未来 inspector 必须使用独立读取实现，runner 必须在隔离权限域中运行；解析代码、形成器内部状态和实时能力句柄都不能跨越边界。当前还没有可执行程序。

计划中的首个可实现范围是什么？

如果项目进入实现，首个范围只包含 form、lint、独立 inspect 和必需的一致性验证。组合、签名集成与受限运行要在前一范围产生安全、复现和独立验证证据后再评估。

项目也不预设自建模型训练平台。数据、训练、评测和量化优先适配成熟外部栈；只有出现 Noemion 特有且不可替代的需求，并有增益实验支持时，才研究新的适配面。

名称何时适合正式发行？

普通名称先确认是一个完整单词，再证明职责明确、与相邻概念没有冲突，并检查词首和关键字语料。只有自造名称还要完成首次朗读、听写回填和成对混淆；真实包名、命令、域名及目标法域权利则按发行时的范围复核。术语简化决定列出现行名称、候选读法和使用限制；设计阶段名称不自动表示软件已经发布。
