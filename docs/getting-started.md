## 先把一句要求写完整

软件工程长期把程序员写的形式代码编译给机器。人工智能系统现在可以接收自然语言、调用工具并改变外部系统，但人的目标仍常常留在临时对话和模型解释中。

假设团队要求 Agent：“把服务依赖更新到安全版本，并确认它可以发布。”这句话适合开始协作，却不足以决定修改和验收。开发者至少还要确认下面五件事。

| 必须确认 | 在这个案例中回答什么 | 缺失时怎样处理 |
| --- | --- | --- |
| 对象 | 哪个服务、仓库、分支和依赖 | 保留为待确认问题，不让模型自行选择 |
| 目标变化 | 当前版本、目标范围和允许修改的文件 | 不生成可使用的目标制品 |
| 行动范围 | 是否允许联网、提交、发布或修改锁文件 | 不授予对应能力 |
| 验收判据 | 在什么环境运行哪些构建、测试和安全检查 | 不能把工具成功写成目标满足 |
| 决定权威 | 谁在什么截止点决定可以发布 | 保持待决定，不由 Agent 自行接受 |

Noemion 研究怎样把这些已经确认的内容形成可检查的目标制品。含来源的形成版保留原始表达、已确认意义和待确认问题；最终发布版按独立 Profile 移除原始自然语言，产生新的内容身份，只保留经过确认的目标结构与必要来源绑定。这里的“编译”不是自然语言生成代码，而是让目标、约束、判据和责任进入一条可复核的形成路径。

最小目标制品的现行名称是 **Endem**。它只回答“一项目标究竟要求什么”，不保存运行权限或最终决定。当前没有可安装编译器或组件，开发者不能把下面的设计职责当作已经可调用的接口。

## 沿一条责任链工作

| 工作阶段 | 必须保留什么 | 不能直接推出什么 |
| --- | --- | --- |
| 人提交自然语言要求 | 实际表达、来源和未确认内容 | 模型解释就是人的确定目标 |
| 形成 Endem | 精确对象、目标变化、约束、判据和待确认意义 | 已获工具权限或工作已经完成 |
| 建立一次会话契约（session contract） | 精确目标版本、当前政策、环境、能力交集、预算和观察责任 | Agent 获得长期或跨会话权限 |
| MCP 或 A2A 处理外部请求 | 协议与版本、对端、任务状态、消息和候选产物 | 外部任务显示 `completed` 就等于目标 `met` |
| 形成有范围证据（evidence entry） | 精确主体、方法、环境、时间、观察和限制 | 记录天然有效、覆盖充分或结论为真 |
| 判断并作出决定 | `met / unmet / undetermined / fault` 满足结果，以及具名权威的 `accepted / rejected / deferred` | 一个“成功”同时代替运行、满足和接受 |

截至 2026-07-16，[NIST AI Agent Standards Initiative](https://www.nist.gov/artificial-intelligence/ai-agent-standards-initiative)仍把产业标准、开放协议、身份基础设施和安全评价作为不同工作轴。[A2A 1.0](https://a2a-protocol.org/v1.0.0/specification/)分别表示 Task、Message、Artifact 和任务状态；[MCP 2025-11-25 的实验 Tasks](https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks)则正在由[官方 Tasks 扩展提案](https://modelcontextprotocol.io/seps/2663-tasks-extension)替代，两者不具备线格式兼容性。这些变化说明协议状态和目标身份必须分开；它们不定义 Noemion 的字段或结果映射。

## “安全使用”必须逐层检查

这里的“安全”不是文件自带的布尔属性，也不表示文件能够自行授权执行。开发者按以下顺序核对；后一步不能弥补前一步的失败。

| 检查点 | 开发者核对什么 | 检查失败后怎样处理 |
| --- | --- | --- |
| 形成与发布 | 形成版是否保留来源、已确认解释和待确认项；发布 Profile 是否移除原文、产生新内容身份 | 无法说明解释或裁剪关系时，不形成可使用的发布制品 |
| 内容完整性 | 实际字节、内容摘要、外部陈述主体、验证政策和截止点是否匹配 | 把内容或陈述视为不适用，不能因文件可读取而继续 |
| 意义确认 | 具名权威是否针对精确来源、候选和语义位置确认了解释 | 不把模型候选写成确定意义；意义确认也不授予动作权限 |
| 动作授权 | 当前主体、对象、动作、目的、范围和截止点是否得到授权，实际能力是否仍在会话上限内 | 拒绝或延后动作；已经运行时停止扩大影响并保留事实 |
| 证据与满足 | 有效记录是否覆盖判据要求的主体、关系、时间和观察责任 | 覆盖不足返回 `undetermined`，观察故障返回 `fault`；工具成功不能改写为 `met` |
| 最终决定 | 具名权威是否依据满足结果、证据限制和决定规则作出结论 | 依据或权限不足时保持 `deferred` |

[SLSA 1.2 制品验证](https://slsa.dev/spec/v1.2/verifying-artifacts)帮助核对签名、主体和来源，[RFC 9334](https://www.rfc-editor.org/rfc/rfc9334.html)把 Evidence、Attestation Results 与依赖方决定分开，[GNU Guix 的 `guix challenge`](https://guix.gnu.org/manual/en/html_node/Invoking-guix-challenge.html)比较独立构建结果。这些机制支持完整性、来源和独立比较，却都不替开发者确认目标意义、动作授权或最终接受。

## 相邻系统继续负责什么

- Prompt、Skill 和模型上下文继续承载来源表达、资料和运行指令；保存为文件不会自动取得目标制品身份。
- MCP、A2A、HTTP 和工作流系统继续交换请求、状态、消息和产物；适配过程必须保留版本、来源和信息损失。
- 身份、政策、凭据与同意系统继续决定谁能代表谁执行什么动作；意义确认不能扩张为行动能力。
- 模型训练、评测、遥测、构建和发布继续由成熟工具承担；Noemion 只在目标、证据和决定发生耦合的边界提出约束。

Noemion 增加的是目标责任层，不是新的通用 Agent 平台、模型服务、工作流引擎或身份系统。

## 当前可以验证什么

项目已经发布架构说明、规范草案、实验性 Endem 字节格式和机器可读测试资料。资料检查可以发现引用、登记和预期分类之间的矛盾，但不能证明解析器、隔离运行、互操作或安全实现已经存在。

目前尚未实现 deterministic producer、independent inspector、bounded runner 与 `endem` CLI。未来首个实现范围只计划验证确定性形成、生产侧分层检查、独立读取和逐字节复现；组合、协议适配和受限运行必须等待前一范围的证据。

Endem、session contract、evidence entry 及动作名称仍是现行设计标识。具体发行拼写和读音仍需完成 ADR-0034 的人类验证；在独立参与者的首次朗读、听写和职责匹配结果形成前，项目不宣称这些名称已经适合传播，也不用临时读法冒充正式读法。读音提示将来只服务人类交流，不会成为第二套命令、机器别名或语义权威。
