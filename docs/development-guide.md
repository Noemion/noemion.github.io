## 先定义变更主张

不要从目录、框架或实现偏好开始。先写一句可被反例推翻的主张：在固定规范、Profile、配置和输入范围后，系统应产生什么结果，或者必须在哪一层拒绝。

| 要回答的问题 | 变更记录必须写清 | 不合格写法 |
| --- | --- | --- |
| 谁需要这项变化 | 真实消费者、输入、输出和停止条件 | “以后可能有用” |
| 什么具有约束力 | CORE 条款、Profile、现行 ADR 或外部版本化规范 | “现有示例就是接口” |
| 失败由谁处理 | 生产层、失败层、稳定诊断和关闭方式 | “失败后自动重试” |
| 什么证据足够 | 支持案例、反例、正反向量、资源上限和人工责任 | “测试都通过了” |
| 最多能声称什么 | 规范一致、组件行为、互操作或研究结论中的一种 | “已经验证安全” |

例如，若要处理外部 A2A Task 的 `completed` 状态，应先把它定位为协议来源状态。开发者随后核对 ADP-CORE、DRO-CORE、IKN-CORE 与 AUT-CORE，并加入“不得直接映射为 `met` 或 `accepted`”的拒绝案例。当前没有适配器实现，因此最多只能声称规范和案例保持一致，不能声称协议接入已经完成。

把这项变更写成一页记录时，至少应得到下面这些可复核结果：

| 记录项 | 本例写法 |
| --- | --- |
| 可证伪主张 | 固定 A2A 版本与映射后，`completed` 保持外部来源；系统不得只凭该状态形成 `met` 或 `accepted` |
| 权威依据 | ADP-CORE 约束外部状态，DRO-CORE 约束会话上限，IKN-CORE 约束证据范围，AUT-CORE 约束决定权威 |
| 支持案例 | 外部状态、候选产物和来源身份被完整保存，后续判断分别形成 |
| 拒绝案例 | 缺少协议版本、对象身份或适用权威时，映射停止并返回所属失败层 |
| 当前证据 | 规范条款、场景和向量的一致性检查 |
| 声明上限 | 可以说明资料保持一致；不能说明适配器、互操作或安全实现已经完成 |

## 当前范围

第一阶段只做规范和最小安全核心，不引入模型、LLVM、MLIR、多格式抽象或大型框架。

ADR-0012 已把 Rust 1.97.0 选为未来 Ktisor 与制品形成侧读取核心的候选语言。当前没有组件实现；禁止 `unsafe`、第三方 crate、工具链锁、`Cargo.lock` 与 release 溢出检查都只是未来实现约束。C/Rust 语言材料只作历史研究依据，不进入生产代码。

当前没有 Ktisor、Theor、CLI 工作区或实现级 fuzz。END-P1 的允许与拒绝字节向量描述预期行为，不能证明解析器、写入器或独立读取路径存在；精确集合以向量源和版本化验证结果为准。未来实现遵循以下开发纪律：

- 锁定 `rhem/semion/skena/telis/krin/apor` 的最小语义、不变量、错误和资源限制。
- 保持规范写入器、制品形成侧结构检查与 `endem ktise/elenk` 的确定性边界。
- 继续让 `endem theor` 独立实现读取、字段与引用验证，禁止共享 Ktisor 形成侧解析代码。
- 规划合法、边界、畸形和对抗语料，以及未来长时公开 CI、故障注入和跨平台复现。

## 规范与 ADR 先行

- 格式、语义、执行、信任边界或应用职责变化必须先更新规范或记录 ADR。
- 开放问题不能因为实现方便而静默变成默认行为。
- 一个新制品或子命令必须说明不可替代价值、上游、下游、失败责任和停止条件。
- 正式接口只描述真实存在且通过验证条件的行为；示例不自动成为 ABI。

## 变更工作流

| 顺序 | 开发者要做什么 | 完成时应留下什么 |
| --- | --- | --- |
| 1 | 确认消费者、输入、输出、权限与失败责任 | 一句可证伪主张和明确停止条件 |
| 2 | 用[规范参考指南](specifications-reference.html)定位唯一条款 | 条款 ID、Profile、适用版本和缺失的权威依据 |
| 3 | 先写支持案例和反例 | 能区分正确解释与危险近似的案例对 |
| 4 | 把案例转成稳定诊断与正反向量 | 失败层、机器码、位置和预期结论 |
| 5 | 只改动承担该责任的资料层 | 规范、ADR、威胁、场景或说明页中的最小变化 |
| 6 | 按变更类型运行验证 | 实际执行结果，以及明确标出的未执行层次 |
| 7 | 写明声明上限并同步公开状态 | 证据能支持的最窄结论；未执行不能写成通过 |

被替代的入口应在同一变更中删除，不保留旧别名、重定向或兼容垫片。当前没有组件实现；进入代码开发阶段后，工作流才会增加最小实现变化和实现级证据。

## 把外部资料送进模型前保留来源与职责

开发者可能让模型阅读依赖库 README、Issue、网页、工具返回、历史记录或附件。这些内容默认都是资料。正文即使自称 `system` 或 `admin`，也不会因此取得指令权。

例如，开发者要求 Agent 调研一次依赖升级。依赖说明和 Issue 可以提供版本、许可与兼容性证据，却不能命令 Agent 上传令牌、执行脚本或忽略项目政策。摘要也只是带有损失的派生候选，不能代替原始资料。

| 装配前要固定 | 开发者必须保留 | 必须停止的情况 |
| --- | --- | --- |
| 输入与来源 | 精确内容、来源身份、取得时间和适用版本 | 无法确认实际送入模型的内容 |
| 用途与指令权 | 政策、请求、资料、工具结果、历史、候选或仅供显示 | 资料靠自称角色取得更高权威 |
| 变换与损失 | 解码、OCR、切块、检索、排序、摘要、去重和脱敏 | 变换后的内容冒充完整原文 |
| 预算与截断 | 模型、tokenizer、顺序、预算和每次删除 | 必需政策、否定条件或决定性证据将被静默删除 |
| 冲突处理 | 同一权威层的冲突、固定裁决规则和责任人 | 没有规则却按到达顺序或模型偏好选择 |
| 缓存边界 | 主体、租户、政策、会话、后端、模型、tokenizer 和截止点 | 跨边界复用旧装配结果 |
| 动作复核 | 模型输出的候选身份，以及动作发生前的精确授权检查 | 模型看到高权威文字就获得工具能力 |
| 不可观察变换 | 可观察请求、供应商声明和无法核对的内部步骤 | 把未知内部处理写成完整上下文证明 |

模型返回工具请求后，控制平面仍要按当前 AUT-CORE 与 Dromen 上限重新检查对象、动作、目的、参数、预算和截止点。工具成功只形成带来源的观察；它不直接成为 `met`、`accepted` 或新的权限。

| 外部依据 | 支持哪项检查 | 不定义什么 |
| --- | --- | --- |
| [NIST 智能体劫持研究](https://www.nist.gov/news-events/news/2025/01/technical-blog-strengthening-ai-agent-hijacking-evaluations) | 分开可信内部指令与不可信外部数据 | Noemion 的消息角色、对象或字段 |
| [MCP 工具规范](https://modelcontextprotocol.io/specification/2025-11-25/server/tools) | 把工具注解视为不可信，并保留拒绝调用的能力 | 本地满足、证据或最终决定 |
| [OpenAI MCP 风险说明](https://developers.openai.com/api/docs/mcp#prompt-injection-related-risks) | 信任工具提供者不等于信任其可访问的全部内容 | Noemion 的权威层级或授权结果 |
| [GNU ld 手册](https://sourceware.org/binutils/docs/ld.html) | 用 map file 和 dependency file 记录输入、输出与依赖关系 | 模型输入是链接对象，或供应商消息角色具有规范权威 |

完整案例、反例与威胁归属见[模型上下文装配边界研究提案](https://github.com/Noemion/noemion.github.io/blob/main/spec/model-context-assembly-proposal.md)。该资料仍是非规范研究，不创建上下文制品、网关、运行时或新的 CORE。

## 建议仓库边界

```text
spec/       权威条款、registry 与错误码
vectors/    规范字节、注释、合法与畸形向量
ktisor/      Rust 写入器、形成侧解析器、ktise/elenk/pleko
theor/      独立只读 Theor，不依赖 Ktisor 形成侧解析器
cli/        endem 调度与独立进程边界
```

没有独立版本、权限、保密或真实消费者时不拆仓库。`theor` 只能共享公开规范与向量，不能共享生成常量、Ktisor 形成侧解析状态机或错误分类实现；`drase` 建立后必须是不同进程并使用最小权限。

## 审查清单

| 层 | 必查项 |
| --- | --- |
| 结构 | magic、版本、偏移、长度、计数、索引、对齐、重叠、环、上限 |
| 语义 | 一个根 skena、rhem/semion 投影、关系角色、kine/mene、krin、apor/agno/aseme 与权威 |
| 组合 | 引用身份、版本、依赖闭包、冲突矩阵、权限交集、顺序无关 |
| 发布 | 版本化裁剪保留关系、调试伴随记录、主体摘要、外部签名、软件物料清单和复现结果 |
| 运行 | 实际字节重验、能力最小化、真实观察、事件完整、验收分层 |

## 模型与协议

模型加入前必须先建立无模型基线。模型输出只作为不可信候选进入系统；MCP 和 A2A 只通过 Drasor 外缘的协议适配器接入。外部参数结构合法不等于语义正确或权限允许。若模型无法在错误确定率、上下文成本或任务成功率上证明净增益，项目就停止自研模型平台投入。

完整完成标准见[开发路线图](../development/implementation-roadmap.html)，变更类型与最低证据见[测试与验证](../development/testing.html)。
