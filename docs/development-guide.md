## 第一阶段范围

第一阶段只做规范和安全二进制核心，不引入模型、LLVM、MLIR 或大型框架。

- 冻结最小 NIR/NOBJ 结构和安全解析规则。
- 实现受检算术（checked arithmetic）、安全读取器、对象写入器、验证器和确定性序列化。
- 建立畸形样例集、基准样例集、模糊测试和跨工具一致性检查。
- 在规范允许的范围内建立对象查看、验证、复制和尺寸分析工具。

## 规范与 ADR 先行

- 实现变更必须先更新或引用规范。
- 格式、类型系统、执行语义和信任边界变更必须记录 ADR。
- 开放问题不能因为实现方便而静默变成默认行为。
- 候选 CLI、扩展名和 ABI 只能作为待验证设计展示。

规范阅读入口见[规范参考](specifications-reference.html)。

## 实现工作流

1. 确认问题、目标使用者、范围、现有技术和验收依据。
2. 确认需求属于背景说明、规范、组件、工具还是开放问题，并建立规范或 ADR 入口。
3. 补充失败用例，证明当前设计或实现确实缺少目标行为。
4. 实现最小确定性变更，不夹带模型或大型依赖。
5. 运行结构、语义、畸形样例、基准样例、模糊测试、复现和浏览器检查。
6. 同步 README 路由、项目规则、相关规范、ADR、测试结果和内容审查清单。

源码仓库建立后，这些检查应进入持续集成。当前还没有可以引用的正式源码仓库或 CI 地址。

## 测试与模糊测试

| 层次 | 重点 |
| --- | --- |
| 结构 | 魔数、版本、边界、计数、索引、对齐和重叠区域 |
| 语义 | 类型、约束、歧义、权限、覆盖和验收契约 |
| 链接 | 符号、版本、归档、ID 重映射、重定位和依赖闭包 |
| 安全 | 签名、Merkle、策略、AI-NX、AI-RELRO 和拒绝路径 |
| 复现 | 相同输入、配置和锁文件产生一致结果 |

完整策略见[测试、模糊测试与验证](../development/testing.html)。

## 贡献与报告

公开贡献渠道、安全报告邮箱和版本治理尚未启用。在正式渠道公布前，请不要向非官方账号发送漏洞细节、对象样本、私钥或其他敏感资料。

- 设计意见应指出涉及的权威定义、成熟度和具体冲突。
- 安全问题应保留最小复现、受影响对象类型、攻击前提和预期拒绝路径。
- 规范建议应区分兼容性变化、语义变化、行为澄清和纯编辑修正。
- 研究报告应包含假设、基线、数据来源、版本、实验条件、失败切片和复现材料。
- 知识产权候选应记录贡献人、现有技术、创新点边界和公开披露状态，不在普通问题讨论中提前披露未审查方案。
- 任何报告都不得包含真实密钥、签名私钥或敏感对象数据。

**尚待确定：**贡献治理、版本策略、安全披露和知识产权审查渠道将在项目基础设施确定后公布。

## 后续开发顺序

**后续计划：**安全对象核心完成实现与验证后，依次推进确定性编译与链接、发布与 Fulfillment Runtime，再进入 Horizon Engine 数据、训练、评测和量化。只有独立实验确认现有底座无法满足需求时，才考虑从头预训练 Horizon Engine 语言底座。

论文、专利、软著与标准化产物应随真实证据形成：研究结论来自可复现实验，专利候选先经过检索与权属审查，软著基于真实软件版本，标准提案基于稳定规范和互操作测试。
