## 直接结论

Noemion 已经用直白职责名称替代了大部分不必要的造词，并建立版本化关键字检查。项目只保留 `Noemion` 与 `Endem` 两个造词；它们的发行读音仍是待定内容。名称必须经过真实人类朗读和听辨；项目自己写出的 IPA、拼读提示、语音合成或自动转写都不能证明名称已经通过。

### 当前资料的阅读约定

面向开发者的入口先写职责，再用代码样式标出现行设计字段。直白解释不是机器别名；研究提案中的英文候选也不能写入来源清单、Profile 或接口。

| 先理解的职责 | 现行设计字段 | 必须保持的区分 |
| --- | --- | --- |
| 来源表达 | `source_expression` | 来源实际出现的记号不同于经确认的意义投影 |
| 已确认意义投影 | `meaning_projection` | 模型候选不同于规则或范围有限具名权威确认的投影；意义确认不授予运行权限 |
| 可能事态 | `situation` | 中性事态不同于目标方向和实际观察 |
| 目标方向 | `goal_direction` | 达到或保持不同于事态本身和满足判断 |
| 满足判据 | `satisfaction_criteria` | 比较方法不同于观察、证据记录和最终决定 |
| 待确认意义 | `unresolved_meaning` | 语义未决不同于观察不足、结构无意义或运行故障 |

这里先用“意义确认”帮助开发者区分职责；AUT-CORE 把精确的外部关系称为语义授权。它和动作授权不是同一件事：前者只允许候选进入确认的 `meaning_projection`，后者才决定某个主体能否针对某个对象尝试某个动作。读音研究必须让参与者能够分别恢复这两项职责，不能把同一个“已授权”标签当作答案提示。

## 读音待定时怎样协作

没有合格读音表时，口头表达只能帮助参与者找到书面标识，不能充当标识本身。开发者在会议、Issue、代码评审和运行手册中遵守以下约定：

1. 先说直白职责，再给出代码样式的精确标识。例如写“有范围的证据记录（现行设计名 `evidence entry`）”，不要只写一个尚未验证读音的专名。
2. 会议中若听者不能立即写回标识，就把精确拼写发到共享文字频道；必要时逐字母确认。团队不得把临时读法、个人口音或一次顺利听写登记成官方读音。
3. 命令、字段、Profile 和规范 ID 必须从书面材料复制或核对。口述的近似声音不能选择命令、修改参数、扩大权限或成为评审记录中的唯一引用。
4. 自动转写只保存一次机器观察。记录者必须对照书面标识确认后再引用；原始转写不能直接改写 schema、命令、授权范围或研究结果。

| 协作任务 | 推荐表达 | 不足以执行或引用的表达 |
| --- | --- | --- |
| 讨论证据对象 | 有范围的证据记录（现行设计名 `evidence entry`） | “那个听起来像 evidence entry 的东西” |
| 讨论会话边界 | 一次会话的只读执行契约（现行设计名 `session contract`） | 只说未经验证的读音 |
| 核对检查动作 | 在文字频道给出精确命令 `endem inspect` | 依靠会议录音猜测动作名或参数 |

Noemion 当前没有语音界面。未来若研究口述动作，界面仍必须显示将要使用的精确机器动作、对象、参数和权限范围，并由适用权威确认；只有声音或转写结果不能触发执行。

读音验证之前还必须先问“是否需要专名”。如果标准术语或简短职责短语已经能准确恢复对象、消费者和失败责任，继续创造新词只会增加记忆、听写和职责匹配负担。[发行术语去专名化研究提案](https://noemion.github.io/spec/release-terminology-simplification-proposal.html)提供了对象、角色和动作的反例；[ADR-0037](../architecture/adr-0037-terminology-simplification.html)已经据此采用直白职责名。

[语义面与观察词去专名化提案](https://noemion.github.io/spec/semantic-facet-terminology-proposal.html)把审查继续应用到 `source_expression/meaning_projection/situation/goal_direction/satisfaction_criteria/unresolved_meaning/structured_observation`。这七项名称已经进入 END-CORE、END-P2、来源清单、向量和测试；提案继续作为迁移证据，不再代表现行词表。

[生命周期与结果词边界提案](https://noemion.github.io/spec/lifecycle-and-result-terminology-proposal.html)把对象边界修正和命名审查分开。当前规范已采用 `formed / resolved`、`undetermined / no_allowed_projection` 与会话终止词 `stopped`，并把外部陈述、验证与依赖方判断移出内容状态。读音流畅不能保住语义错误的名称，普通词也仍须在完整域限定短语中验证。

桌面语义审查和关键字检查已经支持把 `reach/maintain` 作为当前 `goal_direction` 枚举；它们分别表示在判断点成立和在声明时间范围内持续成立，不额外声称本次动作造成了转变。[`goal_direction` 发行词研究](https://noemion.github.io/spec/goal_direction-release-terms-proposal.html)保存了候选比较。两项拼写仍须完成人类口头验证，不能据此宣称发行读音已经通过。

验证分成两个阶段：先让未接触项目的参与者暴露自然读法，再用另一批参与者检验候选读音能否在完整词表中被唯一听出、写回并匹配正确职责。两个阶段不能使用同一批人。

## 证据适用边界

- [ISO 704:2022](https://www.iso.org/standard/79077.html)连接对象、概念、定义和指称，并给出术语与专名形成的一般原则；它不替 Noemion 选择读音。
- [ISO 9241-11:2018](https://www.iso.org/standard/63500.html)把可用性视为具体使用情境中的结果；因此不能只看拼写推断口头可用性。该标准也明确不规定一种唯一评估方法。
- [RFC 5646 / BCP 47](https://www.rfc-editor.org/info/rfc5646/)定义语言标签。项目用它记录测试语言范围，不把 `en`、`en-US` 或 `zh-CN` 混成一个总体。
- [W3C Pronunciation Lexicon Specification 1.0](https://www.w3.org/TR/pronunciation-lexicon/)说明书写形式与读音需要显式映射，并要求支持 IPA。这里借用表示方法，不建立语音组件或运行词典。
- [ITU-T P.800](https://www.itu.int/rec/T-REC-P.800/)与 [ITU-T P.808](https://www.itu.int/rec/T-REC-P.808-202106-I/en)提供主观听测、材料、参与者、环境、可靠性检查和报告方法。它们评估语音传输质量，不直接给出软件名称的通过阈值。
- [NISTIR 7778](https://doi.org/10.6028/NIST.IR.7778)在已发布的指纹图像压缩研究中说明 rule of three：零次观察错误不等于真实错误率为零。Noemion 只借用这个二项比例边界，不借用该报告的生物识别任务、样本设计或结论。
- [NIST Privacy Framework 1.0](https://www.nist.gov/privacy-framework/privacy-framework)把收集、变换、使用、披露、保留与处置视为不同的数据动作。[NIST Research Data Framework 2.0](https://doi.org/10.6028/NIST.SP.1500-18r2)进一步按研究数据生命周期组织责任。它们支持先写数据计划，不替代适用的同意、伦理审查或法律义务。
- [NIST SP 800-88 Rev. 2](https://doi.org/10.6028/NIST.SP.800-88r2)要求清除方法匹配介质与风险。[GNU Coreutils `shred` 手册](https://www.gnu.org/software/coreutils/manual/html_node/shred-invocation.html)也明确指出快照、备份、镜像、写时复制文件系统与 SSD 会破坏“覆盖一次即可删除”的假设。这里借用的是验证边界，不要求研究者使用某个命令。

## 招募前先固定研究数据边界

读音研究会产生录音、转写、语言背景、拼写和自由反馈。它们不是一份可以统一称作“研究结果”的数据，也不会因为参与者同意听测就自动获准用于训练模型、公开语料或长期保存。

研究负责人在招募前先完成一份数据清单。清单逐项说明收集目的、实际处理、接收方、保留截止点、删除范围和仍不可观察的副本。缺少其中任何一项时，不收集对应数据。

| 数据对象 | 只为哪项判断收集 | 默认处理边界 | 结束时要回答 |
| --- | --- | --- | --- |
| 同意与联系记录 | 证明参与范围、补偿和撤回路径 | 与研究响应分库存放；使用独立参与者代号 | 谁可以访问，法定或机构保留要求是什么 |
| 语言、地区或口音自述 | 解释结果适用于哪个人群 | 只收集分析必需的粒度；小样本组合视为再识别风险 | 哪些字段进入聚合结果，哪些按截止点删除 |
| 首次朗读录音 | 发现自然断词、重音与近音 | 只在转写无法回答问题时保留；限制访问并记录每次外发 | 原始音频、工作副本、平台副本和备份分别怎样处置 |
| 听写、拼写与职责选择 | 形成混淆矩阵和职责匹配结果 | 绑定参与者代号，不保存账户标识或无关设备信息 | 哪些原始响应必须保留以复核决定 |
| 自动转写与模型探测输出 | 发现机器识别风险 | 与人类结果分开；固定供应商、模型、版本、请求和已知处理边界 | 供应商是否保留、训练、转交或生成派生数据 |
| 自由反馈 | 解释失败模式 | 提醒参与者不要提交身份、秘密或第三方资料；发布前去标识 | 原文是否必要，摘要损失怎样记录 |
| 聚合混淆矩阵与决定 | 支持保留、改名或重测 | 只发布足以复核结论且通过再识别检查的汇总 | 哪些未覆盖语言、设备和环境必须随结论披露 |

把录音送入外部 ASR、云存储或众包平台会增加新的实际处理者。研究负责人必须分别核对传输、区域、日志、人工访问、训练使用、保留、删除接口、下游服务和不可观察范围。供应商写“零保留”或返回删除成功，只能支持其声明范围内的事实，不能证明备份、镜像和派生物已经不可恢复。

删除计划还要区分逻辑删除、平台删除请求、介质清除和可披露的删除确认。删除方法必须匹配实际存储；本地命令成功不能覆盖云端副本，平台收据也不能覆盖研究者导出的文件。完整判断方法见[软件 Agent 数据使用、保留与删除边界研究提案](https://noemion.github.io/spec/software-agent-data-use-retention-and-deletion-boundaries-proposal.html)。

研究数据计划不是新的 Noemion 制品或接口。它只让负责人在采集前知道需要什么、为什么需要、谁能处理，以及研究结束后怎样证明已完成适用处置。

## 两阶段验证

### 桌面排除不等于人类通过

权威词典中的既有无关词义、无法从名称恢复职责、明显的工程概念冲突或高风险近音，可以在招募参与者前排除候选。排除只说明该拼写不值得进入发行听测；替代词仍必须完成以下两阶段验证。

当前 `goal_direction` 候选还要额外检查语义诱导：听者不能把 `reach` 理解成必须证明一次状态转变，也不能把 `maintain` 理解成一次采样或“没有看到失败”即可证明持续满足。

### 发现阶段

首轮在 `zh-CN` 和一个明确登记的英语变体中分别招募至少 24 名从未接触 Noemion 的参与者。参与者应能阅读拉丁字母技术词，但不能先看到项目读音提示。

每名参与者完成以下任务：

1. 看到一个名称和一句直白职责，首次朗读并说明自然断词与重音。
2. 写下自己认为最自然的普通拼读提示。
3. 指出难读位置、可能省略的辅音、可能插入的元音和近似现有词。
4. 在完整词表中标出看起来或读起来最容易混淆的另一项。

24 人是本项目的发现预算，不是统计通过阈值。结果用于形成读音候选、风险对和改名候选，不能产生“已验证”结论。

### 发行审查阶段

候选读音形成后，必须换用没有参加发现阶段的独立听者。每个目标语言、每个名称和每个测试条件至少收集 60 个来自不同参与者的首次判断；同一参与者不能为同一名称重复贡献判断。

测试至少包含清晰录音和普通远程会议音频两种条件。每个名称由至少两名自然说话者录制，听者只听其中一个随机版本。项目不得用 TTS 取代自然说话者。

若 60 个独立判断中没有关键混淆，rule of three 只能支持“真实关键混淆率的单侧 95% 上界约为 5%”这一有限主张。它不证明所有人、口音、设备和环境下都不会出错。

## 任务与材料

材料不能只比较新造词之间的发音。每组测试都应让参与者在不知道项目偏好的情况下完成对象选择、职责匹配和反例判断。当前优先比较 Noemion、Endem 与 Endem closure，并验证 session contract、evidence entry、deterministic producer、independent inspector、bounded runner 和 `endem form/lint/compose/inspect/run`，以及 `source_expression/meaning_projection/situation/goal_direction/satisfaction_criteria/unresolved_meaning/structured_observation` 的完整语句。语义面组还必须使用来源与投影、事态与观察、方向与判据、待确认意义与观察不足的成对反例。

每个听者先听完整语句，再完成三项相互独立的回答：

1. 从完整当前词表和“无法判断”中选择听到的名称。
2. 不复制选项，手动写回拉丁字母拼写。
3. 从打乱顺序的直白职责中选择该名称负责什么。

语句必须覆盖名称的真实语境。例如动作名放在“运行 endem …”语境，对象名放在“这个记录/契约/闭包…”语境。不能只播放孤立音节，也不能在句中泄露正确答案。

材料顺序、说话者和选项顺序必须随机。每个短任务只包含 5 至 15 个刺激，避免疲劳；不同参与者使用平衡区组覆盖全部词表。正式试验前只用非 Noemion 填充词检查设备和操作理解，不能训练目标名称。

## 通过与停止规则

以下阈值是 Noemion 的发行风险政策，不是外部标准规定：

- 任何名称在任一目标语言或条件中出现一次“误选成另一个现行名称”或“匹配到另一个职责”，当前候选就不能通过。维护者必须分析原因，优先重新命名，再使用全新参与者复测。
- 手写拼写错误单独记录。若错误形成稳定替代拼写，或只能依赖额外字母解释才能恢复，名称进入重新命名审查。
- “无法判断”不是错误名称，却证明传播证据不足；当前候选不能通过。
- 第一版发行研究至少覆盖 `zh-CN` 和明确登记的英语变体。新增主要受众、词表成员、读音、职责或语音交互条件后，相关结果失效并重新测试。
- 不允许通过增加机器别名、兼容发音或隐藏纠错表来保留失败名称。设计阶段不考虑向后兼容，改名应一次性同步全部现行资料。

这个规则故意对关键混淆采用零容忍，因为把动作、对象或证据职责听成另一项可能改变执行与信任边界。普通口音差异可以保留，只要它们仍能唯一指向同一拼写和职责。

## 证据记录

每次研究必须保存以下最小记录：

| 记录 | 必须包含 |
| --- | --- |
| 研究语境 | 研究 ID、日期、负责人、提交标识、当前词表版本、目的和停止规则 |
| 参与者范围 | BCP 47 语言标签、地区或口音自述、技术词阅读经验、是否首次接触项目 |
| 刺激 | 精确书写、候选 IPA、普通拼读、直白职责、语句、说话者范围、条件和音频摘要 |
| 随机化 | 区组、顺序、说话者、选项顺序和可复核随机种子 |
| 原始响应 | 名称选择、手写拼写、职责选择、无法判断、首次朗读转写和自由反馈 |
| 分类 | 关键混淆、稳定替代拼写、允许口音变体、设备失败、无效参与和排除理由 |
| 结果 | 分语言、分条件样本数与混淆矩阵，二项边界、未覆盖总体、决定和复核者 |
| 数据清单 | 每类原始数据、派生数据、接收方、用途、访问者、保留截止点和适用政策 |
| 删除与保留记录 | 逻辑删除、平台请求、备份范围、清除方法、确认、例外和仍不可观察的副本 |

参与者标识必须假名化。公开结果默认只保存聚合数据、刺激摘要和决定依据；原始音频、声音生物特征与自由文本需要单独同意、访问限制和删除期限。若结果用于公开研究而不只是内部可用性审查，还要另行判断适用的伦理审查与数据治理责任。

## 人工智能只做辅助探针

多个 TTS、ASR 和语音模型可以在正式人类测试前发现异常断词、近音和转写失败。每次自动探测必须绑定模型、版本、语言、提示、音频和实际输出；不能只保存“通过率”。

模型结果不能进入人类样本数，不能决定 IPA，不能解释参与者意图，也不能把一次正确转写升级为名称通过。未来若出现语音动作入口，识别结果仍必须安全显示精确机器动作、对象、参数和权限请求，再由适用权威确认。

## 当前状态

截至 2026-07-18，Noemion 尚未执行上述人类研究，也没有合格的发行读音表。当前迁移已经把对象、角色、动作和七个语义面/观察词改为直白职责名称，并用版本化语料排除了已知关键字冲突；完整结果见[术语审查结果](terminology-audit.html)。`Noemion` 与 `Endem` 仍没有人类读音证据，其他普通词在新增主要使用语言或语音交互条件时也必须重新检查口头区分。

在真实结果形成前，公开资料继续先写直白职责，再使用设计阶段名称；不得发布“官方发音”、音频词典或名称已经适合全球传播的结论。
