## 当前可用性

> 项目尚未进入代码开发阶段。当前没有 `endem` 实现、安装包、运行命令、MIME 登记、稳定 CLI 参数或 ABI。

资源状态以[下载与资源](../downloads/)为准。正式制品出现时，安装入口必须同时说明版本、平台、大小、摘要、签名、SBOM、许可证和支持范围。

## 未来职责流程

当前参数和 Synem 接口尚未冻结。以下顺序只描述未来组件的职责关系，不表示对应实现已经存在：

1. 作者提交受控来源表达；模型只能协助提出意义、事态、方向、判据和未决项候选，不能把候选写成规范事实。
2. `ktise` 依据确定性规则与具名语义决定写入 Endem。
3. `elenk` 直接读取实际字节并分层验证；`theor` 以独立实现提供只读视图。
4. 有真实多目标消费者时，`pleko` 才解析引用并形成 Synem。
5. 未来只有在版本化保留关系与正反向量冻结后，对应制品生产者才能执行允许的派生变换；当前没有裁剪等价规范，不能证明保留时必须保留内容或失败。外部签名集成只核对精确请求与响应，私钥不进入 Ktisor。
6. `drase` 在隔离进程重新验证精确制品，并按 DRO-CORE 从政策、环境、能力、预算和证据责任封存 Dromen。独立最小能力域持有实时句柄，运行过程形成 Iknem，具名权威作出最终验收决定。

## 发布原则

- 开发快照与正式版本明确区分，不共享含糊的“最新版”入口。
- 依赖内容寻址并锁定来源；构建固定 locale、时区、路径和构建纪元。
- 候选版本公开前完成双环境复现、安全/模糊测试、许可证核对、SBOM 和跨 Theor 一致性。
- 下载后先验证摘要和签名，再解析制品；签名有效不意味着当前运行策略允许其能力。
- 撤回、漏洞响应、密钥轮换、支持周期和长期归档必须在稳定发布前建立。

## 命名发布条件

Endem 只通过了有日期的精确工程名初筛；它仍有第三方同名对象，也没有通过商标、近音、读音和口头区分门禁。正式发行前必须按[术语与读音验证指南](terminology-and-pronunciation.html)完成人类研究，并在目标司法辖区和类别完成专业权利复核。任一门禁失败都应在 ABI 冻结前通过 ADR 一次性改名，不保留旧命令、格式别名或兼容发音。
