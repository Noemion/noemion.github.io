## 当前可用性

> 当前没有 `endem` 实现、安装包、运行命令、MIME 登记、稳定 CLI 参数或 ABI。

因此，现在不要执行网上出现的同名 `endem` 命令，也不要根据本文编写安装脚本。资源状态以[下载与资源](../downloads/)为准；该页出现可验证制品以前，Noemion 只提供规范、研究记录和开发者指南。

判断一个未来资源是否真的可用时，按下表逐项检查，不要用一个“开放”标签代替不同证据：

| 要回答的问题 | 足够的证据 | 不能据此推定 |
| --- | --- | --- |
| 软件能否自由运行、研究、修改和再分发？ | 对应软件副本的许可，以及可供研究和修改的源码；参见 [GNU 自由软件定义](https://www.gnu.org/philosophy/free-sw.html) | 模型参数、训练数据和托管实例自动具有相同自由 |
| AI 系统的哪些部分开放？ | 对架构、参数、推理代码、训练代码、数据说明、数据和文档逐项列出可访问性与条款；参见 [Open Source AI Definition 1.0](https://opensource.org/ai/open-source-ai-definition) | “API 可调用”“开放权重”或“源码可见”等于完整开放 |
| 用户是否控制实际执行？ | 明确执行位置、操作者、模型与配置、数据外发、保留、下游服务、状态导出和退出路径；GNU 也将[程序副本与网络服务活动](https://www.gnu.org/philosophy/network-services-arent-free-or-nonfree.html)分开讨论 | 源码公开、AGPL、数据驻留或 API 兼容等于服务实例控制 |
| 结果能否复现？ | 锁定版本、输入、环境和随机性，并由独立方重建和比较结果 | 模型卡、签名、SBOM 或许可证本身证明行为可复现 |

当前仓库的 Apache License 2.0 只覆盖实际置于该许可下的源码与文档，不自动覆盖未来模型参数、训练数据、第三方依赖、托管服务或商标。更完整的对象边界见[模型开放性与软件自由研究](https://github.com/Noemion/noemion.github.io/blob/main/spec/model-openness-and-software-freedom-boundaries-proposal.md)，执行与迁移边界见[托管人工智能服务与用户控制研究](https://github.com/Noemion/noemion.github.io/blob/main/spec/hosted-ai-service-and-user-control-boundaries-proposal.md)。两份材料均为非规范研究，不代表已有模型或服务。

## 未来职责流程

当前参数和 Synem 接口尚待确定。未来实现只有按以下顺序留下可核对的输入与输出，才算贯通职责；这不是现有命令清单：

| 步骤 | 输入与动作 | 可交付结果 |
| --- | --- | --- |
| 1. 形成候选 | 作者提交受控来源表达；模型可提出意义、事态、方向、判据和待确认项 | 带来源的候选，不是规范事实 |
| 2. 确定写入 | `ktise` 按确定性规则和具名语义决定形成 Endem | 可重复生成的精确制品 |
| 3. 双路径检查 | `elenk` 读取实际字节并分层验证；`theor` 以独立实现生成只读视图 | 分开的验证结果与检查视图 |
| 4. 组合闭包 | 真实多目标消费者需要组合时，`pleko` 解析引用并形成 Synem | 依赖明确、能力不扩大的组合闭包 |
| 5. 封存运行上限 | `drase` 重新验证精确制品，并按政策、环境、能力、预算和证据责任封存 Dromen | 一个会话可做什么的不可变上限 |
| 6. 执行与决定 | 最小能力域持有实时句柄，运行过程形成 Iknem，具名权威审查证据 | 分开的运行结果、目标判断和最终接受决定 |

ADR-0036 已确定形成版保留原始自然语言，最终发布版移除原文并取得新身份；但发布 Profile 与裁剪实现尚不存在。只有来源引用闭包、版本化保留关系、披露边界和正反向量明确后，制品生产者才能执行派生变换。外部签名集成只核对精确请求与响应，私钥不进入 Ktisor。

## 发布原则

正式版本只有同时提供以下信息，才能进入安装说明：

| 发布证据 | 开发者应能完成的检查 |
| --- | --- |
| 版本、平台、大小、摘要和签名 | 确认下载对象与发布记录完全一致，再解析制品 |
| SBOM、逐对象许可证和来源锁定 | 查明每个依赖与模型相关对象的来源、条款和首选修改形式 |
| 固定 locale、时区、路径和构建纪元 | 在两个独立环境复现并比较输出 |
| 安全测试、模糊测试和跨 Theor 一致性 | 区分结构有效、检查一致与策略允许，不把签名有效当成运行授权 |
| 支持范围、撤回、漏洞响应、密钥轮换和归档 | 判断升级、停用和长期验证路径是否完整 |

开发快照与正式版本使用不同入口，不共享含糊的“最新版”。模型卡、签名、SBOM 或外部开放等级只能证明各自覆盖的事实，不能替代逐对象结论。

## 命名发布条件

Endem 只通过了有日期的精确工程名初筛；它仍有第三方同名对象，也没有通过商标、近音、读音和口头区分审查。正式发行前必须按[术语与读音验证指南](terminology-and-pronunciation.html)完成人类研究，并在目标司法辖区和类别完成专业权利复核。任一审查失败都应在 ABI 锁定前通过 ADR 一次性改名，不保留旧命令、格式别名或兼容发音。
