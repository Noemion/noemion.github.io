Noemion 入门指南

从项目为什么存在开始，建立 GSIR、GOBJ、SSO、确定性编译核心与可信 Runtime 的共同理解。

Getting StartedDesign StageNo Release Yet

从这里开始

已确认原则：Noemion 不是把 Prompt、文本或 Skill 包换一种二进制编码，而是为生成式计算建立可机器分析、验证、重定位、链接和执行的目标与约束对象体系。

它试图解决的不是“如何写出更好的单次提示”，而是目标、约束、证据、歧义、依赖和权限如何成为可检查、可组合、可复现的长期工程对象。如果第一次接触项目，先阅读项目背景与边界，再从本页进入对象、架构和工具内容。

核心对象

GSIR

表达目标、解空间、硬约束、软偏好、歧义、推断权限、证据要求和验收条件。

查看 GSIR 状态

GOBJ

组织可重定位语义对象、符号、类型化引用、完整性信息和链接所需记录。

查看 GOBJ 状态

SSO

以依赖闭包和分层披露图承载可共享语义能力，不等同于压缩提示词。

查看 SSO 状态

信任与确定性

  • Strict 模式无模型、确定且可复现。
  • 只有确定性的 Compiler Core 能生成 GSIR/GOBJ。
  • 模型输出和所有对象输入均视为不可信。
  • 歧义是一等信息；宁可保留未决语义，也不能制造错误确定性。
  • 偏移、长度、计数、索引和对齐计算必须使用 checked arithmetic。

完整生命周期见对象生命周期与信任边界。

推荐阅读路径

  1. 从项目背景理解问题、非目标和可证伪命题。
  2. 阅读架构指南建立端到端系统图和信任边界。
  3. 进入规范参考区分已确认原则、提案和开放问题。
  4. 通过工具参考理解各阶段工具职责。
  5. 准备参与实现、研究或标准化时阅读开发指南及其中的证据和治理要求。

研究者应继续追踪假设、基线、实验和失败案例；实现者应追踪规范、ADR、测试和版本；标准评审者应追踪术语、规范性条款、Profile 与互操作证据。

当前项目状态

设计阶段：当前重点是规范与安全二进制核心，尚未发布可执行编译器、Runtime 或正式安装包。网站中的工具名称、能力边界和流程用于设计评审，不表示已有稳定程序。

证据边界：现有页面能够证明项目已经形成较系统的问题、架构和工具职责设计，但不能证明性能、互操作性、跨模型等价或产业价值。最终对象命名、CLI、扩展名、ABI、重定位编号和模型工程细节仍需规范、ADR、原型与实验冻结。

下一步

未来阶段：规范、安全 Reader/Writer、验证器和确定性 Compiler Core 达到放行门后，项目才会进入可安装工具和 Runtime 的发布准备。

继续阅读架构指南了解对象如何编译、链接、装载和执行。查看使用与获取状态了解当前能获取什么，以及正式发布前不能做什么。
