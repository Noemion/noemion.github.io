项目/开发与贡献/测试与验证

测试与验证

先限定主张，再选择证据。测试只有在对象、失败条件和适用范围明确时，才能支持可复核的工程结论。

主张范围失败条件证据类型

用一次协议映射变更走完整条证据链

假设开发者准备把 A2A Task 的 completed 映射为 Noemion 的 met。这项修改看似只是枚举转换，实际会把远端协议状态提升为本地满足判断。正确做法不是先增加一条通过案例，而是依次固定主张、权威、反例、证据和声明范围。

顺序开发者要留下什么本例的可复核结果
1. 写主张一个会被反例推翻的句子固定 A2A 版本和对端后，completed 只映射为外部 Task 终态，不直接产生 met
2. 定权威本地规范、外部版本和适配责任ADP-CORE 保存外部状态；END-CORE 与 krin 决定满足判断；A2A 只定义自己的协议对象
3. 找反例结构合法但会击穿越级映射的输入远端 Task 已完成，但 Artifact 指向错误版本，或本地安全检查返回拒绝
4. 选证据能够区分成功、拒绝、故障和来源的材料分别覆盖状态保留、版本冲突、产物错绑、取消竞态和本地判据结果
5. 限声明测试通过后最多可以说什么资料与已登记案例保持结果域分离；不能声称适配器已经实现或所有 A2A 对端均可互操作

术语、格式、授权、模型评测和网站解释也遵循这一顺序：先说明测试在回答哪个问题，再讨论测试数量。

变更类型决定证据组合

选择原则：从可能出错的责任层开始。每类变更都需要支持案例和失败案例，但具体证据由对象决定，不能用另一类测试替代。

变更对象需要回答的问题最低证据组合结论上限
公开资料与网站解释是否符合现行规范，页面是否可读、可达并保持状态边界权威链接、资料一致性检查、生产构建、真实浏览器的多视口与交互检查资料和页面在已检查范围内一致；不能证明组件已经实现
语义与结果边界同一输入是否得到规定分类，结果域是否被错误合并规范条款、支持案例、反例、正反向量、稳定失败位置和结果域拒绝当前草案对已登记输入给出一致解释
Profile 与物理格式合法字节能否读取，畸形或超限输入能否确定失败合法与畸形语料、checked arithmetic、字节/节点/深度/内存/时间上限、版本绑定指定 Profile 在已执行实现和配置中符合已测要求；不是普遍安全或稳定 ABI 证明
确定性形成与读取器封闭输入能否复现，独立实现是否给出相同解释精确输入闭包、隔离环境、逐字节比较、两个不共享解析代码的读取器和差异报告已固定范围内的字节复现或互操作；相同输出不证明设计正确
运行器与协议适配器取消、重试、超时、越权和部分副作用是否停在责任边界状态机向量、能力收缩、故障注入、竞态、资源耗尽、未知结果和隔离检查只支持实际执行过的组件、协议版本、平台和故障模型
模型参与的研究测量是否适合目标总体，更新是否可追溯，上线后是否漂移用途与总体、协议、基线、数据资格、不确定性、失败分析、派生身份和运行监测限定实验或监测窗口中的结果；不能替代规范符合性或具名权威决定

不知道从哪份资料开始时，可用规范参考指南的工程问题索引定位权威起点。当前尚未进入组件代码开发阶段，表中的实现级证据是未来组件工作的最低约束，不是已有实现声明。

当前材料与未来证据分开

当前状态：Noemion 目前可以验证规范资料、登记、场景、向量和公开网站之间的一致性。尚无 Ktisor、Theor、Drasor、求值器或决定引擎可供实现级测试。

材料现在能够验证仍然缺少
机器可读条款登记与规范检查规范版本、条款、威胁、验证方式和向量集合的连接关系解析器、运行器或决定引擎的执行结果
自然语言场景支持案例、反例、歧义和责任遗漏规范符合性、性能、安全隔离或互操作结论
语义、专题与字节向量当前草案对已登记输入的允许包络、拒绝结果和失败位置未来组件是否真正消费这些向量并稳定执行
资料一致性、构建与浏览器检查条款、页面、链接、生成站点及目标视口中的阅读和交互表现组件级模糊测试、故障注入、跨平台复现、签名或 SBOM 证据
未来实现符合性材料尚不可验证独立读取器、畸形语料、性质测试、资源边界、隔离环境和发布制品

各向量集合的数量和执行结果以机器可读登记与对应测试输出为准，不在说明页复制容易漂移的计数。资料构建成功只能证明其检查范围，不能被写成组件实现进展。

证据不是一条从弱到强的阶梯

证据类型回答不同问题。数量更多或成本更高的测试，不会自动覆盖另一种证据所处理的风险。

证据类型它回答什么不能替代什么
示例与场景候选含义和责任分工是否可以被具体输入说明不能替代规范条款、反例或实现测试
回归测试已知行为是否被当前变更破坏不能发现所有未知输入，也不能证明安全不变量
性质、安全与对抗测试不变量、资源边界和拒绝路径能否经受生成或恶意输入不能证明两个独立实现采用同一解释
互操作与独立复现固定格式、版本和配置后，独立路径是否产生一致解释或字节不能证明规范目标正确、输入主张为真或系统安全
研究实验与运行监测模型或系统在限定总体、协议和时间窗口中的表现及变化不能替代格式符合性、动作授权或发布决定

一项完整变更可以同时需要多种证据，但报告必须分别写出每种证据的对象、失败条件、未覆盖范围和结论上限。

模型评测、模型更新与上线监测分开

三类活动分别测量能力、改变模型和观察真实运行。共用一组分数或日志会掩盖数据、身份、总体和决定权威的变化。

活动必须固定失败时停止的主张研究入口
模型或 Agent 评测用途、目标总体、任务与采样、提示、模型身份、评分协议、基线和统计不确定性基准分数不能外推为普遍能力、可靠性或发布资格NIST AI 800-3（2026 年 2 月发布）与 NIST AI 800-2 初始公开草案
训练与模型更新数据和反馈资格、基础模型、处理与训练活动、随机状态、派生身份、复现和行为比较训练完成、损失下降或固定种子不能证明改进、安全、可复现或获准发布NIST SP 800-218A（最终版）与模型训练与更新边界研究提案
上线后的运行监测部署版本、实际总体、时间窗口、漂移指标、事件、处置政策和观察缺口离线基准不能覆盖环境变化、交互反馈和低频事故NIST 部署后 AI 监测研究
发布或接受决定适用政策、证据范围、风险容限、具名决定者和决定时间评测者、训练流水线或监测系统不能自行产生 accepted模型参与评测与裁判边界研究提案

两份 Noemion 提案当前均为非规范研究，不创建评测、裁判、训练或模型 CORE，也不证明训练平台、模型组件或运行监测系统已经存在。

比较之前先固定比较域

比较问题可采用的工程纪律结论边界
程序的自测与安装后检查是否通过？GNU 标准目标用 check 检查已构建但未安装的程序，用 installcheck 检查安装状态。两类检查针对不同状态；通过不等于覆盖所有平台、输入和安全风险。
两份独立构建是否逐字节相同？GNU Guix challenge比较独立来源的构建结果；GNU Diffutils区分相同、不同和比较故障。相同字节不证明设计正确；出现差异也不能自行选择可信一方。
规范形式是否保留所需信息？GNU BFD canonical form提示显式处理源格式、目标格式和不可表示信息；RFC 8785为限定 JSON 输入域定义确定表示。规范化或可哈希表示不证明来源、业务语义、权威或跨 Profile 等价。
形成结果能否独立复现？固定解码输入、授权决定、配置、依赖、规范、Profile、工具版本和环境，再分开报告 Endem 字节、工具二进制与发布签名。必须说明比较的是原始字节、规范对象还是派生视图；当前没有通用 round-trip 或跨 Profile 语义等价要求。

这些外部机制只提供测试与比较纪律，不定义 Noemion 的字段、结果域、Profile 或实现状态。任何未执行的层次都必须明确标为尚未验证。
