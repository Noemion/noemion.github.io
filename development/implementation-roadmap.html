开发原则

一个原语

先证明一个 Endem 能稳定表示一个根终态，不为架构对称提前增加格式。

一个入口

公开应用只有 endem；子命令按消费者组织，信任域用独立实现和进程隔离。

失败优先

先定义畸形、冲突、越权、超限和未决输入怎样关闭失败，再实现成功路径。

模型最后

模型只提交候选；无模型路径必须能够形成、检查、组合、装载和验收规则型案例。

代码阶段之前与未来第一条路径

直接结论：当前只完成第 0 阶段的规范与验证方案，不创建实现工作区。未来若用户明确开启代码阶段，第一实现只考虑 ktise、elenk 与独立 theor；一致性验证作为内部门禁，不扩大公开 CLI。

第 0 阶段已经产生 END-CORE、END-FMT、END-P1、SYN-CORE、DRO-CORE、IKN-CORE、DIA-CORE、ADP-CORE、ID-CORE、TEXT-IDENTIFIER-CORE、AUT-CORE、威胁模型和规范向量。ADR-0012 把 Rust 记为未来首版候选语言；ADR-0021 至 ADR-0033 固定组合、证据、内容分层、会话、诊断、适配、身份、文本、授权决定、Endem 内容与授权伴随关系、发布名称门禁及文本标准 ID。这些资料只用于评审设计和形成未来验收条件，不表示任何组件已经开始开发。

阶段、验证条件与停止条件

阶段只交付什么进入下一阶段的验证条件停止或合并条件
第 0 阶段
词汇与规范骨架		Endem、Synem、Dromen、Iknem；六个语义面、一个根 skena、形式显示、三类未知、条款 ID 与正反向量关系角色、投影、满足状态与错误有可复查定义；每项规范要求都有测试或明确人工权威删除不能说明消费者或失败责任的名词
第 1 阶段
最小安全核心		ktise/elenk/theor、规范写入器、生产解析器、独立 Theor 和一致性门禁Ktisor/Theor 差分一致、恶意语料关闭失败、资源有界、双环境逐字节复现若最小字段仍不能表达真实任务，回到规范；符合性检查保持内部责任
第 2 阶段
组合		pleko、引用解析、冲突矩阵、固定依赖与 Synem至少两个真实多 Endem 案例；链接顺序不改变规范结果；权限只能收窄没有真实复用消费者时不建设归档、动态链接或共享披露文件
第 3 阶段
发布		生产者物理封装、调试伴随记录、外部签名请求与响应版本化保留关系、闭包完整、签名主体精确、可复现发行与 SBOM没有真实消费者或无法冻结保留属性、损失和反例时，不执行删除并停止建设通用变换
第 4 阶段
受控运行		drase、隔离 Drasor、Dromen、Iknem 和规则型后端实际字节重验、能力最小化、事件完整、验收权威分离、恢复与升级路径若单一 Iknem 无法覆盖不同权威，再按权限或生命周期拆分
第 5 阶段
模型与协议适配		模型候选封装、MCP/A2A 适配器和版本化遥测导出器相对无模型基线，在错误确定率、上下文成本或任务成功率上证明净增益没有不可替代增益时停止模型训练平台建设，继续使用外部成熟工具

从 GNU 工具链借什么

来源采用不照搬
ELF / 链接器结构视图与装载视图分离、显式引用、类型化绑定、闭包、绑定映射和失败即拒绝地址重定位、机器指令、弱符号默认选择、环境搜索和延迟绑定
GNU readelftheor 必须独立于生产解析库，使一个实现的错误不会污染两条证据链为每种查看方式创建一个稳定二进制
GNU BFD认识统一抽象的价值和信息损失风险第一阶段不建立多格式通用表示层；先把唯一格式做对
GNU GCC 诊断稳定诊断内容与文本、SARIF 等输出出口分开，保留规则与位置不复制编译器严重度、参数或把自由消息当作机器协议
GNU 发行实践固定区域设置、时区、路径与构建纪元；两环境复现；调试信息分离并绑定摘要把构建工具本身当作复现证明

GNU readelf 手册明确说明它独立于 BFD；GNU BFD 手册同时说明跨格式 canonical 转换可能丢失信息。这里借鉴的是验证与职责思想，不是命令清单。Section/Segment、符号、重定位、形成映射、裁剪、调试分离和 Build ID 的逐项边界见非规范 GNU 与 ELF 适用性研究提案；规范化、裁剪和迁移为什么不能自动成为语义等价，见非规范语义等价与迁移研究提案。

面向 Agent 时代的适配边界

完整的跨层映射集中在Agent 系统边界图。路线图只保留等待决定的研究入口和外部版本基线。

  • 候选输入与工作路径分别见上下文装配研究和计划与重规划研究。外部数据不能取得指令权，计划也不能替代 Endem/Synem 目标。
  • 行动解释与人工决定分别见状态变化与因果研究和预览、模拟与批准研究。工具成功不能证明因果、满足或最终接受。
  • 跨会话状态与跨版本关系分别见记忆、检查点与恢复研究和语义等价与迁移研究。恢复不能复活权限，迁移不能继承身份与接受状态。
  • 能力声明、协议协商、授权、会话上限与即时可调用性见能力发现与协商研究。动态工具列表不能扩写旧 Dromen，端点可达和一次调用成功也不能成为权限、满足或接受。
  • 并行分支、推测候选与外部副作用提交见并行与推测提交研究。子 Agent 共享同一能力和预算上限；非幂等写入默认串行，提交前重验强前提，取消不等于回滚，分支完成顺序不产生提交权。
  • 模型、控制面、凭据、适配进程与外部副作用的部署责任见模型、适配器与能力域隔离研究。未来实现必须分别证明文件、网络、秘密、进程、资源、终止和观察限制；容器、seccomp、超时或提示词不能单独成为隔离证据，强制控制不可用时对应能力关闭失败。
  • 开放式输出的测量与模型评审见模型参与评测研究。未来评测必须固定目的、构念、评分说明、协议、模型版本、偏差探针、统计区间和使用权威；模型评分保持候选，多个模型投票和排行榜不产生满足或接受。
  • MCP 工具说明、注解和远端返回都只是 Drasor 外缘的不可信输入；能力授权来自 Dromen 与会话策略。截至 2026 年 7 月 13 日，当前研究基线是官方仍标为 Current 的 2025-11-25 工具规范。官方于 5 月 21 日锁定并公开、计划于 7 月 28 日转为最终版的 2026-07-28 候选版只用于评估版本漂移，不作为当前符合性基线。任何后续正式版本都必须重新完成兼容、安全、错误来源、降级和权限复核。
  • A2A 1.0 的 Task、Message 与 Artifact 适合跨 Agent 传输和记录生命周期，但不能替代 Endem 身份、Synem 闭包或 Iknem 验收。补丁号不进入协议协商；具体映射必须建立遵守 ADP-CORE 的独立 Profile。当前没有协议实现。
  • 可靠 Agent 依赖清晰环境、可读工具、真实观察和反馈循环；这些由 Drasor 及其外侧控制平面负责，不通过增加薄命令包装解决。实践案例见智能体控制平面工程实践。
  • OpenTelemetry GenAI 语义约定已迁入独立仓库，仍在演进且 Schema URL 待定，并可能携带输入、输出和工具数据。它只进入带版本、默认脱敏的导出器，不进入 Endem 编码、Iknem 身份或最终决定。
  • 所有外部协议版本只约束适配器，升级必须重新验证兼容性、安全边界和降级行为。

未来实现基线

当前不创建实现仓库或 Rust 工作区。用户以后明确开启代码阶段时，项目必须重新确认仓库位置，并按 ADR-0012 评估 Rust 1.97.0、Ktisor/Theor 独立实现、禁止 unsafe、依赖边界、工具链锁和 Cargo.lock；这些条件现在只是设计约束。
