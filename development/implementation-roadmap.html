开发路线图

路线图按依赖和证据组织：先让目标与失败语义可检查，再考虑组件、组合、运行和外部 Agent 协议。

方案设计阶段证据驱动尚未实现

四条长期原则

对象先于工具

先证明对象有独立消费者和失败责任，再决定是否需要格式、组件或公开动作。

失败先于成功

先写畸形、冲突、越权、超限和未知怎样关闭失败，再实现成功路径。

独立证据链

制品形成与只读解释使用不同实现；分歧必须进入调查，不能靠多数结果掩盖。

模型最后接入

模型只提出候选。确定性路径、授权、实际观察和最终决定必须在没有模型时仍能成立。

先看依赖顺序

直接结论：当前只有规范和验证材料，没有组件实现。后续工作不会按页面数量或功能清单推进；每一层只有在前一层留下可复核证据、并出现真实消费者后才有建设理由。

工作层要解决的问题进入下一层前应有的证据停止或合并条件
术语与规范目标对象、意义、事态、判据、未知和结果域是否能被不同读者一致定位权威条款、支持案例、反例、威胁、正反向量，以及独立人类读音结果普通工程词已经足够，或专名无法稳定朗读、听写和区分时，不保留专名
确定性形成与独立检查同一封闭输入能否形成同一结果，畸形输入能否有界拒绝两条独立读取路径、资源上限、故障注入、差分调查和双环境逐字节复现若真实案例暴露对象边界错误，返回规范；一致性验证不增加公开动作
多目标组合与发布多个可独立失败的目标怎样固定依赖、权限交集和发布范围真实多目标消费者、顺序无关结果、冲突反例、保留关系、签名主体和可复现发行单个 Endem 足够，或没有复用消费者时，省略组合与通用变换
受控运行与证据一次会话怎样限制对象、政策、环境、能力、预算和观察责任实际字节重验、能力最小化、终止与孤儿副作用测试、有范围证据和决定权威分离控制不可强制或观察责任无法闭合时，对应能力关闭失败
模型与外部协议候选、计划、工具调用和远端状态怎样接入而不取得本地语义与决定权固定协议版本、降级与取消反例、提示注入测试、数据路径、无模型基线和可测净增益成熟外部工具已经满足需求，或模型没有不可替代净增益时，不自建相应平台

GNU 技术提供哪些工程纪律

来源Noemion 借鉴的纪律采用限制
GNU readelf让独立读取路径直接检查实际字节，避免形成侧解析器的同一缺陷污染两条证据链不为每种视图创建一个公开程序，也不把查看结果写回制品
GNU BFD跨格式表示必须显式披露来源域、目标域和信息损失当前不建立多格式通用表示，也不把 canonical form 当作语义等价
GNU Guix challenge从独立来源取得产物并逐字节比较，同时保留差异供调查相同字节不证明设计正确；不同字节也不自动选择可信一方
GNU 文档规范让开发资料同时承担教程与参考职责，并按读者问题而不是实现文件组织不复制 GNU 的命令、格式或项目治理结构

Section、Segment、符号、重定位、裁剪和调试分离的逐项适用性见GNU 与 ELF 适用性研究。它是非规范研究，不定义 Noemion 格式或组件。

外部 Agent 技术只决定适配边界

NIST、MCP、A2A 和遥测规范说明 Agent 生态正在加强互操作、身份、授权和安全，但它们不替 Noemion 决定目标、证据范围或最终接受。完整映射集中在Agent 系统边界指南。

这些研究主题当前都不创建新 CORE、制品、命令或组件。某项责任已经能由现有规范承担时，路线图只把义务归还现有对象，不为研究主题再造专名。

当前限制条件

当前没有 Rust 组件、CLI、协议适配器、沙箱、签名 Profile、安装包或软件发行。开发者现在可以审查规范与验证设计，也可以按贡献指南提交事实、术语、案例和边界修订；不能把路线图当作已承诺日期或已实现功能清单。
