开发原则

一个原语

先证明一个 Endem 能稳定表示一个根终态，不为架构对称提前增加格式。

一个入口

公开应用只有 endem；子命令按消费者组织，信任域用独立实现和进程隔离。

失败优先

先定义畸形、冲突、越权、超限和未决输入怎样关闭失败，再实现成功路径。

模型最后

模型只提交候选；无模型路径必须能够形成、检查、组合、装载和验收规则型案例。

代码阶段之前与未来第一条路径

直接结论：当前只完成第 0 阶段的规范与验证方案，不创建实现工作区。未来若用户明确开启代码阶段，第一实现才考虑 poie、elenk、theor 与 peira；不同时创建工具家族，不接模型，不做通用二进制抽象层。

第 0 阶段已经产生 END-CORE、END-FMT、END-P0、错误目录和语义/结构向量。ADR-0012 把 Rust 记为未来首版候选语言，ADR-0013 以 END-P1 关闭首个字段、排序与引用接口。这些资料只用于评审设计和形成未来验收条件，不表示 Poiet、Theor、CLI 或安全核心已经开始开发。

阶段、验证条件与停止条件

阶段只交付什么进入下一阶段的验证条件停止或合并条件
第 0 阶段
词汇与规范骨架		Endem、Synem、Dromen、Tekmor；六个语义面、一个根 skena、形式显示、三类未知、条款 ID 与正反向量关系角色、投影、满足状态与错误有可复查定义；每项规范要求都有测试或明确人工权威删除不能说明消费者或失败责任的名词
第 1 阶段
最小安全核心		poie/elenk/theor/peira、规范写入器、生产解析器和独立 TheorPoiet/Theor 差分一致、恶意语料关闭失败、资源有界、双环境逐字节复现若最小字段仍不能表达真实任务，回到规范，不叠加补丁格式
第 2 阶段
组合		pleko、引用解析、冲突矩阵、固定依赖与 Synem至少两个真实多 Endem 案例；链接顺序不改变规范结果；权限只能收窄没有真实复用消费者时不建设归档、动态链接或共享披露文件
第 3 阶段
发布		tasse/sphra、调试伴随记录、外部签名请求与响应裁剪等价、闭包完整、签名主体精确、可复现发行与 SBOM通用变换、裁剪或归档没有消费者时，只保留为内部步骤
第 4 阶段
受控运行		praxe、隔离 Praxor、Dromen、Tekmor 和规则型后端实际字节重验、能力最小化、事件完整、验收权威分离、恢复与升级路径若单一 Tekmor 无法覆盖不同权威，再按权限或生命周期拆分
第 5 阶段
模型与协议适配		模型候选封装、MCP/A2A 适配器和版本化遥测导出器相对无模型基线，在错误确定率、上下文成本或任务成功率上证明净增益没有不可替代增益时停止模型训练平台建设，继续使用外部成熟工具

第 0 阶段当前产物

工作包已经形成仍需完成
P0-W1 条款源END-CORE、END-FMT、END-SRCM、43 个唯一条款 ID、END-P0/END-P1 与成熟度登记。代码阶段前继续用真实自然语言案例和反例修订；独立实现验证必须等待用户开启代码阶段。

P0-W3 规范向量6 个已执行语义向量；END-P0 含 1 个结构接受与 5 个拒绝；END-P1 含 2 个语义接受与 11 个拒绝。继续增加累计分配、偏移算术和组合引用边界向量。
P0-W4 历史语言研究C/Rust 双原型、6 个规范向量、144 个差分变异、C Sanitizer、10,000 次 libFuzzer 和两次重复构建的历史记录；ADR-0012 据此记录 Rust 1.97.0 基线。冻结为历史研究资料，不自动运行、不继续扩展，也不视为组件证据。
P1-W1 安全核心尚未开始；当前只有职责分离、候选语言、END-P1 规范向量和未来验证条件。等待用户明确开启代码开发阶段；届时重新确认仓库、范围、依赖、CI 与停止条件。

C/Rust 语言材料只保留为可删除的历史研究证据，不是生产解析器、组件源码或发行物。当前没有 Rust 安全核心工作区，也不得因为规范向量存在就自动开始实现。

从 GNU 工具链借什么

来源采用不照搬
ELF / 链接器结构视图与装载视图分离、显式引用、类型化绑定、闭包、绑定映射和失败即拒绝地址重定位、机器指令、弱符号默认选择、环境搜索和延迟绑定
GNU readelftheor 必须独立于生产解析库，使一个实现的错误不会污染两条证据链为每种查看方式创建一个稳定二进制
GNU BFD认识统一抽象的价值和信息损失风险第一阶段不建立多格式通用表示层；先把唯一格式做对
GNU 发行实践固定区域设置、时区、路径与构建纪元；两环境复现；调试信息分离并绑定摘要把构建工具本身当作复现证明

GNU readelf 手册明确说明它独立于 BFD；GNU BFD 手册同时说明跨格式 canonical 转换可能丢失信息。这里借鉴的是验证与职责思想，不是命令清单。

面向 Agent 时代的适配边界

  • MCP 工具说明、注解和远端返回都只是 Praxor 外缘的不可信输入；能力授权来自 Dromen 与会话策略。当前基线是 2025-11-25 稳定工具规范。2026-07-28 发布候选包含破坏性变化，正式发布并重验前不采用。
  • A2A 1.0 的 Task、Message 与 Artifact 适合跨 Agent 传输和记录生命周期，但不能替代 Endem 身份、Synem 闭包或 Tekmor 验收。页面固定参考 A2A 1.0 版本化规范；补丁号不进入协议协商。
  • 可靠 Agent 依赖清晰环境、可读工具、真实观察和反馈循环；这些由 Praxor 及其外侧控制平面负责，不通过增加薄命令包装解决。实践案例见智能体控制平面工程实践。
  • OpenTelemetry GenAI 语义约定仍处于 Development，并可能携带输入、输出和工具数据。它只进入带版本、默认脱敏的导出器，不进入 Endem 编码、Tekmor 身份或最终决定。
  • 所有外部协议版本只约束适配器，升级必须重新验证兼容性、安全边界和降级行为。

未来实现基线

当前不创建实现仓库或 Rust 工作区。用户以后明确开启代码阶段时，项目必须重新确认仓库位置，并按 ADR-0012 评估 Rust 1.97.0、Poiet/Theor 独立实现、禁止 unsafe、依赖边界、工具链锁和 Cargo.lock；这些条件现在只是设计约束。
