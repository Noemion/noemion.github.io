前端负责产生受限候选或解析受控源语言；Compiler Core 负责绑定、类型与约束检查、覆盖审计和确定性对象生成；链接与装载组件负责对象组合、验证和安全映射；Runtime 才负责生成式求解与 Lowering。

例如，一个“生成报告”的请求只有在目标受众、必需章节、禁止披露内容、证据来源和验收条件都进入受检语义后，才适合成为对象；Runtime 可以选择怎样形成报告，但不能删除硬约束或扩大访问权限。

已确认原则：编译边界

  • Strict 输入通过无模型路径进入确定性 Compiler Core。
  • Assisted 或 Generate-Strict 路径中的模型只能提交不可信候选或源码，不能直接产出 GSIR/GOBJ。
  • Compiler Core 保留歧义、检查类型与约束，并拒绝未覆盖或不可安全解释的输入。
  • 相同受控输入、工具版本和规范化环境应产生可复现的对象结果。

已确认原则：对象与装载边界

  • 一切对象输入均不可信；偏移、长度、计数、索引和对齐计算使用 checked arithmetic。
  • GOBJ 可保留待链接引用；最终执行对象必须满足必需依赖闭包和信任策略。
  • Loader 在映射前验证结构、边界、签名范围和所需完整性证明，验证失败时关闭失败。
  • 重定位完成后，关键符号绑定、入口、策略与权限应被冻结，避免装载后提权。

设计提案：对象视图

当前设计借鉴 ELF 的 Section/Segment 双视图：Section 服务链接、调试与对象工具，Segment 服务验证和装载。具体 Header、记录布局和签名结构仍须由正式格式规范与 ADR 固化。

设计提案：对象语言、元语言与行为边界

思想与方法基础提示我们区分被执行或评价的对象数据，以及描述其类型、验证规则和环境的元层。当前提议要求 GSIR/GOBJ 不得把 Profile、验证器规则或 Runtime 私有假设伪装成对象自身事实；跨层引用必须有显式版本和信任来源。

  • 如果目标要求询问、承诺、授权或执行，候选行为对象应携带行为类型、行动者、权限、前置条件、预期效果和失败语义，而不只是命题内容。
  • 来自字面内容、结构规则、语境推论、工具和模型的结论应保留不同来源等级；可撤销推论不能静默升级为硬约束。
  • 目标满足由明确的评价环境决定，环境包含变量绑定、证据版本、可用能力和 Runtime Profile；未登记背景知识不得参与确定性验证。

这些结构仍须用反例语料、竞争设计、ADR 和一致性测试证明，尚不是对象格式字段。
