前端负责解析受控源语言或封装不可信候选；Noesis Core 负责来源绑定、类型与约束检查、编译来源覆盖和确定性对象生成；链接、发布与对象系统负责组合、覆盖证明、外部签名回填、验证和安全映射；Agent Harness 负责运行会话、能力、观察、证据闭包与验收策略；Fulfillment Runtime 只负责生成式求解和候选评价。

例如，一个“生成报告”的请求只有在目标受众、必需章节、禁止披露内容、证据来源和验收条件都进入受检语义后，才适合成为对象；Fulfillment Runtime 可以选择怎样形成报告，但不能删除硬约束或扩大访问权限。

编译边界

  • Deterministic Profile 下的受控来源通过无模型路径进入确定性 Noesis Core。
  • Model-assisted 或 External-generation 路径中的模型只能提交不可信候选或源码，不能直接产出 NIR/NOBJ。
  • Candidate Envelope 必须绑定来源跨度、替代解释和确认权限；类型检查不能证明模型解释忠实于来源。
  • Noesis Core 只确认可由规则重推导或获得 Source Binding Decision 的主张；其余内容保留歧义或拒绝。
  • 相同受控输入、工具版本和规范化环境应产生可复现的对象结果。

对象与装载边界

  • 一切对象输入均不可信；偏移、长度、计数、索引和对齐计算使用 checked arithmetic。
  • NOBJ 可保留待链接引用；noemlink 可将 NOBJ 与已有 HOBJ 链接为新的 Linked Object 或 HOBJ。
  • Release Object 必须绑定 Release Coverage Proof 与验证结论；noembundle 先生成不可变候选包和 Signing Request，回填时同时核对候选包、原请求与外部 Signature Response，再在被签载荷外附加 Signature Envelope。
  • Noema Object System 在映射前验证 Signed Package 的结构、边界、签名范围和完整性证明，验证失败时关闭失败。
  • 重定位完成后，关键符号绑定、入口、策略与权限应被冻结，避免装载后提权。

智能体执行边界

  • Noema Object System 先产生不可变 Loaded State；Agent Harness 从该状态、显式执行配置和版本化资料构造任务上下文与会话级能力绑定。
  • Runtime 或模型只提出 Capability Request，由 Harness 的确定性策略检查参数、权限、预算、副作用和当前状态。
  • 工具结果、UI 状态、日志、指标和追踪作为结构化观察返回会话；模型输出不能替代真实环境状态。
  • 验收契约、Acceptance Policy、停止条件和人工升级条件在执行前固定；重复失败和预算耗尽不能触发静默提权。
  • Runtime 只产生 Candidate Assessment；Harness 检查证据闭包，最终决定为 accepted、unsatisfied、pending-review、failed 或 interrupted。

当前设计：对象视图

当前设计借鉴 ELF 的 Section/Segment 双视图：Section 服务链接、调试与对象工具，Segment 服务验证和装载。具体 Header、记录布局和签名结构仍须由正式格式规范与 ADR 固化。

当前设计：对象语言、元语言与行为边界

思想与方法基础提示我们区分被执行或评价的对象数据，以及描述其类型、验证规则和环境的元层。当前提议要求 NIR/NOBJ 不得把配置、验证器规则或 Fulfillment Runtime 私有假设伪装成对象自身事实；跨层引用必须有显式版本和信任来源。

  • 如果目标要求询问、承诺、授权或执行，候选行为对象应携带行为类型、行动者、权限、前置条件、预期效果和失败语义，而不只是命题内容。
  • 来自字面内容、结构规则、语境推论、工具和模型的结论应保留不同来源等级；可撤销推论不能静默升级为硬约束。
  • 目标满足由明确的评价环境决定，环境包含变量绑定、证据版本、能力目录和 Fulfillment Runtime 配置；未登记背景知识不得参与确定性验证。

这些结构仍须用反例语料、竞争设计、ADR 和一致性测试证明，尚不是对象格式字段。

尚待确定：运行协议

后端输入适配器、Runtime Request、Capability Request、Capability Observation、状态同步、预算单位和 Acceptance Decision 结构仍未定型。后端输入适配器只是 Runtime 的一种实现方式，不属于 NIR 或对象格式，详见开放问题。
