Endem 生命周期

从 rhem 来源、意义投影和中性事态，到 attested 制品、Dromen、phain、Tekmor 与最终决定。

nascentcoherentattestedDromenTekmor

生命周期不是信任等级流水线

每个阶段只回答一种问题。poie 说明目标契约已经确定性编码；pleko 说明引用和冲突已经解析；sphra 说明发布载荷和签名响应一致；praxe 说明当前环境建立了受控会话；Tekmor 说明判断依据可定位。任何阶段都不能替代其他阶段。

关键边界：Endem 在 nascent、coherent、attested 三个状态之间演进。apor 是语义面，不是生命周期状态；Dromen 是一次会话的实现态，Tekmor 是有限证据记录。

nascent → coherent

  1. elenk 验证 nascent Endem 的结构和语义，不因存在 apor 项就称对象损坏。
  2. pleko 只从显式依赖锁定、授权决定和绑定策略取得候选。
  3. 每个 apor 项必须变成已解析、获授权决定或失败；观察不足只能在运行时产生 agno，不能用 agno 保留语义歧义。
  4. 跨 Endem 引用形成 Synem；没有跨对象消费者时保持单一 coherent Endem。
  5. skena 的极性、telis 的 kine/mene 边界和权限只能保持或收紧，krin 不能因组合变得更容易满足。

coherent → attested

  1. tasse 计算发布闭包和可选 Debug Companion，证明删除内容不改变规范属性。
  2. 待签载荷冻结后生成 Signing Request；私钥始终留在外部签名系统。
  3. sphra 同时读取原候选、原请求与 Signature Response，先验证绑定，再附加 Signature Envelope。
  4. 签名封装不得改写被签载荷；任何载荷变化都要求重新 tasse 和外部签名。
  5. attested 只说明发布身份和字节范围，不证明当前环境可运行。

attested → Dromen → Tekmor

  1. Praxor 对实际输入字节重新执行所需 elenk，不复用过期结论。
  2. Praxor 从对象、当前策略和授权能力建立 Dromen；实时句柄只保留在隔离执行侧。
  3. 后端读取 Dromen 的受限视图，返回候选或类型化能力请求，不能直接调用外部能力。
  4. Praxor 验证请求、执行或拒绝，再把真实观察规范化为 phain 并记录为 Praxe Tekmor。
  5. 决定引擎按照执行前固定的 krin 和权威要求比较 skena 与 phain，形成 Decision Tekmor；观察不足时保持 agno，需要价值判断时保持 pending-review。
  6. 会话结束后 Dromen 销毁，Praxe Record 保留允许公开的 Tekmor 和身份链。

当前实现顺序

第一条切片只实现 Rhem Source → deterministic projection → poie → nascent Endem → elenk/theor → peira。pleko、Synem、tasse、sphra、Praxor、Dromen 和运行 Tekmor 必须等待最小格式、两条读取路径、恶意语料和字节复现稳定后再进入。
