Endem 生命周期

从自然语言目标卡到 sealed 制品、运行 Frame、Witness 和最终决定的完整状态链。

openboundsealedFrameWitness

生命周期不是信任等级流水线

每个阶段只回答一种问题。form 说明目标契约已经确定性编码；bind 说明引用和冲突已经解析；seal 说明发布载荷和签名响应一致；run 说明当前环境建立了受控会话；Witness 说明判断依据可定位。任何阶段都不能替代其他阶段。

关键边界：Endem 在 open、bound、sealed 三个状态之间演进。Frame 不是第四种 Endem 状态，而是一次运行内的只读视图；Witness 也不是状态，而是有限证据记录。

open → bound

  1. check 验证 open Endem 的结构和语义，不因存在 open 项就称对象损坏。
  2. bind 只从显式依赖锁定、授权决定和绑定策略取得候选。
  3. 每个 open 项必须变成已解析、明确保留的运行时未知、获授权绑定或失败；不得静默删除。
  4. 跨 Endem 引用形成 Weave；没有跨对象消费者时保持单一 bound Endem。
  5. 权限和 must 只能保持或收紧，done 不能因组合变得更容易。

bound → sealed

  1. pack 计算发布闭包和可选 Debug Companion，证明删除内容不改变规范属性。
  2. 待签载荷冻结后生成 Signing Request；私钥始终留在外部签名系统。
  3. seal 同时读取原候选、原请求与 Signature Response，先验证绑定，再附加 Signature Envelope。
  4. 签名封装不得改写被签载荷；任何载荷变化都要求重新 pack 和外部签名。
  5. sealed 只说明发布身份和字节范围，不证明当前环境可运行。

sealed → Frame → Witness

  1. Runner 对实际输入字节重新执行所需 check，不复用过期结论。
  2. Runner 从对象、当前策略和授权能力建立 Frame；实时句柄只保留在隔离执行侧。
  3. 后端读取 Frame 的受限视图，返回候选或类型化能力请求，不能直接调用外部能力。
  4. Runner 验证请求、执行或拒绝，再把真实观察记录为 Run Witness。
  5. 决定引擎按照执行前固定的 done 和权威要求形成 Decision Witness；需要价值判断时保持 pending-review。
  6. 会话结束后 Frame 销毁，Run Record 保留允许公开的 Witness 和身份链。
