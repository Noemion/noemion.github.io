项目/架构设计/Endem 生命周期

Endem 生命周期

从人写下目标开始，逐步说明系统何时可以形成文件、谁可以签署或授权，以及一次运行怎样留下可核对的依据。

内容形成外部陈述一次会话有范围证据

先分开内容状态与外部关系

改动目标内容会产生一件新的 Endem；新增签名、撤销授权或更新验证结果不会改写原文件。这种分离让开发者能够判断“目标变了”还是“别人对同一目标作出的声明变了”。

  • 内容形成：ktise 说明目标契约已经确定性编码；pleko 说明引用和冲突已经解析。
  • 外部陈述：签名者针对精确内容和陈述类型产生独立记录，验证者再依据算法、信任根、政策和截止点判断其是否适用。
  • 会话准入：drase 重新核对实际内容、适用外部陈述、当前政策、环境、能力和预算，随后才建立一次 Dromen。
  • 运行判断：Iknem 保存有范围的观察与依据；满足结果、证据覆盖和最终决定继续由各自规则形成。

当前限制：现行 END-CORE 草案仍把 nascent / coherent / attested 列为内容形成分类，DRO-CORE 也使用 attested 表达会话主体准入。这个单值模型不能精确保存多项签名、验证政策、截止点和撤销关系；生命周期与结果词边界研究正在设计迁移，开发者不能把它理解为内容获得了永久可信状态。

nascent → coherent

  1. elenk 验证 nascent Endem 的结构和语义，不因存在 apor 项就称对象损坏。
  2. pleko 只从显式依赖锁定、授权决定和绑定策略取得候选。
  3. 每个 apor 项必须变成已解析、获授权决定或失败；观察不足只能在运行时产生 agno，不能用 agno 保留语义歧义。
  4. 跨 Endem 引用形成 Synem；没有跨对象消费者时保持单一 coherent Endem。
  5. skena 的极性、telis 的 kine/mene 边界和权限只能保持或收紧，krin 不能因组合变得更容易满足。

内容形成与发布陈述不能压成一个状态

  1. 对应制品生产者形成确定性物理封装；若产生派生物，必须保存新身份、变换、保留内容与损失。
  2. 待签陈述必须先绑定精确内容身份、陈述类型和预期签名者；私钥始终留在外部签名系统。
  3. 一项内容可以同时拥有多项外部陈述。每项陈述分别保存签名者、包络、算法和适用范围，不能折叠为内容内的单一布尔值。
  4. 验证记录另行绑定信任根、验证政策、参考值、截止点、撤销状态和实际结果；不同依赖方可以对同一陈述作出不同判断。
  5. 签名封装不得改写被签内容；任何内容变化都产生新身份，并要求外部陈述重新绑定。
  6. 现行草案中的 attested 只能按上述限制阅读，不证明内容真实、当前可运行、目标已满足或权威已经接受。

外部陈述复核 → Dromen → Iknem

  1. Drasor 对实际输入字节重新执行所需 elenk，不复用过期结论。
  2. Drasor 确认每项外部陈述都绑定当前精确内容，并重新评估签名、证明类型、信任根、政策、截止点、撤销和适用范围。
  3. Drasor 按 ADR-0024 把通过当前依赖方政策的结果，同环境、能力、预算和证据责任求交后封存 Dromen；实时句柄只保留在独立最小能力域。
  4. 后端读取 Dromen 的非秘密只读描述，返回候选或类型化能力请求，不能直接取得外部能力。
  5. Drasor 验证请求、执行或拒绝，再把真实观察、规范化变换与 phain 分开记录为符合 IKN-CORE 抽象边界的 Drase 证据。
  6. 若 telis 为 mene，Drasor 还必须按 ADR-0016 绑定 fixed 或 elapsed 范围，记录时钟域、覆盖区间、缺口和违约预算。
  7. 求值器按照执行前固定的 krin 比较 skena 与 phain，形成 met / unmet / agno / fault 中的一个满足结果。
  8. 验证者先在精确政策、参考值、环境与截止点下评估 Iknem 有效性，再相对当前 krin 计算覆盖度；具名权威最后形成 accepted / rejected / deferred 中的一个决定。会话另行记录 completed / failed / interrupted。
  9. 会话结束、失败、中断或实质漂移后，Dromen 与绑定实时能力都不可再用；允许保留的运行记录只进入有范围的 Iknem 或会话事件，不能恢复权限。

现行结果域与草案限制

类别允许状态含义
Endem 内容形成nascent / coherent / attested（现行草案）attested 是正在修正的临时分类；真实签名、证明、验证政策、截止点和撤销必须按外部关系理解。
满足判断met / unmet / agno / faultskena 相对 phain、krin 与求值规则的工程满足结果。
权威决定accepted / rejected / deferred具名规则或主体依据满足结果、Iknem 和覆盖度作出的最终决定。
Drase 会话completed / failed / interrupted只描述一次运行怎样终止，不说明目标是否满足或已被接受。
Elenkpass / fail / not-evaluated每个请求验证层的独立结论；不能压成含义不明的成功布尔值。
Iknem 有效性valid / invalid / revoked验证者针对精确政策、参考值、环境和截止点形成的外部评估；不等同于 claim 为真。
证据覆盖度sufficient / insufficient一组有效 Iknem 是否覆盖当前 krin；不与单份记录有效性合并。

这些结果域由 ADR-0015 分开。外部 Task 的 completed、工具返回成功或会话正常结束都不能跨域改写满足结果与最终决定。

mene 的时间范围、连续性政策与覆盖分类由 ADR-0016 定义。当前没有 mene 线格式、时钟适配器或持续求值实现。

Iknem 的主体、范围、溯源、观察类别、有效性、覆盖度、决定和最小披露由 ADR-0022 与 IKN-CORE 定义。当前没有物理 Iknem 格式、验证器、归并器、撤销服务或决定引擎。

为什么采用两条轴

权威资料可复用的边界Noemion 不据此声称
RFC 9334 RATS Architecture分开 Evidence、Verifier 产生的 Attestation Result、依赖方自己的评估政策和应用决定，并要求考虑 freshness。Endem 等同设备 Evidence，或 RATS 直接定义本项目的内容状态。
in-toto Attestation Statement外部 Statement 以摘要绑定不可变 subject，并用 predicate type 说明陈述类型。存在 Statement 就自动满足本地政策，或签名改变 subject 内容身份。
SLSA 1.2 Verification Summary Attestation验证摘要显式记录 verifier、policy、输入 attestations 和 verification result；消费者仍按自己的目标验证。SLSA 等级等同目标满足、运行授权或最终接受。
GNU Guix substitute authentication
GNU Guix challenge		把制品内容、签名认证和独立构建结果比较分开；差异可以揭示不可复现或潜在篡改。签名有效就证明构建正确，或差异本身决定哪一方可信。
MCP 2025-11-25 Tasks外部任务状态描述被包装请求的执行生命周期，且当前仍是实验能力。远端 completed 能成为 Endem 内容状态、满足结果或本地接受决定。

未来实现顺序

当前没有组件实现。第一条切片限制为“受控来源表达 → 确定性投影 → ktise → nascent Endem → elenk/theor → 差分与复现验证”。最小格式、两条读取路径、恶意语料和字节复现稳定后，项目才评估组合、发布与受控运行。
