ADR-0014
首个 Ktisor 来源清单

用一份有限、显式的来源清单连接人类表达与 END-P1，同时禁止模型直接生成规范对象。

当前草案END-SRCM 0.1实验输入可替换

直接结论

首个 ktise 接受 UTF-8、制表符分列的实验性来源清单。清单同时保存解码并处理转义后的自然语言表达和已经授权的结构投影；Ktisor 只做确定性解析、验证和写入，不调用模型补齐意义。

这项决定关闭“首个候选怎样进入 Ktisor”的实现接口，但不把 .ends 确定为长期源语言。未来正式语言出现时直接删除这一入口，不保留兼容垫片。

为什么不能直接接收 Prompt 或模型 JSON

输入方式关键问题本决定怎样处理
自由 Prompt来源记号、已确认投影和待确认候选混在一起，写入器无法判断谁有权决定。原文进入 rhem；确认关系单独列出；待确认选择进入 apor。
模型 JSON结构合法仍可能事实错误、引用悬空或越权，模型也会影响字段顺序和默认值。模型只能在上游提出候选；Ktisor 重新解析、验证并决定规范字节。
直接手写 CBOR作者可以绕过字段、排序、来源和授权检查。只有 Ktisor Writer 产生 END-P1；来源清单从不携带布局指令。
大型通用 IR引入尚无消费者的类型、依赖和扩展机制，扩大可信核心。只登记首个真实切片需要的十种指令，未知指令关闭失败。

语法边界

每个非注释行由制表符分成一个指令和固定参数。单例指令是 rhem/root/telis/krin；可重复指令是 symbol/relation/situation/phain/iknem/apor。文本只解释四种反斜杠转义，复合标识符集合只使用逗号分隔。

名称边界：“来源清单”是直白工程名称，不再创造新的哲学专名。.ends 只是实验后缀，不是 Endem 对象扩展名或身份的一部分。

保真边界：LF 与 CRLF 等价，四种转义会被解码，所以 rhem.content 不是原始 .ends 文件的逐字节副本。原始来源字节只有在未来另行绑定文件身份、解码 Profile、变换链和损失时才能核对。

完整指令、基数、转义、资源与退出条件见 END-SRCM 条款源。

确定性与授权

  • 输入顺序不决定对象字节；Ktisor 按 END-P1 的稳定 ID 和 CBOR 规则排序。
  • 重复身份、未知引用、越界来源和未登记枚举必须拒绝，不能由后项覆盖。
  • 规则或具名权威确认的关系才能进入 semion；其他可表达候选保留在 apor。
  • 来源清单不包含 profile、offset、length、record ID 或 CBOR 键等布局控制项。
  • 结构接受、语义接受、可信签署和目标满足继续是四种不同判断。

验证与停止条件

最小清单必须先映射到现有语义接受向量，再逐字节生成 END-P1 接受对象。未知指令、错误参数数量、重复单例和无效转义必须稳定拒绝；Ktisor 与 Theor 继续只共享规范和向量。

当正式来源语言具备注释、包含、版本演进、歧义模型、错误恢复和独立语料时，删除来源清单。若清单不能表达新的真实任务，也应回到规范，而不是增加隐式默认或模型补全。

当前状态

ADR-0014 只采用首个实验输入。它不承诺通用自然语言编译、稳定 CLI、包格式或向后兼容；当前只有来源样例，没有 Rust 解析器或实现仓库。
