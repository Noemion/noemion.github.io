项目/架构设计/架构决策/ADR-0022

ADR-0022
证据范围
评估边界

固定 Iknem 的主体、溯源、观察、完整性、有效性、覆盖度、决定和最小披露边界，让证据可以复核而不因签名、数量或模型自述越权。

当前策略IKN-CORE 0.1.0-draftNo Wire Format尚未实现

直接结论

Iknem 是一项有明确主体、范围、生产者、方法、环境、观察、主张和限制的证据记录。它不是日志包装、可信分数、数学证明或最终验收决定。

同一记录必须分开表达原始观察、结构化 phain、派生过程、有限主张、完整性和适用限制。验证者再依据具名政策形成有效性评估，证据集合再相对精确 krin 形成覆盖度；具名权威最后才可能作出决定。

本决定建立 IKN-CORE 0.1.0-draft，但不创建 Iknem 扩展名、物理容器、签名算法、撤销协议或实现。本次工作不表示采集器、验证器、归并器、决定引擎、Theor 或 Drasor 已经实现。

哲学启发与采用限制

“要能看出图像的真假，必须将它同实在比较”

路德维希·维特根斯坦《逻辑哲学论》2.223

这句话提示的是边界：记录不能只凭自身宣告为真。Iknem 因此必须让观察与目标关系可比较，也必须保存比较依赖的方法、范围和限制。

项目不把“图像”和“事实”的哲学关系直接变成软件定理。主体身份、溯源图、证据类别、覆盖计算、撤销和决定权威仍由规范条款、威胁模型与正反向量定义。

一项 Iknem 必须回答九个问题

问题最低责任不能省略的边界
针对什么精确主体身份与声明范围名称、路径和“当前对象”不能替代内容或事件身份
谁产生生产者身份、版本与权限生产记录不自动取得判断或决定权限
怎样产生方法、版本、参数与环境方法覆盖决定主张上限
用了什么精确输入、父记录和派生关系未知输入进入限制，不能藏进环境名称
观察到什么原始引用、变换与结构化 phain推断、模型说明和工具成功不冒充原始观察
实际声称什么有限主张与记录种类不得外推到未声明主体、总体或时间
怎样检查完整性内容身份、认证、序列与签名关系完整性不等于事实正确
什么仍有限时间、丢失、过滤、脱敏和已知缺口限制必须参与有效性与覆盖判断
谁能继续判断验证政策、krin 与决定权威记录不能自填 valid 或 accepted

记录种类与来源类别分别回答不同问题

维度允许值用途
记录种类observation / derivation / attestation / appraisal / decision-record说明这项记录做了什么
来源类别direct-observation / deterministic-derivation / external-assertion / human-judgment / model-candidate说明依据怎样产生

模型可以生成一项带完整溯源的候选说明，但它仍是 model-candidate。签名可以认证外部声明的来源，却不能把它改成直接观察；人工判断也不能因为职位较高就冒充环境事实。

四个相邻结果域不得合并

结果域允许值回答的问题
Iknem 有效性valid / invalid / revoked验证者在指定政策、参考值、环境与截止点下是否接受这项记录
证据覆盖度sufficient / insufficient有效记录集合是否覆盖精确 krin 的全部观察责任
满足判断met / unmet / agno / faultskena 相对 phain 与比较契约得到什么结果
权威决定accepted / rejected / deferred具名权威依据前述结果作出什么决定

valid 不是 Iknem 自带字段，而是带政策与截止点的外部评估。decision-record 只记录已经发生的决定事件；创建这种记录不会产生决定权限。

九条规范规则

  1. IKN-SCP-001：每项主张绑定精确主体和有限范围。
  2. IKN-PRV-001：溯源图完整、有限、无环且不能循环自证。
  3. IKN-OBS-001：原始观察、变换、phain 和主张分别记录。
  4. IKN-CLS-001：记录种类与来源类别不得靠置信度或签名升级。
  5. IKN-INT-001：完整性和生产者认证不替代事实正确与授权。
  6. IKN-VAL-001：有效性评估外置，并绑定政策、参考值和截止点。
  7. IKN-COV-001：覆盖度只相对精确 krin 和不同观察责任计算。
  8. IKN-DEC-001：评估、满足判断和具名权威决定保持正交。
  9. IKN-PRI-001：只做最小披露，不保存实时秘密，并声明信息损失。

开放标准与 GNU 机制提供的边界

  • W3C PROV Data Model分开 entity、activity、agent、派生和 bundle。Iknem 借鉴可追溯的对象—活动—责任图，但不采用 RDF 或 PROV 的物理编码。
  • RFC 9334 RATS Architecture分开 Evidence、验证者的 Attestation Results 和依赖方自己的评估政策。Iknem 同样不允许验证者替代应用权威作出最终决定。
  • SLSA 1.2 Provenance把外部参数视为不可信输入，并要求记录构建定义、运行细节、解析后的依赖与输出主体。Iknem 借鉴精确输入和生产者边界，不把构建证明当成通用事实证明。
  • GNU Guix 的 guix challenge比较本地与替代服务器的独立构建结果。结果不同可以定位不可复现或潜在篡改，却不能单凭差异决定哪一方正确。
  • OpenTelemetry GenAI 语义约定独立仓库正在继续定义 GenAI、MCP、供应商跨度、指标和事件，当前 Schema URL 仍待确定。它只适合作为带版本、默认脱敏的导出映射，不进入 Iknem 身份。
  • MCP Security Best Practices禁止 token passthrough，并强调受众、范围和最小权限。Iknem 可以记录不可逆凭据属性和调用范围，但不得保存令牌正文或实时能力句柄。

规范影响与验证

  • 新增 IKN-CORE 0.1.0-draft 九条规范条款。
  • 新增九项 Iknem 威胁和十四个非规范自然语言场景。
  • 十八个提案向量覆盖九个允许分类与九个确定拒绝；资料验证只证明该版本的矩阵一致。
  • END-CORE、END-FMT、SYN-CORE 与现有字节保持不变。

待定内容

Iknem 物理容器、扩展名、字段编号、摘要与签名算法、透明日志、撤销分发、远程查询、跨生产者时钟归并、隐私策略语言和长期归档仍保持开放。没有真实消费者和新 Profile 前，不为了与 Endem 对称创建格式。
