项目/架构设计/架构决策/ADR-0033

ADR-0033
标准 ID 不能伪装成
文件格式

TEXT-IDENTIFIER-CORE 命名一组跨制品的文本与标识符解释规则，不命名 .txt 文件、Unicode 字符串类型、输入组件或软件包；完整职责名用于阻止实现者选错对象。

当前策略标准 ID 已迁移不新增格式不保留兼容

先确定这个 ID 命名什么

同一段自然语言会同时经过来源文件、解码文本、结构字段、安全显示和模型输入。标准的职责是让每一层声明自己实际处理的码点、变换和比较规则，而不是把这些层统称为“文本”。

名称位置现行身份不能推断什么
规范 IDTEXT-IDENTIFIER-CORE，跨 Endem、Endem closure、session contract、evidence entry、诊断、适配和模型候选约束文本解释不是文件媒体类型、容器、字段或实现组件
条款前缀TEXT-*，标识十二项可独立引用的规范责任不是允许任意 Unicode 标识符，也不声明文本语义等价
规范源文件text-identifier-core.md，承载版本化条款Markdown 文件名不成为运行时对象或稳定 API
现有制品格式仍由 END-FMT 与具体 Profile 决定字节和槽位本决定不创建 .txt、.text 或其他新格式
未来实现与分发当前尚未定义文本处理器、模型输入网关或独立软件包标准 ID 不预留同名包，也不要求组件使用同一名称
绑定原始来源字节
按声明编码严格解码
记录槽位专用变换
按声明域比较与定位
生成安全显示视图
绑定模型实际输入

任何一步发生替换、规范化、大小写折叠、双向重排、截断、脱敏或转义，都必须留下自己的输入、输出、版本和用途；界面看起来相同不能证明机器输入相同。

为什么 TXT-CORE 必须一次性退出

实际误读会导致什么实现错误现行边界
把 TXT-CORE 读成 .txt为并不存在的通用文本文件设计解析器、扩展名或媒体类型它不是 .txt 文件格式；现有格式和 Profile 不变
把规范源当作来源文件层假定解码后的 source_expression.content 等于原始来源字节，丢失编码与变换证据原始字节、解码文本、提交后的来源表达和派生视图分别绑定
把“文本”理解为单一字符串类型让描述文本、结构标识符、登记词和显示标签共享规范化与比较规则每个槽先声明职责，再选择允许码点、比较域和显示策略
只在注释中解释旧缩写代码、导航、Issue 和口述仍持续强化错误对象名称本身已误导职责，注释不能修复机器登记和发现入口

TXT-CORE 已从机器登记、文件、条款引用、威胁、场景、向量、测试和公开入口共同退出；不保留旧路径、别名、重定向、双写或兼容垫片。历史 ADR 只记录迁移原因，不恢复旧标识。

TEXT-IDENTIFIER-CORE 管哪些边界

处理阶段对应条款实现必须保留的区别
槽位与解码TEXT-SLT-001、TEXT-ENC-001先声明字符串职责，再按唯一编码严格解码；错误字节不能静默替换
来源与身份TEXT-SRC-001、TEXT-IDN-001原始字节、解码文本、变换结果和封闭 ASCII 结构标识符不可互换
规范化与比较TEXT-NRM-001、TEXT-CMP-001规范化绑定槽位与版本；比较声明字节、码点、规范化或大小写域
范围与显示顺序TEXT-RNG-001、TEXT-BID-001范围绑定主体、单位和变换；存储逻辑顺序与视觉顺序分别核对
隐藏风险与元数据TEXT-HID-001、TEXT-MET-001不可见字符和同形风险显式处理；语言、方向和 locale 只是有范围声明
模型输入与输出TEXT-AIM-001、TEXT-OUT-001绑定模型实际取得的文本与预处理；显示、复制和导出披露所有变换

Unicode UAX #31为标识符语法、Profile、规范化和稳定性提供通用机制；RFC 8264则分开 IdentifierClass、FreeformClass 及应用 Profile。它们说明不同字符串用途需要不同规则，但不替 Noemion 选择字段、身份、比较域或模型输入。

为什么选择完整职责名

候选或证据核对结果决定上限
TXT-CORE与 .txt 的视觉和口头关联直接诱发文件格式误读退役
TEXT-COREInteredition Text Core 已以精确制品名用于 stand-off 标注文本模型，职责相邻拒绝；同时仍过度暗示通用文本核心
TIB-CORESemEval-2025 LLMs4Subjects 使用 tib-core 数据集进行技术文献主题标注，处于文本与 AI 发现空间拒绝；缩写也无法恢复职责
LEX-CORElex 常指词法分析、词典或词法语义，只覆盖当前职责的一部分拒绝；不能把来源、显示与模型输入收窄为 token 处理
TEXT-IDENTIFIER-CORE直接列出文本和标识符两个治理对象，不伪装成格式、组件或新领域品牌采用为规范 ID；含义仍由十二项条款定义
PyPI、npm 与 GitHub截至 2026-07-16，精确项目或仓库入口为空只说明本次查询未命中，不构成包名、仓库名或权利保留
crates.io本次自动查询返回 403，无法取得可解释结果证据缺失；不能声称同名 crate 不存在

这是标准 ID 的职责可读性决定，不是产品命名或商标意见。名称比缩写更长是有意取舍：开发者应能在登记和错误信息中先看出治理对象，再进入精确条款。

迁移怎样保持规范身份唯一

迁移面现行唯一入口保持不变
规范源spec/text-identifier-core.md版本、十二项职责、规范成熟度和未定义项
配套分析text-identifier-threat-model.md、text-identifier-scenarios.md威胁、十五个场景和失败责任
机器登记TEXT-IDENTIFIER-CORE、TEXT-*、TEXT-THREAT、TEXT-SCEN条款到威胁、场景和验证状态的关系
一致性证据vectors/text-identifier/ 与 text_identifier_vector_test.py十二个允许案例、十二个确定拒绝和诊断责任
公开解释/specifications/text-and-identifiers.html先按开发任务解释，再链接唯一规范源，不复制第二套条款

标准 ID 变化没有改变 END-FMT 字节、END-P2 字段、ASCII 结构标识符、Unicode 文本允许范围、内容身份或授权模型。未来若规范职责变化，必须重新审查名称；不能靠继续扩充“TEXT”一词吞并新对象。

当前状态与继续阅读

TEXT-IDENTIFIER-CORE 0.1.0-draft、TEXT-* 条款、现行规范文件、登记和向量已经完成一次性命名迁移。当前没有文本处理器、Unicode Profile、模型输入网关、新文件格式、稳定 API 或发行包；资料通过只证明规范关系一致，不证明组件已经实现。

文本与标识符边界规范从一段文本进入来源、制品、显示和模型的过程理解十二项条款。 ADR-0028 · 显示相同，不等于输入相同查看原始字节、解码文本、安全显示和模型实际输入为什么必须分开。 规范参考指南按开发问题定位 CORE、Profile、威胁、场景、向量和权威顺序。 ADR-0034 · 读得出，不等于听得清、写得回继续查看机器 ID、书面名称和人类口述怎样保持职责清楚。

口头边界：讨论标准时先说“文本与标识符核心规范”，需要机器精确值时再展示 TEXT-IDENTIFIER-CORE。不建立字母逐读的官方发音，也不让语音近似值选择规范版本、条款或机器输入。
