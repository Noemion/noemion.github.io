项目/架构设计/Agent 系统边界图

Agent 系统边界图

从一次工具调用出发，分开目标、行动者、授权、会话、外部状态、证据与最终决定。

开发者可以据此判断一个框架事件应进入哪个责任域，以及还缺少哪项证据。

非规范说明不增加对象尚无运行组件复核于 2026-07-15

先按一次调用回答五个问题

Agent 框架报告模型、工具、委托、暂停和恢复等运行事件。Noemion 不把这些事件压成一个“成功”状态，而是要求开发者沿下面五个问题逐项核对。

开发者问题责任域需要保留的事实
期望终态是什么？Endem / Synem目标身份、依赖闭包、判据和仍未解决的意义；计划与步骤留在控制平面。
谁代表谁执行什么动作？ID / AUT模型、Agent 定义、工作负载、运行实例、实际行动者、被代表主体、对象、动作、目的和截止点。
本次运行最多能做什么？DRO / ADP已核对的协议、对端、环境、能力交集、预算、凭据域和证据责任；能力声明只作为输入。
外部系统实际报告了什么？ADP / DIA精确调用、Task 状态、返回、错误、取消、交付和已知副作用，不提前改写本地结果。
什么证据支持哪项结论？IKN / END / AUT观察范围、方法、限制、满足判断和具名权威决定；每项结论保留自己的输入。

模型身份不等于实际行动者，认证也不等于逐动作授权。模型输出是候选，不是规范内容；工具或远端 Task 完成，也不证明目标满足。Agent Card、工具列表和 schema 只声明能力，不能扩写已经建立的 Dromen。

同样的分层适用于评测、训练、开放性、托管服务和数据删除。模型评分是有范围的候选测量；反馈记录不是训练资格；参数可下载不说明整个模型栈开放；删除响应也不能概括备份和派生物。

这份说明不增加 ADR、CORE、Profile、对象、命令或组件。现行要求仍以对应 ADR、CORE 和登记为准；其余主题属于正在进行的研究。

用依赖升级看清边界

假设一个 Agent 要把仓库依赖升级到经过允许的新版本。这个例子不需要新的 Noemion 对象，却能展示每一层怎样提供不同答案。

  1. 目标：Endem 描述期望版本、兼容条件和验证判据；它不保存“编辑文件、安装依赖、运行测试”等可变步骤。
  2. 行动者与授权：AUT 绑定当前仓库、分支、依赖、允许的版本范围、实际行动者和截止点。登录成功或 Agent 名称不能代替这项决定。
  3. 会话上限：Dromen 只允许本次会话使用已经核对的文件、网络目标、工具和预算。工具列表后来新增的能力不进入旧会话。
  4. 外部事实：包管理器退出码、远端 Task 状态和生成的锁文件分别记录。退出码为零只说明该次调用怎样结束。
  5. 证据与决定：锁文件差异、构建结果、测试覆盖和来源限制进入有范围的 Iknem；krin 形成满足判断，具名权威再决定是否接受。

如果并行尝试多个版本，分支只产生候选。提交路径仍要重新核对当前分支、授权、预算与外部副作用；取消慢分支也不证明它没有留下远端状态。

为什么现在要明确这些边界

2026 年的人工智能标准化正在从“模型能做什么”转向“Agent 代表谁行动、怎样取得权限、如何互操作、评测结果能支持什么结论”。这与 Noemion 的目标、会话、授权、证据和最终决定分层直接相关，但外部倡议不能替项目决定语义。

  • NIST AI Agent Standards Initiative把身份、授权、安全与互操作列为 Agent 生态的重点。Noemion 因此要求记录请求主体、实际行动者、被代表主体、对象、动作和受众；“Agent 已认证”仍不等于“这个动作已获授权”。
  • NIST AI 800-2 初始公开草案把评测目标、执行和报告分开，并明确这些实践仍会随测量科学发展而更新。Noemion 采用这种可追溯结构，不把一次基准分数或模型裁判输出写成稳定能力、充分证据或最终接受。
  • GNU 自由软件定义说明用户对程序应具有运行、研究、修改和分享的自由；GNU 对他人服务替代用户计算的分析进一步指出，取得源码不等于控制那一次托管执行。Noemion 因而分别记录软件权利、实际执行控制、数据控制和退出能力。
  • Linux Foundation OpenMDW 1.1表明模型发行许可仍在快速发展，覆盖对象可能包括参数、代码、文档和数据。开发者仍须逐对象核对实际提供物、许可、首选修改形式与复现材料，不能用一个许可证标签概括整个 AI 系统。

这些资料说明行业问题正在收敛到身份、权限、测量、供应链和用户控制。它们支持审查方法，不自动引入新的 Noemion 对象、字段、命令或实现。

常见的越级错误

外部数据取得指令权

检索片段、工具返回或远端描述只因为出现在上下文中，就被模型当作授权指令。

显示名称或声明替代授权

Agent 名称、工作负载身份、Agent Card、工具列表、schema、scope 或签名覆盖实际行动者、被代表主体和逐动作决定。

计划或协议状态替代目标判断

步骤、handoff、HTTP、MCP、A2A 或 SDK 的 completed 被直接写成 met 或最终接受。

最快分支或取消替代提交判断

并行候选因为先完成、评分高或多数一致就产生外部副作用；取消又被误读为已经回滚。

记忆、检查点或轨迹替代当前事实

过期指导、恢复状态、Span Ok 或日志数量被当作现行政策、持续权限、完整因果链或充分证据。

隔离机制名称替代有效限制

容器、seccomp、超时或“本地服务”标签被用来概括文件、网络、秘密、资源、终止和信息流控制。

模型评分或反馈替代评测与发布

排行榜、多个模型投票或用户反馈未经构念检查、用途授权、污染分析和独立评测，就改变生产决定。

单项可用性替代完整对象范围

参数可下载、AGPL、数据驻留、导出或删除响应被扩张为全栈开放、实例控制、零外发或全部副本不可恢复。

当前 Agent 技术趋势改变了什么

外部机制现实变化Noemion 保留的边界
OpenAI Agents SDK handoffs 与编排handoff 把当前分支的控制权交给专家；把 Agent 作为工具时，主管仍负责最终答复。高级交接还可以携带结构化元数据或过滤后的历史。答复所有权不等于动作授权。输入变换必须披露来源与损失，handoff 不能隐藏实际行动者或扩大 Dromen。
OpenAI Agents SDK human-in-the-loopRunState 可以暂停、序列化并在工具决定后恢复，审批还能穿过嵌套 Agent。SDK 恢复只说明框架状态可继续。使用时仍须核对对象、政策、权限和精确调用。
OpenAI Agents SDK sessions 与对话状态策略应用自管历史、SDK Session、conversationId 和 previousResponseId 是不同的状态承载策略；混用本地重放与服务端状态可能重复上下文。选择一种策略并显式协调例外。Session 只提供候选输入，供应商 ID 不成为 Noemion 身份、证据或授权。
OpenAI Sandbox Agents 跨运行记忆消息历史、提炼为文件的可复用指导、resume 和 snapshot 保存的工作区状态具有不同身份和生命周期。记忆文件不覆盖当前资料与政策；工作区恢复不复活完整会话、Dromen、秘密、权限或远端状态。
MCP 当前版与5 月 21 日公开、以计划最终发布日期命名的候选版当前正式版仍是 2025-11-25；初始化会协商能力，工具列表还能通过 listChanged 动态变化。名为 2026-07-28 的候选版尚未成为正式版，它转向无状态核心、独立扩展与新版 Tasks 生命周期。声明变化不能扩写旧 Dromen。候选版只用于漂移研究；协议无状态不等于执行无状态，Profile 必须逐版本锁定。
MCP 2025-11-25 Tasks 与 Sampling 草案请求方可以编排并发 Task，模型供应商 API 也已普遍支持并行工具请求；取消后执行仍可能继续。并发和 Task 状态只属于协议域。它们不复制预算、不选择提交者，也不保证副作用回滚。
A2A 1.0 版本化规范Task、Message、Artifact、上下文、流式更新、推送、历史长度与幂等各有明确语义。这些对象只留在外部域。Agent Card、Task 和 Artifact 不替代 Endem、Iknem 或决定。
NIST 软件与 AI Agent 身份授权概念工作、SPIFFE 工作负载身份与 MCP 企业管理授权行业正在把非人主体、工作负载认证、企业接入、最小权限、具体动作、委托与日志透明变成显式机制；同一工作负载还可能覆盖多个弹性副本。模型、Agent 定义、工作负载、运行实例、凭据和实际行动者继续分开。身份认证是 AUT-CORE 的输入，不是意义确认、逐动作授权或责任结论。
MCP 安全最佳实践与 A2A 安全条款token 透传、SSRF、本地服务暴露、多 Agent 凭据链、输入资源上限和日志秘密需要独立控制。模型不接收原始凭据；每次调用绑定资源、受众和实际行动者。协议安全措施不能替代本地授权、满足或接受。
NIST AI 800-2 初稿、NIST AI 800-3与模型裁判研究自动评测逐步固定协议、模型版本、日志、统计假设和限定主张；研究仍发现位置、冗长、格式和来源偏差。模型评审保持候选测量。多个模型票数、一致率和点估计不能替代人工校准、区间、证据覆盖或决定。
NIST AI 600-1、NIST SP 800-218A与偏好训练研究训练、微调和对齐逐步要求追踪数据、反馈、基础模型、训练流程、派生权重、评测、发布与监测；递归模型数据还会形成相关反馈风险。人工在环、模型裁判、训练完成、损失下降或供应商版本名都不能替代数据资格、精确身份、独立评测与发布决定。
OSI Open Source AI Definition 1.0、Model Openness Framework与模型许可演进行业正在把参数、代码、数据说明、数据和文档拆开，并用具名定义与分级框架约束“开放”主张。托管 API、参数可下载、公开仓库、Apache-2.0、模型卡或外部等级都不能单独证明模型全栈开放、可修改或可复现。
GNU 对服务替代用户计算的分析、MCP Sampling与模型服务数据控制托管模型按端点、功能、区域和账户区分数据条件；协议还能把模型选择与凭据留在客户端控制面。软件自由、Sampling、OAuth、零保留或数据驻留不能单独证明用户控制托管实例；调用方只能主张可观察请求、服务语境、返回和限制。
NIST Privacy Framework 1.0、NIST SP 800-88 Rev. 2、RFC 6973与 DPV 2.1访问、使用、披露、存储、删除操作、清除验证与清除确认逐步成为可分开的生命周期事实。工程分类不产生法律结论；策略到期、逻辑删除、工具完成或供应商收据不能证明全部副本与派生物已经不可恢复。
OpenAI Agents SDK tracing模型生成、工具调用、handoff、guardrail 和自定义事件可以进入统一运行轨迹。轨迹先作为有范围运行观察。其完整性、披露和证据覆盖仍须独立评估。

趋势不是“把所有状态放进一个 Agent 对象”，而是让委托、审批、持久历史、跨运行记忆、工作区恢复、任务和可观测性成为显式机制。显式机制更容易审计，也更需要防止跨结果域升级。

GNU 工具与软件自由带来的工程约束

GNU 机制可采用的工程纪律不能照搬的推论
Make target、prerequisite 与 recipe终态、依赖和实现步骤分别表达；实现路径可以变化。文件时间语义不能定义 Endem，recipe 成功也不是目标满足。
Make dry-run预检必须说明它回答的问题，以及仍可能执行的例外。名为 dry-run 不证明零副作用，也不形成授权。
.DELETE_ON_ERROR 与 .PRECIOUS状态被保留和状态有效是两个判断；失败后的部分目标不能冒充最新结果。文件存在、时间更新或中间目标保留不能证明检查点有效。
Guix profile generations状态版本、事务关系和回滚目标应具名并可追溯。回滚配置不会撤销外部副作用，也不会恢复旧凭据和权限。
Autoconf feature checks选择外部程序时核对任务需要的具体特性，不只相信名称、路径或版本字符串。配置期探测、缓存命中或测试成功不能成为运行期授权、持续健康或目标满足。
Make parallel execution、jobserver 与 interrupt cleanup先表达完整依赖，嵌套工作共享有界配额；中断后的目标按可能不完整处理，提交优先采用原子更新。job slot 不是授权，recipe 完成不是提交权，删除临时文件也不是外部副作用回滚。
Guix shell 与 Coreutils timeout从纯净环境开始并逐项开放路径、环境和网络；把期限、信号升级、后代状态和远端效果分别观察。开发容器不是生产隔离证明，主进程超时也不证明子进程、远端任务或副作用已经停止。
Guix channel、manifest 与 time-machine固定完整软件闭包并记录可复制环境，再把数据、随机状态、硬件、数值路径和输出身份分别核对。相同环境或固定种子不能自动证明训练权重逐字节相同、行为等价、用途安全或发布获批。
Diffutils、随机来源与 sort分开相同、不同和比较故障；记录候选排列的随机来源，并固定排序 key、区域设置、稳定性和同分规则。差异不选择正确一方，固定随机源不证明公平，排行榜顺序也不证明统计差异或目标满足。
GNU 自由软件四项自由对实际软件分别核对运行、研究、修改、再分发原版和修改版的权利，并确认源代码是可修改的必要前提。免费、可访问、源码可见或参数可下载不能自动成为自由软件，也不能把软件权利扩张到模型参数、数据和服务。
他人服务替代用户计算与AGPL 边界先区分自己的计算、通信交易和自主管理执行，再分别报告谁运行、谁控制、数据去向与软件权利。不能把所有网络服务都归成同一问题；提供服务端源码也不能证明用户控制正在运行的实例、隐藏软件或数据处理。
Coreutils shred与 Guix 垃圾回收清除方法必须匹配介质；删除前先审查根、引用、快照、备份和派生关系闭包。rm、shred 或垃圾回收成功不能证明 SSD、写时复制、镜像、备份和下游对象均不可恢复。

什么时候才值得增加新对象

Agent 生态每出现一个新 SDK 名词，都新建 Noemion 对象，会让品牌侵入协议和实现边界。新增对象至少要同时满足以下条件：

  1. 拥有不能由现有对象表达的独立身份和不变量；
  2. 拥有独立生命周期、生产者、下游读者和失败责任；
  3. 省略或并入现有对象会造成可重现的安全或互操作失败；
  4. 至少一个真实消费者需要稳定字段、版本和物理表示；
  5. 支持案例、反例、威胁、资源上限和正反向量可以验证。

当前软件 Agent 身份与责任链、上下文、数据生命周期、评测、训练、开放性、托管服务、计划、能力发现、并行候选和隔离责任都不满足这组进入条件。

预览、检查点、Task 和 trace 同样先留在控制平面、外部协议、发布或部署审查和有范围记录中。

怎样使用这张边界图

  1. 在“运行事实应该放在哪里”表中定位外部事件，不先采用 SDK 给出的本地含义。
  2. 固定对象身份、来源、协议版本、对端和实际输入变换。
  3. 分开请求主体、实际行动者、被代表主体、凭据和逐动作授权。
  4. 记录 Dromen 会话上限、调用事实、取消状态、交付和已知副作用。
  5. 让 Iknem 只支持声明范围内的判断，再由 krin 和具名权威分别形成满足结果与最终决定。
  6. 只有现有规范无法表达真实消费者的独立身份、生命周期和失败责任时，才讨论新对象。

专题检查：数据生命周期逐项核对目的、接收方、保留、删除阶段、备份和派生物；模型评测固定构念、协议、偏差探针、区间和用途；训练与更新固定数据资格、基础模型、派生权重、复现、评测和发布；托管服务固定执行者、控制面、数据路径、导出和退出条件。

并行、隔离或外部副作用还要分别核对共享预算、提交选择、秘密、文件、网络、进程、终止、幂等和事后观察。某项事实暂时不可见时，明确记录未知范围，不用产品标签补齐。

架构决策查看当前策略和待定接口。 开放问题查看研究提案仍需关闭的决定与证据。 外部协议适配查看版本、对端、能力、任务、重试和交付边界。 权威与授权决定查看主体、委托、同意、撤销和能力交集。 Dromen 会话契约查看一次运行怎样封闭环境、能力、预算和证据责任。
