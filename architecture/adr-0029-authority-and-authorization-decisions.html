项目/架构设计/架构决策/ADR-0029

ADR-0029
身份已确认
不等于操作已获准

让每次意义确认和 Agent 操作都绑定实际行动者、被代表主体、精确对象、有限范围、政策版本与截止点；登录、签名或界面点击不能自行扩大权限。

当前草案AUT-CORE 0.1.0-draft协议无关尚无政策引擎

用一次生产发布理解授权路径

一项 CI 工作负载请求把精确发布对象部署到生产环境。“发布负责人已批准”无法支持执行；系统必须能够恢复谁请求、谁行动、代表谁、对哪个对象做什么，以及这份决定何时失效。

固定请求与对象
区分三类主体
核对权威与政策
形成有限决定
与本地上限求交
另行判断执行与验收
检查步骤本例必须固定的事实关闭失败条件
固定请求发布对象的精确身份、生产租户、部署动作、目的、受众、次数和截止点对象仍是 latest、名称前缀或可变化集合时停止授权
区分主体提出请求的人或系统、实际运行的 CI 实例，以及它所代表的发布团队只有 Agent 名、服务账户、模型名或密钥时，不能恢复实际行动者
核对权威政策版本、权威范围、角色来源、认证方法及该角色对此对象与动作的决定权登录成功、组织成员资格或签名有效不能补足决定权
形成决定grant、deny 或 defer，以及理由、范围与时效模糊的“已批准”、静默裁剪或 defer 都不能当成允许
派生会话能力决定范围与 Dromen、制品、环境、适配器、预算等全部本地上限的交集任何上限缺失或漂移时关闭对应能力；扩权必须重新请求

当前策略：AUT-CORE 0.1.0-draft 为意义确认与动作授权提供共同检查边界，不增加 END-P1 字段，也不创建政策语言、令牌或执行组件。

实际问题本层结果不能越级推出什么

谁可以尝试一个操作grant deny defer 当前精确请求其他对象、字段、租户、用途或后续版本也获权
当前 Drase 会话能调用什么授权决定与全部本地上限的交集长期权限，或外部协议公开的全部能力
证据是否足以支持目标判断Iknem 在声明范围内的有效性与覆盖度判断授权主体已经接受结果
最终是否接受accepted / rejected / deferred由认证、授权、签名、工具成功或会话完成替代

语言哲学可以帮助区分“一句话表达什么”与“它是否成立”，却不能产生组织角色、法律同意或动作权限。模型置信度、结构合法性和自然语言措辞也都不能自我授权。

一次可复核授权必须保存哪些边界

边界开发者必须得到的答案常见错误
语境与主体谁请求、谁实际行动、代表谁，采用哪份政策和权威目录快照把工作负载身份、Agent 定义或密钥持有者当成被代表主体
对象与范围精确对象、动作、字段、目的、受众、环境、次数、预算和禁止事项使用通配对象、开放字段、权限并集或未声明扩展
决定与部分授予结果、理由和精确限制；部分授予形成明确缩小后的新范围只保存布尔值，或在执行端静默删除未获准部分
时效与撤销生效条件、截止点、政策版本、撤销来源和重新判断条件永久缓存一次批准，或在政策、对象、环境变化后继续复用
重放绑定请求、对象、目的、会话、次数和去重键共同绑定换一个 request ID 重放旧点击、旧签名或旧令牌

grant 只允许声明范围内的一次或一组受限尝试；deny 只拒绝当前请求；defer 表示条件不足，不能默认允许。三者都不证明操作已经发生、目标已经满足或结果已经接受。

委托、多人同意和权限升级怎样停止越权

情形必须保留的关系必须停止的位置
Agent 委托每级授予者、实际行动者、被代表主体、范围、目的、受众、期限、预算和剩余深度链不再有限无环，或任一级扩大对象、权限、期限与预算时
协议允许冒充本地仍保存真实行动者，并由独立政策明确是否允许代表关系无法恢复实际行动者，或把冒充身份写成原始主体时
人类同意安全显示视图与精确机器请求一致，目的、接收方、期限、撤销和拒绝路径清楚默认勾选、捆绑无关权限、隐藏差异或拒绝路径不可达时
多人决定合格主体、不同身份、角色或数量、否决、冲突、缺席和截止规则预先固定重复账户计票、决定后修改门槛或把缺席当赞成时
step-up 或新增 scope新请求、新决定、新能力交集和新 Drase 会话在旧会话中原地增加写权限、资源、预算或凭据时

RFC 8693区分 delegation 与 impersonation；RFC 9470的 step-up 只补充认证强度，不替资源服务器作出授权决定。Noemion 采用这些区分，不采用 OAuth 令牌结构作为本地权威模型。

外部协议、身份和 GNU 信任机制能证明什么

外部机制可以提供不能替 Noemion 决定
RFC 9396 / RFC 9700细粒度授权详情、部分授予、资源与受众限制及 OAuth 安全边界谁能确认自然语言意义、组织角色与最终接受
MCP 2025-11-25受保护资源、scope、访问令牌与资源受众绑定本地对象身份、Dromen 能力上限、目标满足或最终决定
A2A 1.0认证方案、逐请求授权和需要额外认证的协议状态Agent Card 自报能力的真实性或跨系统权威
工作负载与 Agent 身份定位软件、配置、运行实例或一次调用中的部分主体被代表主体、具体动作授权和业务决定权
GNU Guix / GnuPG把签名验证与本地允许的密钥或 trust model 分开把公钥、ownertrust 或 Web of Trust 直接当成语义权威

RFC 9396还要求未知授权类型或字段关闭失败，并允许用户只授予请求子集；RFC 9700要求访问令牌受众与权限收窄。GNU Guix 与 GnuPG 提供的只是“密码证明不等于本地授权政策”的工程参照。

当前还不能承诺哪些授权组件

现有规范源记录 12 类威胁、18 个设计场景和 24 个正反向量。它们验证的是抽象允许与拒绝边界，不证明身份、政策、同意、撤销或能力代理已经实现。

待定内容：主体与角色体系、政策语言、授权事件物理编码、同意 UI Profile、多人算法、凭据格式、协议映射、撤销分发、审计存储和法律效力均保持开放。当前没有权威目录、身份提供方、政策求值器、同意界面、法定人数服务、撤销服务、能力代理、决定服务或协议授权实现。

权威与授权决定规范按十二项责任检查主体、范围、委托、同意、时效、重放和结果分离。 ADR-0024 · 权限只属于这一次会话继续核对授权怎样与环境、预算和能力上限求交。 ADR-0027 · 签名有效，不等于制品获准继续核对签名主体、验证材料和本地依赖方决定。 ADR-0030 · 内容身份不变，不等于授权仍然有效继续核对稳定内容身份与外置授权记录怎样关联。

术语与读音：“身份确认”“意义确认”“动作授权”“会话能力”“最终决定”在职责和口头表达上均可区分；AUT-CORE 是机器文档标识，不作为面向传播的发行名称。任何新角色或公开名称仍需经过朗读、听写回填和成对混淆测试。
