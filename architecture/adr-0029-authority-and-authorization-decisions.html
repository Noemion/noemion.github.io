ADR-0029
权威与授权决定边界

让自然语言候选、委托 Agent、人类同意和运行能力都服从同一套可核对的主体、范围、政策、时效与结果分离规则。

当前草案AUT-CORE 0.1.0-draftProtocol IndependentNo Policy Engine

直接结论

Noemion 采用 AUT-CORE 作为跨制品权威与授权决定标准。模型、签名、登录、界面点击、外部 Agent 状态和凭据都不能自行取得语义或最终决定权。

AUT-CORE 固定十二条抽象规则，关闭“谁有权确认自然语言候选”这一开放问题。它不创建新制品、授权令牌、政策语言或组件，也不增加 END-P1 字段。

为什么现有规范还不够

现有规范已经要求仍缺少的共同定义
END-COREsemion 必须由确定性规则或具名权威确认主体怎样取得权威、范围怎样固定、候选怎样被安全展示
DRO-CORE政策、权威、同意和能力必须在会话前封闭委托、step-up、撤销和能力交集的共同授权语义
IKN-CORE决定记录不能自行创造决定权限决定请求、多人门槛、冲突和重放怎样约束
ID-CORE签名验证与主体授权必须分开角色适用性和当前决定范围由什么证明
ADP-CORE外部协议状态和 scope 不能扩大本地能力链式 Agent 授权怎样回到本地具名权威
TEXT-IDENTIFIER-CORE授权界面要绑定精确机器值和安全显示视图显示一致何时足以成为有效同意的一项依据

授权不是一个布尔字段。请求必须说明谁请求、谁实际行动、代表谁、针对什么对象执行什么动作、出于什么目的、面向哪个受众、使用哪份政策以及在什么截止点判断。

grant 只授予记录中的有限范围；deny 只拒绝当前请求；defer 表示没有获授权答案。它们构成授权决定域，不替代最终 accepted / rejected / deferred、满足结果或会话终止。

自然语言意义为什么不能自我授权

结构合法的候选只说明它能被当前类型系统表达。模型置信度只描述模型输出；高相似度只描述某个比较 Profile。三者都不能说明候选符合来源主体意图，更不能决定谁有权关闭相关 apor。

授权者必须核对精确来源、解码文本、候选、影响的语义面和关系位置、候选生产方式、安全显示视图、决定理由和剩余待确认项。批量批准只有在所有候选属于同一权威范围并完整展示差异时才可能有效。

委托与冒充怎样处理

RFC 8693 明确区分 delegation 与 impersonation。Noemion 默认采用委托：每一级保留原授权主体、授予者、实际行动者、被代表主体、范围、目的、受众、期限、深度和撤销来源。

每一级范围、预算和期限只能保持或继续收窄，链必须有限无环。外部协议即使允许冒充，本地仍保存真实行动者；没有独立政策明确允许时，冒充不能进入授权闭包。

MCP 与 A2A 的授权只覆盖各自协议请求

MCP 2025-11-25 让受保护资源服务器依据资源、scope 和访问令牌处理请求。A2A 1.0 要求服务端依据已认证身份与本地政策逐请求授权，并允许 Task 用 AUTH_REQUIRED 请求额外凭据。它们确实承载动作访问控制，不只是传输状态；但它们不会替 Noemion 确认自然语言意义、定义组织中的最终决定权威，或证明目标已经满足。

因此，外部 scope、凭据、授权结果、任务状态和 Agent Card 都是带来源的适配输入。Drasor 必须固定协议与对端，把外部动作许可同 AUT-CORE 语境、Dromen、制品限制、适配器能力和预算求交。新增权限需要新的本地授权决定和新的 Drase 会话；它不能回写 semion 或改变已有制品身份。

同意、多人决定和重放

RFC 9396 表明细粒度授权需要动作、资源和交易条件，并允许只授予请求子集。AUT-CORE 进一步要求同意界面绑定精确机器请求，披露目的、接收方、期限、撤销和文本变换；默认选中、无关权限捆绑和不可达拒绝路径都不能形成有效同意。

多人决定在请求前固定合格主体、不同身份、所需角色或数量、否决、冲突与缺席规则。一次决定还绑定请求、对象、目的、会话、次数和去重键，防止旧点击、重复签名和新 request ID 重放。

GNU 技术提供的边界参照

GNU Guix 不会仅因替代物带有签名就接受任意服务器。管理员必须把允许的公钥显式加入本地 ACL；这说明密码证明与本地授权政策属于不同判断。

GnuPG 同样区分密钥或签名验证与用户选择的 trust model。Noemion 借鉴“本地显式配置谁被信任”的原则，但不采用 Guix ACL、公钥、GnuPG ownertrust 或 Web of Trust 作为自然语言语义权威。

语言哲学采用边界

《逻辑哲学论》4.022 说：“命题显示其意义。命题显示当它为真时事情是怎样的，而且宣称事情就是这样的。”这支持 Noemion 把候选命题的意义结构与其真假分开。

逻辑形式不会显示组织授权、法律同意或委托链。谁可以确认一种投影、批准一种能力或作出最终决定，必须通过外部政策和主体关系建立。项目不从哲学命题直接推出认证方法、角色体系或授权算法。

权威资料与采用限制

  • RFC 9396 Rich Authorization Requests用于细粒度范围与部分授予参照。
  • RFC 8693 Token Exchange用于委托、冒充、subject 与 actor 分离参照。
  • RFC 9470 Step-up Authentication说明重新认证不等于资源授权政策。
  • RFC 9700 OAuth Security BCP用于受众、资源、动作和发送者限制参照。
  • MCP 2025-11-25 Authorization与A2A 1.0 规范用于 Agent 时代的授权传输与链式请求反例。
  • NIST 软件与 AI Agent 身份授权概念论文用于最小权限、具体动作、代表关系和人机授权绑定的问题参照；该资料是草案概念工作，不是本地语义或身份标准。
  • GNU Guix Reference Manual与GnuPG Configuration Options用于本地显式信任政策参照。

后果与当前状态

  • AUT-CORE 0.1.0-draft定义 12 条跨制品义务。
  • 12 类威胁、18 个设计场景和 24 个正反向量覆盖当前抽象边界。
  • END-CORE 的投影授权、DRO-CORE 的政策与能力闭包、IKN-CORE 的决定权威、ID-CORE 的授权分离和 ADP-CORE 的 step-up 统一引用本边界。
  • 当前没有权威目录、身份提供方、政策求值器、同意界面、法定人数服务、撤销服务、能力代理、决定服务或协议授权实现。

待定内容

主体与角色体系、政策语言、授权事件物理编码、同意 UI Profile、多人算法、凭据格式、协议映射、撤销分发、审计存储和法律效力仍需真实消费者、独立 Profile 与实现证据。
