项目/架构设计/架构决策/ADR-0032

ADR-0032
命令只改大小写
仍然是名称冲突

旧组件名 Poiet 与动作 poie 会同 PFA Open Inference Engine 的缩写 POIE 混淆，因此已经一次性退出；deterministic producer 与 ktise 只接替当前设计职责，不代表发行名称已经通过。

当前策略设计标识已采用发行名称待定不保留兼容

先把动作名称当作公开接口审查

即使动作只出现在 endem 之下，它仍会进入帮助文本、日志、Issue、教程、口头协作和搜索结果。组件名与动作名必须分别审查，不能因为大小写、父命令或一段解释文字而忽略冲突。

发布面开发者实际看到什么必须回答的问题
信任角色deterministic producer，负责规范字节的确定性形成它是否准确表达唯一写入边界，而不是暗示语义决定权
公开动作endem ktise，不是独立的 ktise 程序在帮助、补全、日志和口述中能否同其他动作稳定区分
规范与诊断动作名、错误来源、形成记录和测试说明更名是否保持输入、输出、身份和失败责任不变
未来分发真正需要取得的顶层命令与包坐标仍是 endem子命令无须成为独立包，但仍须通过职责、读音和接口一致性审查
有权主体确认意义投影
绑定精确语义授权
deterministic producer 确定性形成
elenk 分层检查
independent inspector 独立读取
消费者决定是否继续

这条流程描述工程职责，不表示组件已经实现。模型可以提供带来源候选，但不能选择规范意义、写入规范字节或借形成成功取得动作权限。

为什么 Poiet 与 poie 必须共同退出

观察可核对事实工程判断
精确字母组合PFA Open Inference Engine 以 POIE 指代 PFA 规范的参考实现；仓库说明它用于执行预测或分析模型的评分计算大小写不能形成可靠区分；搜索、口述、无障碍输出和大小写折叠环境都可能合并两者
相邻发现空间POIE 属于模型执行与人工智能工程；旧动作也用于模型参与的目标制品形成流程读者可能把实现、依赖、诊断或文档归到错误项目，达到工程停止条件
同一词族Poiet 与 poie 分别承担组件和动作身份只改其中一个会继续保留错误关联，必须作为一次迁移共同退出
法律边界公开资料足以证明发现冲突，但不是商标权或在先权利报告工程更名不宣告外部主体拥有什么权利，也不替代正式法律复核

当前没有组件实现，因此迁移直接同步组件页、动作、导航、图示、规范说明和测试契约；不保留别名、重定向、双写或兼容垫片。历史 ADR 只保存原因，不恢复旧入口。

deterministic producer 能决定什么，不能决定什么

“确定性形成”描述的是相同有效输入产生相同规范字节，不是形成器自行理解自然语言。deterministic producer 只消费已确认且具有精确语义授权绑定的意义投影，再执行规范规定的形成责任。

责任面deterministic producer 可以做什么不得跨越什么
意义输入核对投影已确认且授权关系适用于精确输入与职责不从候选中选择意义，不用模型概率或默认值删除 unresolved_meaning
规范形成依据 END-CORE、END-P2、END-FMT 与适用 Profile 固定顺序、布局和规范字节不把显示文本、原始自然语言或文件名直接当作内容身份
组合形成依据 CLOSURE-CORE 解析精确引用、传递闭包与唯一绑定不通过环境搜索、隐式网络或顺序碰运气补齐成员
检查让形成侧 elenk 路径按请求层次报告稳定诊断不替independent inspector 形成第二条解释路径，也不宣告最终接受
权限只读取形成所需的受控输入不持有模型、签名私钥、实时工具句柄或 bounded runner 能力
结果上限证明当前输入按指定规范确定性形成并通过相应检查层不证明目标已满足、外部陈述有效、操作已获准或制品已接受

唯一写入边界是职责约束，不是实现状态。当前只有规范、登记和向量证据；不存在 deterministic producer、independent inspector、bounded runner、endem CLI 或发行包。

为什么 deterministic producer 与 ktise 仍只是设计标识

证据轴截至 2026-07-16 的观察声明上限
直接相邻用途没有发现足以推翻当前设计决定的精确 AI、形成器或开发工具产品搜索没有命中，不等于名称可以发行
GitHub 精确仓库名deterministic producer 与 ktise 均为零；ktise 的子串查询存在其他较长名称只覆盖本次公开仓库查询，不保留组织、仓库或拼写
PyPI 与 npm两项精确入口均返回 404只说明查询时没有精确项目，不覆盖未来登记和近似名称
crates.io本次自动查询返回 403，无法取得可解释结果证据缺失，不能沿用旧页面的“crate 不存在”结论
专名必要性非规范提案建议比较 deterministic producer 与 form直白候选不是现行名称、别名或兼容入口；必须先做人类职责匹配
读音与听写词首 kt 没有项目登记读音，也没有真实使用者证据不得发布官方读音；持续丢音、插音或拼写回填失败会否决名称

GNU Coding Standards 的命名建议强调名称应传达含义并减少晦涩缩写；其 --version 规则又区分程序的规范名称与可执行文件名。这里借鉴的是可读、稳定和边界清楚的接口纪律，不是把 GNU 的具体命名规则直接变成 Noemion 规范。

名称决定不能替代实现排序

动作名称不等于实现优先级。五项公开职责不是五个已经证明必要的独立程序，现行标识也不是已发布子命令；名称治理只能保持接口清楚，不能制造实现授权。

职责当前成熟度首次发行前还需要什么
ktise、elenk、theor首条候选实现切片有直接消费者，现阶段仍只有规范和向量单独授权组件开发，并证明确定性形成、分层检查和独立读取
pleko职责已定义，依赖真实的多 Endem 组合消费者先证明闭包、引用解析和失败责任需要实现边界
drase职责已定义，依赖受限会话、能力控制和证据闭环先完成平台威胁模型、隔离与授权设计，再单独批准实现
一致性与互操作验证属于质量责任，不是新的公开动作按 ADR-0035 保持内部职责，不扩大公开 CLI
发行名称deterministic producer 与 ktise 只是当前设计标识先比较直白职责名，再完成读音、工程生态、分发坐标和权利复核

即使未来放弃 deterministic producer 或 ktise 的拼写，确定性形成、唯一规范写入、独立读取和最小权限隔离仍必须保持。改名不能暗中重排组件职责。

当前状态与继续阅读

Poiet 与 poie 已完成一次性迁移。deterministic producer 与 ktise 继续作为设计阶段标识，但专名必要性、读音、包坐标和权利均未通过发行门禁；正式发行前必须再次查询，并以新的 ADR 记录保留或改名决定。

deterministic producer · 确定性生产边界查看从已确认意义投影到规范字节的开发者路径，以及唯一写入职责的限制。 术语与读音验证指南查看专名必要性、首次朗读、听写回填、成对混淆和发行复核方法。 名称冲突与读音审计查看包入口、相邻产品、口头风险和设计标识的当前证据上限。 ADR-0035 · 用户任务不是内部职责清单查看五项公开职责为什么不等于五个独立组件或全部内部质量责任。

读音状态：当前没有 deterministic producer 或 ktise 的合格发行读音。现阶段口头说明应先说“确定性生产边界”或“形成动作”，再给出精确书面标识；任何语音输入都不得仅凭近似读音选择动作、参数或权限。
