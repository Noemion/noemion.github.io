## 先用一次发布看懂四类对象

假设开发者写下“把服务依赖升级到指定版本，只有测试通过并确认生产范围后才能发布”。这段自然语言不会沿一条流水线不断升级成“更可信的同一个文件”；系统需要形成四类边界不同的对象。

| 对象 | 保存什么 | 身份怎样处理 | 结论上限 |
| --- | --- | --- | --- |
| 含来源形成版 | 实际进入形成过程的自然语言、来源范围、已确认目标结构和待确认意义 | 按形成 Profile 的精确字节建立内容身份 | 可以复核来源怎样映射为目标；不能据此获得发布许可或运行权限 |
| 裁剪发布版 | 未来发布 Profile 允许的已确认目标结构和来源绑定；不保留原文或可逆重建材料 | 裁剪产生新的精确内容身份，并记录与形成版的类型化派生关系 | 可以检查发布字节与发布 Profile；没有受控伴随资料时，不能证明它忠实对应原始表达 |
| 外部陈述与验证记录 | 被声明的精确对象、陈述类型、签名者、包络、信任根、政策、截止点和撤销状态 | 作为外部关系连接发布身份，不进入 Endem 内容身份 | 只能说明某项陈述在指定验证语境中的结果；不自动授予动作权限或证明目标满足 |
| 一次会话与运行记录 | 只读会话契约（现行名称 contract）绑定的主体、环境、能力、预算和证据责任，以及有范围证据记录（现行名称 evidence） | 会话和记录分别拥有自己的身份，不回写发布对象 | 只服务本次受限运行、观察和判断；会话结束后不能恢复旧权限 |

> **当前状态：**以上是规范与验证方案，不是已经可用的发布系统。END-P2 只服务形成和评审；发布 Profile、裁剪编码、伴随资料格式、签名 Profile、装载器和运行组件尚未形成。

## 开发者按七步处理

1. 确认来源与意义
2. 形成含来源对象
3. 分层独立检查
4. 派生裁剪发布版
5. 验证外部陈述
6. 重新决定运行授权
7. 观察、判断与决定

| 步骤 | 未来责任 | 必须固定 | 停止条件 |
| --- | --- | --- | --- |
| 1 · 确认来源与意义 | 具名规则或权威确认自然语言候选；意义确认不授予工具或运行权限 | 来源身份、候选结构、安全显示、语义位置、主体、范围、截止点和剩余待确认项 | 决定缺失、错绑、拒绝、延后、过期或撤销时，不得关闭待确认意义 |
| 2 · 形成 | 确定性生产者（现行名称 producer）的 `form` 责任按 END-P2 形成含来源对象 | 实际来源文本、显式变换与损失、授权绑定、规范、Profile、配置和依赖闭包 | 来源不可重放、结构不闭合、目标事态不唯一或形成输入不确定时原子失败 |
| 3 · 检查 | `lint` 执行分层检查；独立检查者（现行名称 inspector）从另一读取路径解释相同字节 | 原始字节、规范/Profile、请求层次、资源预算和诊断位置 | 容器、Profile、引用、语义、资源或独立解释失败时，不得继续发布 |
| 4 · 裁剪发布 | 未来生产者按尚未确定的发布 Profile 派生新对象，不新增公开 `strip` 动作 | 保留与删除项、来源引用重写、泄露边界、损失清单、新身份和派生关系 | 原文泄露、引用悬空、身份复用、关系不完整或失败非原子时，不得发布 |
| 5 · 验证外部陈述 | 验证者把签名或证明重新绑定到实际发布字节 | 陈述类型、签名者、算法、信任根、政策、参考值、截止点和撤销状态 | 主体不匹配、材料不可用、策略失败、过期或撤销时，不得把陈述用于准入 |
| 6 · 建立会话 | 受限运行职责（现行名称 runner）另行核对授权，并把本次允许范围固定为只读 contract | 请求主体、实际行动者、对象、动作、目的、范围、环境、能力交集、预算和需要记录的观察 | 授权未授予、能力不能求交、环境漂移、预算无界或观察任务无人负责时，不得运行 |
| 7 · 观察与决定 | 运行产生带来源的观察和 evidence；求值器与具名权威分别形成满足结果和最终决定 | 结构化观察、变换、目标判据、有效性、覆盖度、会话终止和决定依据 | 覆盖不足返回 `undetermined`，观察或适配故障返回 `fault`；两者都不能提升为接受 |

后一步不能弥补前一步的失败。签名不能修复错误目标，授权不能修复不完整身份，会话成功也不能替代现实证据和最终决定。

## 裁剪不是删除一个字段

形成版到发布版是有损、可定位且产生新身份的派生过程。直接删除 `source_expression.content` 会留下来源引用、诊断范围、签名和验证责任不清的问题。

| 常见错误 | 为什么不安全 | 正确边界 |
| --- | --- | --- |
| 发布版复用形成版身份 | 两个不同字节对象被当作同一主体，签名、缓存和政策可能错绑 | 分别计算完整身份，并保存明确的裁剪派生关系 |
| 只删除原文，保留来源区间 | 引用可能悬空，读取器却误以为结构仍然闭合 | 发布 Profile 必须定义不泄露原文的来源绑定，或重写、移除相关引用 |
| 公开低熵原文的裸摘要 | 攻击者可以离线枚举短句，摘要并不自动提供来源保密性 | 来源关联必须经过披露威胁分析，受控伴随资料默认不进入公开包 |
| 发布版继承形成版签名或权限 | 旧陈述绑定的是另一个精确对象，旧授权也可能不覆盖发布用途 | 面向发布字节重新生成陈述、验证记录和使用授权 |
| 运行路径自动读取伴随资料 | 审计材料可能包含原文或敏感范围，读取资格也不等于执行权限 | 来源一致性审计通过独立受控路径完成，运行路径只取得必需最小信息 |

[GNU GDB 的独立调试文件](https://www.sourceware.org/gdb/current/onlinedocs/gdb.html/Separate-Debug-Files.html)与 [GNU Binutils `objcopy`](https://sourceware.org/binutils/docs/binutils/objcopy.html)展示了完整对象、裁剪对象和伴随资料可以分开保存并显式关联。Noemion 只采用这种制品分层，不照搬 ELF、DWARF、`.gnu_debuglink`、CRC、build ID 或字段删除语义。

## 装载前重新决定能否执行

| 核对域 | 必须回答的问题 | 不能代替它的信号 |
| --- | --- | --- |
| 精确对象 | 实际输入是否与允许的身份域、对象种类、规范、物理表示、Profile 和完整摘要完全一致 | 文件名、路径、URL、版本标签、短摘要或 build ID |
| 陈述与验证 | 哪些陈述绑定当前对象，验证者依据什么信任根、政策、截止点和撤销状态接受它们 | 签名存在、密钥 ID、透明日志条目或生产者自述 |
| 动作授权 | 谁允许哪个行动者，以什么目的、范围、条件和期限对当前对象执行什么动作 | 意义确认、登录成功、角色名称、Agent Card 或界面点击 |
| 环境与能力 | 当前环境是否仍匹配，协议、对端、适配器、政策与 contract 的能力交集是否非空且有界 | 远端能力声明、历史会话或外部 Task 状态 |
| 证据责任 | 运行后需要观察什么、由谁产生、如何覆盖目标判据，哪些秘密和实时句柄必须外置 | 日志数量、工具退出码、模型置信度或会话正常结束 |

只有这些问题同时得到允许答案，runner 才能建立一次只读 contract。内容形成状态、外部陈述有效性、动作授权、`met / unmet / undetermined / fault` 满足结果、`completed / failed / stopped` 会话终止、evidence 有效性与覆盖度必须分别保存。

> **当前边界：**END-CORE 的内容状态只保留 `formed / resolved`。SESSION-CORE 要求分别绑定精确内容、必需外部陈述、验证政策、截止点、撤销状态与依赖方适用性；任何单个签名或布尔值都不能表示永久可信、当前获准运行或已经满足目标。

## 规范来源与当前上限

| 资料 | 用于核对什么 | 不能推出什么 |
| --- | --- | --- |
| [END-CORE](https://noemion.github.io/spec/endem-core.html)、[ADR-0030](adr-0030-endem-content-and-authorization-companions.html) 与 [ADR-0036](adr-0036-source-bearing-and-stripped-release.html) | 内容形成、外部授权伴随关系、形成版与裁剪发布版边界 | 发布 Profile、裁剪工具或签名系统已经实现 |
| [ID-CORE](https://noemion.github.io/spec/identity-core.html)、[AUT-CORE](https://noemion.github.io/spec/authority-core.html)、[SESSION-CORE](https://noemion.github.io/spec/session-contract-core.html) 与 [EVIDENCE-CORE](https://noemion.github.io/spec/evidence-entry-core.html) | 精确身份、陈述、授权、会话契约、观察、覆盖度与决定的责任边界 | 密码算法、政策语言、会话格式、证据容器或运行组件已经确定 |
| [RFC 9334 RATS](https://www.rfc-editor.org/rfc/rfc9334.html)、[in-toto Statement](https://github.com/in-toto/attestation/blob/main/spec/v1/statement.md) 与 [SLSA 1.2](https://slsa.dev/spec/v1.2/) | 分开不可变主体、外部陈述、验证者政策、验证结果和依赖方使用决定 | 这些标准直接定义 Endem、来源忠实性、目标满足或运行授权 |
| [MCP 2025-11-25 Tasks](https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks) | 外部任务状态只描述被包装请求的执行生命周期；截至 2026-07-16 仍为实验能力 | 远端 `completed` 成为 Endem 内容状态、本地满足结果或最终接受决定 |

**限制条件：**当前没有 producer、inspector、runner、裁剪发布器、签名器、验证器、授权服务、contract 构造器或 evidence 记录器。页面描述的是开发者应遵守的对象边界和验证顺序，不是安装或运行说明。
