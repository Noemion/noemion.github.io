ADR-0012 · 未来安全核心的 Rust 基线

若用户明确开启代码阶段，Poiet 与生产读取核心以安全 Rust 作为实现基线；当前只保存设计约束和历史研究，不编写组件。

Accepted DesignRust 1.97.0 BaselineHistorical EvidenceImplementation Not Started

决定

如果代码阶段以后开启，第一条“poie → elenk”纵向切片以 Rust 1.97.0 stable 作为重新评审起点。未来首版禁止 unsafe，不引入第三方 crate，所有不可信偏移、长度、计数和对齐使用显式受检算术，发行构建继续启用溢出检查。

这项决定不要求整个 Noemion 生态只使用 Rust。Praxor 适配器、界面、协议桥接和未来外部工具可以采用适合自身消费者的语言。Theor 必须另写解析结构和错误路径；即使以后也使用 Rust，也不得复用 Poiet 解析代码、生成代码或错误分类实现。

未来代码阶段的构建政策

  • 代码阶段开启时先以 Rust 1.97.0 为评审基线；若当时工具链已变化，必须记录升级 ADR 并重新评估规范向量、差分、模糊和重复构建方案。
  • 仓库提交 rust-toolchain.toml 与 Cargo.lock，发布不能依赖网络“最新版本”。
  • 第一纵向切片使用 #![forbid(unsafe_code)]，CI 同时拒绝警告；任何未来 unsafe 提案必须单独 ADR、最小封装和 Miri/Sanitizer/模糊证据。
  • 第一纵向切片不使用第三方 crate。新增依赖必须说明不可替代消费者、许可证、维护状态、供应链边界和删除条件。
  • release profile 保持整数溢出检查；不得用 wrapping 运算解释不可信结构字段。
  • C 原型当前冻结在 experiments/ 作为历史材料，不自动运行或扩展；未来即使复用，也只能消费规范字节并给出差分结果，不能链接进 Poiet、Theor 或 CLI。

权威机制与采用边界

Rust 所有权让编译器在无垃圾回收器的情况下管理内存，但安全代码仍可能包含逻辑错误，所以规范向量、资源上限和独立 Theor 不能省略。语言机制见 《The Rust Programming Language》的所有权章节。

Rust 无符号整数提供显式 checked_add 与 checked_mul；项目采用这些返回失败的运算，而不是依赖调试模式 panic 或发行模式行为。接口见 Rust u64 标准库文档。

GNU GCC 提供 AddressSanitizer、UndefinedBehaviorSanitizer 和覆盖引导插桩；这些继续用于 C 差分原型和未来 FFI 边界。选项见 GCC Instrumentation Options。

LLVM libFuzzer 要求目标直接消费任意字节、保持快速和确定，并建议与 Sanitizer 组合；这与 END-FMT 读取器的输入边界一致。要求见 LLVM libFuzzer 文档。

Cargo 会优先使用 Cargo.lock 中的版本以兼顾复现和依赖解析；这支持锁定依赖，但不自动证明二进制可复现。行为见 Cargo 依赖解析文档。

没有采用的方案

方案当前结论重新考虑条件
C 作为生产读取核心原型正确且更小，但语言不提供所需的默认内存边界；当前风险收益不足。Rust 无法满足已测目标平台、审计或互操作要求，且 C 获得更强形式化与持续模糊证据。
Rust 与 C 同时作为生产解析器会形成两套生产权威和更大的补丁面；C 只作非生产预言机。存在必须由 C ABI 直接承载的真实消费者，并能保持权限和失败隔离。
Swift、Go、C++ 或 Zig没有针对当前字节切片形成等量证据，因此不作优劣结论，也不进入首版核心。Rust 或 C 实验暴露无法解决的消费者要求，再用同一向量补做实验。
Python/JavaScript 生产核心当前 Python 只负责规范资料检查；动态运行时不进入第一阶段可信二进制核心。只用于非权威开发工具或外层适配器，并明确不产生规范字节。

仍需证明什么

  • Rust Poiet 与独立 Theor 必须分别实现完整确定性 CBOR 子集，不能共享解析代码。
  • 需要增加截断、非最短整数、乱序键、深度、字符串、节点、边和累计分配边界向量。
  • 需要 Linux 与 macOS 的规范对象字节复现，以及至少两个隔离构建环境的可执行文件比较。
  • 代码阶段开启后，Poiet 与 Theor 必须建立彼此独立的 fuzz 目标，并以 END-P1 正反向量为初始语料；当前没有实现级 fuzz 结果。
  • 需要在出现真实 FFI、包管理器和发行消费者后再决定 C ABI、动态库或静态链接政策。

撤回条件

若 Rust 原型无法在两个目标平台保持无 unsafe、无第三方依赖、稳定诊断和资源上限，或独立 Theor 无法发现其共同假设，必须重新打开语言选择。不得因为已经编写代码而降低 END-CORE 或 END-FMT 要求。
