## 用一次响应丢失的发布调用理解诊断

bounded runner 已把发布请求交给外部系统，但连接在响应返回前中断。此时既不能证明发布成功，也不能证明发布失败；再次调用还可能重复产生副作用。结构化诊断必须准确说明本地为何停止，并把恢复决定留给具名规则和权威。

1. 固定实际输入与调用身份
2. 保留传输和协议来源
3. 选择唯一主阻断诊断
4. 标记后续层尚未执行
5. 限制披露数量与深度
6. 另行核对副作用再决定恢复

| 处理步骤 | 本例必须记录 | 不能直接推出 |
| --- | --- | --- |
| 固定发生 | 精确发布输入、适配器版本、协议/Profile、本地调用身份和 DIA-CAT 版本 | 不能用文件名、模型会话名或“刚才那次发布”代替 |
| 保留来源 | 连接中断、远端主体、请求关联和是否收到结构化响应 | 没有响应不表示远端没有执行 |
| 选择主诊断 | 唯一登记机器码、适用条款、失败层和类型化位置 | 不能按最后到达的异常或模型认为最重要的消息选择 |
| 关闭本次操作 | 后续检查为 `not-run`，不返回部分可信对象或成功状态 | 不能把未执行写成通过，也不能把诊断变成目标 `fault` |
| 给出恢复条件 | `operator-review`；先取得去重、回执或事后状态证据 | “可以重试”不是自动重试、扩权或重复发布的许可 |
| 控制披露 | 脱敏后的有限上下文、截断事实和独立调试材料入口 | 不得回显令牌、请求正文、租户路径或无限异常链 |

这份诊断可以帮助人、自动化和人工智能系统理解同一失败事实，但诊断本身不拥有目标判断、证据评估、最终决定或动作权限。

## 诊断与结果域必须分别报告

| 输入信号 | 诊断可以说明 | 仍由哪个边界决定 |
| --- | --- | --- |
| 容器目录越界 | `structure` 层阻断，后续 Profile 和语义检查为 `not-run` | 没有形成可用对象；不能凭诊断生成目标 `fault` 或权威 `rejected` |
| MCP JSON-RPC 协议错误 | 协议版本、远端主体、结构化错误码和本地映射 | 适配器决定外部来源映射；目标求值仍独立 |
| MCP 工具执行错误 | 工具调用已经到达业务执行层，并返回带来源的失败内容 | 会话策略决定是否还能继续；工具消息不产生 `unmet` |
| 后端超时 | `backend` 层、截止点、已消费预算和未取得的观察 | 求值规则另行区分 `undetermined` 与 `fault` |
| 政策拒绝 step-up | `policy` 层、具名权威和被拒绝的范围 | 当前 session contract 不变；新授权与新会话必须另行建立 |
| A2A Task 报告 `failed` | 外部任务身份、状态、错误对象和协议来源 | 本地证据充分性、目标满足与最终接受仍分别判断 |

诊断回答“哪个操作因什么停止或提示”；结果域回答目标、证据、会话或权威决定处于什么状态。二者可以互相引用，但不能依靠映射表强制转换。

## 一份可供自动化使用的诊断必须固定什么

| 责任与条款 | 机器内容 | 拒绝条件 |
| --- | --- | --- |
| 身份与语境<br>`DIA-IDN-001` · `DIA-PIN-001` | 唯一登记码、稳定标题、生产者、操作、实际输入、规范/Profile/目录版本和本次发生 | 程序解析本地化消息、颜色或路径名取得类别，或把缓存诊断套到另一输入 |
| 层次与位置<br>`DIA-LAY-001` · `DIA-LOC-001` | 失败来源、实际执行到的层次，以及来源范围、字节范围、语义路径、会话绑定或外部请求关联 | 位置无类型、越界、无限、未转义，或外部文本冒充本地语义路径 |
| 主诊断与恢复<br>`DIA-PRI-001` · `DIA-REC-001` | 唯一主阻断码、稳定相关项顺序、恢复分类、前置条件和影响范围 | 并发完成顺序改变主码，或恢复建议授予权限、修改制品、跳过检查和无限重试 |
| 外部来源与披露<br>`DIA-EXT-001` · `DIA-SEC-001` | 协议版本、远端主体、原始结构化状态、受限本地映射、脱敏政策及其信息损失 | 未知外部码降级为成功，消息文本改变结果，或默认诊断包含秘密和未授权正文 |
| 预算与原子失败<br>`DIA-BND-001` · `DIA-ATM-001` | 诊断数量、因果深度、位置、消息、建议和总输出上限；阻断后的 `not-run` 范围 | 分配后才检查上限、递归自报错误、截断丢失主码，或同时返回部分可信成功 |

机器码说明稳定类别，标题和消息服务人类理解。语言、换行、颜色、建议文本和显示位置可以因出口而变化，但机器码、失败层、阻断状态和生产语境必须保持不变。

## 唯一主错误怎样避免并发结果漂移

同一次操作可能发现多个问题：容器越界、成员绑定缺失、远端超时和披露预算耗尽可以同时出现。若主错误由线程完成顺序、哈希遍历或外部返回顺序决定，同一输入就会得到不同自动化行为。

| 选择顺序 | 开发者判断 | 原因 |
| --- | --- | --- |
| 1. 最早安全失败层 | 结构不安全时停止，后续语义只能标为 `not-run` | 没有安全输入就不能可靠产生更深层结论 |
| 2. 规范固定的规则优先级 | 在同一层按该规范固定的规则优先级选择 | 避免实现者或模型自行排列重要性 |
| 3. 规范位置顺序 | 对类型化位置使用固定、可解释的排序 | 避免图遍历、线程和远端到达顺序进入结果 |
| 4. 稳定机器码 | 仍相同时用目录内机器码完成确定性排序 | 保证重复运行和不同呈现出口得到同一主码 |
| 5. 限制相关项 | 先保存主码，再按上限附加有父子关系的相关诊断和截断计数 | 诊断风暴不能淹没主错误或消耗无限资源 |

阻断诊断使相关操作原子失败。可安全读取的片段只能作为无权威观察，不能成为 Endem、Endem closure、session contract、evidence entry、签名请求、实时能力，或表示检查已经通过的内部引用。

## 可行动建议为什么不等于自动行动

DIA-CORE 只允许六种恢复分类：`fix-input`、`refresh-binding`、`obtain-authority`、`retry-transient`、`operator-review` 和 `do-not-retry`。分类必须同时声明前置条件和影响范围；实际重试继续消耗当前 session contract 预算并受取消约束。

模型可以根据结构化诊断修正候选参数、解释缺少的材料或向操作者提出申请，但不能从人类消息中推导命令。删除、发布、修改已签制品、扩大 OAuth scope 或创建新会话仍须经过原有授权与确认边界。

| 输出或外部规范 | 可以保留什么 | Noemion 不继承什么 |
| --- | --- | --- |
| [GNU GCC 诊断输出](https://gcc.gnu.org/onlinedocs/gcc/Diagnostic-Message-Formatting-Options.html) | 同一诊断可以进入文本、SARIF 和其他输出出口；位置、路径与修复提示可以分别呈现 | 不复制 GCC 选项、编译器严重度或以颜色和文本作为机器接口 |
| [SARIF 2.1.0](https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html) | 规则身份、物理或逻辑位置、相关位置和代码流 | 静态分析结果格式不表达 Noemion 的能力、目标满足或验收权威 |
| [RFC 9457 Problem Details](https://www.rfc-editor.org/rfc/rfc9457.html) | 在 HTTP 中分开问题类型、具体发生、状态提示、标题和详情，并限制错误正文披露 | HTTP 状态不替代本地失败层，程序也不能解析人类 `detail` 取得权限和结果 |
| [MCP 2025-11-25 工具规范](https://modelcontextprotocol.io/specification/2025-11-25/server/tools) | 明确区分 JSON-RPC 协议错误和带 `isError` 的工具执行错误 | 两类外部错误都不能直接映射为目标 `unmet`、最终 `rejected` 或自动重试许可 |
| [A2A 1.0](https://a2a-protocol.org/v1.0.0/specification/) | 不同协议绑定保留 Task、Message、Artifact、状态和错误对象的来源语义 | 远端 Task 状态和消息不取得本地结果、证据或授权决定权 |

**复核日期：**2026-07-16。任何输出适配都必须固定外部规范版本，并证明导出前后的机器码、生产语境、主诊断、披露和阻断状态没有改变。

## 当前还不能承诺哪种诊断格式或自动修复

本决定建立 `DIA-CORE 0.1.0-draft` 和跨对象的诊断目录（`DIA-CAT`），但不冻结 JSON、CBOR、SARIF、Problem Details、JSON-RPC、MCP、CLI 文本、退出状态、媒体类型、持久日志、遥测、堆栈、补丁或跨进程关联格式。当前没有诊断生产器、渲染器、日志服务、重试引擎、协议适配器或 CLI。

现有十条规范要求、十类威胁、十五个自然语言场景，以及十个允许包络和十个确定拒绝向量，只证明资料矩阵一致。它们不能证明诊断格式稳定、秘密已脱敏、主错误算法已实现、外部协议映射正确或自动修复安全。

> **名称与口头边界：**面向开发者首次出现时先写“结构化诊断核心规范（DIA-CORE）”和“诊断目录（DIA-CAT）”。不要只说“错误”“失败了”或“DIA”，应说明“主阻断诊断”“协议错误”“工具执行错误”或“目标求值故障”；机器码用于精确引用，不承担口头传播名称。
