## 用同一份 Endem 理解两种变化

假设一份 END-P2 形成制品在周一获得发布团队确认，周三该授权被撤销。文件字节没有变化，因此内容身份保持不变；周三的“是否仍可用于发布”必须重新评估。

1. 规范字节
2. 内容身份
3. 外部授权绑定
4. 截止点评估
5. 满足结果
6. 最终决定

| 发生的变化 | 内容身份 | 需要产生的新结果 |
| --- | --- | --- |
| 同一 Profile、同一规范字节，由另一名合格主体确认 | 不变 | 新的授权伴随关系及其独立评估 |
| 授权过期、撤销或政策版本失效 | 不变 | 相对新截止点的未满足或未求值结论 |
| 新增签名、evidence 或最终决定 | 不变 | 新的签名、证据或决定关系；各自保留结果域 |
| 修改任一语义记录或规范字节 | 改变 | 新的 Endem 内容身份，并重新绑定全部伴随关系 |
| 未来从形成版派生裁剪发布版 | 改变 | 新 Profile、新身份、来源关系和重新验证结果 |

**当前策略：**END-P2 仍只保存六个语义记录。文件可以独立通过容器与 Profile 检查，但文件内权威名称不能证明授权已经发生。

## 为什么权威名称只能是待核对选择器

| 文件内表达 | 它实际能够说明什么 | 完整内容接受还缺什么 |
| --- | --- | --- |
| `projection.kind` | 候选来自哪类确定性规则或具名权威 | 规则版本、声明输入域、输出位置与实际执行结果 |
| `projection.id` | 指向一个待核对规则、角色或主体名称 | 当前主体资格、精确候选、语义位置、范围和截止点 |
| `decision_authority` | 指出谁应当作出判断 | 该主体是否看过当前安全视图并形成有效决定 |
| `required_evidence` | 指出内容要求的证据选择器 | 实际 evidence、有效性、覆盖度、撤销状态和适用判据 |
| 签名者或本地 ACL 名称 | 为外部验证或本地政策提供查找线索 | 签名陈述、信任政策、授权范围和依赖方决定 |

读取器不得通过当前环境、网络查询、默认目录或缓存命中，把这些名称升级成授权或证据已经存在。名称用于定位待核对对象，不承担认证、授权或最终判断。

## 文件内结论与外部前置条件怎样报告

| 检查层 | 当前字节可以直接证明什么 | 必须单独报告什么 |
| --- | --- | --- |
| 容器接受 | 前导、目录、范围、资源上限和确定性载荷满足 END-FMT | 不解释六个语义记录是否构成可接受内容 |
| Profile 接受 | 记录种类、字段、枚举、排序和引用满足 END-P2 | END-P2 单文件最高只能声称 Profile 接受 |
| 内容接受 | 文件内结构提供待核对候选与选择器 | 外部规则、授权、身份、签名和 evidence 是否取得且精确绑定 |
| 满足判断 | 目标、判据和要求可以被读取 | 现实观察、有效证据、覆盖度和结果计算 |
| 最终决定 | 可以定位具名决定主体 | 有效决定权、输入结果、范围和 `accepted / rejected / deferred` |

外部前置条件缺失、未执行、版本不匹配、错绑、撤销或无法核对时，报告必须明确写成“未满足”或“未求值”。不得省略该层，也不得把调用者断言改写为通过。

## 四项条款怎样关闭原有矛盾

| 条款 | 开发者必须遵守 | 失败后果 |
| --- | --- | --- |
| `END-CON-006` | 分开报告文件内结论与外部前置条件，并声明单文件最高可声称层次 | 缺失或未求值层次不能隐式通过 |
| `END-AUT-002` | 权威、规则和证据名称都只作为待核对选择器 | 缺失、错绑、拒绝、延后、过期或撤销不能产生已授权投影 |
| `END-ID-002` | 授权、签名、evidence、满足与最终决定不进入 Endem 内容身份 | 字节改变必须形成新身份；授权变化只产生新伴随评估 |
| `END-FMT-015` | END-P2 不内联政策、目录、决定、签名、证据或验证材料 | 读取器不能从环境或网络搜索补齐文件含义 |

两个字节完全相同、外部授权证据不同的 Endem 必须进入同一内容身份等价组。相反，只要六个语义面或规范字节改变，就必须建立新身份，不能把语义修订伪装成授权状态更新。

## 外部伴随机制提供哪些工程参照

| 外部机制 | 支持的分层纪律 | Noemion 不直接采用什么 |
| --- | --- | --- |
| in-toto Attestation Framework 1.2 | Predicate、Statement、Envelope 与 Bundle 独立协作，陈述绑定精确主体 | 不复制其 JSON、包络或供应链谓词 |
| The Update Framework 1.0.35 | 角色签署描述目标文件的元数据，而不是目标文件本身；授权还受版本与时效约束 | 不采用 TUF 角色、仓库格式或更新工作流 |
| GNU Guix | 替代物签名必须经过客户端本地授权密钥政策 | 服务器名称或签名存在不能成为语义授权 |
| MCP 2025-11-25 Elicitation | URL 模式中的 `accept` 只表示用户同意进入交互，不表示外部交互完成 | 界面动作不能越级成为现实事实或最终决定 |
| ELF `SHF_OS_NONCONFORMING` | 未知且需要特殊处理的关键语义必须拒绝，不能静默忽略 | 不把 ELF 节、标志或装载语义移植到 Endem |

这些资料共同说明对象、描述对象的陈述、验证材料、本地信任政策和依赖方决定可以保持不同生命周期。它们只提供反例与分层参照，不定义 Endem 的伴随物理格式。

## 当前还不能承诺哪种伴随 Profile

SCN-028 至 SCN-030、语义向量的外部前置条件矩阵，以及缺失和错绑授权诊断已经验证规范关系。它们不能证明伴随资料怎样封装、分发、撤销、缓存或跨系统查询。

**待定内容：**授权伴随关系、权威目录、政策语言、规则登记表、决定事件、撤销分发和符合性报告仍没有物理 Profile。当前改动只修正规范边界，不创建新文件、包络、服务或组件。
