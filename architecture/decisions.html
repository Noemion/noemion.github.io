怎样阅读这些决定

架构决定约束组件职责和后续规范方向，但不等同于已经发布的软件接口。每项决定回答“为什么这样分层、哪些做法被排除、影响哪些页面”；具体字段、编码、ABI 和错误码仍须进入正式规范与一致性测试。

决定发生变化时，相关架构、组件、工具、手册和测试必须在同一次变更中更新。尚无充分证据的问题继续保留在开放问题，不能用实现便利静默替代。

当前决定登记

决定采用的边界直接影响
命名与语义边界Noesis 表示确定性理解与编译活动；Noema 表示机器对象；Horizon 表示未完全展开的背景；Fulfillment 表示受约束求解。NIR、NOBJ、HOBJ、组件名、工具名与公开术语。
Agent Harness 控制平面Harness 位于 Fulfillment Runtime 外侧，负责上下文、会话级能力、策略、观察、预算和人工升级；模型不能绕过它调用能力。Agent Harness、noemexecute、noemobserve、noemcoverage 与 noemevaluate。
工具链产物与外部签名发布链依次形成 Release Object、Release Coverage Proof、Unsigned Package Candidate、Signing Request、Signature Response 与 Signed Noemion Package；私钥只在外部签名系统。noemreduce、noemcoverage、noembundle、noemexecute 与对象生命周期。
候选、能力与验收分层Candidate Envelope 不是 NIR；Capability Requirement 不是实时句柄；Candidate Assessment、Evidence Closure 和 Acceptance Decision 是三个不同结果。Noesis Core、Object System、Agent Harness、Runtime 与所有运行证据工具。

已排除的捷径

  • 模型输出不能直接生成 NIR/NOBJ，也不能因结构合法被当作来源忠实语义。
  • 签名不能替代结构、语义、权限和环境策略验证。
  • Loaded State 不携带可直接执行的会话能力句柄。
  • Runtime 候选通过评价，不等于任务已经最终验收。
  • Release Coverage Proof、Evidence Closure Report、离线评估和跨工具认证都不能替代发布授权或人工专属判断。

尚未冻结的内容

Candidate Envelope、Source Binding Decision、Loaded State Handle、Runtime Request、Capability Request、Capability Observation、Acceptance Policy、Acceptance Decision、签名封装和撤销协议仍需规范、威胁模型、原型和正反测试。当前登记只约束职责，不承诺字段或调用语法。
