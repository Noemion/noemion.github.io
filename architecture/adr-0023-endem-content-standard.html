ADR-0023
Endem 内容标准

把“所有 Endem 都要表达什么”“某一版本允许哪些字段”和“这些内容怎样装进字节文件”分开，避免一处变化牵动全部工具。

当前策略END-CORE 0.1.0-draft分层符合性尚未实现

直接结论

项目需要一份目标文件内容标准，但不需要再造一个与 END-CORE 重复的规范。现有 END-CORE 正式承担通用内容标准，END-P1 承担来源保留的形成 Profile，END-FMT 只承担物理容器。最终发布 Profile 由 ADR-0036 另行约束，当前尚未定义。

这三层必须分别版本化、分别验证。能读出目录不表示内容合格，字段集合闭合不表示语义成立，内容成立也不表示目标已经实现或被权威接受。

ELF 提供的真正借鉴

ELF 不是由 IETF RFC 定义。它主要由 System V ABI 中的通用 ELF 规范，以及面向具体处理器的 psABI 补充共同定义。通用规范说明头部、节、符号、重定位和装载视图；处理器补充再约束机器相关内容。

ELF 分层Endem 对应职责不能照搬
通用对象规则END-CORE 定义所有 Endem 共享的内容语义、不变量与符合性边界机器指令、地址、处理器寄存器和传统链接语义
处理器或平台补充END-P* 固定某一能力集合的字段、枚举、规范顺序、引用和资源上限把运行平台、模型供应商或操作系统变成自然语言语义
文件结构与数据表示END-FMT 定义字节路线图、记录目录、确定性编码和有界读取用容器结构替代语义验证
工具与实现未来 Ktisor 与独立 Theor 共享规范和向量，不共享解析实现把某个实现的内部结构当成标准

权威决定
  • 容器接受只说明头部、目录、范围与编码可以安全解释。
  • Profile 接受只说明字段、枚举、排序、引用和资源约束符合指定版本。
  • 内容接受说明对象满足 END-CORE 的语义与授权义务，但不证明现实中的目标事态成立。
  • END-P1 不承载授权决定；只有精确外部前置条件已核对时，内容接受才能继续。
  • 签名、Iknem、运行会话、满足判断和最终决定继续保持各自独立的结果域。

扩展规则

第一版稳定标准之前不承诺向后兼容。新增 Profile 或字段时直接替换旧设计，不保留含义漂移的别名；但每次改变都必须显式绑定新版本并提供威胁、支持案例、反例和正反向量。

  • Profile 可以收窄能力或补齐物理字段，不能改写六个语义面的职责。
  • 未知必需能力关闭失败；当前没有供应商私有语义区或运行时动态补字段。
  • 没有具名生产者、消费者和失败责任的字段不进入内容标准。
  • 模型候选、提示、计划、推理轨迹、实时凭据和遥测导出字段不进入 Endem 内容身份。
  • 授权、签名、证据、撤销和最终决定也不进入内容身份；它们使用外部伴随关系。

权威资料与采用限制

  • ELF Object File Format 的文件结构说明分开链接视图与执行视图，并让固定头部成为解释文件的路线图。Endem 借鉴分层和可独立解释性，不采用机器程序语义。
  • ELF Sections定义节目录、类型、链接信息与扩展范围。Endem 借鉴显式目录和封闭类型，不把语义面降格成任意节名。
  • PowerPC 64 位 ELF ABI 补充明确区分通用 ABI 与处理器补充。Endem 的 Profile 类比这种职责分层，但不对应处理器。
  • RFC 2119与 RFC 8174只提供规范要求用语，不定义 Endem 内容。

后果与当前状态

  • END-CORE 的标题、范围与符合性条款明确成为内容标准，不新增重复规范 ID。
  • END-P1 继续是实验性设计 Profile；它只支持 nascent、kine 和当前封闭字段，不代表稳定 ABI。
  • ADR-0030 已修正内容身份与授权伴随关系；END-P1 单文件最高只支持 Profile 接受。
  • 当前规范、登记与向量的一致性结果不能证明任何写入器、读取器、CLI 或运行时已经实现。
