ADR-0013
END-P1 封闭内容 Profile

保留 END-P0 作为结构实验，新建 END-P1 作为首个设计 Profile，明确六个记录的字段键、枚举、排序和引用规则。

当前草案END-P1Profile 2接口待定

直接决定

Ktisor 的首个纵向切片不写 END-P0。END-P0 只保留为固定头部、目录和受限 CBOR 的结构实验；生产候选改用 profile_id=2 的 END-P1，并且必须在六个记录内遵守封闭字段 schema。

这避免两种错误：一是把六个空映射误称为 Endem，二是让未来实现私自发明字段。END-P1 仍是实验性设计 Profile，不是稳定 ABI；发现反例时可以直接修订，但必须同步 ADR、Profile、正反字节向量和验证方案。

为什么不能继续复用 END-P0

现有证据能够证明不能证明
END-P0 六个结构记录读取器能安全定位六个关键载荷。载荷内有哪些字段、引用和语义约束。
七个语义 JSON 向量正负极性的两个可接受模型和五种拒绝原因能被实际执行。JSON 是规范字节，或任意 JSON 可以直接写入 Endem。
C/Rust 结构实验两种语言能对同一结构错误给出一致首错。任何原型已经实现 Ktisor、Theor 或 END-CORE。

END-P1 关闭哪些接口

  • 六个记录继续使用 kind 1–6；每个记录的根映射只允许登记键，未知键关闭失败。
  • 符号、关系、角色、事态、观察要求和待确认项使用显式标识与交叉引用，不复制自由文本图。
  • 数组按稳定身份或编码值排序，重复项拒绝；Ktisor 不采用输入顺序或哈希表遍历顺序。
  • 来源范围以 Unicode 标量计数，并在解释投影前检查；媒体类型与语言标签保留原始标准化文本。
  • 第一切片只编码 kine。ADR-0016 已固定 mene 的抽象时间语义，但 END-P1 没有相应字段；它必须等待新的 Profile 与正反字节向量。
  • apor 即使为空也编码为含显式 items 数组的记录；非空项保存候选、冲突、影响范围、允许的解决方式和决定主体。

字段、枚举与上限见机器可读 END-P1 Profile。

外部格式只提供机制

RFC 8949 区分 well-formed、valid 与 application-expected 三层接受，并允许应用定义自己的具体数据模型。END-P1 采用这种分层和确定性编码，不采用通用 CBOR 的全部类型或开放扩展。

语言字段遵守 BCP 47，媒体字段遵守 RFC 6838 媒体类型结构。两者只描述来源，不证明投影正确。

来源位置使用 Unicode 标量范围，不能把字节、UTF-16 code unit 和用户可见字素混为同一单位。单位定义见 Unicode 17 核心规范第 2 章。

明确不关闭的接口

END-P1 不定义量化、时间区间、单位换算、摘要、签名、Semantic Key、授权决定、Synem、coherent/attested、压缩、加密、远程披露或稳定 ABI。这些能力没有真实向量和消费者前，不占用字段键，也不以保留空字段假装完成。

projection.id、decision_authority 与 required_iknem 只是待核对选择器。按照 ADR-0030，END-P1 单文件最高只能声称 Profile 接受，不能自证授权或完整内容接受。

进入实现的条件

END-P1 已覆盖接受、未知字段、悬空引用和来源范围案例。这些材料只满足实现前的规范准备条件；当前没有生产 Ktisor 工作区、Ktisor/Theor 实现或符合性声明。
