项目/架构设计/架构决策/ADR-0027

ADR-0027
签名有效
不等于制品获准

系统必须先确认签名究竟覆盖哪一份精确对象，再分别判断签名者身份、角色权限、陈述有效性、制品接受和动作授权。任何一步都不能由摘要或签名自动代替。

当前策略ID-CORE 0.1.0-draft算法无关尚无密码实现

用一次裁剪发布理解完整判断链

团队先形成一份保留原始自然语言的目标制品，再为公开发布移除原文和可逆重建材料。裁剪发布版已经是另一组字节，不能沿用形成版文件名、摘要、签名、证据或接受状态。

验证陈述签名覆盖对象、陈述类型、用途、受众、主体和受保护上下文签名者有权批准、陈述仍适用或内容必然正确

作出决定具名权威结合对象匹配、适用证据和政策决定是否接受签名数量、证书、透明日志包含或摘要相同自动形成接受

身份相同只说明同一身份语境中的精确字节相同。密码验证、主体认证、角色授权、陈述适用性、目标满足和最终接受继续属于不同结果域。

精确身份为什么不只是一个摘要字符串

身份部分必须固定缺失时的风险
身份域与版本
ID-DOM-001		对象种类、对象规范、物理表示、内容 Profile 和版本化身份域相同摘要文本在 Endem、Endem closure、evidence entry 或签名陈述之间失去语境
字节输入
ID-BYT-001		摘要覆盖的字节范围、顺序、长度、包含项、排除项和规范化规则实现默认规范化、可变输入或自包含摘要导致验证者计算不同对象
算法政策
ID-ALG-001		算法、参数、用途、输出长度、允许状态和未知算法行为试到成功、静默替换、降级或截断长度混淆
完整引用
ID-REF-001		完整身份必须唯一闭合到一个允许对象路径、URL、latest、外部 Task ID 或缓存顺序选择错误对象
显示别名
ID-DSP-001		短摘要只用于显示；歧义时扩展显示或要求完整值首个前缀匹配和短值碰撞进入安全绑定

RFC 6920用摘要算法、长度和二进制值建立名称与数据的完整性绑定，同时把一般对象的具体摘要输入留给采用它的规范。Noemion 因此必须额外固定对象语境与字节边界，不能只复制一种摘要文本语法。

SHA-256 目前只出现在提案向量示例中，不是发行算法决定。精确身份也不定义语义等价：字节相同不证明陈述为真，字节不同也不自动证明目标含义不同。

有类型签名怎样避免跨对象和跨用途重放

签名层次必须保护或固定不能怎样解释
签名陈述
ID-STM-001		载荷类型、对象完整身份、对象种类与 Profile、陈述种类、用途、受众、声明主体、政策和关键字段裸摘要签名不能从公开发布证明变成能力授权，也不能从 Endem 陈述变成 evidence entry 决定
验证包络
ID-ENV-001		签名、证书或公钥材料、可信时间、透明日志与撤销状态；包络拥有自己的身份包络不能改变被签对象身份，缺失材料也不能静默忽略
密码验证受保护算法、实际签名字节、候选密钥和验证结果只证明对应密钥对受保护字节产生了可验证签名
主体与角色
ID-AUT-001		密钥对应谁、该主体在陈述语境中具有什么角色和范围密码正确不自动证明主体有发布、部署或接受权限
适用性与决定陈述用途、受众、对象、政策、截止点和具名决定权威透明日志包含、证书链或多签数量不能直接产生真实、满足或接受

RFC 9052明确说明 COSE 的 kid 可能不唯一，只是筛选候选密钥的提示；DSSE则把 payloadType 与载荷共同纳入签名协议。两者支持上下文与类型分离，但本阶段不选择 COSE、DSSE 或其他物理包络。

证书、时间、透明日志和撤销怎样影响有效性

验证材料可以支持不能支持
证书或公钥材料按具名信任根把验证密钥关联到候选主体主体当前角色、陈述为真或制品已获准
可信时间材料签名或日志事件相对具名截止点的时间关系本地时钟、未认证 signingTime 或上传时间替代可信时间
透明日志签名材料被特定日志接收、包含或一致展示的有限关系日志本身成为信任根、撤销服务或最终接受者
撤销与状态材料
ID-VAL-001		密钥、证书或陈述在截止点的适用状态及材料新鲜度撤销改变历史对象身份，或旧缓存成功永久有效
Sigstore Bundle将签名内容与证书、时间戳和透明日志材料组合为可离线携带的验证包Bundle 存在不等于本地授权、目标满足或最终接受

密钥轮换、证书过期和撤销会改变某项陈述在特定截止点的验证结论，却不会改变已存在对象的内容身份。验证缓存必须绑定信任根、政策、截止点、状态材料及其新鲜度。

可复现构建和派生制品为什么仍需新身份

工程机制可以证明不能证明
SLSA 1.2先匹配陈述主体摘要，再按验证者信任根、构建者、参数和政策检查来源来源验证通过不自动证明业务目标满足或动作获准
GNU ld build ID关联原始链接输出与调试、生产或符号资料不是当前文件校验和；后续工具改变文件时 build ID 可以不变
GNU Guix challenge比较彼此独立的构建产出是否逐字节相同字节不同不能仅凭摘要定位原因，字节相同也不证明输入政策正确
逐字节可复现
ID-REP-001		精确输入闭包、工具、参数、环境约束、独立路径和完整输出比较相同日志、共享缓存、单次重跑或 build ID 冒充独立复现
派生关系
ID-REL-001		来源与派生物各有身份，并记录变换、生产者、保留内容、损失和验证条件裁剪、压缩、迁移或重编码结果继承来源身份、签名、evidence entry、接受状态或能力

含来源形成版、裁剪发布版、受控来源伴随资料和外置验证包络都是不同对象。没有伴随资料的验证者可以核对发布字节与适用外部陈述，但不能声称已独立复核原始自然语言到目标结构的忠实转换。

当前还不能承诺哪些算法、包络和信任设施

本决定建立 ID-CORE 0.1.0-draft 的跨对象边界，但不创建新制品、命令、Semantic Key、摘要语法、算法登记、签名格式、证书 Profile、透明日志、时间戳、撤销分发或来源伴随物理格式。当前没有摘要器、签名器、验证器、透明日志客户端、可复现构建系统或发布组件。

十二项条款、十二类威胁、十八个设计场景，以及十二个允许包络和十二个确定拒绝向量只证明资料矩阵一致。它们不能证明算法安全、密钥管理有效、签名互操作、撤销及时、构建独立或发布授权成立。

名称与口头边界：人类说明先使用“精确内容身份”“定位标识”“有类型签名陈述”“验证材料”和“最终接受”。不要把 identity 与 identifier 都简称为“ID”，也不要用含义过宽的 attestation 概括签名、来源、有效性和授权；ID-CORE 是机器规范 ID，不承担发行读音名称。
