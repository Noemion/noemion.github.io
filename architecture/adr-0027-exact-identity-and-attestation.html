ADR-0027
精确内容身份与签名边界

让每项制品、引用和签名都指向可核对的精确字节，同时不把完整性、来源、有效性、权威和可复现性混成一个结论。

当前草案ID-CORE 0.1.0-draftAlgorithm AgnosticNo Crypto Code

直接结论

Noemion 需要一份跨 Endem、Synem、Iknem、发布签名和伴随制品的共同内容身份标准。否则“精确身份”只是各页反复使用但没有共同定义的承诺。

ID-CORE 固定十二条抽象规则。它不创建新制品或文件格式，也不选择摘要与签名算法。它把对象匹配、签名验证、主体授权、截止点有效性、最终决定和可复现性拆成独立判断。

RFC 6920 用算法、长度和完整摘要值建立名称与数据的完整性绑定。Noemion 还需要对象种类、规范、表示、Profile 与身份域，因为同一摘要文本不能在 Endem、Synem、Iknem 和签名陈述之间失去语境。

文件名、路径、URL、版本标签、latest、build ID、短摘要和密钥 ID 只能帮助人或程序定位候选。任何会影响闭包、证据、签名、政策或执行的引用都必须最终闭合到完整身份。

签名怎样避免越权解释

签名不只覆盖一个裸摘要。签名陈述必须同时覆盖载荷类型、对象身份、陈述种类、用途、受众、声明主体、政策和影响解释的关键字段。这样，公共发布签名不能被搬成能力授权，Endem 陈述不能被重新解释为 Iknem 决定。

证书、公钥提示、可信时间、透明日志和撤销材料放在外置验证包络中。它们帮助验证陈述，却不改变被签对象的精确身份。kid 只是密钥查找提示，透明日志包含只支持有限公开记录关系，两者都不是权威或最终接受。

为什么 build ID 和短摘要不够

GNU ld 明确说明 build ID 标识原始链接输出，不是当前文件内容校验和。后续工具可以改变文件而不改变 build ID。因此 Noemion 把 build ID 保留为未来调试或生产关联候选，不允许它承担精确身份。

短摘要同样只用于显示。发生前缀歧义时，界面必须扩展显示或要求完整值，不能按第一个匹配、缓存顺序或路径选择对象。

可复现性和伴随制品

GNU Guix 的 --check 与 guix challenge 把可复现性落到逐字节比较。Noemion 进一步要求声明精确输入、依赖闭包、工具版本、参数、环境约束和独立产出路径。输出不同只能证明字节不同，不能仅靠摘要定位原因。

未来形成的裁剪制品、调试伴随文件、压缩副本和迁移结果都拥有自己的身份。它们通过版本化关系说明来源、变换、保留内容和损失，不能继承来源身份、签名、Iknem、接受状态或权限。

权威资料与采用限制

  • RFC 6920 Naming Things with Hashes提供完整摘要比较和名称数据完整性边界；项目增加对象域与规范语境。
  • RFC 9052 COSE提供受保护参数、算法认证与密钥提示边界；项目不在本阶段选择 COSE 编码。
  • DSSE 规范展示载荷类型与载荷共同签署的类型分离思想；项目不直接采用其物理包络。
  • Sigstore Bundle Format展示签名、证书、时间和透明日志材料的离线组合；项目不把 Sigstore 身份或日志包含当作本地授权。
  • SLSA 1.2 制品验证要求先匹配制品摘要，再按验证者信任根、构建者和参数判断来源。
  • GNU ld build ID明确区分原始链接标识与当前文件校验和。
  • GNU Guix challenge提供独立产出逐字节比较的实践参照。

后果与当前状态

  • ID-CORE 0.1.0-draft定义 12 条跨制品义务。
  • 12 类威胁、18 个设计场景和 24 个正反向量覆盖当前抽象边界。
  • SHA-256 只用于提案向量示例；发行摘要算法、签名算法、物理包络、证书 Profile、透明日志和撤销协议都待确定。
  • 当前没有摘要器、签名器、验证器、透明日志客户端、可复现构建系统或发布组件。
