ADR-0025
结构化诊断边界

让人、自动化和模型看到同一失败事实，同时阻止消息文本、外部状态和恢复建议改变本地语义或权限。

Accepted DesignDIA-CORE 0.1.0-draftFailure FirstNo Component Code

直接结论

所有 Noemion 工程对象共享一套诊断内容标准，但诊断不成为新的制品、结果域或权限来源。

每项诊断必须保留稳定机器码、生产语境、失败层次、类型化位置、主错误选择、有限恢复分类、外部来源、披露政策、资源预算和原子失败状态。文本、颜色、语言、SARIF、HTTP、MCP 与未来 CLI 只负责呈现或传输，不能改写这些内容。

本决定建立 DIA-CORE 0.1.0-draft 和跨对象 DIA-CAT 目录。它不建立诊断运行时、日志服务、重试引擎、协议适配器、物理格式或可执行 CLI。

现有情况造成的问题本决定

错误层次不能替代结果域

诊断说明某个操作为什么停止或需要注意；结果说明目标、决定、会话或证据处于什么状态。二者可以有关联，但不能相互强制转换。

面向 Agent 的恢复边界

模型需要可行动反馈，但“可行动”不等于“可以自行行动”。DIA-CORE 只登记六种恢复分类：修复输入、刷新绑定、取得授权、短暂错误重试、操作员复核和不得重试。

  • 每项建议必须说明前置条件和影响范围，不能把自由文本当作命令。
  • 重试、子 Agent 和工具回调共同消费当前 Dromen 的有限预算。
  • 取得授权只说明需要哪个具名权威；它不授予能力，也不关闭 apor。
  • 修改已签制品、发布、删除或扩权等外部副作用必须经过原有权限与确认边界。
  • 同一确定性输入重复产生相同拒绝时，恢复只能要求改变输入或停止。

GNU 与开放标准提供的对照

  • GNU GCC 诊断格式选项允许同一诊断输出为文本或 SARIF，并保存位置、规则和相关事件。Noemion 借鉴“诊断内容与输出出口分开”，不复制 GCC 参数或编译器严重度。
  • SARIF 2.1.0提供规则身份、物理/逻辑位置、相关位置和代码流。它适合未来静态检查导出，但不能表达能力、满足或验收权威。
  • RFC 9457 Problem Details区分问题类型、人类标题、具体发生和详情，并警告错误响应可能泄露实现与隐私信息。Noemion 借鉴身份和发生分离，不把 HTTP 状态当成本地结果。
  • MCP 2025-11-25 工具规范区分 JSON-RPC 协议错误和带 isError 的工具执行错误。Praxor 外缘必须保留这种来源差异，再进行本地有限映射。

规范影响与验证

  • 新增 DIA-CORE 0.1.0-draft 十条跨对象诊断内容要求。
  • 把原 Endem 专用目录替换为 DIA-CAT，保留现有 Endem 草案码并补齐 Synem、Dromen、Tekmor 和诊断系统码。
  • 新增十类诊断威胁和十五个非规范自然语言场景。
  • 二十个提案向量覆盖十个允许诊断包络和十个确定拒绝；检查器只证明规范资料一致。
  • END-CORE、END-FMT、END-P1、SYN-CORE、DRO-CORE、TEK-CORE 与任何规范字节保持不变。

仍未冻结的内容

JSON、CBOR、SARIF、Problem Details、MCP、CLI 文本、退出状态、持久日志、遥测、跨进程关联、堆栈和修复补丁格式仍未冻结。当前没有诊断生产器、渲染器、日志服务、重试引擎、协议适配器或 CLI；项目仍未进入组件代码阶段。
