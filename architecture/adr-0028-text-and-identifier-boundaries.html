ADR-0028
文本与标识符边界

让来源字节、解码文字、机器标识符、显示视图和模型实际输入分别可核对，避免相同字形掩盖不同机器内容。

Accepted DraftTXT-CORE 0.1.0-draftUnicode BoundariesNo Unicode Code

直接结论

Noemion 需要一份跨制品的文本边界规范。UTF-8 只能回答字节怎样解码，不能自动回答字符串怎样比较、怎样定位、怎样显示，也不能保证人和模型看到同一内容。

采用 TXT-CORE 0.1.0-draft。它固定十二条抽象规则，不增加 Endem 字段，不创建文本对象，也不实现 Unicode 处理器。当前来源内容不做规范化，END-P1 结构标识符继续只允许 ASCII。

为什么可见文字不能代替机器内容

“记号是一个符号中可以被感官感知到的东西。”

路德维希·维特根斯坦《逻辑哲学论》3.32

这里给出的工程启发很有限但很关键：可见记号只是可以被感知的层，编码序列、结构位置和已授权意义仍须分别确定。3.321 又提醒同一个记号可以被不同符号共有，所以界面中的相同字形不能自动证明机器身份或语义相等。

这些命题不决定 UTF-8、NFC、标识符语法或安全策略。具体规则仍由版本化规范、Profile、威胁模型和向量定义，不能把哲学引文当作实现证据。

必须分开的六个对象

原始来源字节
严格解码文本
显式变换结果
结构字段与范围
人类显示视图
模型实际输入

每一层都有自己的身份和责任。解码文本不能声称保留原文件字节；显示视图不能参与安全引用；模型输入不能只用界面截图或日志摘要证明；规范化结果也不能覆盖来源后再冒充原文。

为何当前结构标识符保持 ASCII

UAX #31 和 RFC 8264 表明，国际化标识符需要明确字符集合、映射、规范化、大小写、双向规则、比较方式和版本稳定性。UTS #39 还能发现部分同形风险，但风险骨架不是身份。

当前 END-P1 已有封闭 ASCII 语法，真实消费者尚未证明 Unicode 标识符的必要性。继续使用 ASCII 可以在解析入口关闭方向控制、混合脚本、默认可忽略字符和 locale 差异。未来如果国际化标识符拥有不可替代的消费者，必须通过独立 ADR 和 Profile 扩展，而不是放宽现有正则。

为何不统一规范化所有文本

UAX #15 定义 NFC、NFD、NFKC 和 NFKD，但不同形式服务不同用途。兼容规范化会主动消除部分字符差异；即使规范等价成立，也不能推出原始字节相同或自然语言意义相同。

因此当前来源内容、精确内容身份和 ASCII 结构标识符一律使用 none。未来任何规范化都必须绑定输入槽、Unicode 版本、形式、输出用途、身份影响和原始输入关系。比较也必须说明是在精确字节、Unicode 标量、规范化结果、区域排序还是同形风险域中进行。

双向文字和隐藏字符怎样处理

UAX #9 负责从逻辑顺序计算显示顺序。Noemion 的身份、范围与解析只使用逻辑顺序；显示器可以重排，却不能把视觉顺序写回对象。安全审查视图必须显示方向控制、零宽字符、变体选择符、连接符、非 ASCII 空白和其他默认可忽略字符的位置。

UTS #39 的同形检测只产生风险提示。提示不能合并对象、自动改写来源或单独证明恶意。普通来源文本可以保存这些码点，但结构标识符由 ASCII Profile 直接拒绝。

模型时代新增的责任

已有研究展示了不可见字符、同形替换和变体选择符怎样改变 NLP 或模型处理，同时逃过普通视觉审查。这些结果会随模型和 tokenizer 变化，因此只作为威胁证据，不冻结成某个攻击成功率或检测算法。

未来模型调用必须绑定实际输入身份、预处理链、不可见字符清单、模型与 tokenizer 身份和人类复核视图。即使这些材料齐全，输出仍是模型候选，不能取得规范写入、授权或最终验收权。

权威资料与采用限制

  • RFC 3629 UTF-8定义合法标量与唯一最短编码；它不定义文本比较或语义等价。
  • Unicode UAX #15定义规范化形式；项目不据此改写全部来源文本。
  • Unicode UAX #31和 RFC 8264 PRECIS提供国际化标识符与字符串 Profile 的机制；当前 END-P1 仍采用 ASCII。
  • Unicode UAX #9定义双向显示；视觉顺序不进入身份或范围。
  • Unicode UTS #39和 Unicode UTS #55提供同形风险与安全审查方法；警告不成为身份或恶意判决。
  • GNU libunistring 手册把编码、规范化、大小写、字素、词与行处理分开；项目只借鉴职责拆分，不引入实现依赖。
  • Bad Characters与 Imperceptible Jailbreaking Against LLMs作为不可见文本威胁证据；论文不定义规范结论。

后果与当前状态

  • TXT-CORE 0.1.0-draft定义 12 条跨制品文本义务。
  • 12 类威胁、18 个设计场景和 24 个正反向量覆盖当前抽象边界。
  • END-SRCM 只承诺映射解码并处理转义后的 rhem.content，不再暗示原始来源文件逐字节保真。
  • 当前没有 Unicode 处理器、规范化器、同形检测器、文本编辑器、模型输入网关或新 Profile。
