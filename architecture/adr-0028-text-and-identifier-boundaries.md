## 用一段发布文本理解六层边界

假设一份 `.ends` 清单包含“将依赖更新到安全版本”、关系标识符 `rel-17`，以及供模型复核的上下文。开发者必须知道每一步取得的究竟是哪一个对象。

1. 原始来源字节
2. 严格解码文本
3. 显式变换结果
4. 结构字段与范围
5. 安全显示视图
6. 模型实际输入

| 处理层 | 本例需要固定的事实 | 不能据此声称什么 |
| --- | --- | --- |
| 原始来源字节 | 文件的完整字节身份，以及声明采用 UTF-8 的解码 Profile | 字节可解码不等于内容已被接受 |
| 严格解码文本 | 按 RFC 3629 得到的 Unicode 标量序列；非法序列在形成对象前原子失败 | 解码后的文本不是原文件的逐字节副本 |
| 显式变换结果 | 转义、换行或具名规范化是否发生，输入、输出和信息损失分别绑定 | 变换结果不能沿用输入身份，也不能冒充来源原文 |
| 结构字段与范围 | `rel-17` 按封闭 ASCII Profile 比较；正文范围绑定精确主体和 Unicode 标量半开区间 | 屏幕列、字素、UTF-16 单元或模型 token 不能替代规范范围 |
| 安全显示视图 | 披露转义、截断、方向控制、不可见字符和混合脚本风险 | 经过重排或标注的显示视图不能参与解析和身份比较 |
| 模型实际输入 | 提交给模型的精确内容、预处理链、模型身份和 tokenizer 身份 | 界面截图或日志摘要不能证明模型实际收到了什么 |

**当前策略：**采用 TEXT-IDENTIFIER-CORE 0.1.0-draft 的十二项跨制品责任。它约束文本怎样进入规范制品、界面和模型，不增加 Endem 字段，也不创建 Unicode 处理器、编辑器或模型网关。

## 五类文本槽为什么不能共享处理规则

“字符串”只说明数据形态，不能说明用途。每个字段先声明文本槽，再选择字符集合、比较方式、变换、长度单位、显示和失败责任。

| 文本槽 | 承担的职责 | 禁止转移的规则 |
| --- | --- | --- |
| `source-content`<br>来源内容 | 保存自然语言的提交后来源表达；允许 Unicode，当前不静默规范化 | 结构标识符的 ASCII 限制不能直接套用到自然语言 |
| `structural-identifier`<br>结构标识符 | 承担稳定机器引用；按登记的封闭 ASCII 语法和精确大小写比较 | 搜索命中、locale、NFKC 或同形骨架不能选择对象 |
| `registered-token`<br>登记词 | 只接受对应登记表中的封闭值，并按该表的比较规则处理 | 登记词合法不能证明它适合其他字段或具有授权意义 |
| `descriptive-text`<br>说明文字 | 供人阅读、翻译和检索；比较与排序必须绑定具体用途 | 语言标签和自然语言相似度不能取得语义权威 |
| `display-message`<br>显示消息 | 用于诊断或界面呈现，可以转义、截断、脱敏和方向隔离 | 短显示、清洗文本和本地化消息不能充当安全引用 |

同一组码点复制到另一字段时，职责不会随字符串一起复制。例如，说明文字进入模型上下文后，必须新建模型文本绑定；错误消息中出现的关系标识符也不能让整条消息取得结构标识符身份。

## 严格解码、变换和范围怎样保持可追溯

| 步骤 | 现行约束 | 失败或变化时的处理 |
| --- | --- | --- |
| 解码 | 先按 RFC 3629 严格验证 UTF-8，再解析字段；拒绝过长编码、代理项和非法标量 | 不得用替换字符、平台编码猜测或部分成功继续形成对象 |
| 来源绑定 | 原始字节、解码文本和提交后的来源表达分别取得身份 | 每次有损变换都披露损失，不能覆盖来源后沿用原身份 |
| 规范化 | 只在具体文本槽的版本化 Profile 中指定 Unicode 版本、NFC/NFD/NFKC/NFKD 形式和输出用途 | 当前来源内容与精确身份域采用 `none`；不得盲目规范化 |
| 比较 | 明确选择精确字节、Unicode 标量、具名规范化结果、区域排序或风险骨架 | 一种比较结果不能自动升级成另一种相等关系 |
| 范围 | 范围绑定精确主体、表示版本、Unicode 标量单位和 `[start, end)` 半开区间 | 主体、表示或单位改变后必须重新定位，不能复用旧偏移 |

[UAX #15](https://www.unicode.org/reports/tr15/)说明规范化会修改文本，而且其适用性取决于上下文；[RFC 3629](https://www.rfc-editor.org/rfc/rfc3629.html)要求解码器防止非法序列造成安全后果。[GNU libunistring](https://www.gnu.org/software/libunistring/manual/libunistring.html)也把编码转换、规范化、大小写、字素、词和行处理分为不同接口。Noemion 借鉴这种职责分离，不据此选定实现库。

## 为什么当前机器标识符保持 ASCII

ASCII 是当前范围的封闭选择，不是对国际化的否定。现有消费者尚未证明 Unicode 结构标识符的必要性，而开放字符集合会立即引入 Profile、迁移和安全责任。

| 需要回答的问题 | 当前决定 | 未来扩展前必须补齐什么 |
| --- | --- | --- |
| 允许哪些字符 | 各结构标识符沿用已登记的封闭 ASCII 语法 | 依据 UAX #31 定义允许、排除和默认可忽略字符规则 |
| 怎样变换 | 不做宽度映射、大小写折叠或 Unicode 规范化 | 固定 Unicode 版本、映射顺序、规范化形式与迁移行为 |
| 怎样比较 | 按原始 ASCII 字节和登记的大小写规则精确比较 | 定义比较域，并证明旧标识符与新标识符不会错误合并 |
| 怎样处置方向与同形 | 非 ASCII 输入在结构入口直接拒绝 | 结合 UAX #9、UTS #39 与实际界面定义方向隔离和风险处置 |
| 怎样发布 | 不接受通过放宽正则实现的隐式扩展 | 真实生产者、消费者、迁移规则、正反向量和独立 ADR |

[Unicode 17.0](https://www.unicode.org/versions/Unicode17.0.0/)调整了部分字符的 `Identifier_Type` 与 `Identifier_Status`。因此，即使未来采用 Unicode 标识符，也必须固定标准和安全数据版本，不能依赖宿主环境的“当前 Unicode”。[RFC 8264](https://www.rfc-editor.org/rfc/rfc8264.html)提供字符串 Profile 的通用框架，但不替 Noemion 选择字符集合、比较规则或迁移策略。

## 双向文字、隐藏字符和模型输入怎样共同审查

| 审查对象 | 开发者应看到什么 | 结论上限 |
| --- | --- | --- |
| 双向文字 | 逻辑存储顺序、隔离后的视觉顺序、方向控制位置和采用的 UAX #9 版本 | 视觉重排只服务显示，不能写回来源、范围或身份 |
| 隐藏与默认可忽略字符 | 码点、位置、名称、所属文本槽和原始内容关系 | 可以警告或按 Profile 拒绝；不能静默删除后冒充原文 |
| 混合脚本与同形 | UTS #39 风险类别、采用的数据版本、字体和使用场景 | 混淆骨架只是风险信号，不是对象身份或恶意判决 |
| 安全显示视图 | 转义、截断、脱敏、方向处理和不可见字符标注 | UTS #55 式审查视图帮助人发现差异，但不取代精确机器内容 |
| 模型实际输入 | 精确输入身份、全部预处理、风险清单、模型、tokenizer 与复核视图 | 模型输出仍是不可信候选，不能写入规范或执行未经授权的越权行为 |

[UAX #9](https://www.unicode.org/reports/tr9/)负责从逻辑顺序计算显示顺序；[UTS #39](https://www.unicode.org/reports/tr39/)的混淆检测可能过度包含合法字符串；[UTS #55](https://www.unicode.org/reports/tr55/)展示双向重排即使只出现在字符串或注释中也会破坏源码审查。关于不可见字符影响 NLP 与大模型的论文只作为威胁证据，不固化攻击成功率、检测算法或特定 tokenizer 行为。

## 当前还不能承诺哪些 Unicode Profile 或文本组件

现有规范源记录 12 类威胁、18 个设计场景和 24 个正反向量。它们验证的是允许包络、确定拒绝和结果域，不证明 Unicode 库、双向编辑器、安全字体、规范化器、同形检测器或模型输入网关已经实现。

**待定内容：**国际化结构标识符、来源字节字段、来源文本重写、语言检测、搜索与区域排序、字素级编辑、字体信任、tokenizer 协议、同形自动处置和双向编辑器均保持开放。当前没有 Unicode 处理器、规范化器、同形检测器、文本编辑器、模型输入网关或新 Profile。

- [文本与标识符边界规范](../specifications/text-and-identifiers.html) — 按十二项责任检查具体文本槽、解码、比较、范围、显示和模型输入。
- [ADR-0037 · 现行术语](adr-0037-terminology-simplification.html) — 区分机器文档 ID、普通职责词和自造名称。
- [ADR-0027 · 签名有效，不等于制品获准](adr-0027-exact-identity-and-attestation.html) — 继续核对文本或制品的精确身份、签名陈述与依赖方决定。
- [ADR-0029 · 权威与授权决定](adr-0029-authority-and-authorization-decisions.html) — 继续核对谁能确认意义、授予动作权限和作出最终决定。

> **名称与读音：**`TEXT-IDENTIFIER-CORE` 是机器文档标识，不是面向口头传播的发行名称。中文“文本”“结构标识符”“安全显示视图”“模型实际输入”职责清楚；该机器 ID 只需保持精确书写、职责和版本化冲突检查，不另设发行读音。
