集中保存尚未通过规范或 ADR 决定的名称、类型、ABI、实现与资源问题。

尚待确定：对象与语义

Noema Object 怎样版本化？

Noema Object（NOBJ）已经作为当前工程名称采用，但文件扩展名、版本协商、能力声明和未知记录处理仍需对象规范与 ADR 确定。

NIR 最小核心类型集是什么？

需要用真实语料和实现原型确定最小集合；“约 20～30 种”只是规划估计。

连续语义残差如何标准化？

Encoder 指纹、维度、码本、量化、相似度与跨模型稳定性仍需证据和互操作测试。

评价器如何成为可链接对象？

确定性、模型与外部评价器的接口、权限和失败边界尚待设计。

四类语义关系是否足够且正交？

Source Expression、Compiled Meaning、Reference Binding 与 Acceptance Contract 当前只表示四类需要分别追踪的关系，不是已证明稳定的四层身份体系；需要以同义改写、跨版本绑定、语境推论和复合验收反例检验是否遗漏或重复。

语义来源和满足环境如何进入对象？

明示内容、结构蕴涵、语境推论、工具推导与模型候选的确认权限尚未定型；评价环境的证据、能力和背景假设也须决定是对象数据、可链接依赖还是 Fulfillment Runtime 配置。

模型候选怎样升级为确认语义？

Candidate Envelope 的最小字段、Source Binding Decision 的授权主体、可由确定性规则重推导的范围，以及必须保留为未决语义的情形仍需 NIR 规范、威胁模型和反例语料。类型正确或模型高置信不能单独完成升级。

HOBJ 与 NOBJ 的格式和链接关系是什么？

当前职责边界采用“共享内容先编译为 NOBJ，再由 noemlink 将 NOBJ 与已有 HOBJ 链接为 HOBJ”；HOBJ 是否共享容器头、怎样递归链接、哪些记录只能出现在 HOBJ，仍需对象规范与 ADR。

尚待确定：语言、ABI 与工具面

NSL 采用何种表面语法？

严格声明式 DSL 是当前优先方向，自然语言化语法糖仍是后续候选。

披露与 Fulfillment Runtime ABI 如何定型？

状态哈希、预算单位、继续句柄、检查点、后端输入适配器槽位和消费者确认协议均未冻结；后端适配不属于对象格式本身。

Agent Harness 的能力与事件结构怎样定型？

上下文清单、能力参数、副作用分类、观察事件、状态快照、预算、重复失败和人工升级结构仍需威胁模型、原型与 ADR；它们不能由模型输出临时定义。

Harness 与 Runtime 怎样交换请求和观察？

Runtime Request、Capability Request、Capability Observation、幂等性、超时、取消、恢复和会话级重试协议尚未冻结。Runtime 不持有实时能力句柄，Harness 也不能改变 Runtime 对象语义。

谁持有并传递 Loaded State？

Loaded State Handle 的不可伪造性、进程边界、生命周期、撤销、并发读取与 Runtime 只读视图仍需规范。当前边界要求 Object System 交给 Harness，Runtime 只能经 Harness 获得请求所需视图。

最终验收由谁决定？

Candidate Assessment、证据闭包、确定性 Acceptance Policy 与人工或外部权威的签署协议仍需规范。noemcoverage 只检查证据是否可定位，noemevaluate 只做离线评估，两者都不能修改单次会话的 Acceptance Decision。

运行产物怎样保持同一身份链？

会话、对象、环境、Runtime 请求、能力调用、观察、候选、Run Result、Trace 与 Acceptance Decision 的稳定身份和防篡改关联尚待事件规范与签名策略确定。

CLI 与重定位标识是什么？

尚未规范化的命令示例和 R_NIR_* 名称只表示候选职责，不构成稳定 CLI 或 ABI。冻结前必须确定命名空间、版本策略、未知类型处理和一致性测试向量。

行为请求需要哪些最小字段？

询问、承诺、授权与执行是否属于核心行为类型，以及行动者、权限、前置条件、效果和撤销边界怎样跨 Compiler、链接器与 Fulfillment Runtime 保持一致，仍需威胁模型和 ADR。

尚待确定：实现与模型

首版实现语言是什么？

Rust 与 C++ 各有工程优势，尚未形成 ADR；第一阶段也不引入 LLVM/MLIR。

Horizon Engine 选用哪个语言底座和训练方案？

候选底座、蒸馏范围、量化方法、阈值与数值门槛必须经过来源隔离评测，并以质量、校准、资源和许可证据决定，不能由未经验证的目标值直接决定。

Coverage Proof 采用哪些自动门槛？

需通过代表性来源语料确定来源单元分类、等价性测试与人工确认风险分级，并区分编译来源覆盖、发布对象覆盖和运行证据覆盖。

Release Object、签名包与撤销状态怎样关联？

Unsigned Package Candidate、Signing Request、Signature Response、Signed Noemion Package、回滚保护、撤销与重新签名的身份关系仍需发布格式和签名 ADR 冻结。

后续计划：资源与性能基线

实现前不承诺特定开发设备足以承担全部阶段。安全对象核心、模糊测试、跨平台构建、Runtime 场景与 Horizon Engine 训练应分别建立可复现的时间、内存、存储和并发基线；只有真实瓶颈形成后才决定本地、持续集成或专用算力配置。

决策方法与关闭条件

开放问题不能仅靠偏好或命名讨论关闭。每项问题应记录负责人、影响的规范和工具、候选方案、现有技术或先例、评价指标、原型或实验、风险、知识产权与互操作影响，以及最终 ADR。

优先级判定标准最迟关闭阶段
阻断型影响安全解析、确定性、对象兼容或第一阶段范围进入相关开发阶段前
架构型影响多个组件、规范层次或长期演进计划评审或首个纵向切片前
接口型影响 CLI、ABI、配置或互操作对外候选发布前
优化型只影响性能、体验或后续模型能力基线与瓶颈证据形成后
