架构

汇总系统关系、对象生命周期、信任边界和开放问题入口，不替代正式规范。

System MapLifecycleTrust Boundary

架构目标与质量属性

架构的首要目标不是提高单次生成效果，而是让任务语义、对象转换、依赖组合、权限变化和运行结果能够被定位、验证与复现。

确定性

相同规范输入、Profile 与锁定依赖应产生相同的规范化对象和诊断结果。

可组合性

符号、类型化引用、约束与依赖闭包能够链接，并明确报告不可合并的冲突。

可验证性

结构、语义、权限、完整性和覆盖要求具有分层检查点，而不是依赖模型自证。

可演进性

格式版本、能力协商和 ADR 管理允许规范演进，同时拒绝未声明的语义漂移。

可以把这条链理解为“提出要求—形成受检对象—组合依赖—验证装载—受约束执行”。前一阶段只能把明确产物交给后一阶段，任何组件都不得越过自己的信任边界替下游做隐式决定。

各职责单元的入口见组件目录，格式与执行约束见规范登记页。

概要设计

层次主要输入主要输出必须保持的不变量
Source Frontend语言、Schema、知识与策略来源候选语义与来源映射保留来源、歧义和未决信息
Compiler Core规范化源表示与 ProfileGSIR / GOBJ无模型、确定、布局受规范控制
Linker / Loader对象、归档、SSO、锁定依赖链接对象、运行 Segment、装载状态冲突显式、权限不静默扩大、完整性可验证
Runtime已验证对象、模型与外部能力绑定受约束求解结果与运行证据Lowering 不改变对象声明的硬约束和权限边界

更细的阶段、信任转换和失败位置见对象生命周期；记录布局、类型系统和执行语义必须进入正式规范或 ADR。

对象生命周期

对象生命周期集中说明从输入、确定性编译、链接、验证到装载执行的阶段与边界。

信任边界

模型输出不得直接生成 GOBJ；对象偏移、长度、计数、索引和对齐均需检查。安全规则的规范状态见规范登记页。

开放问题

尚未批准的名称、ABI、模型与设备决策保留在开放问题中，不在架构概览中固化。
