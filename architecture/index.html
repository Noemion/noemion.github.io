架构设计

说明来源表达如何经过确定性编译、对象链接和安全装载，最终形成可以验收的运行结果。

System MapLifecycleTrust Boundary

架构目标与质量属性

架构的首要目标不是提高单次生成效果，而是让任务语义、对象转换、依赖组合、权限变化和运行结果能够被定位、验证与复现。

确定性

相同规范输入、配置与锁定依赖应产生相同的规范化对象和诊断结果。

可组合性

符号、类型化引用、约束与依赖闭包能够链接，并明确报告不可合并的冲突。

可验证性

结构、语义、权限、完整性和覆盖要求具有分层检查点，而不是依赖模型自证。

可演进性

格式版本、能力协商和 ADR 管理允许规范演进，同时拒绝未声明的语义漂移。

第一条链形成可装载对象；第二条链表示一次运行迭代，Harness 会根据观察、预算和停止条件继续调用 Runtime 或结束会话。每个阶段只证明特定属性：签名证明身份和字节完整性，不证明语义正确；装载证明当前环境允许建立状态，不证明运行结果满足目标；Runtime 产生的候选和外部环境观察始终重新进入不可信域。

各职责单元的入口见系统组件目录，格式与执行约束见规范与成熟度页面。

更细的阶段、信任转换和失败位置见对象生命周期；记录布局、类型系统和执行语义必须进入正式规范或 ADR。

对象生命周期

对象生命周期逐项列出来源、编译、链接、发布签名、装载、会话、能力调用、运行证据和最终验收的产物、失败与审计依据。

信任边界

信任不是单一分数。来源可追溯、结构有效、语义有效、依赖闭合、字节完整、环境授权和结果验收是彼此独立的属性；每个组件只能声明自己已经检查的属性。模型输出不得直接生成 NOBJ；能力调用必须经过参数、权限、预算和副作用检查；对象偏移、长度、计数、索引和对齐均需检查。智能体控制面的职责见Agent Harness，求解与候选评价边界见Fulfillment Runtime，安全规则的规范状态见规范与成熟度页面。

开放问题

当前已采用的跨组件边界集中在架构决策；尚未批准的格式、ABI、模型、运行协议与资源决策保留在开放问题中，不在架构设计总览中固化。
