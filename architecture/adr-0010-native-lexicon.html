ADR-0010
原生词汇与事态分层

把人实际说过的话、系统采用的解释、想达到的状态、判断方法和仍未解决的问题分别保存，避免一次模型解释覆盖原意。

当前策略2026-07-12原生词汇接口待定

直接结论

本决定确定六项语义职责，并采用 rhem / semion / skena / telis / krin / apor 作为现行设计字段。职责分层继续有效；具体发行拼写仍受后续的专名必要性与读音审查约束。

结构底线：一个 Endem 只有一个根 skena。semion、skena 与观察中的 phain 使用可比较的符号和关系位置；形式由结构显示，制品不能用自填标签证明自己有效。

ADR-0034 明确说明：字段较短、词源成组或已经进入设计资料，都不能证明名称适合发行。后续改名若成立，只能替换精确标识，不能合并本决定分开的职责。

事态、目标和观察不再混合

rhem · 来源
semion · 投影
skena · 可能事态
telis · 目标方向
krin · 比较契约
phain · 结构化观察
Iknem · 有范围证据
  • skena 只描述对象怎样关联，不携带“希望、保持、禁止”等力量。
  • telis 第一阶段只区分 kine（使事态达到成立）和 mene（使事态持续成立）。
  • 禁止事项由 skena 的显式否定表达，不再与独立 avoid 值重复编码。
  • phain 是 Iknem 内的结构化观察；无法与 skena 比较的自由文本或模型评分不能独自支持满足判断。

每个专名旁都必须保留直白工程副标题。词源只提供记忆路径，真正含义仍由数据结构、不变量、权限和测试决定。

名称审计

本次对照 C/C++、Rust、Go、Python、Java、ECMAScript、Swift 与 Kotlin 的关键字集合。项目也查询了 PyPI、npm 与 crates.io 的精确包名。

现行领域词没有关键字冲突，查询时也没有形成跨三大注册表的强占用。该结果只降低工程混淆，不构成商标或未来包名可用性保证；日期化查询、风险与停止条件统一维护在 name-audit.md。

关键字集合分别核对 Rust Reference、Go Specification、Python Reference、Swift Reference 与 Kotlin Reference。

待定内容

当前策略包括 ADR-0011 的实验性容器、ADR-0013 的 END-P1 记录字段、ADR-0014 的首个来源清单，以及 ADR-0016、0018 与 0019 的时间、量化和测量规则。正式来源语言、phain 编码、对应物理字段、krin 求值语言、扩展注册表、摘要与签名算法和稳定 ABI 仍需独立规范与对抗测试。
