Noemion

面向生成式计算的、可机器分析、验证、重定位和执行的目标与约束表征对象体系。

GSIRGOBJ / SSODeterministic Core

为什么需要 Noemion

生成式任务不能只依赖不可验证的文本提示。Noemion 旨在把目标、约束、歧义、证据和验收条件组织为可由工具链处理的语义对象。

当前生成式系统常把任务意图、执行权限、外部知识、评价标准和运行记录混合在一次性上下文中，导致目标难以静态检查、依赖难以重定位、失败难以复现、产物难以长期治理。Noemion 的核心研究命题是：生成式计算是否能够像传统对象工程一样，拥有独立于具体模型与提示表达的可验证中间对象和可信装载边界。

了解项目背景、适用范围与非目标。

核心能力

目标与约束表征

GSIR 表达目标、解空间、硬约束、软偏好、歧义和验收条件。

对象化工程

GOBJ 与 SSO 面向组织、链接、重定位、完整性验证和受控装载。

确定性核心

Strict 模式无模型、确定且可复现；只有 Compiler Core 能生成 GSIR/GOBJ。

不可信输入边界

模型输出与对象输入均不可信，Runtime 和对象工具承担验证与安全执行职责。

研究与标准化愿景

科学问题

研究目标、约束、歧义、证据与验收条件如何形成可分析、可组合且可证伪的计算对象。

工程问题

建立确定性编译、对象格式、链接装载、安全验证和运行追溯之间的完整工具链。

知识产权

逐步识别可申请专利或登记软著的创新点候选，但在检索、权属和公开披露审查完成前不作成果声明。

行业规范

只有术语稳定、条款可测试、对象可互操作且安全证据充分后，才把设计提案推进为标准化候选。

当前状态

开发阶段：项目目前处于规范与安全二进制核心开发阶段，尚未发布可供使用的编译器或 Runtime。

第一阶段聚焦规范、确定性 Compiler Core、安全 Reader/Writer、验证与测试基础，不引入模型、LLVM、MLIR 或大型框架。

当前证据边界：已有内容主要是问题定义、架构约束、对象与工具职责设计，尚不足以证明性能、互操作性、模型无关性或产业适用性。相关结论必须等待规范条款、原型、测试矩阵和可复现实验逐步形成。
