# 并行、推测执行与提交边界研究提案

状态：非规范研究提案
日期：2026-07-14
结论状态：桌面审查完成；尚未进入规范
适用范围：非组件研究

## 直接结论

并行调度、分支完成、候选结果、提交授权、外部副作用和目标满足是六种不同事实。Noemion 必须把它们分开：

1. 并行只改变工作何时被尝试，不改变 Endem、Synem、Dromen、授权、证据责任或结果域。
2. 每个分支必须有稳定身份，并绑定它实际读取的输入、快照、行动者、能力子集、预算份额、外部调用和候选结果。
3. 推测分支默认只能产生候选。只有精确指定的提交路径可以在重新核对当前前提后造成外部副作用。
4. “最快完成”“模型评分最高”“多数分支同意”或“任务状态 completed”都不能自行选择提交者、扩大权限或成为最终决定。
5. 取消失败分支只能请求停止尚未发生的工作。它不能撤销已经发送的消息、付款、发布、写入或其他外部效果。
6. 同一对象上的竞争写入必须在提交时核对强前提；前提陈旧时停止并重新观察，不以最后完成者覆盖当前状态。
7. 预算和能力只允许从同一个 Dromen 上限向分支分配，不能让每个子 Agent 重新获得一整份配额或更大的授权。
8. 分支完成和提交成功仍只形成运行事实。后续观察、Iknem 覆盖、`met / unmet / agno / fault` 与 `accepted / rejected / deferred` 继续独立判断。

本提案不构成 ADR、CORE 规范、Profile 或实现要求。它不创建并行制品、事务格式、分支对象、命令、组件、结果域、稳定接口或哲学专名，也不进入 `registry.json`。当前只检验控制平面、DRO、AUT、ADP、ID、IKN、DIA、END 与 SYN 的现有责任能否形成唯一边界。

## 为什么现有取消和重试条款还不够

ADP-CORE 已经规定外部取消、重试、幂等和交付状态；DRO-CORE 已经规定有限预算和会话失效。它们回答单次调用怎样受到约束，却没有完整回答多个同时存在的分支如何竞争同一个提交机会。

若只写“可以并行调用工具”，至少会留下五个危险省略：

- 每个分支是否继承整份权限与预算；
- 分支读取的是不是同一个当前状态；
- 谁可以把候选变成外部副作用；
- 失败或取消能否撤销已经发生的效果；
- 合并、冲突和未知交付由谁承担失败责任。

解决方法不是建立一个总括“并行对象”，而是把分支准入、候选形成、提交选择、提交前重验、外部效果和后验观察放回现有责任域。

## 七个阶段必须分开

| 阶段 | 回答的问题 | 最低绑定 | 不能推出什么 |
| --- | --- | --- | --- |
| 并行意图 | 为什么这组工作可以同时尝试 | 精确目标、计划截止点、依赖图、汇合政策与停止条件 | 分支已获授权或可以写入 |
| 分支准入 | 哪个实际行动者可以尝试哪一部分 | 分支身份、父会话、输入快照、能力子集、预算份额、租户与截止点 | 获得父会话全部资源或独立扩权 |
| 执行尝试 | 分支实际做了什么 | 调用身份、协议、端点、精确输入、幂等或去重条件、原始状态 | 调用一定成功或没有副作用 |
| 候选结果 | 分支产生了什么可比较输出 | 内容身份、来源、变换、生成分支、覆盖范围与限制 | 已提交、事实为真或目标已满足 |
| 提交选择 | 哪个候选有资格进入提交门 | 预先固定规则或具名权威、冲突集合、选择依据与未选分支 | 选择本身就是授权或外部效果 |
| 提交与外部效果 | 当前条件下是否实际发生状态变化 | 当前授权、当前对象身份、强前提、受众、幂等、交付与观察责任 | 目标满足、证据充分或最终接受 |
| 后验观察与判断 | 外部状态现在怎样 | 具名观察方法、范围、时间、损失、Iknem 与 `krin` | 反向改写分支历史或授权 |

这些阶段使用普通职责短语，口头上可以直接区分“分支”“候选”“提交”和“已发生效果”。本主题没有创造发行名称的必要，也不触发新的读音验证。

## 现有责任怎样承接

| 问题 | 唯一主归属 | 相邻域只提供什么 | 当前不新增什么 |
| --- | --- | --- | --- |
| 目标、依赖与可并行性 | END、SYN 与控制平面 | DRO 提供会话上限 | 调度进入 Endem、并行 CORE |
| 分支实际行动者与授权范围 | AUT | DRO 提供能力上限；ADP 保存外部身份 | 子 Agent 自动继承、分支令牌 |
| 分支能力与共享预算 | DRO | AUT 提供决定；控制平面分配份额 | 每分支重置预算、可变 Dromen |
| 调用、幂等、去重、取消与交付 | ADP | DIA 保留失败层次 | 通用事务适配器、取消即回滚 |
| 快照、候选和提交前提身份 | ID | TEXT-IDENTIFIER 绑定实际文本与显示 | `latest` 作为前提、短摘要作身份 |
| 分支观察、重叠范围与覆盖 | IKN | ADP 提供调用事实 | 按分支数计算充分性 |
| 冲突和主阻断诊断 | DIA | 其他域提供原始失败 | 最后错误覆盖其他失败 |
| 满足与最终决定 | END、IKN 与具名权威 | 提交事实只是候选观察 | completed 即 `met` 或 `accepted` |

桌面审查没有发现必须新建 `PAR-CORE`、事务制品或推测执行格式的责任。若后续证据支持并形成正式 ADR，未来义务应优先回到这些现有规范；只有真实消费者证明无法唯一归属时，才重新讨论新对象。

## 当前 Agent 协议给出的证据

### MCP：并发请求与 Task 状态不等于提交权

[MCP 2025-11-25 Tasks](https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks)把 Task 定义为实验性的持久状态机。请求方可以编排并发请求，任务创建响应与后来取得的实际结果分离，相关消息必须绑定同一 Task ID。取消只要求接收方尝试停止；规范明确允许执行在 Task 标为 cancelled 后仍继续。

这支持分支身份、协议状态、结果取得和取消效果分别记录。它不支持把 Task `completed` 当成提交授权，也不支持把 `cancelled` 当成外部副作用已回滚。

[MCP Sampling 草案](https://modelcontextprotocol.io/specification/draft/client/sampling)显示主要模型供应商 API 已能并行发出多个工具请求，但该部分仍是草案，关闭并行调用也不是 MCP 核心机制。它只能作为趋势证据：并行工具调用正在成为常见能力；Noemion 仍须独立冻结版本、授权、预算和提交规则。

[MCP `2026-07-28` 发布候选](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/)已于 2026 年 5 月 21 日公开，正式稿计划在 7 月 28 日发布；当前可用于生产协商的正式版仍是 `2025-11-25`。候选稿把 Tasks 从实验性核心移到扩展，并计划弃用 Sampling 核心能力。这进一步说明 Noemion 只能把当前 Tasks 与 Sampling 当作带版本的外部证据，不能把它们固化成长期并行或提交抽象。

### A2A：外部任务生命周期仍留在协议域

[A2A 1.0 版本化规范](https://a2a-protocol.org/v1.0.0/specification/)分别定义 Task、Message、Artifact、流式更新、取消、订阅、幂等和授权。一个控制平面可以同时处理多个外部 Task，但这些任务的身份、状态和产物仍属于 A2A 域。

Noemion 可以保存并关联这些外部事实，却不能让远端 Task 的完成顺序选择本地提交者，也不能把 Artifact 直接提升为 Endem、Iknem、满足结果或最终决定。

## GNU Make 给出的并行纪律

[GNU Make 并行执行](https://www.gnu.org/software/make/manual/html_node/Parallel.html)用 job slots 限制同时运行的 recipe；默认仍是串行。若 Make 在子进程存在时终止，它会等待这些子进程，而不是假设停止请求已经撤销它们。

[禁用并行执行](https://www.gnu.org/software/make/manual/html_node/Parallel-Disable.html)说明：完整、准确的依赖关系才是安全并行的首选；`.NOTPARALLEL` 和 `.WAIT` 只是需要序列化时的控制机制。Noemion 借用的不是文件时间语义，而是“先证明独立性，不确定时串行”的纪律。

[GNU Make jobserver](https://www.gnu.org/software/make/manual/html_node/Job-Slots.html)要求递归或集成工具共享同一个并行配额，并在错误情况下也归还取得的 slot。Noemion 可借用这一预算纪律：子 Agent 和嵌套工具消费父 Dromen 的同一有界预算，不能通过递归复制资源。

[GNU Make 中断处理](https://www.gnu.org/software/make/manual/html_node/Interrupts.html)会尽力删除中断时可能不完整的目标，但也明确存在无法清理的崩溃情形；原子更新是保留目标的合理前提之一。这支持“候选与已提交输出分开”和“中断后重新核对”，不支持把文件清理规则泛化为跨系统事务或外部副作用回滚。

## RFC 9110 给出的提交前提范例

[RFC 9110 第 13.1.1 节](https://www.rfc-editor.org/rfc/rfc9110.html#section-13.1.1)规定 `If-Match` 使用强比较，状态变更方法必须在执行前评价前提；不匹配时不得执行该方法。它专门用于避免多个主体并行操作同一资源时发生丢失更新。

Noemion 采用的是通用纪律：提交前把候选所依据的对象状态与当前状态做强、可验证的比较，失败就停止并重新观察。它不把 ETag、HTTP 或 412 固化为所有适配器的统一事务层；不同协议仍须由具体 Profile 给出等价而可验证的前提机制。

## 操作类别与并行资格

| 操作类别 | 可以并行尝试吗 | 提交要求 | 失败或取消后的解释 |
| --- | --- | --- | --- |
| 纯计算 | 可以，只要输入与预算固定 | 结果作为候选按精确身份比较 | 丢弃候选不产生外部回滚问题 |
| 只读观察 | 可以，但必须固定观察范围和截止点 | 汇合时处理快照差异、重复与覆盖 | 取消不删除已经取得的观察；披露责任仍在 |
| 隔离候选写入 | 可以写临时或隔离位置 | 只有一个候选通过提交门后原子发布 | 临时结果必须具名清理；存在不等于有效 |
| 有明确幂等语义 | 可以受控重试或并行去重 | 绑定精确幂等键、对象和服务语义 | 同一响应丢失仍产生未知交付，需要查询 |
| 有去重协议 | 可以在服务端强制唯一时尝试 | 去重键必须绑定动作语义和期限 | 去重命中不证明目标满足 |
| 已证明可交换 | 只有在作用域、顺序无关性和合并规则都被证明时 | 保存每个效果和确定性汇合 | “看起来独立”或模型判断不能代替证明 |
| 非幂等副作用 | 默认不得推测并发提交 | 串行提交；提交前重验；提交后观察 | 取消不能撤回已发送、已支付或已发布效果 |
| 交付结果未知 | 不得盲目重试或选另一分支覆盖 | 先按协议查询、对账或人工升级 | unknown 不映射为 failed、`unmet` 或 deny |

“可交换”不是新的布尔标签。未来若采用，必须绑定对象范围、操作语义、合并函数、身份规则、失败责任和正反例；否则按可能冲突处理。

## 支持案例与反例

### 案例一：三个独立只读数据源

三个查询可以共享一个会话预算并行执行。每个分支保存来源、截止点、租户、请求和观察范围；汇合时按内容身份去重。

反例：三份相同观察按分支数计成三份独立证据，进而把覆盖度提升为 sufficient。

### 案例二：两个 Agent 修改同一文件

两个分支可以在隔离工作区生成补丁候选。提交者必须核对共同基线和当前文件身份；若用户或另一分支已修改文件，停止并重新合并。

反例：最后完成的分支直接覆盖当前文件，因为它“更新”。完成时间不是提交前提。

### 案例三：两个发布候选

两个构建分支可以生成不同候选并分别验证。预先固定的确定性规则或具名发布权威只选择一个进入提交门，另一个保持候选身份。

反例：模型评分最高的候选自动发布。模型偏好既不是授权，也不是当前状态重验。

### 案例四：重复付款

付款是非幂等高风险副作用。即使两个分支都获准准备请求，也只能有一个精确提交尝试；响应丢失时先对账，不能让另一分支重试覆盖未知结果。

反例：取消“较慢分支”就认为付款未发生。取消消息可能在外部系统提交之后到达。

### 案例五：并行模型候选

多个模型或同一模型多次采样可以并行生成来源表达、计划或参数候选。每个输出保存实际模型、预处理、输入和生成身份。

反例：多数模型一致就形成语义授权、规范字节或最终接受。

### 案例六：发送邮件后取消

一个分支已向远端邮件服务提交，随后控制平面选择另一个分支并取消前者。系统必须把邮件交付保存为已发生或未知的外部效果，并按需要补偿或人工升级。

反例：把 cancelled 写成“邮件未发送”，或从轨迹删除该分支。

### 案例七：递归子 Agent

父 Agent 把研究拆给两个子 Agent。每个子 Agent只获得精确输入、能力子集和预算份额；全部消耗计入同一个 Dromen 上限。

反例：每次 handoff 都重新获得完整 token、调用、时间和网络预算，从而绕过有限会话。

### 案例八：`any_of` 的决定性短路

一个有效 `met` 叶可以按 ADR-0020 使 `any_of` 决定性短路。控制平面可以停止未开始工作并请求取消其余分支，但仍须保存已经求值的叶、未求值叶、Iknem 与已遇故障。

反例：删除失败分支记录，或把外部取消成功当作所有副作用不存在。

### 案例九：证据范围重叠

两个分支抓取同一日志窗口。Iknem 按主体、方法、范围和溯源识别重叠，不能按记录条数或并发来源数重复计算覆盖。

反例：并行度越高，证据充分性越高。

### 案例十：基线在等待期间变化

分支根据版本 A 生成更新，提交前资源已变为版本 B。强前提失败，候选留存并触发重新观察或人工合并。

反例：因为候选曾经通过验证，就对版本 B 强制应用。

### 案例十一：外部 Task 乱序完成

两个 MCP 或 A2A Task 按不同顺序完成。控制平面按预先固定的汇合与提交规则处理，完成顺序只作为运行事实保存。

反例：第一个 completed 自动赢得权限、预算余额和最终决定。

### 案例十二：不同租户的同名资源

两个分支处理相同资源名但属于不同租户。分支身份和提交前提必须保留租户与受众，候选不得按显示名合并。

反例：同名即同一对象，导致跨租户写入或证据混合。

### 案例十三：已证明可交换的计数增量

只有协议明确给出原子增量、对象范围、去重和失败语义时，多个增量才可能并发提交。仍要保存每次调用和最终观察。

反例：普通 read-modify-write 因为“都是加法”就被当成可交换，造成丢失更新。

### 案例十四：孤儿副作用

控制平面崩溃后，远端分支继续运行并产生效果。恢复流程建立新会话，先查询外部状态和旧调用身份，再决定继续、补偿或升级。

反例：从检查点重放全部分支，认为旧进程消失就等于旧调用没有发生。

## 提交前重新核对矩阵

| 前提 | 读取或候选阶段可以缓存吗 | 提交时必须怎样处理 | 失败落点 |
| --- | --- | --- | --- |
| 精确对象身份与版本 | 可以作为候选基线 | 对当前对象做强比较 | 陈旧候选；不执行写入 |
| 授权决定与撤销状态 | 可以保存决定引用 | 重新核对截止点、主体、动作、对象、目的和撤销 | AUT deny 或 defer；不改写候选内容 |
| Dromen 与能力上限 | 只读绑定 | 确认会话未失效且预算仍足 | DRO 阻断或会话终止 |
| 对端、协议、schema 与媒体类型 | 可以绑定精确版本 | 核对没有实质漂移 | ADP 协商或映射诊断 |
| 幂等、去重或原子条件 | 不能由名称推断 | 由具体协议 Profile 验证 | 未知时按非幂等处理 |
| 受众、租户与外部目标 | 可以形成固定候选 | 与实际调用端点和凭据逐项匹配 | AUT / ADP 阻断 |
| 预算、截止点与速率限制 | 只可预留，不可复制 | 原子消费共享余额并保存实际消耗 | DRO 预算诊断 |
| 提交选择规则 | 计划时固定 | 确认冲突集合、选择依据和具名权威未变化 | 停止、重规划或人工升级 |
| 后验观察责任 | 计划时固定 | 提交前确认仍可执行 | 无可行观察时不得把写入当可验证完成 |

这张表是设计审查材料，不是通用事务算法。具体原子性、锁、条件写入、消息对账和补偿只能由未来协议 Profile 与真实消费者定义。

## 威胁到失败责任的映射

| 威胁 | 失效方式 | 唯一主归属 | 预期处理 |
| --- | --- | --- | --- |
| 权限倍增 | 每个分支继承完整或更大能力 | AUT / DRO | 分支只取父上限内的显式子集 |
| 预算倍增 | 子 Agent 各自重置 token、时间或调用次数 | DRO | 共享有界余额，分配和归还可审计 |
| 丢失更新 | 两个候选覆盖同一当前对象 | ID / ADP | 提交前强前提；失败则重新观察 |
| 重复副作用 | 多个分支执行同一非幂等动作 | ADP | 单一提交路径；幂等或去重必须有协议证据 |
| 陈旧快照提交 | 候选基于旧政策、对象或 schema | AUT / ID / ADP | 提交时重验，不兼容即阻断 |
| 取消洗白 | cancelled 被解释为未发生效果 | ADP / DIA | 保留原始状态与未知交付，查询或补偿 |
| 最快分支洗权 | 完成顺序选择提交者 | 控制平面 / AUT | 预先固定规则或具名权威 |
| 模型仲裁 | 评分或多数票成为写入授权 | AUT / IKN | 只作候选评价，不越过授权与证据要求 |
| 重叠证据膨胀 | 同一观察被多个分支重复计数 | IKN | 按身份、范围和溯源去重 |
| 跨租户合并 | 同名对象或任务被误合并 | AUT / ADP / ID | 绑定租户、受众和精确对象 |
| 分支身份丢失 | 汇合后无法追溯实际行动者 | AUT / ADP / IKN | 贯穿保存父子关系和实际调用身份 |
| 孤儿副作用 | 控制平面终止而远端继续 | ADP / DRO | 新会话先查询旧调用和外部状态 |
| 秘密共享 | 分支复制实时令牌或能力句柄 | DRO / AUT | 句柄留在最小能力域，按调用下发 |
| 错误覆盖 | 胜出分支掩盖其他 fault 或未知交付 | DIA | 主诊断按规范优先级，保留全部遇到的失败 |

## 失败域与结果隔离

| 事件 | 分支与调用事实 | Drase 会话 | 满足与证据 | 最终决定 |
| --- | --- | --- | --- | --- |
| 分支完成候选计算 | completed candidate | 继续 | 尚无新满足结论 | 不变 |
| 分支调用失败 | 保存外部错误 | 依政策继续、失败或中断 | 只有观察契约故障才可能支持 `fault` | 不自动 rejected |
| 分支被取消 | 保存请求与实际状态 | 可继续 | 不证明副作用不存在 | 不变 |
| 提交前提失败 | 不执行该写入 | 重规划、停止或升级 | 旧候选不成为当前观察 | 不自动 unmet |
| 提交响应丢失 | unknown delivery | 依查询预算继续或中断 | 在外部状态确认前为 `agno` 或无判断 | 通常 deferred |
| 提交成功 | 保存精确外部效果 | 可继续 | 仍需后验观察和覆盖评估 | 不自动 accepted |
| 后验观察支持判据 | 保存 phain / Iknem | 可完成或继续 | 按 `krin` 得到四值判断 | 仍由具名权威决定 |
| 失败分支已造成效果 | 保留冲突或补偿责任 | 依政策处理 | 效果本身可成为观察，不能从历史删除 | 由权威决定接受、拒绝或延期 |

## 语言哲学采用边界

路德维希·维特根斯坦《逻辑哲学论》2.04 把世界描述为存在的事态总体。这里的有限工程启发是：一个推测分支描述的可能结果、一个尚未提交的候选或模型对未来状态的预测，都不是已经存在的外部事态；系统必须保存从候选到已发生效果再到观察判断的边界。

该命题不能推出事务、锁、原子提交、补偿、幂等或并行调度算法。Noemion 只借它提醒读者不要把可能事态写成既成事实；具体规则仍由协议、授权、身份、威胁、反例和未来向量证明。

## 可读性与术语边界

本主题优先使用“并行分支”“候选结果”“提交前提”“已发生副作用”“未知交付”和“后验观察”。这些词直接显示职责，也能在中文口头讨论中稳定区分。

不得用 `winner`、`race success`、`best-of-n`、`transactional agent` 或一个新造专名掩盖授权与失败责任。若未来机器接口确实需要字段或枚举，必须单独说明其生产者、消费者、比较规则和失败域，并检查普通词的词首；只有新增自造名称才补充完整读音验证。当前研究提案不预留名称。

## 候选最小规范变化

1. 在 DRO-CORE 明确嵌套执行和分支共享同一能力与预算上限，不能通过递归扩权或重置资源。
2. 在 AUT-CORE 明确分支或子 Agent 的实际行动者、委托和提交选择仍须精确绑定，完成顺序与模型评分不构成授权。
3. 在 ADP-CORE 明确取消不表示已经发生的副作用被撤销，并要求竞争写入绑定协议可验证的提交前提或串行化。
4. 在 ID-CORE 明确候选基线和提交时当前对象必须分别绑定精确身份，位置、`latest` 或时间戳不能代替强比较。
5. 在 IKN-CORE 明确并行分支的重叠观察不能按分支数提升覆盖，外部效果未知时不得伪造缺席。
6. 在 DIA-CORE 明确汇合诊断保留未选分支、未知交付、孤儿效果和提交前提失败，不由胜出分支覆盖。

这些只是责任候选，不是已接受条款。任何规范变更仍须由独立 ADR、唯一条款、威胁更新和正反向量共同完成。

## 进入规范前的条件与停止条件

### 仍需回答

- 哪些真实消费者需要并行只读、隔离候选或并发提交；
- 每种外部协议如何提供强前提、幂等、去重、查询或补偿；
- 分支预算怎样原子分配、消费和报告；
- 谁能选择提交者，选择依据怎样固定并接受审计；
- 控制平面崩溃后怎样发现孤儿副作用；
- 重叠观察怎样进入 Iknem 覆盖计算。

### 不建立新 CORE 的停止条件

若上述义务可以唯一归入 DRO、AUT、ADP、ID、IKN 与 DIA，研究到此停止，不建立 `PAR-CORE`、分支制品或事务格式。只有出现独立持久对象、真实生产者和消费者、现有规范无法承接的不变量、可重现失败和可验证物理需求时，才重新讨论新规范。

## 权威资料与采用限制

- [MCP 2025-11-25 Tasks](https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks)：采用 Task 身份、创建与结果分离、并发编排和取消不保证停止；不把实验性 Task 变成 Noemion 对象或提交权威。
- [MCP Sampling 草案](https://modelcontextprotocol.io/specification/draft/client/sampling)：只作为并行工具调用正在普及的趋势证据；不作为当前稳定符合性基线。
- [MCP `2026-07-28` 发布候选](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/)：采用 Tasks 将移至扩展、Sampling 将弃用和正式稿仍未发布的漂移证据；不把候选协议标识写成当前正式版。
- [A2A 1.0 版本化规范](https://a2a-protocol.org/v1.0.0/specification/)：采用外部 Task、Artifact、流和幂等各有协议语义；不让它们替代本地目标、证据或决定。
- [GNU Make：Parallel Execution](https://www.gnu.org/software/make/manual/html_node/Parallel.html)：采用有界并行与子进程仍可能继续的事实；不采用 recipe 成功即目标满足。
- [GNU Make：Disabling Parallel Execution](https://www.gnu.org/software/make/manual/html_node/Parallel-Disable.html)：采用完整依赖优先、不确定时序列化；不把 `.WAIT` 固化为运行接口。
- [GNU Make：Job Slots](https://www.gnu.org/software/make/manual/html_node/Job-Slots.html)：采用递归工具共享有界配额；不把 jobserver 令牌当作授权能力。
- [GNU Make：Interrupts](https://www.gnu.org/software/make/manual/html_node/Interrupts.html)：采用不完整目标清理、原子更新和清理不保证成功的边界；不把文件清理当作跨系统回滚。
- [RFC 9110 第 13.1.1 节 If-Match](https://www.rfc-editor.org/rfc/rfc9110.html#section-13.1.1)：采用状态变更前的强前提与丢失更新防护；不把 HTTP ETag 变成所有协议的唯一实现。
- 路德维希·维特根斯坦，《逻辑哲学论》2.04：只用于区分可能候选与已经存在的事态，不作为并发算法或事务语义来源。

## 当前决定边界

当前可以冻结的只有审查结论：并行和推测执行不产生新权限，分支默认只产生候选，外部副作用需要单一而重新核对的提交路径，取消不能洗掉已经发生或未知的效果，所有分支共享同一 Dromen 上限，完成顺序不能越过满足与最终决定边界。

六项最小责任只有在形成正式 ADR 后才能写回现有 CORE 并补充向量。在此之前，本文件保持非规范、不得进入登记，也不得被描述为运行时、调度器、事务系统或 Agent 组件已经实现。
