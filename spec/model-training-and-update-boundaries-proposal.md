# Noemion 模型训练与更新边界研究提案

状态：非规范研究提案

日期：2026-07-16

当前策略：正在研究；尚未进入规范

研究范围：训练数据、人工与模型反馈、微调和对齐、基础模型与适配权重、训练环境、评测、发布、回滚与会话漂移

本提案不构成 ADR、CORE 规范、Profile、登记项或实现要求，不进入 `registry.json`。它不创建模型制品、数据集格式、训练清单格式、反馈格式、模型仓库、训练平台、命令、组件、结果域、稳定接口或哲学专名，也不建立 `TRAIN-CORE`、`MODEL-CORE` 或 `FEEDBACK-CORE`。

## 直接结论

模型更新不是“系统记住了新信息”，也不是“证据越来越多”。训练或微调会形成新的模型派生物；会话历史、检索内容、提示、适配权重、基础模型、评测结果和发布决定必须分别识别。

Noemion 当前不需要新的模型对象或训练组件。未来若接入外部训练生态，只应把精确数据、训练活动、派生模型、评测证据和发布决定映射回现有责任：

- TEXT-IDENTIFIER 约束实际文本、变换、显示和模型输入；
- ID 约束数据快照、基础模型、权重、适配器和派生关系的精确身份；
- AUT 约束数据、反馈、模型和目的是否获准使用；
- EVIDENCE 保存有范围的来源、训练、评测和发布依据，但不自证真实性、充分性或接受；
- ADP 固定外部模型、训练服务、仓库和供应商语境；
- SESSION 使模型或适配器实质漂移后的旧会话失效；
- DIA 报告污染、身份不明、配方缺失、环境漂移和评测失败；
- END 与 CLOSURE 继续只表达目标及其组合，不保存权重、训练数据、奖励或优化轨迹。

训练成功、损失下降、偏好胜率提高、模型签名、权重摘要相同或发布流水线完成，都不能自动产生 `met`、充分 evidence 或 `accepted`。

## Noemion 当前语义抽取模型路线

**复核日期：2026-07-16。** 当前候选基础模型是 [Qwen3.5-0.8B-Base](https://huggingface.co/Qwen/Qwen3.5-0.8B-Base)。Noemion 只研究它在自然语言语义抽取中的受限用途，不要求模型承担聊天、开放式推理、授权、目标满足判断或最终决定。模型输出始终是待检查的候选结构；确定性形成、授权、证据评价和最终接受继续由各自责任承担。

当前策略不把设备内算力当作最终训练环境，而是把研发分为两个阶段：

| 阶段 | 当前范围 | 不据此声称 |
| --- | --- | --- |
| 本机流程验证 | 从基础模型导出纯文本路径，删除视觉编码器和视觉投影；保留原始 tokenizer、词表、默认语言覆盖与原生长上下文结构；优先使用 LoRA，只有资源不足时才采用 QLoRA；用常规长度样本依次验证训练、适配器合并、量化、CPU 推理和效果评测 | 已经训练出模型、默认语言能力完整保持、256K 训练已完成，或设备内效果已经满足发布要求 |
| 外部算力训练 | 复用同一数据身份、输出约束和评测协议，比较全参数微调、较长上下文训练与强教师蒸馏；训练服务必须允许固定自有基础模型并导出完整派生权重 | 托管 API 微调等于可下载模型，云端训练成功等于设备内可部署，或更强教师自动保证学生忠实 |

删除视觉部分只减少多模态权重、模型包和相应加载开销，不把约 0.8B 的语言骨干改写成约 0.6B。当前不裁剪多语言词表，也不重建 tokenizer；“保留默认能力”只表示不主动移除其表示范围，不能替代微调后的多语言回归观察。

本机验证优先使用 2K–8K 的常规训练样本和少量较长样本。基础模型的原生长上下文结构继续保留，但本机不承担 128K/256K 全长度训练；超出一次推理预算的文档必须另外验证分段、重叠、跨段引用、结果合并、冲突和遗漏，不能把结构声明当作长文本抽取证据。

本机阶段至少记录并比较：

1. 精确基础模型、文本化变换、适配器目标层、训练配置、数据快照和派生权重身份；
2. 无模型或确定性基线、未微调基础模型、合并前后模型与量化模型在同一隔离测试集上的结果；
3. 输出结构有效率，以及目标字段的精确率、召回率、遗漏、误报、未知和冲突处理；
4. 长文档分段与直接长上下文调用的覆盖差异，不能只报告一次成功样例；
5. 量化前后回归、CPU 延迟、峰值内存、启动开销和资源上限；
6. 少量基座默认语言回归探针，以及微调收益是否伴随非目标能力明显退化。

本机流程验证即使通过，也只说明训练与部署路径可以继续研究。外部算力训练、最终模型选择、权重发布和组件接入仍分别需要精确身份、评测结果、发布决定与明确的实现范围。

## 至少十一种事实必须分开

| 事实 | 回答的问题 | 最低绑定 | 不能自动推出 |
| --- | --- | --- | --- |
| 数据快照 | 实际用了哪些记录 | 精确内容身份、来源、截止点、切片、去重与排除 | 合法、代表性、无污染 |
| 数据资格 | 为什么可以为此目的使用 | 权利、同意、隐私、租户、司法范围、目的与保留政策 | 数据为真、标签正确 |
| 反馈记录 | 谁对什么输出给出什么信号 | 主体、界面、实际显示、候选顺序、规则、时间与限制 | 授权训练、代表所有用户 |
| 数据处理 | 原始数据怎样变成训练输入 | 代码、参数、顺序、过滤、规范化、损失与输出身份 | 处理后语义等价 |
| 基础模型 | 从哪个精确模型开始 | 供应者、对象种类、版本、完整身份、许可与限制 | 供应者名称等于精确权重 |
| 更新方法 | 通过什么活动改变模型 | 目标、算法、损失或偏好规则、超参数、固定层与停止条件 | 方法名称保证对齐或安全 |
| 执行语境 | 在什么计算条件下运行 | 代码、依赖、硬件、分布式拓扑、随机来源、顺序与资源 | 相同种子必然得到相同权重 |
| 派生模型 | 实际产生了什么 | 完整权重或适配器身份、基础模型关系、训练活动与检查点 | 新模型更好、可部署 |
| 评测 | 新旧模型在限定用途上怎样比较 | 目的、构念、数据、协议、统计、不确定度与偏差探针 | 固定基准代表所有用途 |
| 发布决定 | 哪个精确模型可供谁在何处使用 | 权威、政策、用途、受众、部署配置、截止点与撤回条件 | 训练完成或评测高分即批准 |
| 运行观察 | 发布后实际发生了什么 | 精确部署、调用语境、观察范围、延迟与损失 | 单条反馈代表模型能力或因果 |

这十一项可以由外部系统分别保存。它们不需要被压成一个新的 Noemion “模型状态”，也不能用一个 `version`、`latest`、run ID 或模型卡替代。

## 会话、检索、提示与训练不是同一更新

| 变化 | 改变了什么 | 是否产生新模型 | Noemion 边界 |
| --- | --- | --- | --- |
| 会话历史增加 | 后续调用的候选上下文 | 否 | TEXT-IDENTIFIER 与 ID 记录实际输入和变换；不恢复旧权限 |
| 检索索引更新 | 可被选入上下文的外部资料 | 否 | ADP 固定来源和版本；检索命中不取得事实或指令权 |
| 系统提示或工具 schema 变化 | 推理调用的控制输入 | 否 | 旧 contract 因实质漂移失效；实际输入必须绑定 |
| 参数高效微调或适配器替换 | 基础模型上的派生参数 | 是 | 适配器和基础模型组合必须精确绑定；不能只写适配器名 |
| 全量微调、偏好优化或继续预训练 | 模型权重及行为分布 | 是 | 建立新身份和派生关系，重新评测并重新发布决定 |
| 供应商静默更新模型别名 | 远端实际模型可能变化 | 未知 | 无法解析精确身份时不得声称可复现或与旧会话相同 |
| 回滚部署指针 | 当前服务选择的模型变化 | 否，但部署事实改变 | 不撤销已经发生的输出、外部副作用或训练数据使用 |

“记忆”“学习”“更新”“版本”和“恢复”因此不能互相替换。面向读者的页面必须说明究竟变化的是上下文、数据、权重、部署选择还是权威决定。

## 反馈先是记录，之后才可能成为训练输入

人工偏好、用户投诉、模型裁判、自动奖励和工具结果都可以提出候选信号，但它们的来源类别不能因进入训练流水线而升级。

反馈进入数据快照前至少要回答：

1. 反馈者看到的精确内容、顺序、标签和界面是什么；
2. 反馈者是否获知用途并授权将反馈用于训练；
3. 反馈是否包含个人、机密、租户或受限内容；
4. 采样总体、缺失、退出、重复和激励怎样影响代表性；
5. 模型生成、模型裁判和人工反馈怎样区分；
6. 反馈怎样聚合，分歧与无法判断怎样保留；
7. 哪个决定权威允许它进入哪个精确训练目的；
8. 删除、更正、撤销或保留期限怎样传播到后续派生物与风险决定。

模型裁判输出继续属于 `model-candidate`。同一模型既生成候选又给出奖励，或多个相关模型互相投票，会形成相关反馈回路；不能按独立人工证据累计。

## 可复现环境不等于可复现模型

[GNU Guix 参考手册](https://guix.gnu.org/manual/en/guix.pdf)说明，固定 channel commit 与 manifest 可以复制精确的软件环境。Noemion 借鉴完整依赖闭包和可追溯环境，但不把它扩大成模型可复现性证明。

一个训练复现主张还必须固定或披露：

- 精确数据与采样顺序；
- 基础模型与全部适配器；
- 训练代码、配置和停止条件；
- 伪随机算法、种子、每个工作进程的状态与恢复点；
- 硬件、驱动、加速库、数值精度与非确定性算子；
- 分布式拓扑、批次划分、归约顺序与失败重试；
- 产生的每个候选权重身份；
- 独立执行路径、逐字节比较和行为评测。

[GNU Diffutils 的 `cmp`](https://www.gnu.org/software/diffutils/manual/html_node/Invoking-cmp.html)只回答两个文件的字节是否相同、不同或比较失败。字节相同可以支持精确身份复现；字节不同不会说明哪一个模型更好，字节相同也不会证明训练数据合规、模型安全、用途适合或发布获批。

固定种子、相同损失曲线、相同 run ID、相同模型别名或相近评测分数都不是逐字节复现。无法取得精确远端模型身份时，只能形成供应商与调用语境限定的行为观察。

## 支持案例与反例

1. 同一基础模型和数据使用新学习率得到新权重。正确做法是形成新的派生模型身份、评测和发布决定；不能覆盖旧模型的身份或接受状态。
2. 只替换 LoRA 适配器。实际推理对象是基础模型、适配器、合并顺序和推理设置的组合；只记录适配器文件名不能重建行为。
3. 供应商保留同一模型别名却更新后端。旧 contract 不能继续声称模型未变；无法解析精确身份时必须按漂移处理。
4. 用户点击“赞”但界面没有说明用于训练。反馈可以保留为产品观察，不能自动进入训练资格。
5. 模型裁判为自己生成的答案打高分。输出仍是相关的 `model-candidate`，不能成为独立人工偏好。
6. 五个评审模型共享基础模型和训练数据并一致投票。五票不能当五项独立证据，也不能自动成为训练目标。
7. 训练损失下降但固定基准退化。训练活动完成不产生发布批准；评测和用途决定单独处理。
8. 固定基准提高，但目标用户子群显著退化。总体点估计不能覆盖分群风险；发布决定必须保留限定用途和不确定度。
9. 公开评测题被混入训练数据。该分数不能支持未见任务能力，污染范围无法界定时相应推广主张失败。
10. 训练集包含模型生成数据但没有来源标签。无法审计递归比例、代际和尾部覆盖；训练资格或外部有效性不足。
11. Guix 环境和数据摘要相同，两次训练权重不同。环境复现成立，权重复现不成立；继续检查硬件、顺序、随机状态和非确定性算子。
12. 两次权重逐字节相同。只支持精确输出复现，不支持数据权利、模型安全、满足或最终接受。
13. 回滚到旧权重后，系统宣称新版本产生的外部副作用已经撤销。回滚只改变部署选择，旧输出、调用和副作用继续保留。
14. 删除请求到达后只从下一版数据集中移除记录。必须另外评估既有权重、检查点、缓存、评测数据、披露与后续发布；不能声称历史使用被物理撤销。
15. 一个适配器在未经核对的新版基础模型上装载成功。格式兼容不证明语义兼容、用途安全或授权范围相同。
16. 线上投诉触发自动微调并直接发布。观察、训练资格、更新活动、评测和发布决定被压成一条自动回路，应拒绝闭环升级。

## 威胁到失败责任的映射

| 威胁 | 失败方式 | 唯一主责任 | 不能洗成 |
| --- | --- | --- | --- |
| 数据投毒 | 恶意样本、标签或代码改变派生模型 | 数据治理、训练系统、EVIDENCE | 普通模型不确定性 |
| 反馈洗白 | 工具结果、模型评分或未授权点击被称为人工偏好 | AUT、TEXT-IDENTIFIER、EVIDENCE | 已授权训练信号 |
| 隐私与权利越界 | 数据目的、同意、许可或保留条件不成立 | AUT、数据权威 | 模型已签名所以可用 |
| 合成数据递归 | 来源代际不明、尾部和少数模式消失 | 数据治理、评测 | 平均分稳定所以无风险 |
| 训练与评测污染 | 相同题目、近重复或工具答案穿过边界 | 评测治理、ID、EVIDENCE | 未见能力提升 |
| 基础模型替换 | 别名、URL 或仓库标签解析到不同对象 | ADP、ID | 向后兼容更新 |
| 适配器错配 | 适配权重绑定到错误基础模型或顺序 | ID、部署控制 | 能加载所以兼容 |
| 配方缺失 | 数据处理、超参数、随机与停止条件不完整 | 训练系统、EVIDENCE | 可复现训练 |
| 环境和数值漂移 | 硬件、驱动、精度或归约顺序改变结果 | 训练系统、ID | 相同种子保证相同权重 |
| 评测窄化 | 固定基准、单一总体或模型裁判代表全部用途 | 测量程序、EVIDENCE | 模型普遍更好 |
| 奖励投机 | 模型优化评审表面信号而非真实用途 | 训练治理、评测 | 偏好胜率等于目标满足 |
| 自动发布闭环 | 线上反馈未经资格、评测和权威决定直接更新生产 | AUT、发布控制 | 持续改进 |
| 回滚误报 | 部署指针回退被当成副作用、数据使用或风险撤销 | 部署控制、EVIDENCE | 系统恢复原状态 |
| 撤销传播缺失 | 数据删除、模型撤回或政策变化未到达派生物 | 数据权威、ID、AUT | 旧签名仍有效所以可继续 |

## 失败域与结果隔离

| 事件 | 训练活动 | 模型身份 | 评测 | 会话 | 满足 | 最终决定 |
| --- | --- | --- | --- | --- | --- | --- |
| 训练作业正常结束 | completed 记录 | 新候选身份 | 仍待执行 | 未建立 | 不变 | 不变 |
| 权重写入中断 | failed 或 stopped 记录 | 不形成可信完整身份 | 不适用 | 不建立 | 不变 | 不变 |
| 评测数据污染 | 训练事实保留 | 身份不变 | 相应主张无效或不足 | 视政策失效 | 不自动改变 | 发布决定重审 |
| 新模型在当前 contract 中出现 | 外部更新事实 | 新身份或身份未知 | 旧评测不自动继承 | 旧会话失效 | 旧判断不改写 | 重新决定 |
| 回滚部署指针 | 新部署事实 | 两个身份均保留 | 风险与用途重新评估 | 新会话 | 历史满足不改写 | 新发布决定 |
| 用户投诉 | 有范围观察 | 身份不变 | 进入监测材料 | 当前会话按风险处理 | 不自动 `unmet` | 由具名权威判断 |

训练、评测和部署可以各自拥有外部状态，但这些状态不得进入 ADR-0015 的既有结果域并改写其含义。

## 候选责任的唯一主归属

| 义务 | 主归属 | 辅助归属 | 不新增的对象 |
| --- | --- | --- | --- |
| 实际文本、反馈显示和数据变换 | TEXT-IDENTIFIER | ID、AUT | 训练文本制品 |
| 数据和反馈使用资格 | AUT | EVIDENCE、TEXT-IDENTIFIER | 同意令牌格式 |
| 数据、模型、权重与派生关系 | ID | EVIDENCE、ADP | Noemion 模型制品 |
| 外部训练服务、模型仓库和供应商别名 | ADP | ID、AUT、DIA | 训练适配 CORE |
| 来源、训练活动、评测和限制记录 | EVIDENCE | ID、DIA | 训练证据品牌 |
| 模型与适配器漂移后的会话失效 | SESSION | ADP、ID、AUT | 可恢复 contract |
| 训练和评测失败诊断 | DIA | EVIDENCE、ADP | 新结果域 |
| 目标、满足与最终决定 | END、EVIDENCE、AUT | 具名权威 | 模型自我验收 |

若未来真实训练消费者需要物理清单，应优先采用外部成熟模型供应链与数据治理格式，并通过 ADP Profile 固定版本和损失。只有外部格式无法表达 Noemion 特有且不可替代的责任，才重新审查是否需要新对象。

## 证据等级

| 等级 | 能证明什么 | 仍不能证明什么 |
| --- | --- | --- |
| 配置记录 | 声明的数据、方法和参数 | 实际执行、数据资格、输出身份 |
| 训练日志 | 某次运行报告的过程 | 日志完整、权重正确、评测有效 |
| 精确身份与派生图 | 数据、基础模型、权重和活动的绑定 | 数据合法、模型更好或可发布 |
| 独立环境与字节复现 | 在声明条件下得到相同输出 | 用途安全、统计推广、最终接受 |
| 预注册评测与反例 | 限定构念、总体和协议上的表现 | 所有用途、未来漂移、权威批准 |
| 发布与监测决定 | 具名权威在精确范围内的选择 | 目标永久满足、风险已经消失 |

## 术语和读音边界

本提案故意只使用“模型训练”“模型更新”“反馈记录”“派生模型”“发布决定”等普通职责短语，不创造短词、首字母缩写或希腊词根。

原因不是这些职责不重要，而是它们已经有成熟行业称呼，并且当前没有独立 Noemion 物理对象。新专名会让首次读者误以为项目已经建立训练平台、模型格式或自我学习组件，也会增加口头协作负担。

若以后出现不可替代对象，名称仍须通过职责、冲突和词首检查；只有人为创造的名称继续执行完整读音验证：

1. 独立对象、消费者、生命周期和失败责任条件；
2. 软件、AI、标准、包名和权利冲突检索；
3. 普通词逐词检查自然词首，机器值同时核对版本化关键字语料；
4. 自造名称完成中文与英文首次朗读、听写回填、职责匹配以及与 Endem、现行对象和公开动作的口头区分；
5. 自造名称的人类证据失败时直接重新命名，不用发音指南补救接口。

## 接受后的落点与停止条件

若责任分配被现行架构接受，后续只能把唯一义务归还现有规范与页面：

1. TEXT-IDENTIFIER 增加反馈显示、训练输入和变换损失的适用说明；
2. ID 增加外部数据快照、模型、适配器、权重和训练活动派生关系的抽象案例；
3. AUT 增加训练目的、反馈使用、删除和撤销传播的授权案例；
4. EVIDENCE 增加来源、训练、评测、发布与监测记录的范围和覆盖反例；
5. ADP 增加外部模型服务、训练服务、仓库别名和身份解析失败边界；
6. SESSION 明确模型、适配器、提示或推理设置实质漂移使旧会话失效；
7. DIA 只在出现真实消费者后登记有限诊断概念；
8. END 和 CLOSURE 明确排除训练数据、权重、奖励、优化轨迹和发布状态。

以下任一条件成立就停止建设 Noemion 自有训练面：

- 外部成熟工具能保存所需来源、身份、权利、评测和发布关系；
- 没有获准的数据、算力、真实消费者或可证伪增益假设；
- 训练结果不能在无模型基线或外部成熟方案上证明不可替代净增益；
- 隐私、权利、投毒、模型漂移、撤销或评测污染无法形成可接受控制；
- 项目尚未进入相应代码开发阶段。

## 当前策略与待定内容

本提案现在只形成六项候选结论：

1. 会话记忆、检索、提示变化、训练、部署和回滚必须分开。
2. 训练或微调产生新的外部派生模型身份，不能原地修改旧身份。
3. 人工、用户或模型反馈先是有范围记录，只有另行授权和治理后才可能成为训练输入。
4. 环境复现、权重字节复现、行为评测、用途适合和发布决定是五个不同主张。
5. 现有 TEXT-IDENTIFIER、ID、AUT、EVIDENCE、ADP、SESSION、DIA 与 END 足以承接候选责任，不创建新 CORE、制品、组件或专名。
6. Noemion 默认继续适配外部成熟训练生态；没有不可替代消费者与证据时停止自建训练面。

这些结论不改变现行规范、Profile、向量、组件边界或实现状态。责任分配和真实外部训练系统的 Profile 仍是待定内容，须由后续证据支持并成为现行架构结论。
