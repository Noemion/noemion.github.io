# Noemion 状态变化与因果归因边界研究提案

状态：非规范研究提案

日期：2026-07-15

结论状态：正在研究；尚未进入规范

适用范围：Endem 的 `kine`、状态满足、动作发生、状态转变、因果归因、Drase 会话、Iknem 溯源、外部协议和最终决定

本提案不构成 ADR、CORE 规范、内容 Profile 或实现要求，不进入 `registry.json`。它不创建新制品、文件格式、扩展名、命令、组件、结果域、稳定接口或哲学专名。当前只检查现有责任能否区分终态、动作、转变与因果，不声明相应组件已经实现。

## 直接结论

当前体系存在一个需要先澄清、不能靠实现默认值掩盖的语义缺口。

`krin` 能够比较 `skena` 与结构化观察 `phain`，因此能回答“目标事态是否成立”。但 END-TEL-001 又把 `kine` 解释为“使事态达到成立”。中文“使”通常还声称某个动作或主体造成了从未成立到成立的变化。现行六个语义面没有固定基线状态、动作发生、转变窗口、竞争行为者或因果判定规则，所以它们不能单凭终态观察证明这种强因果主张。

建议把 `kine` 稳定解释为目标方向，而不是内置的因果算子：

- 对结果型目标，`kine` 表示“要求事态达到成立”。若判断时事态已经成立，可以得到 `met`，但不能因此声称执行了动作、发生了转变或某主体造成了结果。
- 对动作型目标，动作发生本身必须作为 `skena` 中可观察的关系表达，不能由 `kine`、工具成功、外部 Task 完成或 Drase `completed` 推断。
- 对转变型目标，必须证明声明边界内存在前态、后态和允许的转变关系。现行 END-P1 不能承载这组约束，必须保留为 `apor` 或明确拒绝，不得退化成只看后态。
- 对因果型目标，还必须固定因果主张范围、竞争解释、证据方法和决定权威。Iknem 可以保存有范围的溯源与观察，但活动关联、时间先后和日志共现都不自动证明充分因果。
- 授权、责任、满足、会话终止和最终接受继续保持正交。被授权行动不等于行动发生；行动成功不等于目标满足；终态成立不等于该行动造成了终态。

如果这一方向被接受，END-TEL-001 及所有公开解释中的“`kine` 表示使事态达到成立”应一次性改为“`kine` 表示要求事态达到成立”，再用独立条款说明动作、转变和因果主张。当前提案本身不改写已接受条款。

## 为什么这不是措辞小问题

《逻辑哲学论》命题 2.01、2.031 与 2.1 关注对象配置和事态；命题 4.01 把命题理解为事实的图像。Noemion 从中采用的工程限制是：终态关系必须由结构显示，不能由一句叙述的语气替代。但这些命题没有给出行动理论、责任规则或因果检验方法，不能从“事态成立”推出“某行动者使它成立”。

这一混淆在 Agent 系统中尤其危险。模型很容易把“我调用了工具”“任务返回完成”“日志随后出现目标状态”叙述成“我完成了目标”。如果规范不拆开这些主张，系统会把相关性、时间顺序、协议状态甚至模型自述洗成因果与责任证据。

## 权威资料给出的边界

| 资料 | 它真正定义什么 | Noemion 可以采用什么 | 不能推出什么 |
| --- | --- | --- | --- |
| [GNU Make](https://www.gnu.org/software/make/) 与 [Instead of Execution](https://www.gnu.org/software/make/manual/html_node/Instead-of-Execution.html) | make 先判断 target 是否已是最新；已最新时不执行 recipe，`-q` 还能只查询目标状态 | 目标状态、是否需要更新和更新动作是三个不同事实 | target 已最新不能证明本次执行过 recipe；recipe 返回也不能单独证明目标内容正确 |

| [CloudEvents](https://github.com/cloudevents/spec/blob/ce@stable/cloudevents/spec.md) | Event 是表达一次 occurrence 及其上下文的数据记录；一次 occurrence 可以产生多个 event | 事件记录、事件来源和现实发生应保持可区分并能去重 | 收到事件不能自行证明事件内容真实、消费者造成后续事件或业务目标满足 |
| [OpenTelemetry Tracing API](https://opentelemetry.io/docs/specs/otel/trace/api/) | Span 表示一次操作；插桩库、应用开发者或运维按各自约定设置状态，默认状态是 `Unset` | 操作轨迹可作为有范围的运行观察进入 Iknem | Span 结束或 `Ok` 只描述该操作语境，不能自动证明外部业务终态或因果责任 |
| [RFC 9110 Idempotent Methods](https://www.rfc-editor.org/rfc/rfc9110.html#name-idempotent-methods) | 幂等描述重复相同请求的预期效果，并约束何时适合自动重试 | 重试需要协议语义、去重或事后核对及共同预算 | 幂等不证明第一次请求未发生、第二次产生了变化或终态由该请求造成 |
| [ReAct](https://arxiv.org/abs/2210.03629) | 模型交替生成推理、行动和观察，并据此更新计划 | 计划、动作、观察和重规划应分别记录 | 模型行动轨迹或自述不能成为因果、满足或最终接受的唯一证据 |
| [A2A 1.0 版本化规范](https://a2a-protocol.org/v1.0.0/specification/) | Task 是 A2A 的动作单元，分别保存当前状态、产物与交互历史；`completed` 是 Task 的成功终态 | 外部任务生命周期保持在适配域，并固定所依据的协议版本 | `completed` 不能直接成为 Endem `met`、Drase `completed` 或最终 `accepted` |

共同结论不是“永远不能证明因果”，而是每种主张必须先声明对象、范围、反事实或竞争解释、证据方法与权威。终态观察本身只支持终态判断。

## 五类主张必须分开

| 主张 | 回答的问题 | 最低观察 | 当前主责任 | 不能自动推出 |
| --- | --- | --- | --- | --- |
| 终态满足 | 目标关系现在是否成立？ | 与 `skena/krin` 对齐的 `phain` 和适用 Iknem | END、IKN | 动作发生、状态变化、主体因果、最终接受 |
| 动作发生 | 某个动作是否被请求、开始、执行或结束？ | 有身份的动作事件、输入、主体、时间与结果 | DRO、ADP、IKN | 动作有效、终态成立、该动作造成终态 |
| 状态转变 | 声明边界内是否从前态变为后态？ | 同一对象与关系的前态、后态、顺序、窗口和观察闭合 | END、IKN | 指定主体造成转变、转变被授权、终态持续 |
| 因果归因 | 哪个活动或主体在声明模型中对转变有何影响？ | 有范围的因果规则、竞争解释、干预或其他获授权方法及其限制 | IKN、AUT | 道德责任、法律责任、最终接受、普遍因果 |
| 责任与决定 | 谁被授权行动、承担责任或接受结果？ | 精确政策、委托、主体、对象、动作和决定语境 | AUT | 行动发生、目标满足、因果真实 |

Drase 会话终止、外部 Task 状态、诊断和签名继续位于各自结果域，不加入上述任何一列。

## `kine` 的建议语义

`kine` 应回答“这个 Endem 要求事态朝哪个方向被判断”，而不是“哪种执行历史必须已经发生”。这使它与 `skena` 的中性事态、`krin` 的比较契约和 ADR-0015 的结果正交保持一致。

### 结果型目标

“私有报告不允许匿名读取”若只关心当前安全终态，则目标在会话开始前已经成立也可以是 `met`。此时没有必要强制执行一次撤权动作，也不得生成“本会话撤销了权限”的动作或因果声明。若 `mene` 要求持续成立，还需独立的时间覆盖；最终接受仍由具名权威决定。

### 动作型目标

“轮换签名密钥”不只是期望某个密钥当前有效。若用户要求轮换行为本身，`skena` 必须表达有身份的轮换发生、旧新对象关系、主体和截止点；仅观察到新密钥有效不能证明轮换事件，也不能证明谁执行了它。

### 转变型目标

“把服务从不可用恢复到可用”要求一个前态与后态，而“服务可用”只要求后态。两者不能共用同一个只看终态的 `krin`。当前 END-P1 没有前态、转变窗口或变化归属字段，所以前一种表达仍未获得 Profile 支持。

### 因果型目标

“通过部署版本 A 使错误率下降”还包含活动、候选原因、背景变化和测量构念。部署先发生、错误率后下降，只能形成时间关联。除非授权方法能够排除或建模并发流量、配置、依赖和观测变化，否则因果结论应为未知或受限主张，而不是 `met` 的附赠解释。

## 支持案例与反例

| 场景 | 终态判断 | 动作或会话判断 | 因果边界 |
| --- | --- | --- | --- |
| 服务原本不健康，受权动作后恢复，观察闭合 | 可按 `krin` 得到 `met` | 动作可独立记录为完成 | 仍需有范围方法才能归因 |
| 服务在会话开始前已经健康 | 结果型目标可为 `met` | 不得虚构修复动作 | 没有本会话致因主张 |
| 服务已健康，但用户明确要求重启 | 健康终态不能代替重启事件 | 必须观察重启动作关系 | 重启是否造成健康另行判断 |
| 工具返回成功，服务仍不健康 | `unmet` 或按覆盖情况分类 | 工具调用可以完成 | 工具成功不能提升终态 |
| 工具失败后，另一控制器恢复服务 | 终态可为 `met` | 本动作失败；会话另行终止 | 不得把外部修复归给失败动作 |
| 两个控制器并发修改同一状态 | 终态按观察判断 | 两条活动轨迹分别保存 | 单靠时间顺序不能唯一归因 |
| 动作完成后很久才出现目标状态 | 后态可能成立 | 动作与延迟分别记录 | 必须声明最大影响窗口和替代原因 |
| 状态短暂成立后立即回退 | 瞬时 `kine` 与持续 `mene` 结论不同 | 回退事件保留 | 一次成功不能证明持续效果 |
| 幂等重试命中已经存在的资源 | 终态可成立 | 重试可能是无变化的 no-op | 幂等不能证明哪次请求造成资源 |
| 非幂等请求断连后被重复执行 | 终态可能超出要求 | 两次调用与副作用分别记录 | 不能把重复结果洗成一次成功 |
| 发送取消后远端继续产生副作用 | 终态按真实观察判断 | 取消请求不等于停止或回滚 | 取消不能证明影响链终止 |
| 缺少可靠前态观察 | 后态仍可按结果型目标判断 | 动作可以有轨迹 | 转变与因果默认未知 |
| 模型声称“我修复了问题” | 不影响 `krin` | 只是一条不可信叙述 | 不能成为归因证据 |
| Span 为 `Ok`，业务终态不成立 | `unmet` 或 `agno/fault` | 操作在自身语境可为成功 | 遥测成功不提升业务因果 |
| 部署完成事件作为 `mene` 锚点 | 只确定区间起点候选 | 事件发生需独立验证 | 锚点不证明部署造成持续终态 |
| 撤销匿名读取权限 | 负 `skena` 可按闭合观察判断 | 撤权事件若被要求须单独表达 | 缺席访问不证明特定主体完成撤权 |

## 威胁到失败责任的映射

| 威胁 | 典型攻击或误判 | 必须由谁阻断 |
| --- | --- | --- |
| 因果漂白 | 终态为真就声称 Agent 完成了目标 | END 限定 `kine`；IKN 限定主张范围 |
| 动作成功提升 | HTTP 成功、工具成功或 Task completed 直接产生 `met` | ADP 与 END 保持结果分离 |
| 基线缺失 | 只观察后态就声称发生了转变 | END/IKN 要求前后态与闭合边界 |
| 事后归因 | 因为 B 发生在 A 之后就声称 A 导致 B | IKN 要求具名因果方法和竞争解释 |
| 并发主体遮蔽 | 多个控制器行动却只记录最后一个 | DRO/ADP/IKN 保留全部有身份活动 |
| 幂等漂白 | no-op 被写成“本次创建成功” | ADP 保留调用与事后状态核对 |
| 重试重复 | 断连后重复非幂等副作用 | ADP 用去重证据和共同预算阻断 |
| 取消冒充回滚 | 发出取消即声称副作用消失 | ADP 要求确认和后续观察 |
| 延迟影响扩张 | 任意晚到变化都归给早期动作 | IKN 固定影响窗口和替代解释 |
| 观察缺口消失 | 缺失前态、事件或并发日志仍给确定归因 | IKN 产生覆盖不足；故障按契约分类 |
| 授权继承 | 被授权执行就声称执行并成功 | AUT 与 DRO/IKN 分离 |
| 模型叙事权威 | Agent 自述成为因果决定 | AUT 不授予模型默认决定权 |

## 失败域与结果隔离

- 目标终态已成立时，结果型 `kine` 可以是 `met`；动作未执行不是 `unmet`，除非动作本身进入 `skena`。
- 动作执行成功但终态不成立时，满足结果不能是 `met`；动作结果保留在会话或外部域。
- 后态成立但前态不可知时，结果型目标可以判断，转变主张保持未知；不能用后态补写前态。
- 转变存在但具体原因不明时，状态变化可以成立，因果归因保持未知；不能把“未知原因”改成工具故障。
- 因果检查程序违约、日志损坏或时钟契约失败属于 `fault` 或相应诊断；竞争解释未被排除通常属于因果主张覆盖不足。
- 签名只绑定谁签了哪项精确主张；它不能把错误归因变成事实。
- Iknem 的记录完整、`valid` 或 `sufficient` 只能相对声明范围解释，不能把相关性提升为普遍因果。
- 权威最终 `accepted` 也必须保留原满足、动作和因果结论，不能倒写历史事实。

## 候选责任的唯一主归属

本提案不建议创建“因果制品”“行动对象”或新的 CORE 家族。未来条款应优先归还现有边界：

| 责任 | 主归属 | 说明 |
| --- | --- | --- |
| 终态关系、目标方向与显式动作/转变关系 | END-CORE | `kine` 只固定目标方向；强主张必须由结构显示 |
| 多目标依赖与成员结果 | SYN-CORE | 不把多个独立动作塞进一个 Endem |
| 会话内动作、能力、预算、取消和终止 | DRO-CORE | 行动轨迹不改写 Endem 身份或满足结果 |
| 外部调用、幂等、重试与协议状态 | ADP-CORE | 保留来源域和副作用不确定性 |
| 前后态、活动、主体、溯源、覆盖和因果方法 | IKN-CORE | 证据只支持声明范围，不自授决定权 |
| 主体资格、委托、行动授权、责任与决定 | AUT-CORE | 被授权、实际行动和造成结果保持分离 |
| 失败种类、位置、披露与恢复 | DIA-CORE | 因果未知不伪装成工具错误或 `unmet` |
| 精确对象、活动记录和派生关系身份 | ID-CORE | 名称、路径、trace ID 或短摘要不能绑定安全主张 |
| 可变计划和重规划 | Drasor 外侧控制平面 | 计划只能选择行动候选，不能宣告因果 |

只有真实消费者证明这些责任无法形成唯一主条款时，才重新讨论独立规范。不能为了结构对称创建新专名、命令或组件。

## 进入 ADR 或规范的证据要求

在责任分配形成正式 ADR 前，本提案不修改 END-TEL-001，不进入 CORE、Profile、登记或向量。未来若要形成规则，至少需要：

1. 明确接受 `kine` 是目标方向，不是内置因果算子。
2. 分别给出结果型、动作型、转变型和因果型自然语言支持案例与最小反例。
3. 说明强主张是否能用现有 `skena` 关系表达，还是需要未来 Profile 能力。
4. 固定前态、后态、观察窗口、对象连续性、竞争主体和缺口分类。
5. 为因果主张固定方法、适用域、反事实或竞争解释、权威与限制。
6. 形成终态已成立、外部主体介入、并发控制器、延迟效果、幂等 no-op、重复副作用和取消未回滚的正反向量。
7. 验证任何动作、协议、遥测、模型或会话状态都不能自动提升结果域。
8. 说明省略因果能力、保持 `apor` 或停止建设的条件。
9. 项目进入代码开发阶段并明确实现范围后，才讨论观察器、因果检查器、运行时或性能实验。

## 进入规范前的条件

建议先讨论并接受或修改以下三项方向：

1. 将 `kine` 从“使事态达到成立”澄清为“要求事态达到成立”，把它定义为目标方向而不是因果算子。
2. 默认 Endem 描述终态；只有自然语言明确要求动作、转变或因果时，才把相应关系显式写入 `skena`，当前 Profile 不支持时保留 `apor` 或拒绝。
3. 终态满足、动作发生、状态转变、因果归因、授权责任、会话终止与最终接受始终正交。

这三项若被接受，下一步应先修订 ADR-0010、END-TEL-001、设计场景和公开解释，再判断 IKN、DRO、ADP 与 AUT 是否需要补充唯一条款；不新增术语、制品、命令、组件或规范家族。
