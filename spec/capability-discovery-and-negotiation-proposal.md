# 能力发现、协商与调用边界研究提案

状态：非规范研究提案  
日期：2026-07-14  
当前策略：正在研究；尚未进入规范
适用范围：非组件研究

## 直接结论

“对端声明会做”“双方协议支持”“本地允许做”“本会话可以做”“此刻能够调用”和“调用已经成功”是六种不同事实。Noemion 必须把它们分开：

1. Agent Card、工具清单、schema、描述、签名和注册表条目只能提供带来源的能力声明。
2. 协议初始化或版本协商只能确定双方可以使用哪些协议机制，不能授予业务动作权限。
3. 授权决定必须绑定主体、实际行动者、对象、动作、目的、受众、租户、截止点和政策；OAuth scope、登录或能力描述不能代替它。
4. contract 只封存一次会话建立时已经核对的能力交集。后来新增的工具或技能不能扩大旧会话；移除、撤销或实质变化可以让旧能力立即不可用或使会话失效。
5. 即使动作已获授权，端点故障、预算耗尽、配额、媒体类型、输入 schema、健康状态或依赖缺失仍可能使它此刻不可调用。
6. 一次调用成功只形成外部运行事实和候选产物，不证明未来仍可调用、目标已经满足、证据充分或最终接受。

本提案不构成 ADR、CORE 规范、内容 Profile 或实现要求。它不创建新制品、文件格式、扩展名、命令、组件、结果域、稳定接口或哲学专名，也不进入 `registry.json`。当前只检验 ADP、AUT、SESSION、DIA、EVIDENCE、ID 与 TEXT-IDENTIFIER 的现有责任能否形成唯一边界。

## 为什么现有“能力”一词仍然危险

当前规范已经要求能力取交集，但“能力”在外部协议和 Agent 框架里同时指向至少六个阶段。若页面只写“Agent 有能力调用工具”，读者无法知道这是自报技能、协议支持、授权、会话上限、即时可用还是已经发生的调用。

这种省略会产生三种高影响错误：

- **声明洗权：**远端描述、Agent Card、工具注解或签名被当成授权决定；
- **发现漂移：**工具列表变化后，旧 contract 被静默改写，或已移除的能力仍从缓存可调用；
- **成功洗白：**一次 HTTP 200、`isError: false` 或 Task completed 被提升为长期权限、目标满足或最终接受。

解决方法不是再造一个总括“能力对象”，而是要求每句话和未来字段说明自己属于哪个阶段、由谁产生、在什么截止点有效、失败落在哪个域。

## 六类事实必须分开

| 阶段 | 回答的问题 | 最低绑定 | 不能推出什么 |
| --- | --- | --- | --- |
| 能力声明 | 对端、工具或注册表声称可以做什么 | 声明内容、来源、声明主体、版本、端点、取得时间与完整性 | 已认证主体、授权、真实可用或结果正确 |
| 协议协商 | 双方在精确协议版本下共同支持哪些机制 | 协议、版本、传输、扩展、双方声明与协商结果 | 业务动作获准、凭据有效或端点健康 |
| 授权决定 | 谁可以为谁对什么对象执行什么动作 | AUT-CORE 完整语境、政策、决定、截止点与委托链 | 动作一定可执行、一定成功或目标满足 |
| 会话能力上限 | 本次会话最多可以使用哪些已授权动作 | 精确 contract、能力交集、环境、预算、证据责任 | 动态扩权、跨会话复用或实时秘密进入 contract |
| 即时可调用性 | 这个精确动作此刻是否具备执行前提 | 调用身份、实际端点、schema、媒体类型、健康、配额、预算、截止点与依赖 | 权限长期有效、调用已经发生或产物可信 |
| 调用事实与结果 | 什么请求实际发生，外部系统返回了什么 | ADP 调用身份、精确输入、原始状态、候选产物、错误与观察 | `met`、evidence `sufficient`、`accepted` 或未来仍可调用 |

这里使用的都是普通职责短语，不是新的领域名。公开材料应优先写“能力声明”“协议协商结果”“授权决定”“会话能力上限”“即时可调用性”和“调用事实”，而不是无限定地写“能力状态”。这些短语的普通读音和职责都直接，不需要新增发行读音、缩写或命令名。

## 现有规范已经覆盖什么

| 现有条款 | 已经固定的边界 | 本提案补充的审查问题 |
| --- | --- | --- |
| `ADP-PEE-001` | 发现来源、声明主体、端点、认证主体、受众、租户和政策分开 | 缓存的发现结果何时过期，声明变化怎样触发重验 |
| `ADP-CAP-001` | 有效能力取协议、对端、适配器、contract 与政策的交集 | 交集是会话截止点快照，不是可被动态列表扩大的并集 |
| `ADP-INV-001` | 每次调用绑定精确会话、主体、输入、预算、幂等和观察责任 | 调用前还要区分已授权与此刻可调用 |
| `ADP-MAP-001` | 原始协议事实与映射损失分开 | 工具描述和 schema 变化不能被“兼容”标签静默吸收 |
| `ADP-STA-001` | 外部状态不改写本地结果域 | 调用成功也不反向证明能力声明或授权长期有效 |
| `ADP-SEC-001` | 凭据、受众、网络目标、披露和资源最小化 | scope 与 resource 只是授权输入和限制，不是完整能力结论 |
| `AUT-CAP-001` | 有效授权只取逐项收窄的交集 | 发现、认证、授权和实时可用必须有独立失败责任 |
| `SESSION-CAP-001`、`SESSION-IMM-001` | 会话能力有限且 contract 只读；实质漂移使旧契约失效 | 新增能力需要新决定和新会话；移除不能被缓存掩盖 |
| `DIA-CORE` | 诊断保留生产者、操作、输入与失败层次 | unsupported、unauthorized、unavailable、invalid input 和 remote failure 不能合并 |
| `EVIDENCE-CORE` | 观察、完整性、有效性、覆盖与决定分开 | 能力探测和调用结果只能支持其实际观察范围 |

现有资料没有表明必须新建 `CAP-CORE` 的责任。若后续证据支持并形成现行架构结论，首选仍是把少量唯一义务归还现有规范，而不是建立能力清单格式、能力制品或能力代理的抽象规范。

## 当前协议和 GNU 技术给出的证据

### MCP：协商与动态列表本身就是两个阶段

[MCP 2025-11-25 生命周期](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle)要求初始化阶段先固定协议版本并交换能力，运行阶段只能使用成功协商的能力。[工具规范](https://modelcontextprotocol.io/specification/2025-11-25/server/tools)又把 `tools` 协议能力、`tools/list` 返回的具体工具、`listChanged` 通知、工具 schema、注解和调用结果分开；工具注解默认不可信。

这支持“协议机制、动态声明、具体调用分别绑定”。它不支持把 `tools/list` 当成本地授权目录，也不支持收到 `list_changed` 后直接扩写旧 contract。MCP 当前规范允许列表变化，恰好说明声明快照必须有取得时间、版本和重验策略。

### A2A：声明支持也可能没有配置或没有权限

[A2A 1.0 版本化规范](https://a2a-protocol.org/v1.0.0/specification/)把 Agent Card 定义为身份、能力、技能、端点与认证要求的自描述资料。它要求客户端先按声明检查可选能力，但也定义了“声明支持扩展 Agent Card、实际却未配置”的独立错误；认证后的扩展 Card 还可能因主体不同而暴露不同技能。

这说明 Agent Card 是可验证、可版本化的声明输入，不是完整运行事实。JWS 可以支持来源完整性，却不能证明技能真实可用、调用获权、输出正确或目标满足。A2A 的主次版本协商也不能代替具体技能、媒体类型和安全要求的逐项检查。

### OAuth：scope 与受众必须分开

[RFC 8707](https://www.rfc-editor.org/rfc/rfc8707.html)明确区分访问类型的 scope 与令牌将被使用的目标资源，并建议把令牌限制到特定受众。Noemion 因此不能把一个 scope 字符串解释成“Agent 对所有同名工具都具备能力”，也不能把资源 URI、登录成功或令牌存在解释成业务授权已完整成立。

OAuth 资料只支持受众和授权输入的精确绑定。Noemion 的实际授权仍须按 AUT-CORE 固定行动者、被代表主体、对象、动作、目的、租户、条件和截止点。

### GNU Autoconf：检查需要的特性，而不是只相信名称或版本

[GNU Autoconf 2.73 手册](https://www.gnu.org/software/autoconf/manual/autoconf.html)明确偏好 feature checks 而不是 version checks；`AC_PATH_PROGS_FEATURE_CHECK` 还会对找到的候选程序执行具体特性测试，而不只采用 PATH 中第一个名称。

Noemion 借用的纪律是：选择外部工具不能只看产品名、路径、版本字符串或注册表描述，应核对当前任务需要的精确协议、schema、媒体类型和行为前提。不能照搬的部分同样重要：配置期探测结果不是运行期授权，缓存探测不证明端点此刻健康，执行测试也可能有副作用、跨编译限制或环境漂移。

## 候选责任的唯一主归属

| 问题 | 唯一主归属 | 其他域只提供什么 | 当前不新增什么 |
| --- | --- | --- | --- |
| 发现来源、对端声明和动态工具列表 | `ADP-PEE-001` | ID 绑定声明内容；TEXT-IDENTIFIER 绑定显示和 schema 文本 | 能力目录制品、发现 CORE |
| 协议版本、扩展和机制协商 | `ADP-PIN-001`、`ADP-CAP-001` | DIA 报告协商失败 | 通用协商对象、SDK 类型别名 |
| 动作、对象、主体、目的和截止点授权 | `AUT-CAP-001` | ADP 提供外部 scope 与受众；SESSION 提供会话上限 | scope 即授权、Agent Card 即授权 |
| 会话能力快照、扩大和漂移 | `SESSION-CAP-001`、`SESSION-IMM-001` | AUT 提供决定；ADP 提供声明变化 | 可变 contract、动态能力并集 |
| 调用前即时可用性 | `ADP-INV-001` | SESSION 提供预算；DIA 分类阻断原因 | 全局 `available` 布尔值、永久健康证明 |
| 调用状态、返回和候选产物 | `ADP-STA-001`、`ADP-ART-001` | EVIDENCE 保存有范围观察；DIA 保存失败层次 | 新成功域、远端结果直接成为证据 |
| 调用后的满足与最终决定 | END、EVIDENCE 与具名权威 | ADP 只提供运行事实 | 工具成功即 `met` 或 `accepted` |

## 支持案例与反例

### 案例一：Agent Card 声明付款技能

远端 Agent Card 声称支持付款，但已认证主体只获准查询订单。发现资料可以帮助选择候选对端；有效动作仍只有查询交集。

反例：Card 带 JWS，所以系统自动授予付款。签名只支持陈述来源与完整性，不产生本地授权。

### 案例二：MCP 初始化支持 tools

客户端和服务器成功协商 `tools`，随后 `tools/list` 只返回天气查询。协议支持工具机制，不表示存在任意工具，更不表示模型可以调用所有返回项。

反例：只要初始化含 `tools`，就允许任意名称的 `tools/call`。这把协议机制误作具体操作授权。

### 案例三：工具列表新增写入动作

服务器发送工具列表变化通知，新列表增加仓库写入工具。当前 contract 不扩大；控制平面可以把它作为新会话候选重新做身份、schema、政策和授权检查。

反例：界面自动刷新后模型立即获得新工具。动态发现由此绕过只读会话边界。

### 案例四：工具被移除

旧 contract 允许一个查询动作，但对端撤下工具。系统将该动作标为当前不可用；不得用缓存 schema 继续调用。若它是必需能力，按固定政策停止或使会话失效。

反例：因为移除不能扩大权限，所以完全忽略。权限上限不变不等于运行前提仍存在。

### 案例五：schema 发生实质变化

同名工具把账户参数从稳定标识符改成自由文本，并新增默认副作用。名称没有变，但调用语义、输入和风险已经漂移，旧绑定不能继续使用。

反例：JSON 仍可反序列化，所以视为兼容。结构可解析不能证明语义或权限兼容。

### 案例六：scope 和 resource 不一致

令牌含写入 scope，却绑定另一个 API 受众。适配器拒绝当前调用并报告受众不匹配；它不把令牌存在解释为广域写能力。

反例：scope 文本包含 `edit`，所以任何编辑工具都可用。动作类型、对象和受众被错误合并。

### 案例七：已授权但配额耗尽

动作仍在授权和 contract 上限内，但远端配额耗尽。当前调用不可用；授权决定和历史调用事实仍保留各自有效范围。

反例：把 unavailable 改写为 denied，或把它直接写成目标 `unmet`。这会混淆授权、运行前提和目标判断。

### 案例八：未经授权但技术上可达

端点健康、schema 合法、凭据也能通过认证，但本地政策拒绝该对象上的动作。系统不得因为“技术上能调用”而尝试。

反例：先执行再让人工判断是否保留结果。副作用已经越过授权边界。

### 案例九：Autoconf 式特性探测

两个同名外部程序中，只有一个支持所需参数。候选选择可以执行有界、无副作用或明确授权的特性检查，并保存精确路径、版本、环境和结果。

反例：把探测通过当作后续全部调用授权，或在高风险工具上偷偷运行带副作用的“检测”。特性检查也必须受能力、预算和授权约束。

### 案例十：A2A 声明支持扩展 Card 但未配置

服务器声明 `extendedAgentCard`，调用却返回专门的未配置错误。系统保留“曾声明支持”和“本次不可用”两个事实，不回写历史 Card，也不伪造空技能集。

反例：把这种错误归为身份无效或最终拒绝。错误只落在当前能力提供责任。

### 案例十一：工具调用成功

一次查询返回 `isError: false` 和结构化内容。适配器记录调用事实并验证声明的输出 schema；内容仍是候选观察，必须按来源、方法、范围和 `criteria` 继续评估。

反例：一次成功证明工具长期可靠、证据充分、目标 `met` 和最终 `accepted`。四个结果域被一次外部状态覆盖。

### 案例十二：确定性路径不需要动态发现

未来受控实现若使用构建时固定且由本地政策精确绑定的操作集合，可以省略网络发现；它仍需在会话建立和调用前核对实际实现、版本、环境和授权。

反例：为了架构对称，强制生成持久能力目录和新签名对象。没有独立消费者时，这只增加身份、缓存和撤销负担。

## 变化处理矩阵

| 变化 | 当前调用 | 当前 contract | 新授权 | 新会话 |
| --- | --- | --- | --- | --- |
| 对端新增未请求能力 | 不可自动使用 | 不扩大 | 若未来需要则是 | 若未来需要则是 |
| 对端移除可选能力 | 阻止该动作 | 依政策继续或失效 | 否 | 只有契约要求时 |
| 对端移除必需能力 | 阻止 | 失效或停止 | 否 | 恢复前需要 |
| schema、媒体类型或副作用实质变化 | 阻止旧调用 | 视为环境或能力漂移 | 通常需要重验 | 是 |
| scope 缩小或撤销 | 阻止越界动作 | 相应能力不可用；不得保留缓存授权 | 需要新决定才能恢复 | 是 |
| 端点短暂故障或配额耗尽 | 当前不可用 | 依固定重试和预算政策 | 通常否 | 只有实质绑定变化时 |
| 同一绑定内健康恢复 | 可重新按调用门检查 | 不自动改变 | 否 | 通常否 |
| 调用成功 | 保存事实和候选结果 | 不扩大 | 否 | 否 |

这张表是研究材料，不是运行算法。具体重试、健康、漂移和失效条件仍须在未来协议 Profile 与 runner 实现阶段由真实消费者证明。

## 威胁到失败责任的映射

| 威胁 | 失效方式 | 唯一主归属 | 预期处理 |
| --- | --- | --- | --- |
| 声明洗权 | Agent Card、工具描述或签名成为权限 | AUT / ADP | 保持声明证据身份，重新求交 |
| 动态扩权 | 新工具或扩展进入旧会话 | SESSION | 拒绝扩写；新决定与新会话 |
| 陈旧缓存 | 已移除或撤销能力继续可用 | ADP / SESSION | 固定取得时间与变更来源，关闭失败 |
| 同名漂移 | 名称不变但 schema、媒体类型或副作用改变 | ADP | 视为实质版本变化，拒绝旧绑定 |
| scope 泛化 | scope 字符串跨受众、对象或租户复用 | AUT / ADP | 绑定 resource、audience、主体和对象 |
| 可用性洗权 | 健康端点或成功认证成为授权 | AUT | 独立求值授权；技术可达不准执行 |
| 授权洗可用 | allowed 被当作端点健康或依赖齐备 | ADP / DIA | 调用前检查并分类具体阻断原因 |
| 探测副作用 | feature check 实际执行高风险动作 | AUT / ADP | 探测本身也作为精确调用受限 |
| 成功洗白 | HTTP 200、`isError: false` 或 completed 成为 `met` | ADP / EVIDENCE / END | 保存外部事实，按证据与判据另行判断 |
| 错误压平 | unsupported、unauthorized、unavailable 与 remote fault 合并 | DIA | 保留生产层次、原始错误和唯一主诊断 |
| 能力枚举泄漏 | 动态列表暴露租户、内部端点或敏感技能 | ADP / AUT | 最小披露、主体绑定、分页和资源上限 |
| 无限重探测 | 列表变化或故障触发无界刷新与重试 | SESSION / ADP | 共用预算、截止点、退避与人工升级 |

## 失败域与结果隔离

| 事件 | 声明与协商 | 授权与会话 | 调用与诊断 | 绝不能自动改变 |
| --- | --- | --- | --- | --- |
| 未声明必需协议能力 | 协商失败 | 不形成对应会话能力 | 协议层主诊断 | 目标满足、最终决定 |
| Agent Card 声明技能但认证主体无权 | 声明保持原样 | denied 或不进入交集 | 不尝试调用 | 声明真实性、其他动作授权 |
| 动作获准但当前不可用 | 声明和决定可仍有效 | contract 依政策继续或失效 | unavailable 类阻断 | denied、`unmet`、`rejected` |
| schema 实质漂移 | 旧声明绑定过期 | 旧能力不可继续使用 | 版本或映射阻断 | Endem 身份、已有历史事实 |
| 调用返回远端错误 | 协商不自动撤销 | 授权不自动撤销 | 保存远端层次和重试分类 | `fault`、会话 failed、最终 rejected |
| 调用成功并返回候选 | 不证明长期可用 | 不扩大会话 | 形成调用事实和候选观察 | `met`、`sufficient`、`accepted` |
| 列表新增能力 | 更新候选声明 | 旧 contract 不扩大 | 无调用事实 | 现有授权、结果和身份 |
| 列表移除必需能力 | 更新当前声明 | 依固定政策停止或失效 | 保存变更来源 | 历史调用、旧证据身份 |

## 语言哲学采用边界

路德维希·维特根斯坦《逻辑哲学论》3.32 区分可感知记号与符号使用，3.321 又指出同一个记号可以被不同符号共有。这里的工程启发是：`capability` 或“能力”出现在 Agent Card、协议初始化、授权政策、contract 和调用界面中，并不表示这些位置表达同一种事实；职责必须由生产者、关系、截止点和失败域显示。

这些命题不能推出协议协商算法、权限交集、健康检查、缓存期限或错误分类。Noemion 只借它们识别同词跨域造成的概念混淆；具体规则仍须由外部协议、现有规范、威胁、反例和未来向量证明。

## 可读性与术语边界

本主题不需要创造新的希腊化名称、缩写或结果枚举。`capability` 在外部协议里有既定但不完全相同的含义，公开中文若只写“能力”仍会隐藏责任。写作时应遵守三条规则：

1. 首次出现时写完整职责短语，例如“对端能力声明”或“本会话能力上限”。
2. 表格和诊断必须带域，不使用无主体的 `available/unavailable` 或“已支持”。
3. 口头讨论优先使用“声明、协商、授权、会话上限、当前可调用、调用结果”这组普通词；它们音节和职责差异清楚，不需要发行读音研究。

若未来确需机器枚举，名称仍须先绑定唯一生产者、消费者、失败责任和结果隔离，再检查普通词的词首与关键字冲突；只有新增自造名称才进入完整读音验证。读音流畅不能为一个合并错误对象的短词辩护。

## 当前结论

当前研究可以形成的结论是：

1. 对端声明、协议协商、授权、contract 上限、即时可调用和调用结果必须分开。
2. 新能力不能扩写旧 contract；能力移除、撤销或实质漂移必须阻止陈旧调用。
3. scope、Agent Card、工具 schema、签名、健康检查和一次成功调用都不能单独成为完整能力结论。
4. 现有 ADP、AUT、SESSION、DIA、EVIDENCE、ID、TEXT-IDENTIFIER 与 END 足以承载候选义务；当前不创建 `CAP-CORE`、能力制品或新专名。
5. 这些结论当前不会改变任何现行规范条款、登记、向量、contract 字段或结果值。

若责任分配被现行架构接受，下一步应先用现有规范修订逐项吸收唯一义务，再设计协议专属场景与正反向量。只有真实消费者证明这些责任无法由现有规范唯一承担时，才重新讨论独立标准。

## 权威资料与采用限制

- [MCP 2025-11-25 生命周期](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle)：用于核对版本初始化、能力交换、协商与运行阶段分离；不作为本地业务授权模型。
- [MCP 2025-11-25 工具规范](https://modelcontextprotocol.io/specification/2025-11-25/server/tools)：用于核对工具列表、变更通知、schema、注解和调用结果；工具说明与注解不能取得本地权威。
- [A2A 1.0 版本化规范](https://a2a-protocol.org/v1.0.0/specification/)：用于核对 Agent Card、技能、协议版本、扩展 Card 和能力专属错误；Card 与签名不能替代授权、可用性或结果判断。
- [RFC 8707 Resource Indicators for OAuth 2.0](https://www.rfc-editor.org/rfc/rfc8707.html)：用于分开 scope 与目标资源、限制令牌受众；不定义 Noemion 的语义权威或完整动作范围。
- [GNU Autoconf 2.73 手册](https://www.gnu.org/software/autoconf/manual/autoconf.html)：用于借鉴特性检查优先于版本猜测，以及对候选程序执行具体 feature test；配置期探测和缓存不能成为运行期授权或持续可用证明。
