# 预览、模拟与批准边界研究提案

状态：非规范研究提案
日期：2026-07-13
决定状态：等待决定
研究范围：Agent 工具调用前的预览、dry-run、模拟、授权、执行与事后观察

## 治理边界

本提案不构成 ADR、CORE 规范、内容 Profile 或实现要求，不进入 `registry.json`。
它不创建新制品、文件格式、扩展名、命令、组件、结果域、稳定接口或哲学专名。
当前项目尚未进入代码开发阶段；本文只检查现有责任能否承载未来 Agent 的预览与批准流程。

如果用户接受本提案，后续也应优先把唯一义务归还 TEXT-IDENTIFIER、AUT、DRO、ADP、IKN、END 与 DIA。
只有真实消费者和验证证明这些责任无法分担时，才重新讨论独立规范。

## 直接结论

预览只能说明系统向决定者展示了什么。
模拟只能说明某个模型在声明条件下预测了什么。
授权只能说明具名权威允许哪个精确请求在什么范围内尝试。
执行记录只能说明某个动作被请求、开始、拒绝、中断或返回。
观察只能说明声明范围内实际看到了什么。
满足与最终接受仍必须由 `krin`、适用 Iknem 和具名决定权威产生。

因此，Noemion 不应建立一个把“预览一致、dry-run 通过、用户点击批准、工具成功返回”压成单一成功的流程状态。
这五类事实必须分开，并以精确身份、时间、环境、能力和损失关系连接。

## 发现的真实缺口

AUT-CORE 已要求同意绑定安全显示视图与精确机器请求。
TEXT-IDENTIFIER-CORE 已要求显示与机器值分开。
DRO-CORE 和 ADP-CORE 已要求执行前重新核对环境、能力、协议、预算和幂等条件。
IKN-CORE 与 END-CORE 已把观察、满足和决定分开。

现有条款仍没有在一个可读模型中回答以下横切问题：

1. 预览展示的是精确请求、摘要、计划，还是模拟结果；省略了什么？
2. dry-run 是否真的无副作用，使用了哪个环境快照和替身？
3. 批准后请求、工具 schema、目标对象或政策变化时，旧批准是否仍适用？
4. 执行偏离预览时由哪个责任域阻断，怎样显示差异？
5. 工具返回成功后，谁观察真实终态并决定目标是否满足？

如果这些问题不显式分开，Agent 界面很容易制造一种错误印象：
“我看过计划并点击批准，所以系统一定按这个计划完成了目标。”

## 六类事实必须分开

| 类别 | 它回答什么 | 最低绑定 | 不能推出 |
| --- | --- | --- | --- |
| 预览 | 决定者实际看到了什么 | 显示视图身份、机器请求身份、变换、隐藏与截断、主体、时间 | 模拟忠实、请求获准、动作发生 |
| 模拟或 dry-run | 在什么假设下预计会发生什么 | 模拟器、版本、输入、环境快照、替身、非模拟边界、输出 | 无副作用、真实执行相同、目标满足 |
| 授权决定 | 谁允许哪个请求怎样尝试 | AUT 完整语境、主体、对象、动作、范围、次数、截止点、决定 | 动作已经发生、工具成功、终态成立 |
| 执行尝试 | 实际发起了什么 | Dromen、调用身份、精确参数、能力、环境、开始与终止事件 | 副作用已提交、结果由该动作造成 |
| 事后观察 | 实际环境显示什么 | Iknem 主体、方法、范围、时间、限制与 `phain` 对齐 | 观察者有最终决定权、动作具有因果责任 |
| 满足与决定 | 目标和权威怎样裁定 | `krin`、适用证据、结果域、政策和最终权威 | 反向证明预览、模拟或工具声明真实 |

## 来自 GNU Make 的可用边界

GNU Make 提供两个重要反例。

第一，`--dry-run` 是执行模式的一个选项，不是对所有 recipe 的纯度证明。
递归 make 和显式 `+` 行会越过通常的 no-op 行为。
因此未来任何 Noemion 适配器若声称“模拟”，都必须列出仍可能发生的真实读取、网络调用、缓存写入、锁、时间推进和副作用例外。

第二，`--question` 的退出码只回答目标是否需要更新，并把工具错误保留为另一状态。
它不证明更新配方会成功，更不证明业务目标已经满足。
这支持 Noemion 继续把预检、执行终止、满足判断和最终决定分开。

本提案不把 GNU Make 的 recipe、退出码或文件时间语义写入 Endem。
它只借鉴“预检模式必须说明仍执行什么，以及它回答的精确问题”的纪律。

## Agent 时代为什么会放大问题

模型生成的预览通常是自然语言摘要，而实际工具调用是结构化参数。
两者可能因为截断、默认值、重规划、schema 漂移、并发、工具内部展开或环境变化而不同。

长时间审批还会扩大时序差异：

- 目标对象在批准前后发生变化；
- 价格、权限、租户、分支、版本或工具列表漂移；
- 模型或控制平面在恢复后生成不同参数；
- 一个批量批准覆盖后来出现的新调用；
- 预览只显示高层计划，实际动作包含额外网络目标或接收方；
- dry-run 使用替身、缓存或只读接口，真实路径使用另一身份和副作用语义。

OpenAI Agents SDK 在执行前重新运行时效性检查，是现实框架对这个时序问题的直接承认。
Noemion 不能复制 SDK 状态机，但应保留同一不变量：使用决定时重新核对决定仍适用于将要执行的精确请求。

## 支持案例与反例

### 案例一：精确参数预览后立即执行

系统显示工具、目标对象、全部参数、接收方、预计副作用和不可逆边界。
显示视图绑定机器请求身份；具名权威只授权该调用一次。
执行前对象、政策、能力与参数未变化，调用随后产生独立观察。

支持结论：预览可成为有效同意的一项依据，授权可允许一次执行尝试。
它们仍不替代事后观察、满足判断和最终决定。

### 案例二：摘要隐藏默认接收方

预览只写“发送报告”，机器请求却包含外部邮箱和完整附件。

反例：显示视图与机器请求不一致，AUT 必须拒绝授权。
模型声称“已概括关键内容”不能关闭披露缺口。

### 案例三：GNU Make dry-run 仍执行递归行

调用者把 `make -n` 记录为“零副作用验证”，但 Makefile 中含 `$(MAKE)` 或 `+` recipe。

反例：模拟声明隐藏已知例外。
该记录不能支持无副作用结论，发生的实际动作必须独立记录。

### 案例四：批准后对象漂移

用户批准删除对象 `artifact@a`，执行前同一路径已指向 `artifact@b`。

反例：路径不是精确对象身份。
旧决定不能用于新对象；系统必须拒绝或形成新请求与新决定。

### 案例五：价格或权限在长等待中变化

审批界面显示的价格、scope 或目标域在恢复执行前已经变化。

反例：批准时的预检不能替代使用时核对。
扩大范围必须重新授权；收窄也要按政策决定是否需要新请求。

### 案例六：批量预览与并发工具调用

界面一次展示三个精确调用，权威只批准其中两个。
第三个保持 pending，后来出现的第四个调用不能继承该批准。

支持结论：批量界面可以存在，但决定必须逐项或按冻结的同质集合绑定。
并发和到达顺序不能扩大范围。

### 案例七：模拟成功、真实执行失败

只读模拟使用静态快照通过，真实执行因权限撤销而被拒绝。

正确分层：模拟记录保持“在快照下预计可行”；执行会话可以 failed；满足结果依据观察保持 `agno` 或其他适用结果。
不得回写模拟失败，也不得把模拟通过写成 `met`。

### 案例八：工具成功、终态未成立

API 返回成功，但异步处理稍后失败，目标资源没有形成。

正确分层：调用可以 completed，外部 Task 有自己的终态，实际 `phain` 反驳目标时 `krin` 才得到 `unmet`。
工具成功不能越级成为满足。

### 案例九：用户接受 URL 交互

MCP URL elicitation 返回 `accept`，用户随后在外部页面取消付款或 OAuth 流程。

正确分层：accept 只证明同意进入该交互。
外部结果仍需协议事件和有范围观察，不能由 accept 推断完成。

### 案例十：模型解释与机器请求冲突

模型声称操作只读，工具 schema 和本地政策却把它分类为破坏性写入。

反例：模型解释是候选显示内容，不是分类权威。
冲突必须阻断并显示，不得以自然语言更“易懂”为由覆盖机器边界。

### 案例十一：预览之后发生重规划

已批准步骤不可执行，模型提出另一个工具或不同参数。

正确分层：重规划产生新候选调用。
除非旧决定明确覆盖冻结的等价请求集合，否则新调用必须重新核对授权。

### 案例十二：拒绝不等于目标失败

权威拒绝一次高风险工具调用，但仍有另一条已授权路径可以实现目标。

正确分层：该调用为 deny，计划可以重规划，Drase 会话按规则继续或终止。
目标结果不能仅因一次拒绝被写成 `unmet` 或 `rejected`。

## 威胁到失败责任的映射

| 威胁 | 典型表现 | 唯一主责任 |
| --- | --- | --- |
| 显示替换 | 用户看到摘要，执行隐藏参数 | TEXT-IDENTIFIER 固定视图、机器值、变换和损失；AUT 绑定决定 |
| 模拟洗白 | dry-run 名称掩盖真实副作用 | ADP 固定模拟方法、例外和调用事实 |
| 批准重放 | 旧点击用于新对象、新租户或新请求 | AUT 固定对象、次数、会话、截止点和去重 |
| 检查后变化 | 批准后政策、对象、价格或 schema 漂移 | DRO 在使用时重新验证并使旧会话失效 |
| 批量放大 | 部分批准扩展到未显示调用 | AUT 固定集合与逐项决定；控制平面不得并集 |
| 工具自述 | 注解称只读或幂等便自动放行 | ADP 把注解保持为不可信声明 |
| 成功洗白 | HTTP、MCP、A2A 或 SDK 成功变成 `met` | END/IKN 保留观察与满足责任 |
| 人工洗白 | 有按钮就声称 human-in-the-loop 风险已关闭 | AUT 核对权威与理解边界；DIA 报告缺失 |
| 预览过期 | 长等待后仍按旧快照执行 | DRO/AUT 核对时效、撤销与精确请求 |
| 结果反写 | 真实执行失败后篡改原模拟记录 | ID/IKN 保留各记录身份与关系 |

## 失败域与结果隔离

| 事件 | 预览或模拟 | 授权 | 会话或调用 | 满足 | 最终决定 |
| --- | --- | --- | --- | --- | --- |
| 显示与请求不一致 | invalid candidate view | `deny/defer` | 不得开始 | 不改变 | 不自动 rejected |
| dry-run 发现副作用例外 | 限制或失败 | 重新请求范围 | 例外动作独立记录 | 不改变 | 不改变 |
| 批准后对象漂移 | 旧预览过期 | 旧 grant 不适用 | 中断或新会话 | 依据既有证据 | 可以 deferred |
| 实际调用被拒绝 | 原预览仍是历史记录 | grant 不等于执行成功 | failed 或受控返回 | 通常仍需观察 | 不自动 rejected |
| 调用成功但无终态观察 | 模拟不变 | grant 已消费 | completed | `agno` | 可以 deferred |
| 有效观察反驳目标 | 不回写预览 | 不回写授权 | completed/failed 按事实 | `unmet` | 由权威决定 |

任何诊断都不能把这些状态压成一个“审批失败”或“执行成功”。
DIA 必须选择当前主阻断问题，同时保留上游预览、决定和调用的独立身份。

## 候选责任的唯一主归属

| 责任 | 主归属 | 说明 |
| --- | --- | --- |
| 显示了什么、怎样变换和省略 | TEXT-IDENTIFIER-CORE | 预览是显示视图，不是新制品 |
| 谁能批准什么、决定是否仍适用 | AUT-CORE | 绑定精确请求、范围、次数、时效和撤销 |
| 执行前环境、能力与政策是否漂移 | DRO-CORE | 使用时重新核对；实质变化关闭旧会话 |
| dry-run、工具声明、调用与外部状态 | ADP-CORE | 保留来源、模拟方法、例外、调用和副作用不确定性 |
| 预览、决定、执行和观察的记录关系 | IKN-CORE / ID-CORE | 每项记录有自己的身份、范围与有限溯源 |
| 目标是否满足 | END-CORE | 只由 `krin` 与适用观察判断 |
| 主阻断问题和人类解释 | DIA-CORE | 不改写任何结果域或授予恢复权限 |

控制平面负责把这些已有责任编排成可审计流程，但不因此成为新组件或权威域。
模型只能提出预览文案、模拟计划和调用候选，不能给自己的解释授予更高可信度。

## 语言哲学边界

维特根斯坦《逻辑哲学论》2.12 把图像称作现实的模型，2.225 则否定图像可以先天地为真。
这支持一个有限工程类比：预览和模拟是对可能执行或可能结果的表示，不是现实已经如此的证明。

该类比不能决定授权法、用户界面、仿真算法、TOCTOU 防护或结果域。
正式边界仍必须由精确请求、时间、环境、威胁、反例和验证证据定义。

## 若接受后的最小规范动作

1. 在 TEXT-IDENTIFIER-CORE 的安全显示责任中明确预览必须绑定机器请求、变换、截断和隐藏内容。
2. 在 AUT-CORE 中明确 grant 只允许一次或声明次数的精确执行尝试，并在使用时重新核对适用性。
3. 在 DRO-CORE 中明确批准等待期间的对象、政策、工具 schema、环境和能力漂移处理。
4. 在 ADP-CORE 中明确模拟或 dry-run 必须声明方法、替身、快照、仍执行的动作和非模拟边界。
5. 在 IKN-CORE 中明确预览、模拟、决定、调用和观察是不同记录主张，不能相互升级。
6. 形成显示替换、dry-run 副作用、批准重放、时序漂移、批量放大和成功洗白的正反向量。

这些动作都要求先由用户接受责任分配，再通过 ADR 或现有规范修订进入权威材料。
不得直接把本提案文字复制为条款。

## 采用门槛与停止条件

在以下条件满足前，本提案保持非规范：

- 用户接受六类事实分离和现有 CORE 的责任分配；
- 至少一个真实高风险工具消费者需要预览或批准；
- 预览与机器请求的身份关系可以无歧义表达；
- 模拟方法能声明全部真实副作用例外和未模拟边界；
- 批准后的漂移、撤销、重放、批量与并发反例有正反向量；
- 执行结果能由独立观察路径核对，而不是只信任工具自述；
- 人工批准界面经过可用性、安全显示和自动化偏见风险审查。

若真实消费者只需要 AUT-CORE 的精确授权而不需要预览或模拟，本提案可以完全省略。
若预览不能绑定精确机器请求，系统必须拒绝以该预览取得授权，不能用免责声明降级继续。
