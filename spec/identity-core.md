# Exact Content Identity and Attestation Core Specification

- 规范 ID：`ID-CORE`
- 版本：`0.1.0-draft`
- 日期：2026-07-13
- 状态：草案；条款化表达 ADR-0027 已接受的跨制品精确身份与签名边界
- 物理编码：未定义；本规范不创建文件、对象类型、扩展名、媒体类型、签名格式、证书 Profile 或稳定 ABI
- 实现状态：仅有规范提案向量检查器；摘要器、签名器、验证器、透明日志客户端和发布系统均未实现

## 1. 范围

本规范定义 Endem、Synem、Iknem、发布签名陈述以及调试或裁剪伴随制品之间共享的精确内容身份、引用、签名验证、撤销判断和可复现性边界。它回答的是“哪些精确字节被指向、哪些声明被签署、验证者在什么语境下可以得出什么有限结论”。

`ID-CORE` 不是新的制品、文件格式、命令或组件，也不定义语义等价。文件名、路径、URL、版本标签、build ID、短摘要、密钥提示、签名存在和透明日志条目都不能替代精确内容身份。身份相同只说明同一身份域中的精确字节相同，不说明主张为真、目标满足、制品被接受或两个不同字节语义等价。

## 2. 要求用语与符合性

大写的 `MUST`（必须）、`MUST NOT`（不得）、`SHOULD`（应当）、`SHOULD NOT`（不应）与 `MAY`（可以）按 BCP 14 解释：

- RFC 2119：https://www.rfc-editor.org/rfc/rfc2119.html
- RFC 8174：https://www.rfc-editor.org/rfc/rfc8174.html

符合性声明 `MUST` 固定 `ID-CORE`、对象规范、物理表示、内容 Profile、身份域、摘要算法政策和签名陈述 Profile 的精确版本。签名主体、验证身份或密钥适用性被用于授权时，还 `MUST` 固定并符合 `AUT-CORE` 的精确版本；签名有效不能替代授权决定。当前向量以 SHA-256 作为已知算法示例，只检查字段与边界，不为未来发行选择摘要或签名算法。

## 3. 精确身份与证明内容

### ID-DOM-001 — 身份必须绑定显式域与对象语境

**要求：**精确身份 `MUST` 由身份域、对象种类、对象规范版本、物理表示版本、适用内容 Profile 以及摘要元组共同解释。比较只有在这些字段均精确匹配时才可以得出身份相同；同一摘要值出现在不同域、对象种类、规范或 Profile 中 `MUST NOT` 合并。身份域 `MUST` 稳定、版本化并说明自己的摘要输入边界。

**失败：**仅比较摘要文本、把 Endem 摘要当作 Synem 身份、忽略 Profile 或在域定义变化后沿用旧身份时，比较无效。

**验证：**`ID-SCN-001`、`ID-SCN-002`；`vectors/identity/cases.json`；未来 `conformance:identity-domain-binding` 组件测试。

### ID-BYT-001 — 摘要输入必须是精确且非循环的字节范围

**要求：**身份域 `MUST` 指定摘要覆盖的精确字节、顺序、长度、表示版本和包含或排除项。输入可以是完整制品字节或另一规范定义的规范字节，但 `MUST NOT` 由文件读取器、可变路径、隐式规范化或验证时猜测决定。身份值本身、签名包络和其他后附可变材料 `MUST NOT` 形成自包含循环；任何排除项 `MUST` 有唯一规范依据。

**失败：**发布者与验证者对换行、压缩、尾随数据或包络覆盖范围理解不同，摘要把自身字段作为输入，或只写“对内容求哈希”而没有定义字节边界时，身份无效。

**验证：**`ID-SCN-003`；`vectors/identity/cases.json`；未来 `conformance:identity-byte-domain` 组件测试。

### ID-REF-001 — 安全引用必须携带完整不可变身份

**要求：**用于绑定、闭包、签名、证据、政策或执行的引用 `MUST` 携带完整身份域、对象种类、对象规范、物理表示、Profile、算法标识、输出长度和完整摘要值。名称、版本范围、文件名、路径、URL、仓库坐标、`latest`、build ID、密钥 ID 或外部 task ID `MAY` 帮助定位候选，但 `MUST NOT` 选择或替代目标。解析结果必须恰好匹配一个允许身份。

**失败：**Synem 通过搜索路径绑定成员、Dromen 通过发布标签选择主体、Iknem 只保存文件名，或签名陈述只包含 URL 时，引用不完整。

**验证：**`ID-SCN-004`、`ID-SCN-005`；`vectors/identity/cases.json`；未来 `conformance:identity-reference-closure` 组件测试。

### ID-ALG-001 — 摘要与签名算法必须由版本化政策登记

**要求：**每个摘要或签名算法使用 `MUST` 绑定登记身份、用途、参数、输出长度或密钥要求、允许状态、最低安全强度和政策版本。未知、禁用、撤销、参数不完整或用途不匹配的算法 `MUST` 关闭失败；消费者 `MUST NOT` 静默替换算法、尝试算法列表直到成功，或把算法名称当作安全证明。算法迁移 `MUST` 建立新的身份或签名评估，并保留旧材料的明确状态。

**失败：**省略算法、接受任意长度、把 SHA-1 与 SHA-256 当作同一身份空间、把摘要算法用于未登记签名用途，或验证器自动降级时，处理无效。

**验证：**`ID-SCN-006`；`vectors/identity/cases.json`；未来 `conformance:identity-algorithm-policy` 组件测试。

### ID-DSP-001 — 截断值只能显示不能绑定

**要求：**安全比较与引用 `MUST` 使用登记输出长度的完整摘要值。截断、前缀、省略号、二维码缩写或其他短形式只能是显示别名，并 `MUST` 明确指向完整身份；发生前缀歧义时必须扩展显示或要求完整值，不能由顺序、缓存或“第一个匹配”选择对象。若未来登记截断算法套件，它必须拥有独立算法身份且不得自动满足本规范的安全绑定要求。

**失败：**界面复制短摘要用于 Synem 绑定、日志只保存十二位前缀、碰撞时选择缓存对象，或把 build ID 当作当前字节校验和时，绑定无效。

**验证：**`ID-SCN-007`；`vectors/identity/cases.json`；未来 `conformance:identity-full-digest-binding` 组件测试。

### ID-EQV-001 — 精确身份与语义等价必须分开

**要求：**精确身份相同只允许推出同一身份域和版本语境中的精确字节相同。不同精确身份既不能自动推出语义不同，也不能自动推出语义等价。Semantic Key、跨版本等价、裁剪前后等价、不同编码等价或规范迁移等价，在各自规范化规则、观察等价、正反向量和失败责任冻结前 `MUST` 保持未定义。

**失败：**摘要相同被解释为命题为真、不同编码被默认合并、裁剪后对象继承原身份，或发布者自行声明 Semantic Key 时，推导无效。

**验证：**`ID-SCN-008`、`ID-SCN-009`；`vectors/identity/cases.json`；未来 `conformance:identity-equivalence-separation` 组件测试。

### ID-STM-001 — 签名必须覆盖有类型的完整陈述

**要求：**签名陈述 `MUST` 使用版本化 Profile 和不可混淆的上下文，覆盖精确对象身份、对象种类、规范与 Profile、声明种类、签名用途、预期受众或依赖政策、声明主体和必要外部证据引用。载荷类型、算法和所有影响解释的关键字段 `MUST` 受到签名或等价认证保护。原始签名值不得在不同陈述类型、包络结构或用途之间重放。

**失败：**只对裸摘要签名、载荷类型未覆盖、发布签名被当作授权签名、算法字段可被替换，或同一签名从单签结构搬到多签结构仍被接受时，陈述无效。

**验证：**`ID-SCN-010`、`ID-SCN-011`；`vectors/identity/cases.json`；未来 `conformance:signed-statement-context` 组件测试。

### ID-ENV-001 — 验证包络必须外置并完整携带验证材料

**要求：**签名、证书或公钥材料、时间证明、透明日志包含材料、撤销或状态材料和签名 Profile `MUST` 位于不改变被指对象身份的验证包络或外部引用中。包络 `MUST` 绑定精确签名陈述并说明每项材料的来源、版本、适用范围和缺失影响。公钥标识或 `kid` 只能帮助查找候选密钥，不能成为唯一身份或权威。包络自己的字节可以拥有独立精确身份，但 `MUST NOT` 继承被指对象身份。

**失败：**追加证书改变 Endem 身份、包络只保存 `kid`、离线验证缺少必要时间或日志材料，或把多个签名数量当作覆盖增加时，包络不足。

**验证：**`ID-SCN-012`；`vectors/identity/cases.json`；未来 `conformance:attestation-envelope-materials` 组件测试。

### ID-AUT-001 — 密码验证不能替代身份授权与主张判断

**要求：**验证者 `MUST` 分开计算签名字节验证、签名主体认证、主体角色与范围授权、陈述适用性、政策满足和最终决定。有效签名最多支持“声明密钥对这份受保护陈述产生了可验证关系”；它 `MUST NOT` 自动证明声明真实、内容安全、语义正确、目标满足、能力获授或制品被接受。信任根、主体匹配和授权政策 `MUST` 由验证者显式配置。

**失败：**供应商签名被当作事实正确、透明日志包含被当作批准、证书有效被当作当前授权，或模型签名后候选直接成为 Endem 时，权威推导无效。

**验证：**`ID-SCN-013`；`vectors/identity/cases.json`；未来 `conformance:signature-authority-separation` 组件测试。

### ID-VAL-001 — 有效性必须绑定截止点与状态证据

**要求：**签名有效性评估 `MUST` 绑定具名截止点、信任根与政策版本，并分别核对签名时刻依据、证书或密钥适用期、撤销状态、透明日志或时间证明的有限作用和状态材料新鲜度。密钥轮换、证书过期或撤销不会改变被签字节、内容身份或历史签名值，但会改变特定截止点的有效性结论。离线评估 `MUST` 携带足够且有界的验证材料，不能把缓存成功无限延长。

**失败：**未写截止点、只看当前时间、忽略撤销、把日志集成时间当作真实签署时间，或旧缓存继续授权新会话时，评估无效或不足。

**验证：**`ID-SCN-014`、`ID-SCN-015`；`vectors/identity/cases.json`；未来 `conformance:signature-validity-cutoff` 组件测试。

### ID-REP-001 — 可复现性必须由独立产出与逐字节比较证明

**要求：**可复现声明 `MUST` 绑定精确输入、依赖闭包、工具与版本、构建或封装参数、环境约束、至少两个被声明为独立的产出路径、每个输出身份和逐字节比较结果。相同输出支持有限的重复性或可复现性结论；不同输出只证明字节差异，不能单独定位原因。build ID、相同日志、相同源版本、单次成功或发布者自述 `MUST NOT` 替代输出比较。

**失败：**只重跑一次、两个结果来自同一缓存、输入不完整、只比较 build ID，或输出不同仍声称 bit-for-bit reproducible 时，声明不成立。

**验证：**`ID-SCN-016`；`vectors/identity/cases.json`；未来 `conformance:reproducible-independent-outputs` 组件测试。

### ID-REL-001 — 派生与伴随制品必须各有身份和显式关系

**要求：**裁剪制品、调试伴随文件、索引、传输包络、压缩副本、格式迁移结果和其他派生物 `MUST` 各自拥有精确身份，并以版本化关系种类指向精确来源身份。关系 `MUST` 声明变换规则、生产者、保留内容、删除或降精度清单和可验证条件；它 `MUST NOT` 让派生物继承来源身份、签名、Semantic Key、Iknem、接受状态或权限。

**失败：**派生处理后的文件沿用原摘要、调试文件只按文件名配对、压缩包继承内部制品签名，或迁移结果自动继承原 Iknem 时，关系无效。

**验证：**`ID-SCN-017`、`ID-SCN-018`；`vectors/identity/cases.json`；未来 `conformance:artifact-relation-binding` 组件测试。

## 4. 权威依据与采用边界

- RFC 6920 把基于哈希的名称比较建立在摘要算法、长度与完整值上，并明确哈希只能支持名称与数据的完整性绑定，不能证明权威或保密性。本规范增加对象域、规范与 Profile 语境，避免跨制品混淆。
- RFC 9052 将受保护和不受保护参数分开，要求在可行时认证算法字段，并明确 `kid` 只是密钥查找提示。Noemion 借鉴这些不变量，不在本阶段选择 COSE 作为签名物理格式。
- DSSE 通过把载荷类型和载荷共同纳入待签名字节防止类型混淆。Noemion 采用有类型签名陈述和用途分离，不直接采用 DSSE 包络或序列化。
- Sigstore Bundle 把签名、证书、公钥提示、透明日志条目和时间戳作为验证材料组合，并要求验证时提供或计算制品摘要。Noemion 借鉴离线验证材料边界，不把 Sigstore 身份或日志包含当作本地授权与验收。
- SLSA 1.2 先核对陈述主体摘要，再根据验证者配置的信任根、构建者和参数判断来源。Noemion 同样分开对象匹配、签名验证和政策判断。
- GNU `ld --build-id` 明确说明 build ID 标识原始链接输出，但不应作为当前文件内容校验和。GNU Guix 的 `--check` 与 `guix challenge` 则要求对独立构建结果做逐字节比较。本规范据此分开构建关联标识与精确内容身份，并把可复现性写成有证据的比较结论。

## 5. 当前未定义

本规范不冻结摘要或签名算法基线、算法登记表、摘要文本语法、签名陈述物理编码、COSE/DSSE/Sigstore Profile、证书体系、透明日志、时间戳协议、撤销分发、Semantic Key、跨版本等价、物理 Synem 或 Iknem 格式、调试伴随格式、长期归档和密钥管理。它们必须在真实生产者与消费者、独立 ADR、威胁分析、正反向量、规范字节，以及进入代码开发阶段并明确实现范围后取得的实现证据共同支持时另行建立。
