# Noemion 托管人工智能服务与用户控制边界研究提案

状态：非规范研究提案

日期：2026-07-14

结论状态：桌面审查完成；尚未进入规范

研究范围：第三方托管模型、自主管理执行、通信服务、模型与配置漂移、数据外发、保留与二次使用、凭据、状态、导出、切换、停服、观察、复现和发布决定

本提案不构成 ADR、CORE 规范、Profile、登记项、许可选择、部署决定或实现要求，不进入 `registry.json`。它不创建服务制品、会话导出格式、模型网关、代理、云平台、运行时、命令、组件、结果域或哲学专名，也不建立 `SERVICE-CORE`、`CLOUD-CORE`、`PORTABILITY-CORE` 或新的 Noemion 部署对象。

## 直接结论

“软件可以取得”和“实际计算由用户控制”是两个不同问题。自由软件可以运行在他人控制的服务器上；非自由程序也可以运行在用户控制的设备上。前者不让用户控制那一次服务执行，后者不让用户研究和修改程序。两条边界都必须说明，不能用“开源”“本地”“私有云”或“企业版”互相替代。

托管人工智能服务可以是有用的外部能力，但 Noemion 不能把它描述成已经拥有、复现或控制模型。未来若接入，只能在精确政策允许的范围内记录调用方能够观察的请求、服务语境、返回、限制和外部效果。服务端隐藏提示、模型替换、路由、缓存、安全处理、数据保留和内部工具不能被猜成已知事实。

Noemion 当前没有模型网关、托管模型、远端 Agent 服务或自主管理推理组件。本提案只修复未来表述和验证边界，不选择“全部本地”或“全部托管”，也不授权建设任何服务。

## 先分开三种网络关系

| 关系 | 谁的活动 | 典型例子 | 本提案怎样处理 |
| --- | --- | --- | --- |
| 他人服务替代自己的计算 | 用户本可在自己控制的环境执行，却把输入交给他人运行的服务处理 | 文本生成、翻译、总结、代码分析、图像编辑、私有资料检索 | 重点审查执行控制、数据外发、隐藏变换、可替代路径和退出条件 |
| 与他人共同完成的通信或交易 | 活动本身必须包含另一方或公共信息源 | 发布内容、访问他人网站、向商店下单、与远端主体通信 | 不能因为使用网络就自动归入“软件替代”；仍单独审查隐私、授权和协议 |
| 自主管理但不在本机的执行 | 用户或组织控制软件负载、配置、密钥、数据路径和停启，即使机器是租用的 | 自管服务器、受控集群、隔离租户 | 不称为“本地”；按实际控制和可验证范围逐项说明 |

“本地”只表示某个组件位于当前设备或局域环境，不能证明整条路径没有远端调用。“云”也不能说明控制者、数据位置或服务边界。公开页面优先使用“第三方托管执行”“自主管理执行”“设备内执行”和“通信服务”等直白短语。

## 至少十六种事实必须分开

| 事实 | 回答的问题 | 最低绑定 | 不能自动推出 |
| --- | --- | --- | --- |
| 任务性质 | 这是用户自己的计算，还是必须包含他人的通信/交易 | 任务、输入主体、外部参与者与不可替代关系 | 只因联网就属于同一伦理或工程分类 |
| 实际执行者 | 谁运行决定输出的程序和模型 | 法人或组织、服务、租户、区域、截止点 | 名义供应商等于唯一处理者 |
| 控制平面 | 谁能选择软件负载、模型、配置、停启和更新 | 主体、管理权限、变更路径与限制 | 机器归属等于执行受控 |
| 计算位置 | 输入、缓存、推理、工具和日志在哪里处理 | 区域、设施类别、处理阶段、例外与转移 | 数据驻留等于全程区域内处理 |
| 模型与配置 | 实际使用哪个模型、tokenizer、提示、工具和参数 | 可观察身份、快照或别名、设置、时间与未知范围 | 模型名未变等于实现未漂移 |
| 服务端隐藏变换 | 供应商可能添加、删除、路由或缓存什么 | 供应商声明、不可观察范围、政策与影响 | 调用方请求等于模型最终看到的内容 |
| 请求数据 | 哪些原始、派生、元数据和秘密离开控制域 | 精确字段、来源、最小化、目的、受众与分类 | TLS 等于接收方用途获准 |
| 保留与二次使用 | 数据、状态和日志保存多久、用于什么 | 端点、功能、账户设置、默认值、训练选择、删除和法律例外 | “不用于训练”等于零保留或零人工访问 |
| 凭据与能力 | 哪个主体把什么权限交给哪个服务 | 实际行动者、资源、受众、scope、期限、委托与撤销 | API key 可用等于动作获权 |
| 外部工具与下游服务 | 托管模型还把什么发送给谁 | 工具、MCP/A2A 对端、子处理者、参数、凭据和各自条款 | 主服务政策自动覆盖第三方服务 |
| 持久状态 | 历史、文件、向量、缓存和远端 Task 存在哪里 | 对象身份、所有者、保留、删除、导出和恢复语义 | 会话 ID 等于完整历史或可移植状态 |
| 导出与可移植性 | 可以带走哪些内容和关系 | 格式、版本、范围、遗漏、来源、身份与验证 | 能导出文本等于能迁移同一系统行为 |
| 替代与切换 | 换模型、供应商或自管路径时改变什么 | 新身份、映射、损失、权限、评测与重新决定 | API 兼容等于语义等价 |
| 连续性与退出 | 配额、涨价、区域故障、停服或撤回时怎样停止 | 限额、SLA 的有限作用、失败模式、缓存、降级和退出触发 | 状态页正常等于调用必然可用 |
| 观察与审计 | 调用方能证明哪部分实际发生 | 请求、响应、时间、服务声明、外部收据、未知范围与损失 | 日志完整等于内部执行可见 |
| 复现与决定 | 能否重现限定行为，谁允许继续使用 | 重放材料、快照、统计差异、风险评估、权威与截止点 | 一次相同输出产生 `met` 或 `accepted` |

这些事实可以由外部合同、供应商文档、服务配置、调用日志、数据清单、DPA、状态记录和独立测试分别保存。它们不需要压成新的 Noemion “云状态”“主权等级”或“可移植分数”。

## GNU 对他人服务替代用户计算的质疑

[GNU《Who Does That Server Really Serve?》](https://www.gnu.org/philosophy/who-does-that-server-really-serve.html)把“由他人运行的服务替代用户运行自己的程序”称为 Service as a Software Substitute。其核心问题不是服务器是否收费或源码是否公开，而是用户没有控制那一次实际计算：服务运营者可以改变程序，用户必须把数据交给运营者，也难以验证或修改真实执行路径。

Noemion 采用这个问题框架，但不把 GNU 的全部伦理结论静默写成规范：

1. 先判断活动是否确实是用户自己的计算，而不是通信、发布或必须包含另一方的交易；
2. 把“谁运行”“谁控制”“谁取得数据”和“用户拥有哪些软件自由”分别报告；
3. 不把所有网络服务统称为 SaaSS，也不把 `SaaSS` 变成 Noemion 领域词；
4. 不因为第三方托管就自动判定结果为假、证据无效或目标未满足；
5. 只把不可观察、不可修改、不可迁移和不可持续的部分写成明确限制与决策输入。

GNU 的批评提醒项目：便捷、性能和低初始成本不能替代用户控制说明。但实际采用仍由用途、风险、资源、可行替代、用户选择和具名权威决定。

## AGPL 提供源码，不提供服务实例控制

[GNU《Why the Affero GPL》](https://www.gnu.org/licenses/why-affero-gpl.html.en)说明，AGPL 要求向通过网络与修改程序交互的用户提供对应源码；同一页面也明确指出，AGPL 不解决他人服务替代用户计算的问题。得到服务端程序源码，仍不能证明：

- 当前实例运行的就是该源码及声明构建；
- 没有其他程序查看、修改或保留数据；
- 用户能选择更新时点、模型、配置或停服策略；
- 模型参数、训练数据和第三方依赖处于相同许可；
- 用户能导出状态并在自己控制的环境继续同一任务。

Noemion 因此不会把 AGPL、Apache-2.0 或其他许可证自动写成“服务受用户控制”。未来许可证选择必须另行审查对象、消费者、协作模式和兼容性；本提案不推荐或排除具体许可证。

## 托管人工智能的当前变化怎样影响边界

### 模型身份和服务行为会漂移

模型服务可能同时提供滚动别名与有日期快照。供应商也可能更新安全处理、工具、系统内容、路由和基础设施；即使 API schema 保持兼容，行为仍可能改变。公开表述必须区分：

- 可定位的服务模型 ID；
- 供应商承诺的快照或版本；
- 调用方实际提交的设置；
- 供应商不公开的路由与内部变换；
- 当前时间、区域、账户和功能资格。

若只有滚动别名，就只能形成该供应商、别名和发生时间限定的行为观察。旧 Dromen 不得因别名相同继续有效；切换到新快照、其他供应商或自管模型必须重新核对身份、评测、权限和用途决定。

### 数据控制是逐端点、逐功能的

[OpenAI API 数据控制文档](https://platform.openai.com/docs/models/default-usage-policies-by-endpoint)作为一个供应商实例，明确把训练使用、滥用监测日志、应用状态、零数据保留资格、区域存储和区域处理分开，并指出远端 MCP 服务受第三方自己的保留政策约束。Noemion 只采用这种“逐端点、逐功能、逐区域披露”的结构，不复制其产品字段，也不把一家供应商政策推广成行业事实。

“不用于训练”“零数据保留”“数据驻留”“区域处理”和“删除”必须分别核对。账户设置、端点、工具、后台模式或第三方连接变化时，旧结论失效。供应商政策是合同和配置证据的一部分，不是物理删除、内部零访问或全部子处理者行为的独立证明。

### 协议可以重排控制位置，但不能创造控制

[MCP 2025-11-25 Sampling](https://modelcontextprotocol.io/specification/2025-11-25/client/sampling)允许服务器经客户端请求模型生成，使客户端保留模型访问、选择和权限，并避免服务器持有模型 API key。它展示一种有价值的拓扑：外部工具不必取得模型凭据，模型选择可以留在用户侧控制面。

但这不证明用户控制了托管模型本身，也不保证客户端 UI、人工复核、数据最小化或供应商行为。Sampling 请求仍须绑定实际提示、上下文、工具、模型服务、权限和返回；规范中的 `SHOULD` 级人工参与不能被写成所有实现必然存在。

[MCP 2025-11-25 Authorization](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization)又要求用 Resource Indicators 把令牌绑定到目标资源，并禁止把入站令牌原样透传给下游服务。Noemion 采用受众隔离和独立凭据域，但不把 OAuth 成功、scope 或资源可达写成用户控制、目的合规或最终授权。

## NIST 供应链要求不等于用户控制结论

[NIST AI 600-1](https://doi.org/10.6028/NIST.AI.600-1)把生成式 AI 的价值链与组件集成列为独立风险来源，指出预训练模型、数据和其他第三方组件会让责任归因更困难。[NIST SP 800-218A](https://doi.org/10.6028/NIST.SP.800-218A)要求对取得的模型、数据和其他 AI 组件核对完整性、来源与安全，并在可行时追踪未知来源。

Noemion 借此要求记录供应者、对象、处理链、外部依赖和未知范围，但不让供应链审查自动产生：

- 软件自由或服务控制；
- 数据使用授权；
- 行为可复现；
- 模型安全或用途适合；
- Iknem 充分覆盖、`met` 或 `accepted`。

第三方服务通过安全审查，只说明具名检查在特定截止点通过。服务行为、条款、模型、下游处理者或区域变化后必须重新评估。

## “自主管理”“设备内”和“私有”也不能自证

1. **设备内应用仍可能远端调用。** UI 或主进程在本机，不表示模型、检索、遥测、审核或工具都在本机。
2. **租用机器也可能由用户自主管理。** 物理所有权不是唯一控制边界；仍要核对云管理员、镜像、密钥、网络和硬件信任。
3. **私有部署不等于自由软件。** 用户可以控制启停和数据路径，却仍不能研究或修改闭源程序。
4. **自托管不等于安全。** 漏洞修复、密钥、备份、日志、访问控制和运营能力可能更弱。
5. **离线不等于完整。** 下载参数但缺少 tokenizer、代码、许可或硬件路径，仍不能完成可修改执行。
6. **本地替代不等于同一结果。** 更小模型、不同量化或不同工具链会改变行为与证据，必须建立新身份和新决定。

公开页面只能按实际路径说“设备内推理已验证”“自主管理服务”“第三方托管模型”或“混合路径”。“本地优先”“私有 AI”“主权 AI”若无精确控制、处理与退出定义，只能保留为供应商原始说法，不能成为 Noemion 事实。

## 支持案例与反例

1. 桌面应用界面全部在本机，生成请求仍发送到第三方 API。正确分类是设备内客户端加第三方托管执行；不能写“本地模型”。
2. 模型运行在组织租用的云主机，组织控制镜像、参数、密钥、网络和更新。可以写“自主管理执行”，不能仅因机器不在办公室写成第三方模型服务。
3. 服务器程序采用 AGPL 并公开源码，但运营者还使用未披露的过滤器和日志服务。源码提供义务可能成立，当前服务实例和数据路径仍未完整可见。
4. 供应商声明 API 输入不用于训练，但默认保留滥用监测日志。训练使用和保留是两个事实，不能合并成“数据不保存”。
5. 账户启用零数据保留，随后调用不符合资格的工具。必须按具体端点和功能重新判断，不能沿用组织级标签。
6. 数据存储在指定区域，推理在其他区域完成。只能分别报告区域存储和处理，不能概括为“数据始终留在本地”。
7. 托管模型别名未变，供应商更新后端快照。旧评测、Dromen 和复现主张不能自动继承。
8. 导出对话 JSON 后导入另一供应商。文本历史可迁移，不表示隐藏提示、工具状态、缓存、模型行为和权限已经迁移。
9. 两家供应商兼容同一 API schema。结构兼容只证明请求形状可映射，不能证明模型语义、错误、保留、价格和限制相同。
10. 主服务把文件发送给远端 MCP 工具。主服务的数据政策不能覆盖 MCP 运营者；每个下游受众、字段和保留条件单独核对。
11. MCP Sampling 让工具服务器请求客户端模型。服务器不持有模型 key 是有益隔离，但客户端仍需核对提示、模型、工具和用户批准。
12. 自管模型停机后切换到托管模型作为降级。执行者、数据边界、模型身份和条款都改变，必须建立新会话并重新授权。
13. 托管服务返回同样答案两次。只能支持两次观察相同，不能证明模型快照、隐藏上下文或未来行为相同。
14. 服务状态页显示正常，当前租户已超配额。全局状态不能证明精确账户、区域、模型和请求即时可用。
15. 删除 API 返回成功。它支持供应商接受删除请求的协议事实；物理删除、备份清除和法定保留范围仍按证据说明。
16. 用户取得完整代码和模型参数，却没有足够硬件运行。软件和模型可得事实不变；实际可行替代和成本限制必须另外披露。
17. 自管闭源模型没有数据外发。运行控制和数据路径可能较强，软件研究与修改自由仍不成立。
18. 第三方托管自由软件模型提供可验证构建和远程证明。证据可以缩小未知范围，但用户仍不能直接控制运营者的更新、数据处理和停服决定。

## 威胁到失败责任的映射

| 威胁 | 失败方式 | 唯一主责任 | 不能洗成 |
| --- | --- | --- | --- |
| “本地”洗白 | 本机 UI 隐藏远端模型、检索或遥测 | 发布治理、TEXT-IDENTIFIER、ADP | 设备内完整执行 |
| 服务控制洗白 | 开源或 AGPL 服务被写成用户控制实例 | 发布治理、AUT、IKN | 实际执行可修改 |
| 模型别名漂移 | 同名端点解析到不同快照或路由 | ADP、ID、DRO | 无缝兼容更新 |
| 隐藏上下文 | 服务端添加、删除或压缩内容却声称完整可见 | ADP、TEXT-IDENTIFIER、IKN | 精确模型输入 |
| 保留概念合并 | 不训练、零保留、删除和区域驻留互相替代 | 数据治理、AUT、IKN | 全程零访问 |
| 下游政策外溢 | 主服务条款被扩张到工具和子处理者 | ADP、AUT | 全链路合规 |
| 凭据透传 | 模型或中间服务取得原始通用令牌 | AUT、ADP、DRO | 便捷委托 |
| 状态锁定 | 远端历史、文件、向量或 Task 无法完整导出 | 服务治理、ID、IKN | 可恢复会话 |
| API 兼容洗白 | schema 相同被写成语义、错误和政策等价 | ADP、ID、DIA | 无损替代 |
| 降级静默换域 | 本地与托管路径切换却沿用旧权限和证据 | DRO、AUT、ID | 连续同一会话 |
| 停服与配额失明 | 状态页、SLA 或缓存成功替代即时能力检查 | ADP、DIA | 持续可用 |
| 删除误报 | 接受删除请求被写成所有副本物理消失 | 服务治理、IKN | 已证明完全删除 |
| 自管安全洗白 | 控制软件负载被写成安全、私密和可靠 | 安全治理、IKN | 风险已接受 |
| 托管结果升级 | API 成功或相同输出成为满足与最终决定 | END、IKN、AUT | `met`、`accepted` |

## 失败域与结果隔离

| 事件 | 服务事实 | 会话 | 证据 | 满足 | 最终决定 |
| --- | --- | --- | --- | --- | --- |
| API 返回成功 | 调用和返回存在 | 继续受当前 Dromen 约束 | 形成有范围观察 | 不自动改变 | 不自动改变 |
| 模型快照或隐藏政策漂移 | 服务配置变化或未知 | 旧 Dromen 失效 | 旧评测不自动继承 | 旧判断不改写 | 重新决定未来使用 |
| 数据保留设置变化 | 新处理条件 | 视政策阻止或新建会话 | 保存变化与未知范围 | 不自动 `unmet` | 重新授权调用 |
| 导出成功 | 指定内容已导出 | 不恢复旧会话 | 支持导出范围 | 不证明迁移等价 | 不产生接受 |
| 供应商停服 | 当前能力不可用 | 会话失败或转入具名降级 | 保留调用与失败 | 依据目标另行判断 | 权威决定是否切换 |
| 切换自管或其他供应商 | 新执行路径 | 新 Dromen | 新身份、评测和限制 | 重新求值 | 重新决定 |

服务可用性、会话终止、证据覆盖、目标满足和最终决定继续遵守 ADR-0015 的正交结果域，不能因“云端成功”或“本地失败”互相改写。

## 采用托管路径前的最小问题清单

1. 这项活动是用户自己的计算、共同通信，还是两者混合；
2. 谁运行、谁控制软件负载、谁能静默改变模型和配置；
3. 哪些输入、元数据、秘密和派生内容离开控制域；
4. 存储、处理、日志、训练、人工访问和删除分别是什么条件；
5. 是否存在下游工具、子处理者或跨区域转移；
6. 模型快照、提示、工具和参数能否精确绑定，哪些部分不可观察；
7. 凭据是否绑定精确资源、受众、scope、期限和实际行动者；
8. 历史、文件、向量、远端 Task 和审计材料能导出什么、缺失什么；
9. 切换供应商或自管路径会损失哪些语义、权限、证据和能力；
10. 配额、涨价、停服、撤回和区域故障触发什么关闭失败或降级；
11. 自主管理替代是否真实可行，资源、硬件和运营成本怎样披露；
12. 哪个权威在什么截止点接受剩余不可观察性和锁定风险。

任一高风险问题未知时，页面和控制面关闭失败或明确降级主张。不得用“企业级”“可信云”“隐私模式”“本地优先”或“可移植”营销词填补证据空白。

## 术语和读音边界

本提案故意使用“第三方托管执行”“自主管理执行”“设备内执行”“数据外发”“导出”“切换”和“用户控制”等普通职责短语，不创造首字母缩写、短词或希腊词根。

`SaaSS` 只在引用 GNU 思想来源时出现，不成为 Noemion 术语、字段、状态或界面标签。“本地”“云”“私有”和“主权”都太容易掩盖实际控制者与处理阶段；面向首次读者必须展开为谁执行、谁控制、数据去向和仍不可观察什么。

如果未来真实部署对象确实需要独立名字，仍须先证明独立身份、消费者、生命周期和失败责任，再通过中文与英文朗读、听写、职责匹配、冲突检索和权利复核。当前没有这种对象，不进入命名流程。

## 接受后的落点与停止条件

若责任分配经正式 ADR 接受，后续只把唯一义务归还现有规范和页面：

1. ADP 增加托管模型端点、区域、供应商别名、下游服务、导出和切换损失案例；
2. AUT 增加数据外发、保留、训练用途、下游受众和降级路径授权案例；
3. DRO 明确模型、提示、工具、保留政策或执行控制实质漂移时旧会话失效；
4. ID 增加服务快照、远端状态、导出和替代关系的有范围身份案例；
5. IKN 增加调用方可观察请求、服务声明、删除响应和未知内部路径的覆盖限制；
6. TEXT-IDENTIFIER 限制“本地”“私有”“云”“主权”和“可移植”的无定义使用；
7. DIA 只在真实消费者出现后登记有限的服务漂移、保留冲突、导出损失和停服概念；
8. END 与 SYN 不新增供应商、云、区域、许可或服务状态字段。

以下任一条件成立就停止建设 Noemion 自有模型网关或服务控制面：

- 外部客户端、协议和部署工具能满足版本、授权、导出与失败要求；
- 没有真实托管模型消费者、获准数据或可证伪净增益；
- 供应商不可观察性超过具名政策允许范围；
- 无法形成可接受的数据、凭据、保留、下游处理者和退出控制；
- 切换只剩表面 API 兼容，不能保存语义、证据和权限边界；
- 项目尚未进入对应代码或部署阶段。

## 当前决定边界

本提案现在只形成七项候选结论：

1. 软件自由、实际执行控制、数据控制和服务可移植性是四个不同主张。
2. “本地”按组件判断；第三方托管、自主管理、设备内执行和通信服务必须分开。
3. 托管模型只形成调用方可观察的请求、服务语境、返回和限制，不能声称掌握隐藏内部执行。
4. 不训练、零保留、数据驻留、区域处理、删除和下游服务政策必须逐项核对。
5. 导出、API 兼容或降级切换不保持模型身份、会话、语义、证据、权限和最终决定。
6. AGPL 源码提供义务、NIST 供应链审查和 MCP 控制拓扑都不能单独产生用户对服务实例的控制。
7. 当前不创建新 CORE、Profile、登记、制品、网关、组件、命令或专名；只改善公开表述和未来采用门禁。

这些结论形成正式 ADR 后，项目再判断是否把唯一义务补入现有规范案例。在此之前，本提案保持非规范研究资料。
