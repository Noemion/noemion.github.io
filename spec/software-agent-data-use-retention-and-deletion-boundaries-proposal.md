# Noemion 软件 Agent 数据使用、保留与删除边界研究提案

状态：非规范研究提案

日期：2026-07-15

结论状态：桌面审查完成；尚未进入规范

研究范围：数据访问、目的授权、实际处理、变换与派生、披露、存储、保留、删除请求、删除操作、介质清除、备份与派生物传播、审计保留、证据和未知范围

本提案不构成 ADR、CORE 规范、Profile、登记项、法律结论或实现要求，不进入 `registry.json`。它不创建数据制品、数据目录、隐私服务、删除服务、清除证明、命令、组件、结果域或新专名，也不建立 `DATA-CORE`、`PRIVACY-CORE` 或 `DELETION-CORE`。

## 直接结论

访问、使用、保留与删除不是一条状态线。有权读取某项数据，不等于有权把它发送给模型、工具或下游服务；一次处理得到授权，不等于可以写入日志、跨会话记忆或训练集；删除接口返回成功，也不等于主存储、缓存、索引、备份和派生物已经完成有效清除。

软件 Agent 会让同一输入沿模型上下文、工具参数、返回值、日志、缓存、记忆、检索索引、嵌入、训练数据、派生模型和外部服务复制或变形。每一条路径都需要精确主体、目的、接收方、截止点和证据。不能观察的服务端处理、备份清除和派生模型影响必须明确写成未知范围，不能由政策标签或成功状态补全。

Noemion 当前没有数据目录、隐私政策引擎、删除执行器、存储组件或模型训练组件。本提案只规定未来文档与验证应怎样分开主张，不选择具体法律依据、保留期限、存储介质、云服务或清除技术。

## 至少十八种事实必须分开

| 事实 | 回答的问题 | 最低绑定 | 不能自动推出 |
| --- | --- | --- | --- |
| 精确来源与数据范围 | 哪些原始字段、附件、元数据或派生值进入路径 | 对象身份、字段、来源、版本、主体和截止点 | 页面可见内容等于全部发送内容 |
| 数据类别与敏感性 | 哪类风险需要更强限制 | 分类依据、适用范围、未知项和复核时间 | 工程分类等于法律定性 |
| 访问请求 | 谁请求读取什么 | 请求主体、实际行动者、对象、动作和时间 | 请求已经获准或已经发生 |
| 授权决定 | 哪个权威允许哪项处理 | 主体、对象、动作、目的、接收方、条件、期限和决定 | 数据已经使用、保存或删除 |
| 声明目的 | 为什么计划处理 | 用途、任务、消费者、禁止用途和截止点 | 实际处理符合声明 |
| 实际处理 | 什么计算真正接触或改变了数据 | 执行者、程序或服务、输入、时间、环境与结果 | 处理获准、正确或用途适合 |
| 变换与派生数据 | 摘要、嵌入、索引、标注或模型怎样由来源形成 | 来源身份、变换、损失、派生身份与关系 | 派生物不再敏感或可忽略删除 |
| 披露与接收方 | 数据被发送给谁 | 精确字段、接收方、受众、协议、下游路径与未知范围 | 加密传输等于用途获准 |
| 存储位置与控制者 | 哪个系统在什么控制域保有数据 | 存储对象、区域、租户、控制者、副本类别和访问面 | 数据驻留等于只在该处处理 |
| 保留政策与截止点 | 数据预计保留多久、按什么条件到期 | 对象类别、起点、期限、例外、复核和销毁路径 | TTL 到期等于物理清除完成 |
| 删除请求 | 谁要求对哪些对象停止可用或清除 | 请求者、对象、范围、理由、时间和接收方 | 请求合法、已接受或已执行 |
| 删除授权 | 哪个权威允许删除哪些内容 | 对象、关系、法定或审计保留约束、决定和期限 | 所有关联副本都可删除 |
| 删除操作与状态 | 哪个系统执行了什么逻辑或物理动作 | 操作、目标、方法、开始与结束时间、返回和失败 | 成功返回等于不可恢复 |
| 清除验证 | 预定技术是否按声明完成 | 方法、工具、目标范围、完成记录和异常 | 方法对实际介质有效 |
| 清除确认 | 结果是否在风险与约束下足够 | 风险、介质、复原能力、残留范围、审批者和结论 | 一次覆盖或 API 结果足够 |
| 备份、副本与派生物传播 | 删除怎样传到镜像、快照、索引、嵌入和下游 | 关系闭包、各对象状态、传播期限、阻塞与未知范围 | 主对象删除等于闭包清空 |
| 审计或记录保留例外 | 哪些最小记录仍需保留、为什么 | 记录类别、必要字段、用途、权限、期限和删除条件 | 例外允许保留原始秘密或无限日志 |
| 证据与未知范围 | 当前能够证明什么、还不能观察什么 | 生产者、方法、对象、时间、限制、来源类别和截止点 | 日志存在等于处理链完整 |

这些事实不需要压成一个“隐私状态”“删除等级”或新对象。未来政策可以引用它们，但必须让每项决定和观察回到现有身份、授权、适配、会话、文本、证据与诊断责任。

## 权威资料怎样约束这组边界

### NIST Privacy Framework 分开处理阶段

[NIST Privacy Framework 1.0](https://nvlpubs.nist.gov/nistpubs/CSWP/NIST.CSWP.01162020.pdf)在控制与沟通活动中分别讨论数据审查、传输或披露、变更、删除访问、按政策销毁、权限、日志最小化、披露记录和更正或删除通知。Noemion 采用这种阶段分离，但不复制框架分类，也不把它当成法律合规结论。

截至本提案日期，[Privacy Framework 1.1](https://www.nist.gov/privacy-framework/new-projects/privacy-framework-version-11)仍处于初始公开草案阶段，因此当前稳定基线仍采用 1.0。未来版本成为最终版后应重新核对术语和控制关系。

### NIST AI 600-1 让撤回与未来使用可定位

[NIST AI 600-1](https://nvlpubs.nist.gov/nistpubs/ai/NIST.AI.600-1.pdf)要求生成式 AI 风险管理记录来源与隐私，并在适用时提供对当前和未来使用撤回或撤销同意的机制。撤回会改变未来允许的使用，却不会使历史处理从未发生，也不自动清除已经产生的日志、派生物或模型影响。

### RFC 6973 分开收集、使用、披露与保留

[RFC 6973](https://www.rfc-editor.org/rfc/rfc6973.html)把数据最小化放在收集、使用、披露、存储与保留等多个阶段，并把未经相应同意的二次使用作为独立风险。协议通常不能强制下游真正遵守用途和保留承诺；因此协议字段或加密成功只能支持传输事实，不能证明接收方的后续处理。

### W3C DPV 提供分类纪律，不提供 Noemion schema

[W3C Data Privacy Vocabulary 2.1](https://w3id.org/dpv/2.1/changelog.html)（DPV 2.1）分别描述访问、使用、存储、逻辑删除、销毁、擦除、派生数据、存储期限和恢复关系。这种词汇分离有助于避免“删除”一词过载，但 DPV 是 W3C 社区组成果，不是 W3C Standards Track 标准。Noemion 只借用分类纪律，不导入 DPV 类、URI、schema 或法律结论。

### NIST SP 800-88 Rev. 2 分开清除验证与确认

[NIST SP 800-88 Rev. 2](https://csrc.nist.gov/pubs/sp/800/88/r2/final)把介质清除定义为使目标数据在给定努力水平下不可访问，并明确覆盖逻辑和云存储。它还分开 sanitization verification 与 validation：前者检查预定技术是否完成，后者判断结果在风险、介质和约束下是否有效、可接受。

成功执行错误方法、遗漏不可访问区域、快照或副本，仍可能得到无效结果。Noemion 中文页面使用“清除验证”和“清除确认”表达这两个阶段，首次出现时保留英文原词，避免把两个都机械译成“验证”。

### MCP 可以保护输入交互，不能定义完整生命周期

[MCP 2025-11-25 Elicitation](https://modelcontextprotocol.io/specification/2025-11-25/client/elicitation)要求用户可以复核、拒绝或取消请求，并禁止用表单模式索取敏感凭据；URL 模式还能让秘密不经过客户端或模型。它约束交互和秘密路径，却不定义一般数据保留、备份传播或清除证明。

协议日志和可观察性结构还会演进。Noemion 因而固定采用的协议版本，并把协议返回、调用日志和实际数据生命周期分开；协议的成功状态不会成为本地删除或清除结论。

## GNU 技术给出的两个具体反例

[GNU Coreutils `shred`](https://www.gnu.org/software/coreutils/manual/html_node/shred-invocation.html)依赖文件系统原地覆盖，但日志文件系统、写时复制、快照、SSD 重映射、备份和镜像都可能使旧数据仍然存在。它表明“命令成功”“文件名消失”和“所有副本不可恢复”是不同主张；Noemion 不把 `rm`、`unlink` 或 `shred` 退出状态当作完整清除证明。

[GNU Guix 垃圾回收](https://guix.gnu.org/manual/devel/en/guix.pdf)按根的可达性区分仍存活与可回收对象，活动引用或根会阻止删除。Noemion 只借用“先枚举关系闭包再判断哪些对象仍被引用”的工程纪律，不把软件存储回收模型当成隐私删除或介质清除模型。

## 至少十六个支持案例与反例

1. 工具把文件内容返回给模型。工具读取获准，不表示模型接收和后续记录也获准。
2. 文档被加入检索索引并生成嵌入。源文件删除后，索引条目、嵌入和缓存仍是具名派生物。
3. 提示正文未保存，调用元数据和错误日志仍包含用户标识与片段。应分别报告各记录范围。
4. 服务声明“不用于训练”，但保留滥用监测日志。训练用途和保留期限是两个事实。
5. 删除接口接受异步请求。成功响应只说明请求被接收；完成、范围和异常需要后续证据。
6. 主数据库删除记录，备份按周期过期。只能报告主库状态和备份清除计划，不能立即声称全部删除。
7. 原始文本被删除，摘要与嵌入仍可定位其内容。派生关系必须进入删除传播检查。
8. 训练样本从未来数据集移除。它限制后续训练，不会让过去训练出的模型自动回到未见过该样本的状态。
9. 安全审计需要保留最小事件记录。例外必须删去无关正文和秘密，并固定用途、权限与期限。
10. 用户撤回同意。未来处理需要停止或重新授权，历史处理事实与必要审计记录不能被改写成从未发生。
11. 桌面应用调用云端 OCR。设备内界面不表示图像、元数据和识别结果没有离开设备。
12. 自主管理模型关闭远端推理，却仍启用外部遥测。执行控制和数据披露需要逐路径说明。
13. TTL 已到期，但清理任务失败。政策到期不等于数据已不可访问。
14. 在 SSD 或快照文件系统上运行 `rm` 或 `shred`。命令状态不能覆盖介质、快照和备份限制。
15. 生成的合成样本没有复制原句，却可能由敏感来源派生。仍要记录来源、变换和用途，不可仅凭“合成”解除限制。
16. 主服务把内容交给未知下游提供者。调用方只能记录已知接收方与未知范围，不能替下游证明保留和删除。
17. 缓存键删除后，共享缓存节点仍有副本。必须分开逻辑不可见、节点传播和介质清除。
18. 清除工具按计划完成，但选错卷或漏掉不可寻址区域。验证可以通过，确认仍应失败。

## 威胁到失败责任的映射

| 威胁 | 失败方式 | 唯一主责任 | 不能洗成 |
| --- | --- | --- | --- |
| 用途外溢 | 一次访问许可扩张到日志、记忆、训练或下游 | AUT | 已同意全部使用 |
| 隐藏模型输入 | 变换、工具返回或服务端内容未记录 | TEXT-IDENTIFIER、ADP | 精确输入已知 |
| 派生物失明 | 摘要、嵌入、索引或模型不在关系闭包中 | ID | 源删除即闭包清空 |
| 接收方漂移 | 新工具、子处理者或遥测端点沿用旧决定 | ADP、AUT | 同一服务政策覆盖全链路 |
| TTL 洗白 | 到期时间替代实际清理和观察 | DRO、IKN | 已删除 |
| 删除响应升级 | 接收请求或逻辑删除被写成物理清除 | ADP、IKN | 所有副本不可恢复 |
| 方法错配 | 覆盖操作不适用于快照、SSD 或云副本 | IKN、DIA | 清除技术有效 |
| 验证与确认合并 | 工具完成记录替代风险判断 | IKN | 清除结果可接受 |
| 审计例外扩张 | 必要记录成为无限保留原文和秘密的理由 | AUT、DRO | 合规要求无限保存 |
| 未知范围消失 | 服务内部或下游不可观察处理被写成零处理 | ADP、IKN | 全链路已经证明 |
| 历史重写 | 撤回或删除被描述成数据从未使用 | ID、IKN | 过去事件不存在 |
| 结果域污染 | 删除失败直接改写目标满足或最终决定 | DIA、END、AUT | 自动 `unmet` 或 `rejected` |

## 失败域与结果隔离

| 事件 | 授权事实 | 生命周期事实 | 证据 | 目标满足 | 最终决定 |
| --- | --- | --- | --- | --- | --- |
| 访问获准 | 精确请求可尝试 | 尚未证明已处理或存储 | 保存授权决定 | 不自动改变 | 不自动接受 |
| 处理完成 | 旧授权不扩张 | 记录实际处理和派生关系 | 形成有范围观察 | 依目标另算 | 不自动接受 |
| 删除请求被接收 | 请求进入对端流程 | 尚未证明执行完成 | 保存协议收据与限制 | 不自动改变 | 不自动接受 |
| 主对象逻辑删除 | 该对象当前不可见或不可检索 | 副本、备份和派生物分别待查 | 保存操作与范围 | 不自动改变 | 不自动接受 |
| 清除验证通过 | 不产生新权限 | 预定技术完成 | 支持技术执行事实 | 不自动改变 | 仍需清除确认 |
| 清除确认失败 | 未来处理授权不变 | 残留或方法不适用 | 记录原因和未知范围 | 依据目标另算 | 权威决定补救或接受风险 |
| 最小审计记录保留 | 仅按具名例外获准 | 原始内容与无关字段不得沿用 | 保存例外依据和期限 | 不自动改变 | 不自动接受 |

访问、处理、删除、证据、满足和最终决定继续遵守正交结果域。删除故障需要诊断和补救，却不能静默改写已经发生的处理历史或让一个协议状态替代权威决定。

## 候选责任的唯一主归属

| 义务 | 主归属 | 辅助归属 | 不新增的对象 |
| --- | --- | --- | --- |
| 数据、请求、副本、索引、嵌入、摘要和模型派生关系的身份 | ID | TEXT-IDENTIFIER、IKN | 数据目录制品 |
| 实际模型输入、脱敏、截断、摘要和有损变换 | TEXT-IDENTIFIER | ID、IKN | 输入生命周期格式 |
| 对象、动作、目的、接收方、保留、删除和例外授权 | AUT | ADP、DRO、IKN | 隐私令牌或删除权威 |
| 端点、接收方、下游处理、协议状态和不可观察范围 | ADP | AUT、ID、DIA、IKN | 数据网关 CORE |
| 一次会话的能力、存储预算、有效期限和漂移失效 | DRO | AUT、ADP、ID | 保留会话对象 |
| 处理、删除、清除验证与确认的观察、来源和限制 | IKN | ID、ADP、DIA | 清除证明制品 |
| 生命周期冲突、失败、残留、传播超时和未知范围诊断 | DIA | IKN、ADP、ID | 删除结果域 |
| 目标是否满足 | END | IKN | 隐私目标类型 |
| 多目标、依赖与复合责任 | SYN | END、IKN | 数据工作流对象 |

END 与 SYN 只表达目标及其组合，不容纳保留状态、隐私分类或删除流水线。AUT 允许某项处理也不证明处理发生；IKN 记录观察也不授予处理权限。

## 开发者最小检查清单

1. 精确列出进入模型、工具、日志、缓存、记忆、索引和外部服务的字段，而不是只写“用户数据”。
2. 分开访问请求、授权决定、声明目的、实际处理和外部披露。
3. 为摘要、嵌入、索引、数据集和派生模型建立来源关系与独立身份。
4. 对每个接收方固定受众、下游路径、保留条件和不可观察范围。
5. 让 TTL、退出状态、HTTP 成功和供应商收据只支持它们实际观察到的阶段。
6. 删除前枚举主对象、副本、缓存、备份、快照、索引、派生物和下游对象。
7. 分开删除请求、删除授权、删除操作、清除验证和清除确认。
8. 若必须保留审计记录，固定最小字段、目的、权限、期限和最终删除条件。
9. 记录传播失败、残留、未覆盖介质和第三方未知范围，不能用默认成功补齐。
10. 政策、账户设置、端点、模型、工具、存储拓扑或供应商条款变化后重新判断。
11. 不把撤回、删除或回滚写成历史处理从未发生。
12. 不让数据生命周期状态自动成为 `met`、`accepted` 或组件实现证据。

## 术语与读音边界

本提案不创建新专名。公开页面优先使用“访问请求”“授权决定”“实际处理”“保留期限”“删除请求”“逻辑删除”“介质清除”“清除验证”“清除确认”“副本传播”“派生物”和“未知范围”等职责短语。

这些短语在普通话中音节清楚，口头区分度高于新增音译词；“删除”“清除”和“销毁”仍必须附带对象、方法与证据，不能只追求顺口。英文 sanitization verification 与 sanitization validation 首次出现时保留原词，避免中文近义词造成误听。若未来形成规范字段或 CLI，仍需按独立人类读音测试验证，不能把本次桌面判断写成人类研究结果。

## 语言哲学边界

维特根斯坦在《逻辑哲学论》4.022 指出，命题显示它为真时事情会是怎样。对数据生命周期而言，“删除请求已接收”“主记录不可检索”“预定清除技术已完成”和“所有具名副本在给定努力下不可恢复”显示的是不同可能事态。

这段哲学提示只支持主张边界清晰，不定义隐私权、法律义务、保留期限或清除技术。工程结论仍必须由对象身份、授权、实际操作、介质条件、观察方法、未知范围和具名权威决定。

## 采用门槛与待定内容

只有同时出现真实消费者、明确政策语境、存储与下游拓扑、可识别副本和派生物、清除方法、证据生产者、威胁模型、正反场景与规范字节需求，才应考虑把其中某项义务写回现有规范。即使接受本提案，也优先修订 ID、TEXT-IDENTIFIER、AUT、ADP、DRO、IKN、DIA、END 或 SYN，不新建数据生命周期 CORE。

法律依据、数据主体请求流程、司法辖区、法定保留、具体期限、存储服务、密钥销毁、介质分类、远程证明、训练数据去除和模型反学习仍保持开放。政策或法律效力不属于本提案结论。
