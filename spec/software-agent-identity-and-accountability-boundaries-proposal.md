# 软件 Agent 身份、委托与责任链研究提案

状态：非规范研究提案

日期：2026-07-15

结论状态：桌面审查完成；尚未进入规范

适用阶段：规范、资料与网站设计阶段

## 直接结论

“使用哪个模型”“运行哪套 Agent 定义”“哪个工作负载通过认证”“哪一个运行实例发起调用”“使用什么凭据”“代表谁行动”“谁授予权限”和“哪一次动作实际发生”是不同事实。任何一个名称、令牌或日志字段都不能同时回答这些问题。

Noemion 当前应保持以下边界：

1. 模型身份只定位模型或托管端点及其版本语境，不能自动成为 Agent、实际行动者或权威。
2. Agent 定义身份只定位软件、配置、提示、工具声明与政策闭包，不能证明某个副本正在运行或拥有权限。
3. 工作负载身份只说明哪个受管理软件主体通过指定信任域认证；共享工作负载名可能覆盖多个副本，不能替代精确运行实例。
4. 运行实例与 Drase 会话必须分别绑定启动、环境、实际软件闭包、Dromen、截止点和销毁状态。进程重启、弹性副本或恢复操作不能继承旧会话身份。
5. 凭据和身份陈述只支持发行者、主体、受众、期限与验证范围内的认证主张。它们不自动产生业务动作授权，也不能隐藏实际持有者。
6. 请求主体、实际行动者与被代表主体继续由 AUT-CORE 分开。服务账户、企业单点登录或 Agent Card 都不能把三者压成一个 `user` 字段。
7. 委托必须绑定具体活动、逐级收窄并保留每个中间行动者。控制权转交、模型路由或子 Agent 调用不等于授权转移。
8. 每次外部动作必须拥有独立调用身份，绑定实际行动者、代表关系、对象、动作、输入、会话、授权决定、幂等与观察责任。
9. 日志、trace、签名和 Iknem 只能支持其实际保存的责任链。主体名称相同、认证成功或记录不可变都不能证明意图、授权、因果、满足或最终接受。

本提案不构成 ADR、CORE 规范、内容 Profile 或实现要求。它不创建 Agent 身份制品、责任制品、目录、凭据格式、命令、组件、结果域、接口或新专名，也不创建 `AGENT-IDENTITY-CORE`，不进入 `registry.json`。当前只检查 ID、AUT、DRO、ADP、IKN、DIA 与 TEXT-IDENTIFIER 的现有责任是否足以承载这些区分。

## 为什么“Agent 身份”仍然过于宽泛

开发者常用一个产品名或服务账户回答“谁做了这件事”。在单次演示中，这种简写可能足够；在多模型、弹性副本、handoff、企业授权和异步任务中，它至少会造成四类错误：

- **模型替代行动者：**日志只写模型名，隐藏真正编排请求、选择工具和提交调用的软件实例；
- **工作负载替代实例：**多个副本共享同一服务身份，事故后无法定位哪一个实例使用了哪份配置和会话；
- **认证替代授权：**有效证书、令牌或单点登录被当成当前动作已经获准；
- **代表关系消失：**服务账户或 impersonation 只显示被代表用户，隐藏自动系统、委托链和实际调用者。

解决方法不是建立一个覆盖全部事实的“Agent 身份对象”。每种身份必须说明它定位什么、由谁发行、何时变化、能支持哪些判断，以及失败落在哪个责任域。

## 九个身份与关系层次必须分开

| 层次 | 回答的问题 | 最低绑定 | 不能推出什么 |
| --- | --- | --- | --- |
| 模型身份 | 哪个模型、权重或托管端点参与候选生成 | 供应商或发布者、精确版本或快照、配置、取得时间与已知限制 | Agent 软件身份、实际行动者、授权或输出正确 |
| Agent 定义身份 | 哪套软件与声明规则构成可部署 Agent 定义 | 软件闭包、配置、提示与策略身份、工具声明、构建或发布来源 | 当前正在运行、凭据有效或动作获准 |
| 部署身份 | 哪次部署选择了什么定义和运行环境 | 定义身份、部署配置、平台、租户、区域、变更与截止点 | 某个副本已经执行动作或状态未漂移 |
| 工作负载身份 | 哪个受管理软件主体通过信任域认证 | 身份命名空间、信任域、发行者、验证材料、受众与有效期 | 单个进程、代表的人类、角色权威或业务授权 |
| 运行实例身份 | 哪个具体副本在何处、何时运行 | 工作负载、进程或等价实例、启动、镜像或软件闭包、环境、终止状态 | 旧实例重启后仍是同一会话，或实例名称证明行为 |
| 会话身份 | 哪次受限运行在什么上限内发生 | 精确 Dromen、运行实例、主体、政策、能力、预算、截止点与销毁 | 跨会话权限、恢复旧能力或目标满足 |
| 凭据身份 | 哪份陈述或能力被什么持有者出示 | 发行者、主体、持有者、受众、资源、scope、有效期、撤销与令牌绑定 | 实际使用者等于声明主体、权限覆盖任意动作或最终权威 |
| 主体与委托关系 | 谁请求、谁实际行动、代表谁、谁授予 | AUT-CORE 语境、每一级行动者、代表关系、范围、目的、期限与撤销 | 隐藏中间 Agent、继承全部权限或把认证当成授权 |
| 动作身份 | 哪一次请求、尝试和外部效果实际发生 | 调用身份、会话、实际行动者、代表主体、对象、动作、精确输入、授权、幂等与观察 | 成功等于因果归属、`met`、证据充分或 `accepted` |

这些都是直白职责短语，不是新的领域对象。模型、定义、部署、工作负载和运行实例可以拥有各自精确身份，但“身份存在”只说明能够定位和比较，不能自动形成权限、责任或信任。

## 身份、认证、授权与责任不是一条等级线

| 判断 | 需要的证据 | 失败时记录 | 不应改写 |
| --- | --- | --- | --- |
| 内容身份相同 | ID-CORE 规定的身份域、表示、算法和完整摘要 | identity mismatch | 主体、权限、语义等价 |
| 工作负载已认证 | 信任域、验证材料、持有证明、受众和期限 | authentication failure | 动作授权、角色资格 |
| 当前动作已授权 | AUT-CORE 精确语境、政策、决定与截止点 | `deny` 或 `defer` | 身份真实性、目标满足 |
| 实际调用已发生 | ADP 调用身份、对端事实、输入和返回 | 调用或交付诊断 | 授权范围、因果归属 |
| 记录可用于判断 | Iknem 主体、范围、方法、环境、溯源与限制 | `invalid`、`revoked` 或覆盖不足 | 内容为真、最终接受 |
| 责任可以归属 | 实际行动者、委托关系、活动、政策与适用制度 | 责任关系未确定 | 技术认证自动产生法律责任 |

认证和授权不是“先低后高”的同一种可信度。一个工作负载可以被准确认证，却没有当前对象上的权限；一次动作也可能被正确授权，却因凭据持有者、实例绑定或输入漂移而不能执行。

## 现有 Noemion 责任已经覆盖什么

| 现有责任 | 已覆盖 | 本提案发现的补充问题 |
| --- | --- | --- |
| ID-CORE | 精确内容身份、引用、签名陈述、验证包络、截止点和派生关系 | Agent 定义、部署、工作负载、实例和调用必须使用不同身份域，不能只存显示名 |
| AUT-PRN-001 | 请求主体、实际行动者、被代表主体、认证、角色与决定权分开 | 实际行动者还要能绑定精确工作负载与运行实例，不能只写 Agent 产品名 |
| AUT-DEL-001 | 委托保留双方身份并逐级收窄 | handoff、服务账户和企业授权仍要保留具体活动与全部中间行动者 |
| AUT-TIM-001 | 使用前核对主体、政策、状态、截止点与撤销 | 工作负载凭据轮换、部署变更和实例重启都可能触发重验 |
| DRO-SUB-001、DRO-ENV-001 | 会话绑定精确主体、政策、环境和观察支持 | 运行实例、Agent 定义与模型版本的实质漂移必须使旧会话失效 |
| ADP-PEE-001、ADP-INV-001 | 对端声明、认证主体、调用主体、输入和会话分别绑定 | 协议对端、实际本地调用者和被代表主体不能合并成一个 peer |
| IKN-SUB-001、IKN-PRO-001 | 证据绑定主体、范围和有限溯源 | trace 必须说明记录的是模型、编排器、工作负载、实例还是外部对端 |
| DIA-CTX-001 | 诊断绑定生产者、操作、输入与失败层 | 主体不明、实例不明、凭据不适用、委托断裂和授权失败要保持不同层次 |
| TEXT-IDENTIFIER-CORE | 标识符、显示、范围和来源分开 | 可读 Agent 名、服务账户名和身份 URI 不能通过字符串相同合并身份域 |

桌面审查没有发现必须建立 `AGENT-IDENTITY-CORE` 或新制品的责任。若这些边界被接受，首选是补强现有条款的场景、威胁和未来验证条件。

## 权威资料给出的边界

### NIST 2026 概念工作：问题已经明确，答案尚未标准化

[NIST NCCoE 软件与 AI Agent 身份授权概念论文](https://www.nccoe.nist.gov/sites/default/files/2026-02/accelerating-the-adoption-of-software-and-ai-agent-identity-and-authorization-concept-paper.pdf)把区分人类与 Agent 身份、Agent 授权、访问委托、日志透明和数据流追踪列为拟研究范围。论文还直接询问 Agent 身份元数据应当固定还是随任务变化，以及怎样把 Agent 身份绑定回人类授权。

这证明行业正在处理同一缺口，不证明已经存在统一的 Agent 身份模型。该文件是概念论文，不是正式实践指南；Noemion 只采用它提出的问题分解，不把拟议架构或候选标准写成本地规范。

[NIST AI 800-5](https://www.nist.gov/publications/summary-analysis-responses-request-information-regarding-security-considerations-ai)总结的公众意见认为传统网络安全原则仍然相关，但需要适配 Agent 风险。该结论支持从既有身份、授权和审计标准出发，也要求保留“仍待验证”的范围，不能以 Agent 热点为由新建无消费者对象。

### SPIFFE：工作负载身份不是运行实例或授权决定

[SPIFFE 身份与可验证身份文档标准](https://spiffe.io/docs/latest/spiffe-specs/spiffe-id/)使用信任域与路径定位工作负载，并由 SVID 让工作负载向资源证明身份。标准明确把路径含义留给管理者，并提醒不同信任域对同一属性的解释不能自动互通。[SPIFFE 核心概念](https://spiffe.io/docs/latest/spiffe-about/spiffe-concepts/)进一步说明，工作负载可以分布在多个节点，并且 SVID 本身不授予应用权限。

SPIFFE 的“工作负载”还可以覆盖弹性运行在多个节点上的服务。因此一个 SPIFFE ID 可以成为工作负载认证输入，却不一定唯一定位单个进程、副本、Drase 会话或动作。SVID 有效也只支持指定信任域、受众和期限内的身份主张，不自动形成 AUT-CORE 的角色资格、委托或动作授权。

### RFC 8693 与 W3C PROV：行动者、代表关系和活动必须关联

[RFC 8693 Token Exchange](https://www.rfc-editor.org/rfc/rfc8693.html)用 subject 与 actor 表达委托，并区分 delegation 和 impersonation。Noemion 继续要求即使外部协议只暴露被代表主体，本地记录也保存实际行动者和完整委托链。

[W3C PROV-DM](https://www.w3.org/TR/prov-dm/)把 SoftwareAgent、Activity、Association、Attribution 与 `actedOnBehalfOf` 分开；委托关系还可以绑定具体活动。Noemion 采用“责任关系必须指向精确活动”的结构纪律，但不把 PROV 对象直接写入 Iknem、AUT 事件或物理格式，也不让 provenance 关系自动决定法律责任。

### MCP 企业授权：组织访问不等于每次工具动作的责任

[MCP Enterprise-Managed Authorization](https://modelcontextprotocol.io/extensions/auth/enterprise-managed-authorization)让组织身份提供方集中决定员工能否访问 MCP 服务器，并支持统一撤销和审计路径。它解决组织级接入，不说明模型、编排器、运行实例或下游工具中谁实际发起某次动作。

Noemion 因此把企业身份陈述作为授权输入：本地仍要绑定实际行动者、被代表主体、精确对象与动作、资源受众、当前 Dromen 和一次调用身份。单点登录和服务器接入成功不能替代这组判断。

### GNU 身份工具：真实身份、有效身份与安全上下文可以同时存在

[GNU Coreutils `id`](https://www.gnu.org/software/coreutils/manual/html_node/id-invocation.html)会分别报告真实用户 ID、有效用户 ID、组和安全上下文，而不是只输出一个“当前用户”。Noemion 借用的是表达纪律：起源、当前有效权限语境和补充上下文应分别可见。

Unix 用户与进程模型不能直接定义 Agent 身份。工作负载、容器、服务账户、模型、组织委托和跨系统令牌拥有不同生命周期；GNU 工具只说明把多种身份压成一个显示名会丢失安全信息。

## 十二个支持案例与反例

### 案例一：同一个模型服务被两个 Agent 定义调用

两个 Agent 都使用同一托管模型端点，但提示、工具、政策和运行软件不同。模型身份相同不产生同一 Agent 定义身份。

反例：日志只保存模型名，因此把两个系统的工具调用归给同一行动者。

### 案例二：同一 Agent 定义部署到测试和生产

软件与配置模板相同，部署租户、秘密、网络和政策不同。定义身份可以相同，部署身份和允许能力必须不同。

反例：因为版本号相同，测试环境的授权决定和证据被生产复用。

### 案例三：一个工作负载身份覆盖多个副本

弹性服务的三个副本共享工作负载身份。每个副本仍需自己的启动、软件闭包、环境、会话和调用关联。

反例：事故日志只写服务身份，无法判断哪个副本使用了陈旧策略。

### 案例四：实例重启但名称不变

编排器用同一名称重建实例。新实例重新取得凭据并建立新 Drase 会话；旧 Dromen、预算、幂等状态和实时句柄不复活。

反例：把稳定实例名当成连续进程身份，恢复旧能力。

### 案例五：有效工作负载凭据没有当前动作权限

工作负载通过 SVID 或等价机制认证，但 AUT 政策只允许读取。写入动作被拒绝；认证事实保持有效。

反例：因为证书有效且服务属于生产域，所以自动允许写入。

### 案例六：服务账户代表员工调用企业 MCP

企业 IdP 允许员工访问服务器，客户端后台工作负载实际发起工具调用。记录同时保留员工、客户端工作负载、运行实例、授权决定和具体工具动作。

反例：只记录员工邮箱，使自动系统和中间委托从责任链消失。

### 案例七：handoff 到子 Agent

上游 Agent 把摘要任务交给子 Agent。子 Agent 的工作负载与实例成为实际行动者，被代表主体和原始请求仍保留；权限、预算和期限只能收窄。

反例：控制权转移后，子 Agent 继承上游的全部凭据与授权。

### 案例八：模型路由发生变化

控制面把候选生成从一个模型切换到另一个模型。若 Dromen 把模型与配置列为实质环境绑定，旧会话失效；否则也必须在 Iknem 中保存实际模型与已知限制。

反例：因为 Agent 产品名未变，模型漂移被日志隐藏。

### 案例九：令牌主体正确但持有者错误

攻击者取得可重放 bearer token。令牌中的主体没有变化，实际持有者与调用环境已经不符合绑定。适配器拒绝或触发撤销责任。

反例：只验证签名和 `sub`，不核对受众、期限、发送者约束与实际调用者。

### 案例十：一次调用因重试产生两个外部效果

同一高层请求产生两个调用尝试。两次尝试拥有不同调用身份，共享显式幂等关系；不能因请求主体相同而合并已经发生的效果。

反例：日志以 Agent 名聚合，第二次写入被误认为第一条记录的重复展示。

### 案例十一：提示注入没有改变认证身份

工作负载身份、用户会话和凭据都有效，但不可信工具内容诱导了越权动作。身份链可以完整，授权输入和意图来源仍然无效。

反例：认证、签名和不可变日志齐全，因此认定动作受用户授权。

### 案例十二：责任关系尚不能确定

日志证明某实例发起调用，也证明它收到一个用户请求，但缺少把该精确动作绑定到用户授权的关系。系统可以记录技术行动者，不得填补“用户负责”结论。

反例：为了让审计表完整，默认把登录用户写成所有下游动作的决定主体。

## 威胁与失败责任

| 威胁 | 典型错误 | 首要责任 | 预期处理 |
| --- | --- | --- | --- |
| 模型身份洗白 | 模型名替代 Agent 与行动者 | ID、IKN | 分开模型、定义、实例与调用身份 |
| 产品名洗实例 | “assistant”或 Agent Card 名替代精确副本 | ID、DRO | 绑定部署、运行实例与会话 |
| 共享工作负载模糊 | 多副本共享身份却无法定位动作 | DRO、ADP、IKN | 每次调用绑定具体实例或可验证等价关联 |
| 凭据洗权 | 有效令牌或 SVID 成为业务授权 | AUT、ADP | 独立求值对象、动作、目的与范围 |
| subject/actor 混淆 | 被代表主体覆盖实际行动者 | AUT | 保留双方和完整委托链 |
| 企业身份泛化 | 组织接入授权扩张为任意工具权限 | AUT、ADP | 逐动作求交并绑定受众与资源 |
| 重启复活 | 同名实例继承旧 Dromen 和句柄 | DRO | 新实例、新会话、重新验证 |
| 配置漂移隐藏 | Agent 名不变但软件、提示、策略或模型变化 | ID、DRO | 实质变化建立新身份或使会话失效 |
| bearer 重放 | 令牌主体正确但实际持有者错误 | AUT、ADP | 发送者约束、受众、期限、撤销与重放检查 |
| 委托链截断 | handoff 隐藏中间 Agent | AUT、IKN | 有限无环并逐级记录实际行动者 |
| 调用聚合洗白 | 多次尝试按 Agent 名合并 | ADP、ID、DIA | 独立调用身份与幂等关系 |
| 注入洗责任 | 认证身份有效被当作意图有效 | TEXT-IDENTIFIER、AUT | 保存输入来源与授权语境，拒绝越权动作 |
| 日志自证 | Agent 自报身份和行为成为完整证据 | IKN、DIA | 独立完整性、范围、来源与覆盖评估 |
| 责任过推 | 技术行动者自动成为法律或最终责任主体 | AUT、具名制度权威 | 保留未知关系，不生成超范围结论 |

## 候选义务的唯一主归属

| 候选义务 | 唯一主归属 | 接受后可能补强的位置 |
| --- | --- | --- |
| 每类身份使用显式身份域，不以显示名合并 | ID-CTX-001、ID-REF-001 | 身份场景与威胁模型 |
| Agent 实际行动者绑定工作负载与运行实例 | AUT-PRN-001 | 权威场景与公开授权说明 |
| 委托关系绑定精确活动并保留中间行动者 | AUT-DEL-001 | 权威场景与 Agent 边界指南 |
| 会话绑定 Agent 定义、实例、模型与环境的实质版本 | DRO-SUB-001、DRO-ENV-001、DRO-IMM-001 | Dromen 场景和未来 Profile |
| 工作负载凭据不替代动作授权 | AUT-SEP-001、ADP-PEE-001、ADP-SEC-001 | 适配与授权威胁模型 |
| 每次尝试拥有独立调用身份与幂等关系 | ADP-INV-001、ADP-RET-001 | 具体协议 Profile |
| trace 指明记录主体层次、来源与观察范围 | IKN-SUB-001、IKN-PRO-001 | Iknem 场景与诊断语境 |
| 身份、委托或实例绑定缺失时关闭失败 | DIA-CTX-001、AUT-DEC-001 | 诊断目录与未来错误 Profile |

现有责任能够承载这些义务，因此当前不创建 Agent 身份标准、Agent 目录、责任制品或身份服务。只有真实跨实现消费者证明某项身份拥有独立物理交换需求，并且现有 ID、AUT、DRO、ADP 与 IKN 无法唯一承担时，才重新讨论新标准。

## 失败域与结果隔离

| 发生事实 | 直接结论 | 允许的后果 | 绝不能自动推出 |
| --- | --- | --- | --- |
| 模型版本无法精确定位 | 模型身份范围不足 | 候选拒绝、限制声明或人工升级 | Agent 未认证、目标 `unmet` |
| Agent 定义或配置漂移 | 旧定义绑定不再适用 | 新身份或旧会话失效 | 新定义错误、最终拒绝 |
| 工作负载认证失败 | 不能确认当前软件主体 | 拒绝连接或调用 | 被代表用户无权、目标不成立 |
| 运行实例无法定位 | 责任链与会话绑定不足 | 阻止高风险动作或降低可声明范围 | 工作负载身份虚假 |
| 凭据过期或受众不匹配 | 当前凭据不可用 | 重获凭据或拒绝调用 | 原授权决定自动撤销、目标 `unmet` |
| 委托链断裂 | 无法证明代表关系 | `deny`、`defer` 或人工升级 | 实际行动者不存在、内容为假 |
| 调用身份重复或不明 | 无法安全归并效果 | 查询外部状态、停止盲重试 | 已回滚、已成功或已失败 |
| trace 缺失实例或输入来源 | 观察范围不足 | Iknem 覆盖降低或保持未知 | 动作未发生、责任已经确定 |
| 身份链完整但输入被注入 | 认证仍可有效，授权输入不可信 | 拒绝越权动作并记录来源 | 用户授权、Agent 有恶意或目标失败 |

## 未来验证证据

当前没有 Agent 身份目录、工作负载身份集成、Drasor、协议适配器或运行时。本提案只能形成资料审查问题，不能证明任何身份或审计机制已经实现。未来进入获准代码范围后，至少需要：

1. 固定模型、Agent 定义、部署、工作负载、运行实例、会话、凭据和调用的身份域及变化规则；
2. 用多副本、重启、轮换、撤销、跨租户、handoff、impersonation、重试和提示注入反例验证责任链；
3. 证明日志与 Iknem 能定位实际行动者、代表关系、精确活动和输入来源，同时不保存原始秘密；
4. 对同名不同实例、同实例不同配置、同主体不同凭据和同请求多次尝试进行负向比较；
5. 在协议、策略、模型、部署和身份提供方漂移后验证旧 Dromen 关闭失败且历史记录不被改写。

一次单点登录、证书验证、SPIFFE 演示、trace 查询或审计报表都不能代替这组证据。

## 哲学与语言边界

维特根斯坦《逻辑哲学论》3.203 说：“名称意谓对象。对象是它的意义。”工程上的限制性启发是：一个名称只有在确定的身份域和关系中才能指向对象；相同显示名不能让模型、软件定义、工作负载、进程和主体成为同一对象。

这条命题不能决定 Agent 身份 schema、认证协议、委托政策、法律责任或审计保留。Noemion 只用它提醒开发者：关系必须由结构显示，不能让 `agent_name`、`user` 或 `model` 一个字符串承担全部身份与责任。

本主题全部使用“模型身份”“Agent 定义身份”“工作负载身份”“运行实例”“实际行动者”和“被代表主体”等直白职责短语，不创建需要读音验证的新造词、缩写或产品名。若未来外部标准形成稳定术语，仍须先核对其精确职责和口头可读性，再决定是否进入公开词表。

## 决定与停止条件

进入规范前仍需解决以下问题：

1. 是否接受九个身份与关系层次的分离；
2. 是否要求实际行动者始终绑定工作负载与运行实例，而不只保存 Agent 产品名；
3. 是否要求委托绑定精确活动并保留每个中间行动者；
4. 是否把企业身份、工作负载凭据和协议身份都限制为 AUT-CORE 的输入，而非完整授权；
5. 是否把上述唯一义务分别补回 ID、AUT、DRO、ADP、IKN、DIA 与 TEXT-IDENTIFIER 的场景和威胁模型。

在形成正式 ADR 前，本提案保持非规范，不修改 CORE、ADR、登记、向量、对象、接口或实现路线。形成决定后也应优先补强现有责任；若没有真实多实例、跨 Agent 或跨协议消费者，不为架构对称建立新的身份对象或目录。

## 参考资料与采用限制

- [NIST NCCoE · Accelerating the Adoption of Software and AI Agent Identity and Authorization](https://www.nccoe.nist.gov/sites/default/files/2026-02/accelerating-the-adoption-of-software-and-ai-agent-identity-and-authorization-concept-paper.pdf)：采用人类与 Agent 区分、授权、委托、日志和数据流问题；明确它仍是概念论文，不当作正式实践指南。
- [NIST AI 800-5](https://www.nist.gov/publications/summary-analysis-responses-request-information-regarding-security-considerations-ai)：采用传统安全原则需要适配 Agent 风险的有限结论；不把公众意见摘要当成身份 schema。
- [SPIFFE Identity and Verifiable Identity Document](https://spiffe.io/docs/latest/spiffe-specs/spiffe-id/)与[核心概念](https://spiffe.io/docs/latest/spiffe-about/spiffe-concepts/)：采用信任域、工作负载身份、SVID 与属性语义边界；不把工作负载身份提升为实例、委托或授权。
- [RFC 8693 OAuth 2.0 Token Exchange](https://www.rfc-editor.org/rfc/rfc8693.html)：采用 subject、actor、delegation 与 impersonation 分离；不采用 token 结构作为 Noemion 物理格式。
- [W3C PROV-DM](https://www.w3.org/TR/prov-dm/)：采用 SoftwareAgent、Activity、Association、Attribution 与 Delegation 的关系分离；不让 provenance 自动决定授权或法律责任。
- [MCP Enterprise-Managed Authorization](https://modelcontextprotocol.io/extensions/auth/enterprise-managed-authorization)：采用组织级身份提供方、接入政策和集中撤销；不把服务器接入变成逐动作授权或实际行动者身份。
- [GNU Coreutils · `id`](https://www.gnu.org/software/coreutils/manual/html_node/id-invocation.html)：采用真实身份、有效身份、组与安全上下文分别可见的表达纪律；不把 Unix 进程身份模型直接移植为 Agent 身份标准。
