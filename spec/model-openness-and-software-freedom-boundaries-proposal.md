# Noemion 模型开放性与软件自由边界研究提案

状态：非规范研究提案

日期：2026-07-14

结论状态：桌面审查完成；等待用户决定

研究范围：托管服务、代码、模型架构、参数、训练数据说明、训练数据、文档、许可、使用限制、可修改性、可复现性与外部符合性主张

本提案不构成 ADR、CORE 规范、Profile、登记项、法律意见或实现要求，不进入 `registry.json`。它不创建开放模型制品、许可证制品、模型发行格式、合规服务、命令、组件、结果域或哲学专名，也不建立 `OPEN-MODEL-CORE`、`LICENSE-CORE` 或新的 Noemion 模型对象。

## 直接结论

“可以调用”“可以下载”“可以阅读源码”“允许修改”“能够重建”和“符合某个开放定义”是六个不同主张。公开 API 不等于拥有软件或模型；可下载权重不等于自由软件或 Open Source AI；公开仓库、模型卡和 Apache-2.0 标记也不能自动覆盖未来模型参数、数据、第三方依赖、外部服务或商标。

Noemion 当前仓库的源码和文档在文件未另行说明时采用 Apache License 2.0。这个事实只约束实际落入该许可范围的仓库材料。当前没有模型权重、训练数据、推理服务、训练平台或模型发行物，因此网站不能声称 Noemion 已经提供开放权重、开放模型、自由软件模型栈或可复现模型。

未来每个发布主张必须先列出实际对象及各自条款，再选择准确短语：

- 只有远端调用条件时，写“托管 API 可用”；
- 参数可以取得时，写“模型参数可下载”，并紧接许可证与使用限制；
- 只有部分源文件可见时，写“源码可见”，不能自动写“自由软件”或“开源软件”；
- 只有逐项满足 GNU 自由软件或 OSI Open Source AI 定义时，才引用相应定义和版本；
- 只有独立重建证据支持时，才写“可复现”，并说明复现的是环境、字节、行为还是研究结论。

托管访问还必须继续区分软件权利与实际服务控制。谁运行计算、哪些数据离开控制域、服务端隐藏什么、状态能否导出以及停服或切换时损失什么，见[托管人工智能服务与用户控制边界研究提案](hosted-ai-service-and-user-control-boundaries-proposal.md)。

## 至少十二种事实必须分开

| 事实 | 回答的问题 | 最低披露 | 不能自动推出 |
| --- | --- | --- | --- |
| 托管服务访问 | 谁能在什么条件下调用什么服务 | 供应者、端点或产品、区域、账户条件、价格、用途、数据条款、版本与撤回 | 用户拥有软件、模型或修改权 |
| 模型参数 | 哪些最终或中间参数可以取得 | 精确身份、格式、大小、来源、许可、限制、完整性与撤回 | 训练过程、数据或代码开放 |
| 模型架构 | 网络结构和配置怎样表达 | 精确代码或配置、版本、许可、依赖与参数绑定 | 能重建相同模型或行为 |
| 推理代码 | 怎样从输入和参数得到输出 | 源码、构建依赖、tokenizer、运行配置、许可与限制 | 训练代码或参数已提供 |
| 训练与评测代码 | 怎样处理数据、训练、验证和评测 | 完整源码、参数、配置、依赖、许可与运行条件 | 实际训练数据可得、结果可复现 |
| 训练数据说明 | 数据来自哪里、怎样选择和处理 | 来源、范围、特征、取得、选择、标注、过滤和不可共享部分的说明 | 数据本身可以取得或合法使用 |
| 训练与评测数据 | 哪个精确数据快照可以取得 | 身份、来源、许可、同意、隐私、范围、截止点与撤回 | 数据代表性、正确性或无污染 |
| 文档 | 使用、局限、评测和风险怎样说明 | 模型卡、数据卡、技术报告、版本、许可与证据范围 | 对代码、参数或数据授予权利 |
| 许可与其他条款 | 每个对象允许谁做什么 | 对象、版权或其他权利主体、许可证版本、附加条款、例外与冲突 | 技术完整、来源可信或用途安全 |
| 首选修改形式 | 使用者实际怎样研究和改变系统 | 参数、代码、数据说明、配置、工具链与必要文档 | 成本可承受或逐字节可复现 |
| 可复现性证据 | 独立主体能复现什么 | 输入闭包、环境、活动、输出身份、比较方法、差异与失败 | 获得许可、开放符合性或用途适合 |
| 外部分类与决定 | 谁依据哪个定义、版本和证据作出什么主张 | 定义、版本、检查范围、决定者、日期、例外、到期与复核条件 | 永久合规、风险可接受、`met` 或 `accepted` |

这些事实可以由外部许可证清单、模型仓库、数据目录、SBOM、AI BOM、模型卡和评测报告分别保存。它们不需要压成新的 Noemion “开放状态”，也不能用一个 `open`、`public`、`community`、`research` 或 `Apache-2.0` 标签替代。

## 术语只描述已经证明的范围

| 公开短语 | 允许表达 | 必须避免的越界 |
| --- | --- | --- |
| 托管 API 可用 | 用户可按明确服务条款远端调用 | 写成已取得软件、参数或修改权 |
| 模型参数可下载 / 开放权重 | 精确参数可被下载；后接许可和限制 | 自动写成 Open Source AI、自由软件或可复现模型 |
| 源码可见 | 指定源文件可以阅读 | 自动声称允许任意使用、修改和再分发 |
| 开源软件 | 精确软件在 OSI 批准许可证下发布 | 把模型参数、数据、服务和商标一并纳入 |
| 自由软件 | 用户获得 GNU 所述运行、研究、修改、再分发原版和修改版的自由 | 把“免费”“公开仓库”或“非商业可用”当作自由 |
| Open Source AI | 精确 AI 系统或组件符合具名版本的 OSI 定义 | 只因权重可得、代码公开或供应商自称而采用 |
| 开放科学模型 | 依据具名框架披露完整研究链和许可范围 | 与 OSI Open Source AI、自由软件或可复现性互换 |
| 可复现模型 | 在声明条件下有独立重建与比较证据 | 只凭固定种子、容器、模型卡或相近分数声称 |

“开放模型”本身过于含糊。面向首次读者时，应优先列出“参数、推理代码、训练代码、数据说明、数据、文档和各自许可”，然后才引用具名分类。若页面必须使用“开放模型”，同一句必须说明采用哪个定义、版本和覆盖对象。

## GNU 软件自由适用于软件，不自动解决模型全栈

[GNU 自由软件定义](https://www.gnu.org/philosophy/free-sw.en.html)要求用户能够为任何目的运行程序、研究和修改程序、再分发副本，并分发修改版；研究和修改以获得源代码为前提。Noemion 采用这四项自由作为软件许可表述的用户权利门禁，而不是把“自由”降为零价格或公开可读。

模型系统同时包含软件、参数、数据、文档和服务。GNU 定义可以直接审查推理代码、训练代码和工具，但不能仅凭代码自由推出参数、数据或托管服务也处于相同权利边界。未来必须对每个对象分别核对：

1. 用户取得的究竟是副本还是临时服务访问；
2. 哪一种形式是实际修改该对象所需的首选形式；
3. 使用、研究、修改和再分发是否对所有用户和用途成立；
4. 非商业、研究限定、用途政策或地域限制是否使某项自由不成立；
5. 必需依赖、tokenizer、训练框架和服务端组件是否改变实际可修改范围。

GNU 的价值主张与 OSI 的工程定义不能静默互换。页面可以同时报告“该软件采用 OSI 批准许可证”和“该软件满足 GNU 自由软件四项自由”，但必须分别给出依据；其中一项也不能自动扩张到模型全栈。

## OSI Open Source AI Definition 1.0 的采用边界

[Open Source AI Definition 1.0](https://opensource.org/ai/open-source-ai-definition)把使用、研究、修改和分享的自由应用到完整 AI 系统及其离散元素，并把“首选修改形式”拆为三组：

- 足够详细的训练数据说明，包括来源、范围、特征、取得、选择、标注、处理和过滤；
- 用于处理数据、训练、验证、评测与运行系统的完整代码；
- 模型参数及必要配置。

该定义要求数据说明，不等于要求所有训练数据都能够公开。Noemion 不把“训练数据未公开”单独写成必然不符合，也不把“有数据卡”写成数据说明已经充分。符合性必须依定义 1.0 的具体条款逐项检查，并保存不可共享数据的说明质量、可获得数据列表、代码完整性、参数条款与实际修改任务的反例。

OSI 还明确指出，“Open Source models”和“Open Source weights”必须包括推导参数所需的数据说明和代码。因此 Noemion 在没有完整检查前只使用“模型参数可下载”或“开放权重”这种事实短语，不把它升级成“开源模型”或“开源权重”。

## 开放完整性、许可和复现是三条证据轴

[Linux Foundation Model Openness Framework Specification](https://lfaidata.foundation/resources/whitepapers/)用 Open Model、Open Tooling Model 和 Open Science Model 三层检查模型架构、参数、代码、数据和文档的发布完整性及相应开放许可。它适合回答“发布包还缺哪些组成”，不能替代 OSI 或 GNU 的定义，也不能替代逐项法律与权利审查。

截至 2026 年 7 月，模型许可仍在演进；Linux Foundation 已发布 OpenMDW 1.1 等面向模型分发的许可方案。Noemion 把这视为“单个软件许可证不足以描述整个模型分发”的趋势证据，而不是自动采用决定。任何未来选择都必须比较适用对象、版权、专利、数据库权、输出、附加材料、兼容性、司法范围和撤回条件，并接受专业法律复核。

开放完整性、许可与复现分别回答：

| 证据轴 | 核心问题 | 典型证据 |
| --- | --- | --- |
| 发布完整性 | 必要组成是否已经提供 | 对象清单、精确身份、模型卡、数据卡、代码与参数链接 |
| 权利与许可 | 用户对每个组成可以做什么 | 许可证原文、权利主体、例外、兼容性与限制审查 |
| 技术复现 | 独立主体能否重建声明结果 | 输入闭包、活动记录、环境、输出比较与差异归因 |

三条轴都通过，也只支持具名范围内的开放与复现主张；它们不证明模型安全、公平、无侵权、适合目标用途或已被 Noemion 的最终权威接受。

## 安全来源与开放主张不能互相替代

[NIST AI 600-1](https://doi.org/10.6028/NIST.AI.600-1)要求在生成式 AI 风险管理中处理价值链、组件集成、来源、第三方资源、模型选择和持续监测。[NIST SP 800-218A](https://doi.org/10.6028/NIST.SP.800-218A)又把模型、训练库、框架、数据与开发活动纳入 AI 安全开发实践。

Noemion 采用这些资料约束来源、供应链与风险证据，但不让 NIST 记录替代许可证分类：

- SBOM 或 AI BOM 可以列出对象，不授予使用或修改权；
- 摘要和签名可以绑定字节与陈述，不证明许可证兼容或开放符合性；
- 模型卡可以说明用途和局限，不是许可证；
- 来源记录可以支持审计，不证明数据取得、同意与再分发权成立；
- 开放许可可以授予权利，不证明依赖完整、制品未投毒或用途安全。

## 支持案例与反例

1. 某模型只有付费 API。正确表述是“托管 API 可用”；用户没有取得服务端代码、参数或修改形式。
2. 权重可下载，但许可证限于非商业研究。可以说明“参数可下载且限非商业研究”，不能称其为自由软件或按 OSI 定义的 Open Source AI。
3. 权重采用宽松条款，但训练与推理代码缺失。开放权重事实成立；完整模型、首选修改形式和复现主张仍不成立。
4. 推理代码采用 Apache-2.0，模型参数使用另一份自定义许可。两者必须分别列出；仓库根许可证不能覆盖参数。
5. 仓库公开了训练代码和模型卡，没有任何文件说明模型卡的许可。可读不等于允许再分发或修改，文档许可状态必须写成未知或另行核对。
6. 数据不能因隐私和合同公开，但提供完整来源、特征、选择、处理和可取得数据列表。是否符合 OSI 1.0 取决于数据说明是否足够使熟练人员建立实质等价系统，不能仅因原始数据不公开直接否定。
7. 数据卡只有数据集名称和规模。它不能替代来源、取得、选择、标注、过滤和不可共享部分说明。
8. 训练数据、代码和权重都公开，但部分依赖只提供封闭服务。必须说明依赖怎样限制研究、修改和重建，不能声称完整首选修改形式已经提供。
9. 两个团队用公开材料得到相近基准分数但不同权重。这支持有限行为复现，不支持逐字节模型复现。
10. 两次训练得到逐字节相同参数。只支持声明条件下的输出复现；不证明许可、数据资格、安全或用途接受。
11. 一个外部工具给出 MOF 等级。该等级必须保存工具版本、输入清单、缺失对象和复核日期；不能自动成为 OSI、GNU 或法律结论。
12. 模型发布页写“open source”，却没有定义版本或对象清单。Noemion 适配器只能保留供应商原始声明，并将本地分类标为待核对。
13. 公开仓库后来删除权重或变更条款。历史取得事实、当前可用性、既有副本权利与新分发条件必须分别记录。
14. Apache-2.0 仓库引用第三方数据、字体或模型。第三方对象继续受自己的条款约束；聚合页面不能把仓库许可证扩张到这些对象。
15. 模型卡声称“responsible use only”。这可能是使用建议、可接受使用政策或合同限制，必须确定其法律位置；不能既忽略它，也不能未经审查把它当成模型许可。
16. 托管 API 返回可导出的适配器参数。服务访问、用户输入权利、派生参数身份、参数许可证、基础模型限制和再分发条件必须分别核对。

## 威胁到失败责任的映射

| 威胁 | 失败方式 | 唯一主责任 | 不能洗成 |
| --- | --- | --- | --- |
| API 开放洗白 | 远端可调用被写成系统开放 | 服务与发布治理、ADP | 用户拥有和可修改 |
| 权重开放洗白 | 参数可下载被写成 Open Source AI | 发布治理、IKN | 完整首选修改形式 |
| 仓库许可外溢 | 根许可证被扩张到参数、数据和第三方材料 | 发布治理、AUT | 全栈统一授权 |
| 源码可见洗白 | 可阅读代码被写成允许使用、修改和再分发 | 发布治理、AUT | 自由软件或开源软件 |
| 模型卡代替许可 | 文档说明被当作权利授予 | 发布治理、AUT | 合法再分发 |
| 数据说明不足 | 不可共享数据只留下名称或营销摘要 | 数据治理、IKN | OSI 1.0 数据说明充分 |
| 依赖闭包缺失 | 必需 tokenizer、服务或工具未披露 | ID、ADP、供应链治理 | 完整可修改形式 |
| 用途限制隐藏 | 非商业、研究或可接受使用限制未随对象显示 | AUT、发布治理 | 可用于任何目的 |
| 分类版本漂移 | 定义、检查器或许可证更新后沿用旧结论 | IKN、发布治理 | 永久符合 |
| 复现洗白 | 容器、固定种子或相近分数被写成模型可复现 | ID、IKN、训练治理 | 逐字节重建 |
| 开放替代安全 | 开放许可被当成未投毒、无漏洞或用途安全 | 安全治理、IKN | 风险已接受 |
| 安全替代自由 | 签名、SBOM 或安全审计被当成用户权利 | AUT、发布治理 | 自由或开放符合性 |

## 候选责任的唯一主归属

| 义务 | 主归属 | 辅助归属 | 不新增的对象 |
| --- | --- | --- | --- |
| 软件、参数、数据、文档和服务的精确身份 | ID | IKN、ADP | Noemion 开放模型制品 |
| 每个对象的主体、动作、目的、受众和限制 | AUT | IKN、ADP | 许可证令牌格式 |
| 供应商 API、模型仓库和外部分类语境 | ADP | ID、AUT、DIA | 开放模型适配 CORE |
| 许可证、来源、检查清单和外部结论记录 | IKN | ID、AUT | 合规证书制品 |
| 页面实际显示与供应商原始声明 | TEXT-IDENTIFIER | IKN、ADP | 开放性标签对象 |
| 缺失许可、冲突、漂移和不可解析身份诊断 | DIA | ID、AUT、ADP | 新结果域 |
| 模型发行、用途与最终接受 | 具名发布权威 | AUT、IKN | 自动开放决定服务 |

开放性不进入 END 或 SYN 的目标身份，不进入 Dromen 伪装成会话权限，也不产生 `met`、`sufficient` 或 `accepted`。如果未来某个目标要求使用特定开放材料，Endem 只能表达可观察的目标关系和判据；实际许可证证据与决定仍由外部权威和 Iknem 保存。

## 发布清单与失败关闭

未来任何 Noemion 模型相关资源在公开前至少要给出：

1. 发行包包含和不包含的对象清单；
2. 每个对象的精确身份、来源、权利主体和许可证版本；
3. 附加使用条款、用途政策、地域或账户限制；
4. 参数、推理代码、训练代码、数据说明、数据和文档的可得状态；
5. 修改所需却缺失或封闭的依赖；
6. 采用 GNU、OSI、MOF 或其他分类时的精确版本和逐项检查；
7. 可复现主张的对象、条件、独立执行和比较结果；
8. 安全、隐私、供应链和用途证据与许可结论的分离；
9. 撤回、条款变化、镜像、长期归档和复核时间；
10. 具名发布权威、法律复核范围和仍未关闭的问题。

无法确定对象或条款时，页面关闭失败：写“许可状态待核对”“参数身份未知”或“仅供应商自述”，不猜测“开放”。许可证冲突、必需对象缺失或用途限制与公开主张矛盾时，不发布相应符合性标签。

## 术语和读音边界

本提案故意使用“托管 API”“模型参数”“训练数据说明”“软件自由”“发布完整性”“许可”和“可复现性”等普通职责短语，不创建缩写、短词或希腊词根。

这些职责已有成熟行业用语，且当前不存在独立 Noemion 物理对象。新增专名会隐藏最需要读者看见的对象差异，并让口头讨论中的“开放”继续承担过多含义。未来页面第一次出现“开放权重”“自由软件”或“Open Source AI”时，都必须紧接直白定义或具名外部标准；不靠专名暗示权利。

## 接受后的落点与停止条件

若用户接受责任分配，后续只把唯一义务归还现有页面和规范责任：

1. 下载页逐项展示对象、许可、限制、完整性与撤回；
2. ID 只在真实模型资源出现后增加对象和发行包精确身份案例；
3. AUT 增加许可证、服务条款、用途限制和再分发动作的适用说明；
4. IKN 增加外部定义、版本、清单、决定者和复核日期的有范围记录；
5. ADP 保留供应商原始开放声明，并与本地核对结论分开；
6. TEXT-IDENTIFIER 约束页面不得省略限制或把供应商营销词升级成本地事实；
7. DIA 只在出现真实消费者后登记有限的许可缺失、冲突和版本漂移概念；
8. END、SYN 与 DRO 不新增开放状态、许可证字段或模型发布权限。

以下任一条件成立就停止建设 Noemion 自有开放性对象或分类器：

- 外部成熟定义、清单与许可证工具能够表达所需责任；
- 没有真实模型、数据、代码或服务发行消费者；
- 项目无法取得逐项权利信息或专业法律复核；
- 分类不能保存定义版本、证据范围、例外和复核条件；
- 用户没有明确开启相应代码或发行范围。

## 当前决定边界

本提案现在只形成六项候选结论：

1. 托管 API、参数、代码、数据说明、数据、文档和许可证必须分别披露。
2. “开放权重”只描述参数可得事实，不自动等于 Open Source AI、自由软件或可复现模型。
3. GNU 软件自由、OSI Open Source AI、MOF 发布完整性、许可证法律审查和技术复现是不同判断。
4. Apache-2.0 只覆盖实际置于其下的仓库材料，不自动扩张到未来模型全栈和第三方对象。
5. 模型卡、SBOM、签名、来源记录和外部分类都不能互相替代，也不能产生 `met` 或 `accepted`。
6. 当前不创建新 CORE、Profile、登记、制品、组件、命令或专名；只改善公开表述和未来发布门禁。

用户接受这些结论后，项目再决定是否把唯一义务补入现有规范案例。接受之前，本提案保持非规范研究资料。
