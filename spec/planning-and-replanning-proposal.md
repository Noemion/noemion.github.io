# 目标、计划与重规划边界研究提案

状态：非规范研究提案  
日期：2026-07-13  
适用阶段：规范、资料与网站设计阶段

## 直接结论

Noemion 当前不应再创造一个“计划制品”、计划格式、计划组件或计划命令。目标、计划、外部任务和满足判断属于四种不同责任：

1. Endem 或 Synem 固定要达到什么，不固定模型准备怎样做。
2. Dromen 固定一次会话允许在什么边界内做，不保存可变计划。
3. 计划是 Drasor 外侧控制平面在该边界内维护的会话状态；模型只能提出不可信计划候选。
4. MCP Task、A2A Task、工具调用、handoff 和本地步骤只描述工作怎样推进，不能直接成为 `met`、`accepted` 或 Iknem 充分覆盖。
5. 重规划可以改变步骤、顺序和分工；只有目标、政策、环境、能力、预算或证据责任发生实质变化时，才必须回到相应身份或会话边界。

本提案不构成 ADR、CORE 规范、内容 Profile 或实现要求。它不创建新制品、文件格式、扩展名、命令、组件、结果域或稳定接口，也不进入 `registry.json`。当前只判断现有 END、SYN、DRO、IKN、AUT、ADP、DIA 与 TEXT-IDENTIFIER 责任是否足以承载未来规划行为。

## 为什么这是底层问题

Agent 系统越来越常见地交替推理、调用工具、请求补充信息、委托其他 Agent 和依据观察重规划。若项目只保存一段用自然语言表达的目标和最后的“完成”状态，会出现三类结构性混淆：

- 把模型生成的步骤当成目标本身，使重规划无意改变 Endem 身份；
- 把外部任务或步骤的 `completed` 当成目标 `met`，绕过 `krin` 和证据责任；
- 把“为完成目标需要做某事”误写成独立目标，造成 Endem 数量随实现策略膨胀。

Noemion 的价值不在于替模型设计更漂亮的计划语言，而在于让任何规划器都不能越过目标、授权、能力、证据和最终决定边界。

## 五个对象不能混为一谈

| 对象或状态 | 回答的问题 | 可以变化什么 | 不能推出什么 |
| --- | --- | --- | --- |
| Endem / Synem | 期望终态和精确目标闭包是什么 | 只有形成新的内容身份或闭包身份后才能改变 | 具体步骤、工具、Agent、完成时间或执行成功 |
| Dromen | 本次会话允许在什么主体、政策、环境、能力、预算和证据责任内行动 | 只读；实质漂移使旧会话失效 | 计划正确、目标满足或权限可以扩大 |
| 会话计划 | 当前准备以哪些有界步骤、顺序、分支和检查点推进 | 可在 Dromen 上限内重排、替换、细化或放弃 | Endem 身份改变、步骤获权或目标已满足 |
| 外部 Task / 工具调用 / handoff | 某个外部工作单元处于什么生命周期 | 按精确协议和对端更新 | 本地会话终止、目标满足、Iknem 有效或最终接受 |
| Iknem 与最终决定 | 哪些观察在什么范围内支持什么主张，谁作出决定 | 新记录和新决定保持各自身份 | 计划文本、任务数量、签名或执行轨迹自行成为充分证据 |

“计划”在这里是普通工程词，不是新的哲学专名或正式对象名。只有未来出现跨会话引用、独立签名、长期保存或跨实现交换的真实消费者，才有理由重新讨论物理 Profile；目前没有这种证据。

## 计划步骤何时不是 Endem

一个步骤默认只是达到目标的候选手段。它同时满足以下条件时，才可能拆成独立 Endem：

1. 步骤本身描述可独立判断的期望终态，而不是动作名称或内部控制流；
2. 它有独立 `krin`、证据责任和失败责任；
3. 它可能被不同上层目标复用，或需要独立授权、接受、撤销或审计；
4. 省略它会改变外部可观察承诺，而不只是换一种实现路径；
5. 它与其他目标的组合关系可以由 Synem 精确表达。

“搜索三个来源”“调用付款工具”“让第二个 Agent 检查”“重试一次”通常不是 Endem。若真正承诺的是“退款已进入指定账户”或“发布包通过独立安全审查”，这些才可能成为独立终态。

## 候选责任分配

| 规划问题 | 唯一主归属 | 其他域只引用什么 | 当前不新增什么 |
| --- | --- | --- | --- |
| 目标与计划分离 | END-CORE / SYN-CORE | DRO 绑定精确主体；IKN 绑定观察主体 | 计划字段、计划制品、目标步骤语法 |
| 计划候选能否成为动作 | AUT-CORE | DRO 提供能力上限；ADP 描述外部调用 | 模型自授权、计划签名即授权 |
| 步骤调用与外部状态 | ADP-CORE | DIA 保存来源错误；IKN 保存有范围观察 | 把 MCP/A2A 状态写进目标结果域 |
| 会话内计划与重规划上限 | DRO-CORE | AUT 处理新增授权；TEXT-IDENTIFIER 处理模型实际输入 | 可恢复 Dromen、跨会话权限、可变 Dromen |
| 计划和行动溯源 | IKN-CORE | DIA 保存诊断；TEXT-IDENTIFIER 保存实际输入边界 | 全量思维链强制披露、新证据对象 |
| 步骤失败、停止和升级 | DIA-CORE 与 ADR-0015 | DRO 负责预算与失效；AUT 负责人工决定 | 新的通用 `success/failure` 结果域 |

责任分配的关键不是让每个域都复制“计划”字段，而是让每项义务只有一个主条款。若未来进入规范，候选条款应分别回到现有规范；只有跨域引用仍无法形成唯一失败责任时，才讨论新的横切规范。

## 来自 GNU Make 的可用边界

GNU Make 把 target、prerequisite 与 recipe 分开：target 表示要更新的对象，prerequisite 说明依赖，recipe 说明怎样更新。这为 Noemion 提供了有用的反混淆框架，但不能直接复制其语义。

| GNU Make 机制 | 可借鉴的不变量 | Noemion 必须拒绝的照搬 |
| --- | --- | --- |
| target / prerequisite / recipe 分离 | 期望对象、前置关系和实现步骤不是同一种信息 | 把文件时间戳、recipe 退出码或文件存在直接当 `krin` |
| 普通与 order-only prerequisite 分离 | “会影响结果身份”和“只约束执行顺序”的依赖必须区分 | 把所有先后步骤都变成 Synem 成员或 Endem 内容 |
| phony target | 动作名可以组织入口，但不等于产生同名制品 | 把 `clean` 一类过程动作当成已达到的终态 |
| `--keep-going` | 一个分支失败后可以在预算内继续收集其他诊断 | 其他分支完成后把整体目标判为成功 |
| 默认 goal 与显式 goal | 调用方必须明确最终请求；内部规则顺序不能偷偷改变目标 | 依赖首条规则、目录顺序、隐式规则或环境搜索选择语义目标 |

GNU Make 自己允许隐式规则、时间戳判断、环境影响和忽略错误。这些是构建工具的具体策略，不适合定义以自然语言表达的目标怎样成立或获得行动授权。

## 当前 Agent 技术给出的信号

| 来源 | 当前可核对事实 | 对 Noemion 的采用边界 |
| --- | --- | --- |
| ReAct | 推理、行动与观察交替，可用于更新计划和处理异常 | 证明动态重规划是现实需求；不证明模型计划正确、获权或可作为证据 |
| A2A 1.0.0 | Task 是带状态、history 和 artifacts 的工作单元；有 submitted、working、completed、failed、canceled、input-required、rejected、auth-required 等状态 | 作为 ADP 外部状态输入；任何状态都不能直接映射本地满足、决定或会话结果 |
| MCP 2025-11-25 Tasks | Task 是用于轮询和延迟取回结果的持久状态机，但官方仍标为 experimental | 只能作为版本固定的候选适配资料；不能冻结稳定 MCP Task Profile 或把 durability 引入 Dromen |
| OpenAI Agents SDK | tracing 记录模型生成、工具调用、handoff、guardrail 和自定义事件；handoff 可过滤输入并把控制交给另一 Agent | 轨迹有助于调试和溯源，但不是 Iknem 自动充分证明；handoff 仍需 AUT 委托与逐动作能力检查 |
| Agent SDK guardrails | 输入、输出和工具 guardrail 的触发范围不同，handoff 不等于普通工具调用 | 不能用一个“全局安全检查”标签洗白所有步骤；每个执行边界必须说明检查实际覆盖哪里 |

这些资料共同说明 Agent 运行时正把长期任务、委托、工具调用和轨迹变成一等运行概念。Noemion 不应与每个框架竞争这些机制，而应规定它们不能跨越哪些目标与信任边界。

## 支持案例与反例

### 案例一：确定性单步路径

一个规则型目标可以直接通过一次确定性动作形成可比较观察。控制平面可以不建立持久计划，只记录精确动作、输入、输出和 Iknem 责任。

反例：为了架构对称，仍创建“计划文件”和新的签名对象。没有独立消费者时，这只增加身份和撤销负担。

### 案例二：模型提出多步计划

模型建议先读取库存、再选择仓库、最后提交调拨。控制平面逐步核对能力、预算与授权；计划文本保持候选身份。

反例：因为模型输出符合 JSON schema，就把全部步骤视为已授权。结构合法没有授权权威。

### 案例三：观察后重规划

首选仓库库存不足，计划在同一 Dromen 允许的仓库集合、预算和证据责任内改用第二仓库。目标和 Dromen 不变，计划历史保留替换原因。

反例：新仓库不在原受众或能力范围内，系统仍以“只是重规划”为由继续。扩大对象或能力必须获得新授权，并可能开始新会话。

### 案例四：用户澄清改变目标

用户把“本周发货”改成“今天十八点前发货”。这改变时间范围和满足判据，必须形成新的 Endem 内容身份；不能只改计划截止时间。

反例：控制平面保留旧 Endem，却在计划里偷偷提高时限要求。执行记录与目标身份由此分裂。

### 案例五：内部步骤不是独立目标

为验证发布包，系统选择运行两个解析器和一组畸形向量。这些是验证策略；最终承诺是精确发布包在指定规范和环境下通过检查。

反例：每条测试命令都被拆成 Endem，导致更换测试框架也改变目标闭包。

### 案例六：真正的子目标

“完成安全审查”和“获得发布授权”拥有不同权威、证据责任和接受决定，即使常按顺序进行，也应是两个 Endem，由 Synem 组合。

反例：把两者压成计划步骤，最终只保留一个“流程完成”布尔值，丢失独立拒绝和延期状态。

### 案例七：并发分支

三个独立数据源可以并行查询。计划记录分支身份、预算和汇合规则；分支完成顺序不改变目标身份、授权或聚合判据。

反例：最早完成的分支自动成为事实来源，或并发竞态改变最终绑定。

### 案例八：失败后继续收集诊断

一个分支已确定失败，但其他独立分支仍可在总预算内完成，以便形成完整诊断和人工材料。整体结论仍依据原 `krin`。

反例：因为两个分支成功、一个失败，就按“多数步骤成功”把目标判为 `met`。

### 案例九：外部 A2A Task 完成

外部 Agent 返回 `TASK_STATE_COMPLETED` 和一个 Artifact。适配器保存协议版本、对端、任务身份、原始状态和候选产物；本地仍要验证内容、来源、授权和 `krin`。

反例：把 completed 直接映射为 Drase `completed`、目标 `met` 或最终 `accepted`。

### 案例十：MCP Task 仍在演进

一个长时间工具调用使用 MCP 2025-11-25 experimental Task。适配器可以轮询和取消，但必须把实验版本、TTL、访问控制和结果取回限制写入适配语境。

反例：把当前 experimental 状态机固化成 Noemion ABI，或因为任务可持久化就把 Dromen 做成可恢复文件。

### 案例十一：handoff 与委托

主 Agent 把有限分析请求交给专门 Agent。AUT 记录实际行动者、被代表主体、范围、截止点和逐级收窄；控制平面过滤交接输入，并重新核对工具能力。

反例：handoff 名称、Agent Card、模型选择或一次登录自动授予子 Agent 原主体的全部权限。

### 案例十二：计划耗尽但目标未知

所有允许步骤都已尝试，仍缺少覆盖关键关系位置的观察。计划可以终止为“无更多允许步骤”，Drase 会话可以 failed 或 interrupted；满足判断仍是 `agno`，最终决定可以 deferred。

反例：把“没有更多计划”翻译成目标 `unmet`，或把“步骤全部执行”翻译成 `met`。

## 重规划触发矩阵

| 变化 | 只改计划 | 新授权 | 新 Dromen | 新 Endem / Synem |
| --- | --- | --- | --- | --- |
| 同一能力范围内调整步骤顺序 | 是 | 否 | 否 | 否 |
| 用同范围内已允许的替代工具 | 可以，但要保留工具身份与损失 | 依既定政策 | 工具 schema 或环境实质变化时需要 | 否 |
| 增加副作用、对象、受众或代表主体 | 否 | 是 | 通常是 | 目标承诺变化时是 |
| 扩大总预算或重试上限 | 否 | 是 | 是 | 否 |
| 修改终态、时间范围、量词、阈值或 `krin` | 否 | 是 | 是 | 是 |
| 仅获得新观察并选择另一允许路径 | 是 | 否 | 否 | 否 |
| 模型、tokenizer、后端、政策或能力集合实质变化 | 否 | 依政策 | 是 | 否，除非目标也变 |
| 把一个内部步骤提升为独立可接受承诺 | 否 | 是 | 是 | 是，并重新审查 Synem |

这个矩阵只提供设计审查问题，不是规范算法。边界不确定时必须停止在 `apor`、授权 defer、会话失效或诊断，而不是选择最方便的列。

## 威胁到失败责任的映射

| 威胁 | 攻击或失效方式 | 唯一主归属 | 预期处理 |
| --- | --- | --- | --- |
| 目标漂移 | 计划逐步改写终态或判据 | END / SYN | 发现内容变化，要求新身份 |
| 计划洗权 | 模型把未授权步骤写进“完成目标所必需” | AUT | 对每项精确动作重新求交 |
| 完成洗白 | 外部 task、工具或步骤 completed 直接成为 met | ADP / ADR-0015 | 保留外部状态，拒绝跨域映射 |
| 委托放大 | handoff 后权限、受众或预算扩大 | AUT | 委托逐级收窄；扩大必须新决定 |
| 无限重规划 | 失败后循环改写步骤和重试 | DRO | 共用有限预算、重试和截止点 |
| 竞态选择 | 并发完成顺序改变绑定或证据选择 | DRO / IKN | 固定汇合规则，保留全部决定依据 |
| 陈旧计划 | 环境、政策、工具 schema 已漂移仍继续执行 | DRO | 使旧会话失效或阻止具体动作 |
| 隐藏步骤 | SDK、子 Agent 或供应商执行未披露副作用 | ADP / AUT | 阻止不透明能力；记录不可观察限制 |
| 轨迹过度收集 | 为“可解释”保存秘密、私有推理或无关正文 | IKN / TEXT-IDENTIFIER | 最小披露；行动证据不要求完整思维链 |
| 分支结果洗白 | 成功分支覆盖失败、agno 或 fault | IKN / ADR-0015 | 依原判据聚合，保存未覆盖和故障 |

## 失败域与结果隔离

| 事件 | 计划状态 | 会话影响 | 满足判断 | Iknem / 决定影响 |
| --- | --- | --- | --- | --- |
| 某步骤成功 | 可推进 | 无自动变化 | 无自动变化 | 可形成有范围观察 |
| 某步骤失败且有允许替代 | 重规划 | 预算继续扣减 | 通常仍未知 | 保存失败来源和替代依据 |
| 无允许替代 | 停止或请求输入 | failed / interrupted 由会话规则决定 | 仍按证据得出 met/unmet/agno/fault | 决定可以 deferred，不自动 rejected |
| 外部 task completed | 标记外部工作结束 | 不自动 completed | 不自动 met | Artifact 仍是候选输入 |
| 计划全部步骤执行 | 计划耗尽 | 不自动 completed | 必须重新按 `krin` 求值 | 覆盖不足仍 insufficient |
| 目标内容改变 | 旧计划失配 | 旧 Dromen 失效 | 旧结果只适用于旧目标 | 新目标、新授权与新决定链 |
| 能力或预算扩大 | 原计划越界 | 必须新授权和新会话 | 无自动变化 | 旧记录保留历史范围 |

## 轨迹与思维链边界

为了审计规划，不要求保存模型的完整私有推理文本。未来最低可审计内容应优先是：

- 精确目标与 Dromen 身份；
- 计划候选的生产者、模型和输入边界；
- 被批准的动作、参数、能力决定和调用结果；
- 重规划触发观察、被替换步骤和替代依据；
- 分支、重试、预算、取消、handoff 和人工升级；
- 与 `krin` 对齐的 phain、覆盖缺口和最终决定依据。

自由推理文本可能包含秘密、个人数据、供应商不可观察变换或无法稳定解释的内容。它可以作为受政策限制的调试材料，但不能成为目标身份、授权依据或唯一审计证据。

## 生产者、消费者与省略条件

| 关系 | 当前研究结论 |
| --- | --- |
| 生产责任 | Drasor 外侧控制平面在 Dromen 上限内维护计划状态；模型、规则和人工都只能提出或确认各自权限内的候选。 |
| 直接消费者 | 能力门、协议适配器、预算器、Iknem 溯源和人工升级。当前没有实现。 |
| 失败责任 | END/SYN 负责目标漂移，AUT 负责动作与委托，DRO 负责会话上限，ADP 负责外部状态，IKN 负责观察与覆盖，DIA 负责诊断。 |
| 省略条件 | 单步确定性路径可以完全省略显式计划；没有模型、分支、重试、委托或人工升级时不持久化计划状态。 |
| 拆分条件 | 只有计划需要跨会话独立引用、签名、交换、撤销或长期保存，且现有 Iknem 与 Dromen 关系无法表达时，才重新评估物理 Profile。 |

## 语言哲学采用边界

维特根斯坦在《逻辑哲学论》3.32 与 3.321 区分记号和记号在符号中的使用：同一可感知记号可以承担不同符号职责。这里的工程启发是，“完成”出现在计划步骤、外部任务、会话和目标判断中，并不表示四者属于同一个结果域；职责必须由结构关系显示。

这些命题不能推出规划算法、Agent 分工、重试次数、并发策略或工具选择。哲学只帮助识别概念混淆；正式规则仍必须由规范条款、威胁、反例和向量证明。

## 权威资料与采用限制

- [GNU Make 规则与 prerequisite 类型](https://www.gnu.org/software/make/manual/make.html#Rule-Syntax)用于核对 target、prerequisite、recipe、普通依赖和 order-only 依赖的分离。它不定义以自然语言表达的目标或满足语义。
- [GNU Make Phony Targets](https://www.gnu.org/software/make/manual/html_node/Phony-Targets.html)用于说明动作入口不等于真实制品；[Errors in Recipes](https://www.gnu.org/software/make/manual/html_node/Errors.html)用于说明继续其他分支不等于整体成功。
- [ReAct 论文](https://arxiv.org/abs/2210.03629)用于确认推理、行动、观察和计划更新是 Agent 的现实运行模式；论文效果不能成为 Noemion 的安全或正确性证据。
- [A2A 1.0.0 规范](https://a2a-protocol.org/v1.0.0/specification/)用于核对 Task、Message、Artifact、context 和生命周期；这些对象只属于外部协议域。
- [MCP 2025-11-25 Tasks](https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks)明确把 Tasks 标为 experimental。本提案只记录版本事实，不建立稳定适配 Profile。
- [OpenAI Agents SDK tracing](https://openai.github.io/openai-agents-python/tracing/)、[handoffs](https://openai.github.io/openai-agents-python/handoffs/)与 [guardrails](https://openai.github.io/openai-agents-python/guardrails/)用于核对当前框架会记录运行、委托和检查边界；供应商 API 设计不进入 Noemion 规范身份。

## 进入规范前的条件

本提案目前不关闭以下问题：

- 计划状态是否需要跨会话独立身份；
- 何种用户修改构成目标变化，何种只构成计划调整；
- 并发分支、人工输入和长期任务的物理事件格式；
- A2A 1.0、MCP Tasks 或其他协议的具体映射 Profile；
- Drasor 控制平面的规划器、预算器、能力门或运行实现。

在出现真实消费者、正反向量和代码阶段授权前，默认不增加对象、格式、组件或命令。
