# Noemion 语义等价与迁移边界研究提案

状态：非规范研究提案  
日期：2026-07-13  
结论状态：正在研究；尚未进入规范
适用范围：Endem、Synem、文本变换、裁剪、重编码、跨规范迁移、模型相似度和未来 Semantic Key

本提案不构成 ADR、CORE 规范、内容 Profile 或实现要求，不进入 `registry.json`。它不创建新制品、文件格式、扩展名、命令、组件、结果域或稳定接口。当前项目尚未进入代码开发阶段；本文只检查现有责任是否足以承载未来的等价与迁移规则。

## 直接结论

Noemion 不应建立一个跨对象、跨 Profile、跨版本通用的“语义等价”布尔值，也不应现在定义万能 Semantic Key。

“相同”至少包含五种不同问题：字节是否相同、结构是否只换了表示、指定观察是否无法区分、迁移保留了什么、一个目标是否强化或弱化了另一个。它们的方向性、传递性、权威、失败责任和安全后果不同。把它们压成一个 `equivalent=true`，会让裁剪、迁移、缓存、签名、权限和最终决定互相越权。

当前首选方案是使用有范围的关系，而不是创造新对象：

1. 精确字节相同继续由 ID-CORE 负责。
2. 同一封闭 Profile 内的结构同构，只能由该 Profile 明确列出的无关表示差异决定。
3. 观察等价必须绑定有限观察集合、判据、权威、时间与失败责任。
4. 迁移是有方向的派生关系，必须记录映射、保留、增加、删除、降精度和失败，不自动成为等价。
5. 强化、弱化与细化是有方向的偏序候选，不得伪装成对称等价。
6. 文本相似度、向量距离和模型裁判只提供候选证据，永远不能单独建立上述关系。

因此，当前任何候选派生处理都不能“证明裁剪等价”。在属性级保留关系、观察集合和正反向量冻结之前，生产者只能保留内容或失败。未来若允许删除，输出仍获得新的精确身份，并以版本化派生关系连接来源。

## 为什么这是语言哲学问题也是安全问题

《逻辑哲学论》命题 2.1–2.15 用要素间关系解释可能情况，命题 2.17–2.172 强调可对应的形式由结构显示，命题 3.203–3.21 又区分名称与对象配置。Noemion 只采用其中可测试的工程限制：表面文字相似、名称相同或编码相近，不能替代 `semion`、`skena`、`telis`、`krin` 与 `apor` 的关系检查。

这不是说只有一种合法语义理论。相反，它阻止项目把某一种文本规范化、模型向量或发布者声明偷偷提升为全部语境中的“同义”。后期语言使用、翻译不确定性、言语行为和权威关系继续构成反例：两个句子可以在一种任务中可互换，在另一种权限、时间或社会语境中不可互换。

安全上，错误等价会产生四类直接后果：

- 把较弱目标替换为较强或较宽目标，改变要执行的行为；
- 让迁移对象继承来源签名、证据、授权或接受状态；
- 用相似文本命中错误缓存、依赖或发布对象；
- 删除 `apor`、来源、披露或调试信息后仍宣称“完全相同”。

## 权威资料给出的边界

| 资料 | 它真正定义什么 | Noemion 可以采用什么 | 不能推出什么 |
| --- | --- | --- | --- |
| [W3C RDF Dataset Canonicalization](https://www.w3.org/TR/rdf-canon/) | RDFC-1.0 在一个精确定义的 RDF 数据集域中规范空白节点标识；同一规范结果对应图同构 | 只有先定义封闭对象域和同构关系，规范形式才可能成为可重复比较依据 | 图同构不能自动成为自然语言意图、授权、满足或跨 Profile 语义等价 |
| [RFC 8785 JSON Canonicalization Scheme](https://www.rfc-editor.org/rfc/rfc8785.html) | 为受限 JSON 数据产生可哈希表示，并要求字符串保持原样、数组顺序不变 | 规范字节必须绑定精确输入域和不改变内容的规则 | 属性排序或可哈希表示不能证明业务语义相同 |
| [Unicode UAX #15](https://www.unicode.org/reports/tr15/) | 在指定 Unicode 版本与规范化形式下定义规范等价或兼容等价 | 文本比较必须声明 Unicode 版本、形式和槽位 | NFC/NFKC 不能证明 `semion` 等价；NFKC/KD 还可能删除有语义价值的区别 |
| [RFC 7950 YANG 1.1](https://www.rfc-editor.org/rfc/rfc7950.html#section-11) | 限定已发布模型可做的更新；语义改变通常要求新标识 | 迁移规则应区分编辑变化、允许变化与新定义，并显式绑定版本 | 成功解析新版本或保留名称不能证明旧定义与新定义相同 |
| [GNU BFD canonical object-file format](https://sourceware.org/binutils/docs/bfd.html) | 为多对象格式提供通用内部表示，同时明确跨格式转换可能丢失信息 | 转换必须声明源格式、目标格式和信息损失 | 进入同一 canonical form 不能证明所有格式特有含义被保留 |
| [GNU objcopy 调试信息分离](https://sourceware.org/binutils/docs/binutils/objcopy.html) | 可以把调试内容外置、裁剪主文件并建立 debug link | 主对象、裁剪对象和伴随文件各自有身份，并记录精确关系 | “仍可运行”不能证明调试、来源、披露和审计行为完全等价 |
| [Sentence-BERT](https://aclanthology.org/D19-1410/) | 学习可用余弦距离比较的句向量，服务相似度检索和聚类 | 模型距离可以排序需要人工或确定性复核的候选 | 高相似度不是身份、逻辑等价、授权或可替换性的证明 |
| [NeurIPS 的 LLM-as-a-Judge 研究](https://proceedings.neurips.cc/paper_files/paper/2023/hash/91f18a1287b398d378ef22505bf41832-Abstract-Datasets_and_Benchmarks.html) | 研究模型裁判并报告位置、冗长、自增强和推理限制 | 模型评价可成为有版本、有不确定度的候选测量 | 单次模型判断不能建立语义等价或高风险迁移决定 |
| [NIST AI 800-3](https://nvlpubs.nist.gov/nistpubs/ai/NIST.AI.800-3.pdf) | 要求显式定义评估目标、统计假设与不确定度 | 模型相似度和裁判必须先定义构念、总体、程序与不确定度 | 一个分数或点估计不能成为普遍语义事实 |

共同结论不是“规范化无用”，而是规范化只在自身封闭关系内有效。离开对象域、版本、变换规则和观察集合以后，相同输出不再拥有可解释的含义。

## 五类关系必须分开

| 关系 | 问题 | 方向 | 最低证明条件 | 明确不能继承 |
| --- | --- | --- | --- | --- |
| 精确内容身份 | 两个引用是否指向同一身份域中的同一规范字节？ | 对称 | 完整身份语境与完整摘要；由 ID-CORE 定义 | 真值、满足、授权、签名主体、接受 |
| 封闭结构同构 | 同一对象域中，是否只改变了该域声明为无关的表示？ | 对称 | 同一关系规范、同一适用 Profile、规范解析、结构同构和反例向量 | 跨 Profile 意义、外部授权、运行行为 |
| 有范围观察等价 | 对明确列出的观察，两对象是否不可区分？ | 对称，但只在给定观察集合内 | 精确观察集合、`krin`、环境、权威、时间、容差、不确定度与失败政策 | 未列出的性质、未来观察、不同环境结论 |
| 版本化迁移 | 目标对象怎样由来源对象映射而来？ | 有方向 | 源/目标规范与 Profile、映射域、保留/新增/删除/降精度清单、失败和回转性质 | 来源身份、签名、Iknem、接受状态、权限 |
| 强化/弱化/细化 | 一个对象是否比另一个要求更多、允许更少或提供更具体结构？ | 有方向，通常不对称 | 明确偏序、适用域、反例、`telis/krin/apor` 变化和权威 | 反向可替换、普遍等价、自动授权 |

“文本相似”“模型认为相同”“翻译看起来一致”“两次都得到 `met`”不属于这五类中的任何一种。它们只能触发候选比较。

## 一个关系声明最少需要回答什么

未来若真实消费者需要保存或验证关系，提案必须先固定以下语义；本节不是字段设计：

1. 关系种类以及它是否声称自反、对称、传递或有方向。
2. 来源与目标的完整精确身份，而不是名称、路径、短摘要或 `latest`。
3. 来源和目标各自的对象规范、内容 Profile、物理表示与版本。
4. 关系规范自身的身份、版本、适用域和明确排除项。
5. 要保留的 `semion/skena/telis/krin/apor` 属性与允许变化的表示属性。
6. 对观察等价而言，完整观察集合、环境、时间、容差、不确定度和权威。
7. 生产者、核验者、实际检查步骤、Iknem 范围与失败责任。
8. 新增、删除、降精度、不可逆和未判断内容的显式清单。
9. 是否允许复合、怎样证明传递、怎样避免关系链扩大原始适用域。
10. 任何安全后果都必须另经 AUT-CORE、IKN-CORE 与最终决定核对。

缺少任一会改变结论的项目时，关系保持未定义或检查失败，不能退回文本猜测。

## 派生处理的当前边界

现有资料中“发布处理证明裁剪等价”的说法过强，因为 ID-CORE 已明确 Semantic Key、裁剪前后等价和跨版本等价尚未定义。当前可辩护的规则只有：

- 当前不设通用裁剪动作，也没有真实裁剪消费者。
- 在版本化保留关系与正反向量冻结前，对应制品生产者不得删除会影响内容、来源、披露、诊断、证据或未来核验的信息；无法证明保留时保守保留或失败。
- 未来产生的裁剪对象、调试伴随文件和发布包各自获得新精确身份。
- 派生关系记录保留、外置、删除、降精度和不可逆内容；它不让输出继承来源签名、Iknem、权限或接受状态。
- “运行观察相同”最多支持一个已声明观察集合内的等价，不支持来源、调试、审计、披露和未来行为都相同。

因此未来失败诊断也不应写成抽象的 “changes semantics”。它应定位具体违反的保留属性、未覆盖观察、未知损失或关系规范缺失。

## 迁移规则

当前设计阶段不承诺向后兼容，名称、路由和草案结构变化继续直接替换；历史 ADR 只解释迁移，不提供旧入口或兼容垫片。这不等于未来迁移可以省略语义分析。

任何未来规范或 Profile 迁移至少需要：

| 项目 | 必须回答的问题 | 失败示例 |
| --- | --- | --- |
| 方向 | 从哪个精确版本迁到哪个精确版本？是否允许逆向？ | 把双向转换器存在误写为双向等价 |
| 定义域 | 迁移对全部对象还是封闭子集有定义？ | 遇到未知关键内容仍输出部分对象 |
| 映射 | 每个来源结构进入哪个目标结构？角色与顺序怎样保持？ | actor/resource 对调但字段仍齐全 |
| 新增 | 目标版本新增的必需意义从哪里获得？ | 用默认值或模型猜测关闭 `apor` |
| 删除 | 哪些信息消失、外置或降精度？谁批准？ | 删除判据或权威后仍称等价 |
| 身份 | 新对象、伴随物和关系怎样获得独立身份？ | 复用来源摘要或签名 |
| 回转 | `source → target → source` 保持哪些属性？哪些明确不保持？ | 只因回转字节相同就宣称全部语义相同 |
| 安全 | 签名、证据、权限、撤销和最终决定怎样重新核对？ | 迁移自动继承 grant 或 accepted |
| 失败 | 未知、歧义、资源超限、损失和规则缺失怎样分类？ | 静默跳过无法映射的记录 |

成功迁移只证明迁移规范对这对精确对象完成了声明的映射。它不证明来源与目标可在所有消费者、时间、环境和权威下互换。

## 支持案例与反例

| 场景 | 正确分类 | 不能宣称什么 |
| --- | --- | --- |
| 同一 Profile 的对象属性顺序被规范化，而数组顺序保持 | 可能是封闭结构同构，需 Profile 明确授权 | 不是跨 Profile 普遍语义等价 |
| RDF 空白节点只换标识，关系图同构 | 在 RDFC-1.0 对象域中可规范化 | 不能推出 Endem 的 `telis/krin/apor` 相同 |
| 只改变空白和排版的来源文本 | 来源表示发生变化；意义是否相同仍需关系规则 | 可见相同或解析成功不是语义证明 |
| NFC 前后码点不同但属于 Unicode 规范等价 | 只在声明的文本槽和 Unicode 版本内成立 | 不能改写来源身份或证明 `semion` 相同 |
| NFKC 把全角、圈号或兼容字符折叠 | 有损文本变换候选 | 不能默认用于任意来源或安全标识符 |
| 两句自然语言互为常见释义 | 模型或人工可提出候选关系 | 不能跳过指称、作用域、否定、时间和权威复核 |
| 关系相同但 actor 与 resource 互换 | 明确反例 | 词汇相同不构成结构等价 |
| `krin` 阈值从 0.80 改为 0.90 | 目标判据强化候选 | 两次都 `met` 也不使目标等价 |
| 增加一个未解决 `apor` | 新内容身份，且确定性降低 | 不能继承原 coherent/accepted 状态 |
| 删除一个 `apor` 但没有授权决议 | 无效变换 | 模型高置信度不能补授权 |
| 两个不同目标在一次观察中都得到 `met` | 结果偶合 | 相同结果不证明目标相同 |
| 同一 Endem 在两个权威域中获得不同授权 | 内容身份可以相同，授权语境不同 | 内容相同不产生跨域 grant |
| `all_of` 叶只改变规范允许的无序排列 | 只有组合规范明确声明无序时才可能同构 | 不能对任意数组重排 |
| 主对象外置调试内容并绑定伴随文件 | 新主身份、新伴随身份和派生关系 | 仍可运行不表示审计行为等价 |
| 模型向量余弦相似度很高 | 候选检索证据 | 不能产生 Semantic Key |
| 两种语言的翻译都经人工认可 | 对精确用途的授权翻译关系候选 | 不能推出所有语境、双关和法律效果相同 |

## 威胁到失败责任的映射

| 威胁 | 典型攻击 | 必须由谁阻断 |
| --- | --- | --- |
| 等价漂白 | 发布者自填 `equivalent=true` | ID 拒绝未登记关系；AUT 不授予语义权威 |
| 关系域扩大 | 一个测试环境的观察等价被用于全部环境 | END/IKN 固定观察与主张范围 |
| 静默弱化 | 迁移降低阈值或删除约束 | END/SYN 比较 `telis/krin` 并失败 |
| `apor` 擦除 | 模型或默认值关闭未决项 | END/AUT 要求具名授权决议 |
| 角色置换 | 名词相同但关系角色对调 | END 结构比较与反例向量阻断 |
| 规范化损失 | Unicode 或 BFD 通用表示删除重要区别 | TEXT-IDENTIFIER/ID 记录变换与损失，未知时失败 |
| 缓存投毒 | 相似文本或短摘要命中错误对象 | ID 只接受完整精确身份 |
| 迁移降级 | 新格式不支持关键字段但仍产生输出 | 迁移关系声明定义域并原子失败 |
| 回转幻觉 | round-trip 成功被提升为普遍等价 | 关系规范只声明经过证明的保留属性 |
| 权限继承 | 派生对象沿用来源 grant | AUT 对新精确对象重新决定 |
| 证据继承 | 派生对象沿用来源 Iknem 或 accepted | IKN 固定主体与范围；最终决定重新形成 |
| 模型裁判偏差 | 位置、长度或模型自偏好决定等价 | 模型只出候选；测量固定程序与不确定度 |
| 关系链放大 | 多段局部关系组合成超出各段范围的全局等价 | 组合前求关系域交集；无法证明传递时失败 |

## 失败域与结果隔离

等价或迁移检查的结论不得改写现有结果域：

- 关系规范缺失、映射无定义、损失未知或验证器故障，是关系检查失败，不是 `unmet`。
- 目标结构有效但观察不足，仍是 `agno`，不能用相似度补齐。
- 迁移完成不表示 Drase 会话 `completed`，更不表示最终 `accepted`。
- 签名验证成功不表示关系主张真实；签名只绑定声明者与精确声明。
- Iknem 可以记录检查步骤、输入、结果和限制，但不能自行授予等价权威。
- 外部模型、MCP/A2A Task、工具或工作流的 `completed` 不能建立等价或迁移接受。

## 候选责任的唯一主归属

本提案不建议创建“等价组件”或“迁移对象”。未来义务应优先归还现有边界：

| 责任 | 主归属 | 说明 |
| --- | --- | --- |
| 精确来源/目标身份与派生关系 | ID-CORE | 新对象总有新身份；关系不继承信任 |
| Endem 结构、方向、判据和未决项比较 | END-CORE | 只比较规范明确拥有的语义属性 |
| Synem 闭包、成员与权限变化 | SYN-CORE | 不能用聚合等价洗白成员差异 |
| 文本规范化、显示、翻译与模型输入 | TEXT-IDENTIFIER-CORE | 每种变换绑定文本槽、版本和损失 |
| 语义确认、迁移批准与能力授权 | AUT-CORE | 模型、签名或发布者自述不取得权威 |
| 检查记录、覆盖和限制 | IKN-CORE | 证据只支持声明范围 |
| 外部格式与协议映射损失 | ADP-CORE | 保留来源域，不把外部状态提升为本地事实 |
| 主失败、位置、恢复和披露 | DIA-CORE | 关系失败不改写目标结果 |
| 一次运行允许使用的关系与迁移 | DRO-CORE | 只在封闭会话上限内使用，不扩大能力 |

如果这些责任无法形成唯一主条款，才重新讨论独立规范；不能为了名称对称先创建 `EQV-CORE`、新命令或新组件。

## Semantic Key 的进入条件

当前结论是继续禁用 Semantic Key。未来只有同时满足以下条件，才可提出一个有范围的键：

1. 存在不能只用精确身份和显式关系解决的真实消费者。
2. 对象域、源/目标 Profile、关系和版本完全封闭。
3. 已证明该关系在声明范围内需要的自反、对称与传递性质。
4. 规范形式不会删除安全、授权、`apor`、判据或披露相关区别。
5. 有正向同构、近似但不等价、对抗碰撞和资源耗尽向量。
6. 键值包含关系规范身份，不能跨关系、Profile 或版本比较。
7. 完整精确身份继续保留，签名与授权仍绑定精确对象。
8. 关系误判的失败责任、撤销、缓存清理和升级策略已经确定。

即使这些条件满足，Semantic Key 也只能是一个封闭关系的索引，不是自然语言意义的万能摘要。

## 进入 ADR 或规范的证据门

在责任分配形成正式 ADR 前，任何结论都不进入 CORE、Profile、登记或规范向量。未来提出规范关系时，至少需要：

- 一个真实上游生产者和一个真实下游消费者；
- 对象域、观察集合、关系代数和失败责任；
- 支持案例、最小反例、对抗案例和资源上限；
- 源/目标 Profile 与跨版本映射；
- 属性保留、增加、删除、降精度和不可逆清单；
- 正反向量与至少两条独立读取或检查路径的验证方案；
- 对身份、签名、证据、授权、缓存、撤销和最终决定的影响；
- 省略、合并或停止建设的条件；
- 项目进入代码开发阶段并明确实现范围后，才可讨论实现、性能或可执行迁移器。

## 进入规范前的条件

进入规范前需先验证以下三项方向性结论：

1. 拒绝通用语义等价布尔值和万能 Semantic Key，改用有范围、有类型的关系。
2. 把派生处理从“证明裁剪等价”改为“生产者只在冻结的保留关系内执行允许变换；否则保留或失败”。
3. 迁移、相似度、签名、证据、授权和最终决定始终保持正交，任何派生对象都重新取得精确身份与外部决定。

只有这三项经过讨论形成可辩护决定后，才判断哪些义务应写回 ID、END、SYN、TEXT-IDENTIFIER、AUT、IKN、ADP、DIA 或 DRO；当前不新增术语、制品、命令、组件或规范家族。
