# Authority and Authorization Decision Natural-Language Design Scenarios

- 文档 ID：`AUT-SCEN`
- 版本：`0.1.0-draft`
- 日期：2026-07-13
- 状态：非规范设计审查语料
- 适用范围：自然语言投影授权、权威语境、作用范围、决定、委托、多人规则、同意、撤销、重放、能力与结果分离

这些场景不是政策语言、访问令牌、同意界面、权威目录或组件测试。它们用支持案例、反例和边界场景检查 AUT-CORE 能否解释现实授权问题；只有唯一条款、登记验证和正反向量共同覆盖的内容才可能成为符合性义务。

### AUT-SCN-001 — 授权绑定精确政策而不是最新政策

- **场景：**一项 `meaning_projection` 投影昨天按政策 3 获准，今天政策 4 收紧了高风险关系的决定主体。
- **审查结论：**历史决定保留政策 3 与截止点；新形成或新会话必须按政策 4 重新决定，不能让 `latest` 静默改写旧记录或延长旧授权。

### AUT-SCN-002 — 同一角色不能跨租户和目的复用

- **场景：**主体在测试租户拥有“发布负责人”角色，却请求批准生产租户的能力写入。
- **审查结论：**角色名称相同不等于权威语境相同；租户、对象、动作和目的不匹配，因此请求必须拒绝或转交正确权威。

### AUT-SCN-003 — 已认证用户不自动成为语义权威

- **场景：**一名已通过高强度认证的观察者打开候选投影页面。
- **审查结论：**认证只确认当前主体；只有权威目录与适用政策证明该主体能决定当前关系位置时，确认才有效。

### AUT-SCN-004 — 签名者和实际行动者必须分别保存

- **场景：**自动化服务使用组织密钥签署一项由人工负责人发起的批准。
- **审查结论：**签名服务是实际行动者或认证机制，人工负责人是声明决定主体；两者身份、角色、范围和依据都必须保存，签名本身不能补足权限。

### AUT-SCN-005 — 字符串前缀不能扩大字段授权

- **场景：**政策允许修改 `meaning_projection.relation.read`，界面据此尝试批准所有以 `meaning_projection.relation` 开头的字段。
- **审查结论：**范围展开必须由封闭政策定义；字符串前缀不能隐含 `write`、角色或量化位置。

### AUT-SCN-006 — 模型候选保持未授权

- **场景：**模型以 0.97 置信度把“它”解析为生产数据库，并给出结构合法的 `meaning_projection`。
- **审查结论：**模型输出仍是候选；具名权威必须看到精确来源、候选、影响关系与风险后决定，否则候选继续保存在 `unresolved_meaning`。

### AUT-SCN-007 — 批量批准不能跨越高风险差异

- **场景：**页面同时列出十个拼写修正和一个会改变付款接收方的指称变更，并只提供“全部接受”。
- **审查结论：**授权必须按影响范围和风险展示精确差异；一个批量点击不能把未展示或不同权威范围的候选一起确认。

### AUT-SCN-008 — 部分授予必须形成缩小后的明确结果

- **场景：**请求要求读取和删除两个资源，权威只允许读取其中一个。
- **审查结论：**决定记录 `allowed` 的精确子范围，并让调用方核对；不得把原请求标为全部允许，也不得把未授部分静默删除后继续执行。

### AUT-SCN-009 — 委托保留被代表主体与实际 Agent

- **场景：**用户授权控制平面让 Agent A 读取一份报告，Agent A 又委托 Agent B 做摘要。
- **审查结论：**链保留用户、A、B 和每一级范围；B 只能得到读取该报告所需的严格子集，不能伪装成用户或 A。

### AUT-SCN-010 — 外部 impersonation 不能隐藏真实行动者

- **场景：**OAuth token exchange 允许服务 A 在有限语境中冒充用户 B 调用服务 C。
- **审查结论：**Noemion 仍保存 A 是实际行动者、B 是被代表主体，并要求独立政策明确允许该语境；外部 token 不能抹去责任链。

### AUT-SCN-011 — 多人批准按不同主体计数

- **场景：**发布政策要求安全负责人、数据负责人和产品负责人三种角色分别批准。
- **审查结论：**三项决定绑定不同合格主体和角色；同一人持有多个账户或重复签名不能满足三个独立责任位置。

### AUT-SCN-012 — 冲突决定不会由到达顺序解决

- **场景：**一名权威 `allowed`，另一名拥有预注册否决权的权威 `denied`，允许事件更晚到达。
- **审查结论：**按预先固定的否决与冲突政策处理；事件顺序不能覆盖否决，未能解析时结果保持 `pending`。

### AUT-SCN-013 — 同意视图必须与机器请求一致

- **场景：**界面显示“读取日历”，机器请求还包含写入联系人和向第三方发送内容。
- **审查结论：**显示与机器对象不一致，同意无效；所有动作、资源、接收方、期限和撤销后果必须在绑定视图中可理解地呈现。

### AUT-SCN-014 — 同意者不一定拥有授权范围

- **场景：**团队成员点击允许处理一份由客户控制的数据，但政策只允许客户代表授权该用途。
- **审查结论：**点击可以成为一项记录，却不能替代角色和范围核对；请求仍需正确权威决定。

### AUT-SCN-015 — 撤销改变后续适用性但不删除历史

- **场景：**资源所有者撤销仍在有效期内的读取授权。
- **审查结论：**后续使用必须拒绝，当前 session contract 失效并按政策中断；原决定、已发生调用和 evidence entry 保持追加记录，不能被改写成从未授权。

### AUT-SCN-016 — 一次批准不能用于另一个对象或第二次副作用

- **场景：**用户批准一次付款，适配器超时后以新 request ID 再次发送同一金额。
- **审查结论：**授权绑定原请求、收款方、金额、用途和一次使用；重试共享原身份与去重证据，不能获得第二次消费机会。

### AUT-SCN-017 — Step-up 必须建立新会话

- **场景：**运行中的 MCP 工具返回写 scope 要求，而当前 session contract 只有读取能力。
- **审查结论：**旧会话不能原地扩权。控制平面可以提出新授权请求；只有新决定、新能力交集和新 run 会话建立后才可写入。

### AUT-SCN-018 — 已授权操作不等于目标满足或最终接受

- **场景：**恢复测试获得执行权限并正常完成，但恢复出的数据损坏。
- **审查结论：**授权决定可以是 `allowed`，run 会话可以 `completed`，满足结果仍为 `unmet`，最终权威决定可以 `rejected`。任何一层都不能洗白另一层。

## 仍需设计

权威目录、角色和政策语言、授权事件编码、同意 UI Profile、多人独立性证明、撤销分发、外部 OAuth/MCP/A2A 映射与能力代理仍未冻结。代码阶段开启前，项目只维护条款、威胁、场景、提案向量和未来验证条件。
