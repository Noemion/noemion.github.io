## 诊断目标

诊断既服务开发者，也服务构建自动化、测试和安全审计。它必须解释哪一个输入对象、哪一项规则、哪一组候选导致失败，并给出稳定的机器类别；不能只输出易变的自然语言，也不能为了“继续构建”把安全失败降级。

## 必须失败

- 未解析强符号、版本不匹配或多个强定义。
- 硬约束冲突。
- 需要扩大权限但缺少显式授权。
- 不允许的循环依赖、图规模或重定位数量超过配置限制。
- 偏移、长度、计数、对齐、索引或 ID 运算越界或溢出。
- 未知必需特性、目标类型不匹配或产物自检失败。

## 冲突报告

报告至少定位输入成员、符号或节点、冲突类型、涉及规则和拒绝原因；不得以普通 warning 掩盖安全失败。

## 结构化诊断记录

| 字段 | 用途 |
| --- | --- |
| 诊断类别与版本 | 供自动化稳定识别；具体编号尚待诊断规范冻结。 |
| 严重度 | 区分致命错误、错误、可行动警告和说明；安全不变量失败只能是致命错误。 |
| 主体定位 | 输入内容摘要、归档成员、Section、符号、节点、边或重定位。 |
| 规则引用 | 关联规范条款、配置条款或 ADR，而不是只引用实现函数。 |
| 参与对象 | 列出全部冲突候选、期望值、实际值和选择阶段。 |
| 因果链 | 从顶层失败追踪到依赖、引用或权限来源，避免重复孤立报错。 |

## 聚合、顺序与隐私

- 诊断采用规范稳定顺序，输入排列或并发执行不得改变首要失败类别。
- 同源错误可聚合，但必须保留总数、样例和被截断标记；诊断数量受配置限制。
- 路径可归一化或映射为构建根相对位置，受限符号、策略内容、密钥材料和私有数据默认脱敏。
- 提供修复提示时只描述满足契约的方向，不自动修改约束、权限或对象语义。

## 验证要求

每个“必须失败”规则都应有最小反例、复杂组合反例和确定性快照测试。机器字段需要兼容性政策；人类文本可独立演进。安全评审还应检查诊断本身不会触发二次解析漏洞、资源放大或敏感信息泄漏。
