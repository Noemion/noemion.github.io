noemobserve · 运行追踪与调试映射

采集装载、能力调用、环境观察、策略拒绝、约束求解和产物访问事件，并映射回会话、对象、NIR 节点与来源单元。

Phase 6Release & Fulfillment RuntimeTrace Stream → Diagnostic ReportDesign Stage

它解决什么问题

noemobserve 定义运行采集协议，并离线规范化装载、符号绑定、披露、Capability Request / Observation、策略拒绝、Candidate Assessment 和产物访问事件，再映射回会话、对象、NIR 节点与来源单元。

生成式运行包含对象系统、Agent Harness、Fulfillment Runtime、模型和外部能力等多条路径，普通日志难以重建因果且可能泄露敏感内容。采集模式负责产生带身份和完整性元数据的 Trace Stream；规范化模式负责排序、关联和脱敏，但绝不能把来源不明、事件丢失或不可信自述自动升级为验收证据。

当前状态

当前状态：尚未发布可执行版本，命令行界面、参数和文件扩展名仍在设计中。当前公开信息包括计划用途、输入输出和安全边界；安装与调用说明将在首个可用版本发布时提供。

开发顺序：先稳定事件结构定义、生产者身份、哈希链、时钟与序号模型、事件丢失/采样声明、Debug Companion 和脱敏策略；随后验证并发排序、篡改、丢事件、状态关联、敏感字段和低开销采集。

  • 定义稳定、版本化、机器可解析的追踪事件和生产者等级
  • 核对 Session、对象、环境、序号、事件哈希链、丢失计数、采样和截断声明
  • 关联 Agent Harness 会话、能力请求/观察、工具结果、对象、NIR 节点和来源单元

  • 按策略脱敏敏感产物和自然语言片段，同时记录脱敏造成的证据强度变化

它读取什么，产生什么

方向对象作用
输入Trace Stream + Integrity Metadatanoemexecute 产生的事件、生产者、序号、时钟域、哈希链、丢失、采样和截断声明
输入Session State + Run Report绑定会话、对象、环境、Runtime 与能力调用的预期身份
输入Debug Companion独立保存的源映射、符号名和开发调试 Section
输入Redaction Policy字段级保留、哈希、隐藏、导出和证据强度规则
输出Normalized Trace稳定排序、身份关联、已脱敏的事件序列；不因规范化自动获得可信状态
输出Trace Integrity Report生产者等级、哈希链、丢失、采样、截断、时钟不确定与可用于哪些证据规则
输出Diagnostic Report按事件、对象、源单元或因果链组织的报告

它不会做什么

  • 不改变 Fulfillment Runtime 决策或重放时注入新语义
  • 不要求发布包保留全部开发调试信息
  • 不把日志文本作为正式事件协议
  • 不把 Runtime、模型或工具自述当作真实环境观察
  • 不绕过访问策略读取被剥离内容

追踪不是确定性重放证明；事件丢失、采样、时钟不确定、生产者信任和脱敏必须显式记录。Trace Integrity Report 未满足 Coverage Policy 时，Normalized Trace 不得进入验收证据闭包。
