项目/工具/noemdecode

noemdecode · NIR 反汇编器

把运行 Section 反汇编为可再次汇编的规范文本，并按请求附带调试、覆盖和来源注释。

Phase 2Text IRNOBJ / HOBJ → 文本 NIRDesign Stage

它解决什么问题

noemdecode 把运行 Section 反汇编为可再次汇编的规范文本，并按请求附带调试、覆盖和来源注释。

对象必须能被独立审计、评审和重建；只显示字节或摘要不足以验证语义。反汇编器提供对象到规范文本的可追溯映射，同时明确哪些信息已被 strip 或无法恢复。

当前状态

当前状态：尚未发布可执行版本，命令行界面、参数和文件扩展名仍在设计中。当前公开信息包括计划用途、输入输出和安全边界；安装与调用说明将在首个可用版本发布时提供。

开发顺序：先完成安全读取器、规范文本语法和稳定命名规则；随后验证合法对象可规范化再汇编、损坏对象能够安全拒绝、注释视图不改变运行文本语义。

它怎样工作

结构验证
读取语义记录
建立稳定名称
分离运行与注释
输出规范文本
  • 反汇编多层 NIR、符号引用、重定位和验收契约
  • 使用稳定名称替代布局相关 ID
  • 可选显示来源映射、证据账本和 Disclosure 注释
  • 输出 canonical、pretty 或 JSON 视图

它读取什么，产生什么

方向对象作用
输入Object通过结构验证的 NOBJ/HOBJ
输入Layer Filterhigh-level/canonical/backend/behavior 或 all；名称随分层规范冻结
输入Output + Annotation Policycanonical/pretty/json，以及 debug/coverage/source/relocation 注释开关
输出Canonical Text NIR在信息无损时可被 noemassemble 接受的规范文本
输出Pretty / JSON View面向阅读或机器处理的视图，不默认保证可再次汇编
输出Symbol Map对象局部索引到稳定文本名称映射
输出Diagnostics无法表示、信息损失或损坏记录的位置与原因

它不会做什么

  • 不恢复已被 strip 删除的源文本
  • 不把调试注释混入运行语义
  • 不将损坏对象猜测为合法 NIR
  • 不宣称反汇编摘要等同原始自然语言

所有对象输入均视为不可信；偏移、长度、计数、索引和对齐计算必须使用 checked arithmetic。输出必须标记不可表示记录、未知扩展和信息损失，不能用占位文本制造可再次汇编的假象。
