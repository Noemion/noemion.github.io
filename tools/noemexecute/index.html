项目/工具/noemexecute

noemexecute · 可信装载与执行

验证 Signed Noemion Package，建立 Harness 与 Runtime 循环，输出可核验运行证据，并在证据闭合后形成明确验收决定。

Phase 6Release & Fulfillment RuntimeSigned Package → Run Evidence → DecisionDesign Stage

它解决什么问题

noemexecute 重新验证 Signed Noemion Package 的实际字节、闭包和策略，建立不可变 Loaded State 与 Agent Harness 会话，再驱动 Fulfillment Runtime 求解和候选评价。

签名包仍是不可信输入；预先验证也不能替代实际装载边界的检查。noemexecute 统一驱动对象系统、Harness 和 Runtime，但 Runtime 只产出 Candidate Assessment。运行结束后，Trace 先由 noemobserve 规范化，再由 noemcoverage 检查证据闭包；noemexecute 的 finalize 阶段才按预注册 Acceptance Policy 形成最终决定。

当前状态

当前状态：尚未发布可执行版本，命令行界面、参数和文件扩展名仍在设计中。当前公开信息包括计划用途、输入输出和安全边界；安装与调用说明将在首个可用版本发布时提供。

开发顺序：先稳定包验证、Loaded State、Execution Profile、Capability Request / Observation、事件结构定义、Trace 完整性、Evidence Closure 与 Acceptance Decision；随后测试篡改、回滚、越权披露、工具污染、预算耗尽、状态漂移和模型或外部能力失败。

  • 验证包签名、Manifest、SBOM、Section 摘要、Coverage Proof 和依赖闭包
  • 按 Segment 与权限建立只读、可执行和私有映射，冻结为 Loaded State
  • Harness 根据能力需求和 Execution Profile 建立会话级 Capability Binding；实时句柄不进入 Loaded State 或 Runtime
  • 对象系统按策略建立披露视图；Runtime 返回 Candidate Assessment 或 Capability Request，Harness 校验、执行并返回 Capability Observation
  • 运行阶段输出 Run Record、Session State、Run Report、Trace Stream 与 Trace Integrity Metadata，状态保持 pending-evidence
  • finalize 阶段读取 noemcoverage 的 Evidence Closure Report，按预注册 Acceptance Policy 输出 Acceptance Decision；需要外部判断时保持 pending-review

它读取什么，产生什么

方向对象作用
运行输入Signed Noemion Package由 noembundle 核对外部签名响应后封装的最终发布包
运行输入Execution Profile + Acceptance Policy目标环境、能力、预算、披露、停止、证据、人工升级与最终决定规则；只能收紧包内上限，不能扩大权限
运行输入Runtime Binding + Capability Catalog模型或求解后端、评价器和 Harness 可以绑定的环境能力；身份和版本必须满足签名包锁定条件
运行输出Run Record + Session State候选、Candidate Assessment、对象与环境身份、披露、预算、能力绑定和观察；初始状态为 pending-evidence
运行输出Run Report + Trace Stream + Integrity Metadata验证链、装载映射、能力请求/观察、策略决定、序号、事件哈希、丢失和采样声明
回验输入Evidence Closure Report由 noemcoverage 绑定同一会话、对象、结果、Trace Integrity Report 和契约项形成
回验输出Run Result + Acceptance Decisionaccepted、unsatisfied、pending-review、failed 或 interrupted，以及决定策略、证据和权威身份

它不会做什么

  • 不修复损坏或不兼容的对象
  • 不绕过 noemvalidate 的拒绝结论
  • 不让模型或 Runtime 直接持有能力句柄、修改二进制布局或加载策略
  • 不把 Candidate Assessment、Trace 规范化或 Evidence Closure Report 单独当作最终验收
  • 不把运行期产物静默写回签名包

所有对象、模型输出、外部能力结果和运行上下文均视为不可信。运行失败、未满足结果和策略拒绝必须分类记录，不能把“模型生成了内容”当作验收契约已满足。
