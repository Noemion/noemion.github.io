noemevaluate · 模型与智能体任务评估

分别评估 Horizon Engine 模型候选和已完成的 Agent 场景证据，为模型资格、回归与发布判断提供离线结论。

Phase 7 / Phase 8Offline EvaluationTarget + Evidence → QualificationDesign Stage

它解决什么问题

为什么平均分和模型自述都不够

Horizon Engine 的主要风险不是单纯“预测错”，而是在证据不足时给出错误确定结论，或在关键语言、结构与风险切片上悄然退化。Agent Harness 的主要风险则是结果看似合理，但能力调用越权、观察失真、恢复循环失控或验收证据缺失。

noemevaluate 使用两类明确分开的离线目标：模型评估测量 checkpoint 或部署模型包的错误确定、校准、OOD、切片和量化退化；场景评估消费已经完成的 Run Result、Acceptance Decision 与证据闭包，测量结果、过程合规、恢复、预算和人工升级。两类结论不能互相替代，也不反馈成为同一次运行的覆盖输入。

当前状态

当前状态：尚未发布可执行版本，命令行界面、参数和文件扩展名仍在设计中。当前公开信息包括计划用途、输入输出和安全边界；安装与调用说明将在首个可用版本发布时提供。

正式评估准备

模型评估前需要固定数据切片、指标、阈值、基线、用途和统计方法，并把盲测数据与训练调参隔离。任务场景评估前需要固定 Signed Package、Execution Profile、环境快照、能力策略、扰动、验收契约和人工升级条件。评估报告只回答预先声明的问题。

它怎样工作

  • 模型模式评估 checkpoint 或量化后 Model Package Candidate 的跨度、关系、模态、歧义、路由和封闭任务头，并保留逐样本证据
  • 模型模式测量错误确定、拒绝质量、校准、来源覆盖影响和部署退化，并按用途输出 Model Qualification Record
  • 场景模式可以编排固定场景，但执行、追踪和覆盖分别委托 noemexecute、noemobserve 与 noemcoverage；noemevaluate 只在它们完成后消费证据
  • 场景模式比较最终 Acceptance Decision、真实结果、过程合规、恢复、预算和人工升级，不修改单次会话决定
  • 两种模式都与批准基线比较，并保留无法比较或证据不足的明确结论

概要处理流

  1. 核验目标：确认 checkpoint、模型包或场景，以及数据、对象、策略、环境和评测套件的不可变指纹。
  2. 取得证据：模型模式生成逐样本预测；场景模式等待 execute → observe → run coverage → Acceptance Decision 链完成。
  3. 分层计量：计算质量、校准、任务结果、过程合规、恢复、升级与资源表现。
  4. 对照分析：与批准基线和预声明消融或场景对照比较，定位差异来源。
  5. 形成结论：输出通过、拒绝、需复核或证据不足，并精确列出失败切片和契约项。

它读取什么，产生什么

方向对象作用
模型输入Checkpoint Candidate / Model Package Candidate绑定 noemtrain 清单、数据快照、量化谱系、模型与任务头身份
场景输入Agent Scenario + Final Run Evidence绑定 Signed Package、Execution Profile、Run Result、Acceptance Decision、Run Report、Normalized Trace、Trace Integrity Report 与 Evidence Closure Report
输入Evaluation Suite + Baseline Registry冻结数据/场景、切片、扰动、指标、阈值、用途、执行后端和可比基线
输出Evaluation Bundle逐样本或逐场景记录、指标、置信区间、契约结论与环境指纹
模型输出Model Qualification Record绑定目标、训练/量化谱系、套件和阈值，标明 eligible-for-quantization、eligible-for-bundle、rejected 或 review-required
场景输出Scenario Evaluation Decision离线通过、拒绝、需复核或证据不足；不修改原 Acceptance Decision
输出Regression Report相对基线的模型质量、任务行为、资源和过程合规变化

它不会做什么

  • 不以平均准确率或单一任务成功率掩盖高风险切片退化
  • 不在评测集上调参后继续把同一结果作为盲测结论
  • 不把模型指标等同于 Noesis Core、NOBJ、Agent Harness 或 Fulfillment Runtime 的正确性证明
  • 不把 Agent Harness 自述成功当作任务完成证据
  • 不向 noemcoverage 反向提供同一次运行的必需证据，也不修改单次 Acceptance Decision
  • 缺少数据血缘、环境身份、运行证据或污染检查的结果不能用于模型资格或场景结论
  • 不把统计显著性自动解释为工程价值、发布授权、原创性或标准化成熟度

模型、模型包、工具输出和运行证据一律按不可信输入验证。Model Qualification Record 只证明指定用途的评测要求已经满足，不能替代 noembundle 的发布闭包、外部签名或发布授权。

尚未冻结的评估契约

模型指标、任务指标、数据与场景分区、统计置信、过程合规、回归标准和结果解释仍需分别规范，并明确哪些结论只支持探索分析，哪些可以用于工程验收或标准一致性判断。
