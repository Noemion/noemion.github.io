noesis · 确定性语义编译器

接收受控 NSL、Candidate Envelope 或外部候选来源，再由 Noesis Core 完成来源绑定、语义检查和 NOBJ 写入。

Phase 3 / Phase 7Compiler & LinkSource / Candidate → NOBJDesign Stage

它解决什么问题

noesis 统一驱动 Deterministic、Model-assisted 与 External-generation 前端；只有确定性 Noesis Core 能完成语义绑定、覆盖审计和 NOBJ 写入。

模型候选、受控 DSL 和对象写入如果处于同一信任层，系统无法证明布局与约束由规范决定。noesis 将多种前端统一收敛到同一个无模型、确定性 Noesis Core。

当前状态

当前状态：尚未发布可执行版本，命令行界面、参数和文件扩展名仍在设计中。当前公开信息包括计划用途、输入输出和安全边界；安装与调用说明将在首个可用版本发布时提供。

开发顺序：Phase 3 先实现 Deterministic 前端、类型/约束/覆盖检查和确定性对象写入器。Model-assisted 与 External-generation 将在 Deterministic 拒绝行为稳定后开发，并验证关闭模型时对象格式、核心语义和测试保持不变。

它怎样工作

选择前端配置
构造 Candidate Envelope
核对来源绑定权限
确定性类型与约束检查
编译来源覆盖
写入可重定位对象
  • Deterministic Profile 解析 NSL 并由版本化规则确定性生成 NIR
  • Model-assisted Profile 使用锁定的 Horizon Engine Model Package 产生 Candidate Envelope，不直接产生 NIR
  • 每项确认语义必须能从受控规则重推导或绑定获授权的 Source Binding Decision；否则保留未决或失败
  • External-generation Profile 对外部模型生成的 NSL 或 Candidate Envelope 应用同一边界
  • 构建来源图、证据账本、验收契约与可重定位 NOBJ

它读取什么，产生什么

方向对象作用
输入NSL SourceDeterministic Profile 的受控声明式源
输入Source Package + Candidate Envelope模型辅助或外部生成路径的来源、跨度、替代解释、证据、置信信息和未决状态
输入Frontend Binding + Model Qualification Record模型辅助时绑定已取得 eligible-for-bundle 资格的 Model Package、阈值、本体库和候选协议
输入Source Binding Decision对不能由受控规则重推导的候选主张提供获授权、可追溯的显式确认
输入Compiler 配置规范版本、Profile、限制、目标 ABI 和复现环境
输出Relocatable NOBJ规范化 NIR、符号、重定位、产物与调试 Section
输出证据账本每个来源单元的推导依据、绑定权限、未决、精确保留、丢弃或阻断状态
输出Build 清单输入哈希、工具/模型指纹、Source Binding Decision、依赖与可复现构建字段

它不会做什么

  • 模型输出不能直接成为 NIR/NOBJ
  • 不把结构合法、类型正确或高置信候选自动升级为来源忠实语义
  • 不在来源绑定或覆盖缺口存在时静默发布
  • 不承担跨对象符号解析和最终链接
  • 不接受 Agent Harness 以运行便利为由跳过类型、来源或覆盖检查

所有源、候选、资源和对象输入均视为不可信；编译必须记录规范版本、配置、依赖与工具/模型指纹。模型质量或置信度不能替代来源绑定、类型、覆盖、权限和安全证明。
