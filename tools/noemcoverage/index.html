noemcoverage · 语义与执行证据覆盖工具

分别建立 Release Coverage Proof 与 Run Evidence Closure；它证明映射和证据可定位，不替代最终验收判断。

Phase 5 / Phase 6Evidence CoverageRelease Proof / Run ClosureDesign Stage

它解决什么问题

noemcoverage 在发布前逐来源单元证明输入已结构化、精确保留、显式丢弃或阻断；在运行后检查验收契约的必需证据是否真实出现且能绑定同一会话、对象和环境。

字符覆盖率无法说明语义去向，模型自述成功也无法说明任务结果得到验证。noemcoverage 因此提供两种互不替代的结果：Release Coverage Proof 面向发布对象，Run Evidence Closure 面向一次运行。证据齐全只说明可审查，不自动说明证据内容为真或任务最终 accepted。

当前状态

当前状态：尚未发布可执行版本，命令行界面、参数和文件扩展名仍在设计中。当前公开信息包括计划用途、输入输出和安全边界；安装与调用说明将在首个可用版本发布时提供。

开发顺序：先稳定来源图、Assembly/Compiler Evidence Ledger、Debug Companion、Coverage Policy 与验收契约；随后验证跨文件条件、否定、例外、重复、冲突、不可安全摘要材料，以及 Run Report、Normalized Trace 或 Trace Integrity Report 缺失、伪造、采样不足或无法关联的情况。

它怎样工作

锁定身份
选择 Release / Run 模式
对齐对象或运行证据
检查完整性与必需项
应用 Coverage Policy
输出 Proof / Closure
  • Release 模式关联 Source Package、最终 Release Object、Debug Companion 和 Assembly/Compiler Evidence Ledger
  • 检查目标、条件、例外、硬约束、权限和精确载荷是否在发布对象中有明确去向
  • Run 模式关联 Run Record、Run Report、Normalized Trace、Trace Integrity Report、Candidate Assessment 与确定性验证结果
  • 先判断事件链是否完整、是否采样或丢失，再检查验收契约的必需证据是否出现；不把 Runtime 评价当作离线 noemevaluate 结果
  • 分别生成发布证明、运行证据闭包、风险分级和缺口列表，不用任一层替代另一层

它读取什么，产生什么

方向对象作用
发布输入Source Package + Evidence Ledger原始来源、稳定来源图，以及 noemcompile 或 noemassemble 产生的映射账本
发布输入Release Object + Debug Companion最终发布对象，以及由 noemreduce 分离的来源、覆盖与调试映射
运行输入Run Record + Run Report会话、对象、环境、候选、Candidate Assessment 与确定性检查身份
运行输入Normalized Trace + Trace Integrity Report由 noemobserve 形成的事件序列，以及生产者、哈希链、丢失、采样和完整性结论
输入Coverage Policy风险级别、允许丢弃类别、人工确认、产物和验收证据规则
发布输出Release Coverage Proof绑定来源、Release Object、Evidence Ledger 与策略的可验证摘要，供 noembundle 使用
运行输出Evidence Closure Report每个验收契约项的证据存在性、身份、完整性、强度与缺口，供 noemexecute finalize 使用
输出Gap Report未覆盖、冲突、可疑摘要、跨文件断链、事件丢失和缺失运行证据

它不会做什么

  • 不证明任意自然语言与 NIR 数学等价
  • 不把 token/字符覆盖率当语义覆盖率
  • 不自动批准有损摘要
  • 不允许模型高置信或自述成功掩盖缺失证据
  • 不把运行覆盖通过当作发布对象覆盖通过，反之亦然
  • 不签署发布包，也不单独形成 Acceptance Decision

Release Coverage Proof 与 Evidence Closure Report 只证明给定 Policy、绑定身份和可用证据下的映射与审计过程。未解决、冲突、高风险有损映射、事件完整性不足或必需证据缺失必须阻断对应发布或验收结论。
