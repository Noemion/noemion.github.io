noembundle · 可验证发布包工具

冻结发布对象、覆盖与验证证据、依赖、模型和策略，并通过外部签名回填形成可离线验证的发布包。

Phase 5Release & Fulfillment RuntimeRelease Closure → Signed PackageDesign Stage

它解决什么问题

noembundle 冻结 Release Object、Release Coverage Proof、验证结论、强依赖、合格模型、本体库、策略和构建证明，并通过外部签名系统返回的签名结果形成最终发布包。

对象本身有效并不代表分发闭包完整；浮动依赖、模型资格、策略、覆盖证据或许可证缺失都会让接收方无法复现和验证。私钥也不应进入普通打包进程，因此候选包、签名请求、外部签名和最终封装必须具有明确边界。

当前状态

当前状态：尚未发布可执行版本，命令行界面、参数和文件扩展名仍在设计中。当前公开信息包括计划用途、输入输出和安全边界；安装与调用说明将在首个可用版本发布时提供。

开发顺序：先确定 HOBJ 闭包、Release Coverage Proof、分层验证结论、Model Qualification Record、SBOM、Signing Request、Signature Response 与 Signature Envelope 规范；随后验证离线重建、依赖替换、路径穿越、压缩炸弹、许可证缺失、签名回填错配和清单篡改。

它怎样工作

验证发布输入
解析锁定闭包
构建候选包
生成签名请求
核对三项回填输入
封装并离线复验
  • 收集最终 Release Object、Release Coverage Proof、分层验证结论和锁定强依赖闭包
  • 模型包必须绑定 noemevaluate 的 eligible-for-bundle Model Qualification Record，再嵌入或内容寻址关联词表、本体库与策略
  • 生成 Manifest、SBOM、不可变 Unsigned Package Candidate、Signing Request 和构建元数据
  • 回填时同时读取原候选包、原 Signing Request 和外部 Signature Response，核对包身份、算法、签名范围、签名者和策略版本
  • 最终包只在不可变候选载荷外附加 Signature Envelope，不重写任何被签字节；离线验证器从 Envelope 重建同一待签消息
  • 签名验证通过后才输出 Signed Noemion Package，并再次执行离线完整性检查

它读取什么，产生什么

方向对象作用
构建输入Release Object + Evidence已裁剪的发布对象、Release Coverage Proof、分层验证结论和等价证据
构建输入依赖与模型锁定强依赖、本体库、策略，以及 Model Package Candidate 与 eligible-for-bundle 资格记录
构建输入Package 配置平台、嵌入策略、压缩、签名算法、信任策略和预算
回填输入Unsigned Package Candidate首次构建输出的不可变候选载荷；身份必须与 Signing Request 一致
回填输入Signing Request首次构建输出的候选身份、签名范围、算法和待签消息
回填输入Signature Response外部签名系统返回的签名、签名者、请求身份和策略版本
构建输出Unsigned Package Candidate内容已冻结但尚不可作为执行发布物的候选包
构建输出Signing Request候选包身份、签名范围、算法和待签消息
回填输出Signed Noemion Package原候选载荷、Signature Envelope、Manifest 与 SBOM；被签载荷字节保持不变

它不会做什么

  • 不在打包时解析浮动版本
  • 不自动下载锁文件之外的依赖
  • 不替代 noemvalidate、noemcoverage 或许可证审查
  • 不允许 thin/external 强依赖进入离线发布配置
  • 不持有私钥、不替签名系统授权，也不把未签名候选标记为可执行发布包

所有对象、证据、模型、清单和签名响应均视为不可信；路径、压缩、计数、签名范围和解包预算必须受限。打包只冻结已批准闭包，不能替代组件本身的许可、安全或语义验证。
