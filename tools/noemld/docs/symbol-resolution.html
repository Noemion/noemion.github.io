符号解析

强弱绑定、版本、可见性与归档成员选择

noemld Phase 4 / Phase 5

使用者判断：符号名称相同不代表定义等价。解析必须同时考虑命名空间、类型、版本、可见性和绑定强度，并把选择理由写入 Link Map。

设计目标

符号解析负责把“谁引用了什么”转化为唯一、可解释的定义选择。名称相同并不代表语义相同；版本、命名空间、类型、可见性和绑定强度都属于解析键。任何未写进规则的输入顺序偏好都会破坏复现，也会让攻击者通过排列对象影响最终行为。

解析规则

  • 扫描显式对象和归档索引，建立全局符号表。
  • 按可见性、版本和强弱绑定选择定义。
  • 只拉入满足当前未解析强引用所需的归档成员。
  • 链接顺序不得改变规范允许范围之外的结果。

候选解析键与选择顺序

下列字段与优先级是详细设计候选，最终形式必须由符号与版本规范冻结。
  1. 以命名空间、规范化名称、符号种类、版本要求和目标 Profile 构造查询键。
  2. 过滤不可见、版本不满足、类型不兼容或来自未准入依赖的定义。
  3. 按强定义、弱定义、仅声明等规范类别形成候选集合；不能用文件遍历顺序打破平局。
  4. 若规范规则仍不能产生唯一选择，则生成冲突；只有规范明确允许的可替代弱定义才能用稳定键选择。
  5. 将选择、被拒候选和原因写入 Link Map，再驱动归档成员闭包扩展。

冲突矩阵

情况候选结果必须记录
一个兼容强定义选择该定义。定义来源、版本、可见性和类型。
多个强定义失败，除非规范声明它们是可合并的同一内容身份。全部候选和冲突字段。
只有兼容弱定义按冻结的稳定规则选择或合并。所有候选与排序键。
弱引用无定义仅按 Profile 定义的缺失语义继续。缺失状态及下游行为。
强引用无定义失败。引用链、搜索过的依赖和版本约束。
同名但类型不兼容失败，不允许名称匹配掩盖类型冲突。期望类型、实际类型和来源。

失败条件

  • 未解析强符号必须失败。
  • 版本不匹配必须失败。
  • 多个强定义不得静默选择最后写入者。

归档成员选择应以不动点算法终止，并由 Profile 约束扫描轮数、成员数和符号数。循环别名、版本回退链和恶意碰撞必须有专门的负向测试向量。

验证证据

解析算法放行需要证明：输入排列不改变结果；显式对象与归档索引得到一致的定义语义；强弱、版本、可见性和类型组合的冲突矩阵全部覆盖；每次选择都能从 Link Map 回溯到规范规则。若未来提供增量链接，其结果还必须与完整重链接等价。
