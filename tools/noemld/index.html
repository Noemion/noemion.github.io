noemld

Noemion Linker：面向 GOBJ、归档与 SSO 的确定性链接工具项目。

Phase 4 / Phase 5Compiler & LinkDesign Stage

工具简介

noemld 负责解析符号、重映射对象内部 ID、应用类型化重定位，并按语义规则组合约束、权限和依赖闭包。它不调用模型解决冲突，也不承担 Runtime 求解。

问题背景：生成式对象的链接不仅合并字节和符号，还会组合硬约束、权限、Artifact Contract、披露图和依赖。普通最后写入者规则会隐藏语义冲突或扩大权限，因此 noemld 必须把组合规则和失败证据结构化。

当前状态

设计阶段：当前未发布可执行程序；命令行接口、参数和文件扩展名尚未冻结。当前页面仅描述已经确认的职责边界和仍待规范或 ADR 冻结的接口。

阶段门：进入 Phase 4 前必须稳定符号、归档、重定位和冲突规则；进入 Phase 5 前必须稳定 SSO 闭包、权限与披露合并。放行证据包括链接顺序不影响规范结果、冲突矩阵完整和相同输入产生相同输出。

主要能力

扫描输入与归档
解析符号
映射 ID 与重定位
合并约束/权限/闭包
提交输出与报告
  • 解析强弱符号、版本、可见性与所需归档成员。
  • 合并兼容 Section、GSIR 图与 Contract，并重映射节点和 Artifact ID。
  • 应用类型化重定位，构建强依赖闭包、Link Map 与结构化冲突报告。
  • 按语义类型处理约束、权限和披露规则冲突，不采用静默覆盖。

输入与产物

方向对象关键契约
输入GOBJ、归档、SSO结构已验证，但符号、约束与依赖仍待全局解析
输入Link Profile 与 Dependency Lock声明输出形态、目标 ABI、限制、冲突策略和内容寻址依赖

产物Link Map记录输入成员、符号选择、重定位、Section 和摘要计划
产物Conflict Report定位未解析、重复、类型、约束、权限与依赖冲突

处理边界

  • 不调用模型解决符号、类型、约束或权限冲突。
  • 不承担 Runtime 的生成式求解与 Lowering。
  • 未知强制 Section、越界索引、非法对齐和溢出必须稳定拒绝。
  • 歧义和冲突必须保留为结构化未决信息，不得静默覆盖。

所有对象输入均视为不可信；偏移、长度、计数、索引和对齐计算必须使用 checked arithmetic。

链接采用先计划、后提交的事务边界；任一强制检查失败时不得输出看似可用的部分链接对象，也不得调用模型猜测冲突选择。
