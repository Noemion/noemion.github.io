工具简介

为什么部署转换需要独立验证

浮点 checkpoint 通过评估，并不意味着导出、算子替换或低精度表示后仍保持相同行为。量化可能对任务头、置信校准、OOD 拒绝和长结构样本产生非均匀影响，而桌面基准也不能代表目标设备的内存、延迟和能耗边界。

noemquant 将通过评估的 NSFE checkpoint 导出为目标后端模型，执行可诊断量化、端侧基准与模型包封装，并保留从浮点基线到部署包的完整精度谱系。

当前状态

当前状态：尚未发布可执行版本，命令行界面、参数和文件扩展名仍在设计中。当前公开信息包括计划用途、输入输出和安全边界；安装与调用说明将在首个可用版本发布时提供。

部署转换准备

转换前需要确定浮点基线、独立校准集、目标硬件与运行时画像、误差预算以及分层质量阈值。候选部署包必须同时满足模型质量和真实设备资源要求；后端或硬件变化后需要重新验证。

主要能力

  • 导出 ONNX 或经规范批准的端侧执行格式，并验证计算图与原模型的契约对应
  • 执行动态 INT8、静态校准或混合精度策略，保存所有回退与例外规则
  • 定位高误差层、算子、任务头和风险切片并生成诊断
  • 封装 tokenizer、schema、阈值、许可、目标画像和模型指纹

概要处理流

  1. 绑定基线：核验 checkpoint、浮点评估、校准集和目标画像。
  2. 导出验证：转换计算图并检查算子、形状、任务头和输出协议。
  3. 量化校准：应用配方，记录逐层误差、回退精度和不可量化节点。
  4. 双重评估：复跑质量切片，并在真实目标设备测量性能与资源。
  5. 封装交付：仅在双门通过后生成带签名清单和完整谱系的候选模型包。

输入与产物

方向对象作用
输入Approved Checkpoint绑定 noemtrain Manifest 和浮点评估基线
输入Quantization Recipe目标后端、精度、校准集、算子和误差预算

输出NSFE Model Package模型、tokenizer、schema、阈值、许可和摘要
输出Quantization Report逐层误差、回退精度和质量差异
输出Target Benchmark真实目标设备的延迟、吞吐、内存和能耗数据

处理边界

  • 不以不可解释的全局量化替代逐层、逐任务头和逐风险切片诊断
  • 不在质量门或真实设备资源门失败时强行发布
  • 不以模拟器或开发机基准替代目标硬件数据
  • 不修改 Compiler Core 或 GOBJ 格式，不让后端模型参与确定性布局
  • 不把模型包视为可信对象输入；Runtime 必须独立校验其清单、大小和兼容性

模型输出和模型包一律视为不可信；它们不能决定二进制布局，也不能直接生成 GOBJ。量化报告只对绑定的模型、校准集、后端和目标设备组合有效。

尚未冻结的部署契约

后端画像、校准协议、误差诊断、设备基准与模型包结构仍需规范化；ONNX、INT8 等只表示候选技术路径，不构成冻结格式或互操作承诺。
