项目/工具/noemcompile

noemcompile · 确定性语义编译器

接收受控 NSL、模型辅助候选或外部生成来源，再由 Noesis Core 完成绑定、覆盖检查和 NOBJ 写入。

Phase 3 / Phase 7Compiler & LinkNSL / 自然语言 → NOBJDesign Stage

它解决什么问题

noemcompile 统一驱动 Deterministic、Model-assisted 与 External-generation 前端；只有确定性 Noesis Core 能完成语义绑定、覆盖审计和 NOBJ 写入。

模型候选、受控 DSL 和对象写入如果处于同一信任层，系统无法证明布局与约束由规范决定。noemcompile 将多种前端统一收敛到同一个无模型、确定性 Noesis Core。

当前状态

当前状态：尚未发布可执行版本，命令行界面、参数和文件扩展名仍在设计中。当前公开信息包括计划用途、输入输出和安全边界；安装与调用说明将在首个可用版本发布时提供。

开发顺序：Phase 3 先实现 Deterministic 前端、类型/约束/覆盖检查和确定性对象写入器。Model-assisted 与 External-generation 将在 Deterministic 拒绝行为稳定后开发，并验证关闭模型时对象格式、核心语义和测试保持不变。

它怎样工作

选择受信任前端模式
构造候选与来源
确定性绑定和检查
覆盖与契约审计
写入可重定位对象
  • Deterministic Profile解析 NSL 并确定性生成多层 NIR
  • Model-assisted 模式接收 Horizon Engine 封闭候选并逐项验证或拒绝
  • External-generation 模式检查外部模型生成的 NSL
  • 构建来源图、证据账本、验收契约与可重定位 NOBJ

它读取什么，产生什么

方向对象作用
输入NSL SourceDeterministic Profile的受控声明式源
输入Natural Language PackageModel-assisted 模式的文件、资源、目录和策略
输入Compiler 配置规范版本、模式、本体库、限制和目标 ABI
输出Relocatable NOBJ多层 NIR、符号、重定位、产物与调试 Section
输出证据账本每个来源单元的结构化、精确保留、丢弃或阻断状态
输出Build 清单输入哈希、工具/模型指纹、依赖与可复现构建字段

它不会做什么

  • 模型输出不能直接成为 NIR/NOBJ
  • 不在覆盖缺口存在时静默发布
  • 不把低置信候选自动升级为唯一语义
  • 不承担跨对象符号解析和最终链接
  • 不接受 Agent Harness 以运行便利为由跳过类型、来源或覆盖检查

所有源、候选、资源和对象输入均视为不可信；编译必须记录规范版本、配置、依赖与工具/模型指纹。模型质量或置信度不能替代类型、覆盖、权限和安全证明。
