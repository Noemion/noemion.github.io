Phase 3Compiler & LinkGSL / GSIR 诊断Design Stage

工具简介

noemlint 在不生成对象的情况下检查未绑定概念、约束冲突、权限扩大、不可达披露节点和不稳定依赖。

问题背景：完整编译和验证成本较高，而许多语义、策略和维护问题可以在编辑期发现。noemlint 提供快速反馈，但必须与规范错误和发布门清楚区分。

当前状态

当前状态：尚未发布可执行版本，命令行界面、参数和文件扩展名仍在设计中。当前公开信息包括计划用途、输入输出和安全边界；安装与调用说明将在首个可用版本发布时提供。

开发计划：每条规则需要稳定 ID、适用语法或规范版本、严重度依据、正反样例和误报处理方式；接入 CI 前将验证 Baseline 不会隐藏新增高风险问题。

主要能力

快速解析
建立局部语义环境
运行规则集
应用受审查 Baseline
输出可定位诊断
  • 快速解析 GSL 或文本 GSIR 并运行静态语义规则
  • 检查模态、否定、条件、例外和作用域可疑组合
  • 检测未使用符号、遮蔽、循环依赖和不可达节点
  • 支持规则级启停、严重度 profile 与 CI 基线

输入与产物

方向对象作用
输入GSL/Text GSIR单文件、目录或 stdin
输入Lint Profile启用规则、严重度、领域策略和忽略清单
输入Ontology Snapshot概念、关系和弃用规则
输出Diagnostics规则 ID、严重度、源范围、证据和建议
输出SARIF/JSON Report供 CI、编辑器与审计系统消费
输出Baseline受审查的既有问题指纹

处理边界

  • 不自动消除语义歧义
  • 不改写用户约束以通过检查
  • 不替代完整 noemc 类型检查和 noemverify
  • 不把 warning 当作发布通过证据

规则关闭、严重度降低和 Baseline 变更必须可审计；过期 ontology 或 Profile 应产生明确状态，不能让“无诊断”被误解为语义有效。
