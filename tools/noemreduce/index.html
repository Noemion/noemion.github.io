noemreduce · 发布对象裁剪工具

从开发对象中移除不参与运行的调试信息、训练溯源和开发决策记录，同时证明运行语义、依赖和披露行为保持不变。

Phase 5Release & Fulfillment RuntimeDevelopment NOBJ / HOBJ → Release ObjectDesign Stage

它解决什么问题

noemreduce 移除非运行时 .debug.*、训练溯源和开发决策记录，同时证明运行语义、依赖和披露行为保持不变。

开发信息可能占用大量空间或包含敏感来源，但“删除后还能解析”不能证明发布语义不变。noemreduce 的成功标准是运行闭包、约束、权限、依赖和披露行为等价，而不只是文件更小。

当前状态

当前状态：尚未发布可执行版本，命令行界面、参数和文件扩展名仍在设计中。当前公开信息包括计划用途、输入输出和安全边界；安装与调用说明将在首个可用版本发布时提供。

开发顺序：先完成 Section 分类、规范属性比较、披露图，并接入 noemcompare 与 noemvalidate；随后验证多种 Strip 配置下的等价性、分离调试关联和旧签名失效。

它怎样工作

验证开发对象
计算运行闭包
分离可移除信息
重建索引与对象摘要
证明规范属性等价
  • 按发布配置选择可移除调试、来源和构建 Section
  • 生成分离 Debug Companion 与关联标识
  • 重建索引、偏移、哈希和供发布阶段使用的完整性输入材料
  • 明确使任何覆盖旧字节的签名失效；最终签名范围由 noembundle 建立
  • 调用 noemcompare/noemvalidate 证明目标、约束、权限、依赖和披露属性等价

它读取什么，产生什么

方向对象作用
输入Development Object包含运行与 .debug.* 的 NOBJ/HOBJ
输入Strip 配置release/minimal/separate-debug 与保留清单
输入Equivalence Policy必须保持的目标、约束、权限、依赖、验收与披露属性；语义哈希需等待等价规则冻结
输出Release Object仅含运行闭包与必要验证摘要的对象
输出Debug Companion可选分离调试、来源和 Coverage 证据，并以完整内容摘要绑定主对象
输出Strip 清单移除项、保留原因、摘要与等价证明

它不会做什么

  • 不删除运行闭包、硬约束或验收契约
  • 不以文件变小作为成功标准
  • 不保留覆盖已变化字节的旧签名
  • 不恢复或重新生成缺失调试信息

所有对象输入均视为不可信；无法证明某 Section 与运行语义无关时必须保留或拒绝。输出对象和 Debug Companion 需要不可混淆的关联标识与访问策略。
