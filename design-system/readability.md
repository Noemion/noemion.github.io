# 长内容可读性

适用范围：技术说明、规范解读、架构文章、工具项目介绍和手册正文。页面所属角色的设计文档继续决定内容结构；本文件只定义跨角色的阅读尺度、信息栏职责和版式边界。

## 首要原则

- 阅读路径优先于版式变化。连续论证不得为了填满画布被拆成无上下文的卡片，也不得为了视觉对称被拉成超长文本行。
- 1200px 是页面画布，不是段落宽度。普通正文保持约 700–760px 的舒适阅读列；中文正文不得长期占满 1000px 以上宽度。
- 桌面正文基线为 17–18px、约 1.75–1.8 行高。长篇技术页的二级标题以 30–34px 为主，避免标题远大于正文却没有形成更清楚的层级。
- 当前共享实现以桌面 18px / 1.78 为长正文默认值；工具状态栏、导航、徽标、表格元数据等辅助界面可以保持 13–16px，但不能反向缩小主阅读列正文。
- 每段只承担一个主要论点。定义、原因、机制、限制和例子按自然论证顺序连续出现；只有精确映射、比较、状态和短步骤才改用表格、卡片或横向行。
- 首次出现的缩写或项目术语先给直白解释，再给精确定义；提高可读性不能删除规范性强度、不变量、失败语义、成熟度或权威来源。

## 1200px 桌面阅读框架

- 长篇技术页与工具项目页优先使用约 800px 主阅读列和约 380px 信息栏，中间以一条低对比分隔线建立连续画布。
- 信息栏从首屏开始承担真实职责，不是为消除空白而添加的装饰。可包含主题封面、成熟度、可用性、关键对象、章节目录、输入输出摘要和下一阅读入口。
- 信息栏在桌面使用 `position: sticky`，顶部与 64px 导航对齐；自身内容不能超过可视高度，超出时只允许信息栏内部滚动。
- 信息栏内容不得重复主栏整段正文，也不得包含内部制作说明。没有足够有用信息时回到单列阅读，不保留空栏。
- Hero 与正文使用同一列线：标题和摘要对齐主阅读列，主题图形或状态摘要对齐信息栏。不能在 Hero 结束后突然改变左右栏比例。
- 宽表格、对象布局、流程图和大型对照可以临时跨越完整画布；其前后解释仍回到主阅读列，并提供局部横向滚动而不是扩大所有段落。

## 普通专题

- `content-stack`、`content-band`、`content-split`、`content-grid`、`content-rows` 和 `content-wide` 只决定章节组织，不得解除正文最大行宽。
- `content-rows` 的整行分隔可以横跨画布，但每行使用独立的序号/标签列与受限文字列；正文文字不直接铺满 1080px。
- `content-split` 的标题侧栏用于章节定位，右侧论述保持约 680–760px；镜像版式不得让读者在左右两边来回寻找下一段。
- 普通专题中由直接语义章节产生的 `h2` 达到 6 个时，共享脚本必须在 Hero 后自动生成一份页内目录；少于 6 个时不生成，避免短页面增加无意义的导航层。
- 目录只读取正文中真实存在的章节标题。作者只需写有意义的 `h2`，可在需要长期外链时提供稳定 `id`；生成器必须保留已有 `id`，缺失时才使用确定性的顺序标识，不得要求作者在正文之外手抄第二份链接清单。
- 自动目录必须带有可理解的导航标签，链接文字与对应标题一致，跳转位置避开固定页头。窄屏时目录只在自身横向滚动，不得撑宽页面，也不得改变章节的 DOM 阅读顺序。
- 自动目录适用于没有手册布局的 `page_role: content` 页面。手册使用自身动态目录；工具项目页使用固定的工具信息栏和六章阅读结构，不叠加这份自动目录。

## 工具项目页

- 六个语义章节继续位于主阅读列，形成从定位、状态、能力、输入输出、边界到资源的连续说明。
- 右侧信息栏从 Hero 起形成一块连续工具面板：上部显示工具专属封面与签名，下部显示可用状态、开发阶段、关键输入输出和手册入口。
- “当前状态”仍是正文第二章的语义内容；桌面可在信息栏中呈现，移动端必须按第二章顺序回到主内容，不得只留一份脱离阅读顺序的副本。
- 工具正文使用 17–18px、约 30–32px 行高；单段建议控制在约 700–740px。输入输出表可以使用主栏完整宽度。
- 信息栏固定期间，后续主栏章节自然滚动；相关资源结束后不再用空白高度维持两栏等高。

## 手册

- 手册保留左侧动态目录与右侧阅读区，不再叠加第二个固定信息栏。
- 阅读区内普通正文继续限制在约 720–760px；表格、代码和对象结构可以使用阅读区完整宽度。
- Markdown 作者只负责语义内容；行宽、字号、目录固定、当前章节提示和响应式行为由 Jekyll 布局及共享样式统一生成。
- 每张由 Markdown 生成的表格必须拥有自己唯一的 `.table-wrap.manual-table-wrap` 滚动容器。增强脚本只包装尚未包装的表格，保留原生 `table` 语义，不产生嵌套滚动层，也不要求作者书写 HTML 容器。
- 手册表格以约 680px 的最小内容宽度保存列关系；容器不超过阅读区宽度，并使用局部 `overflow-x: auto` 与横向过滚动约束。桌面能完整容纳时不制造额外滚动，窄屏上的触摸、触控板或滚轮操作只移动该表格，不得带动整页横向位移。
- 1218px 及以上可显示 320px 动态目录和约 880px 阅读区。1000–1217px 必须隐藏目录侧栏，把手册变成最大 1000px、两侧至少 18px 留白的居中单列阅读页；该区间不是把双栏硬塞进窄画布，也不能为隐藏侧栏保留空白占位。1000px 以下继续沿用移动目录和单列正文。

## FAQ

- FAQ 的问题标题保持清楚的可操作层级，答案正文使用 17px、约 1.75 行高，最大行宽 760px；展开后不得为了与问题行等宽而把答案拉满画布。
- 答案中的段落、列表与链接沿用正文对比度和自然换行。折叠组件只负责渐进披露，不得用更小字号隐藏重要限制，也不得把一个需要连续论证的长篇章节拆成大量问答。

## 响应式与验收

- 1000px 以下取消粘性信息栏，按语义顺序合并为单列；移动端正文使用 16–17px、至少 1.7 行高。
- 390px 下不通过缩小字体维持桌面行数；表格和代码只在自身容器滚动。
- 桌面验收必须测量普通段落实际宽度、字号和行高，不能只检查页面是否溢出。
- 每类页面至少检查首屏、长段落、列表、表格、最后一个章节和粘性信息栏的起止状态；FAQ 还要检查展开答案宽度，长专题要分别检查 5 个与 6 个 `h2` 的目录阈值，手册要覆盖 999px、1000px、1217px 与 1218px 的布局边界。深浅主题均须保持正文与次级文字对比度。
