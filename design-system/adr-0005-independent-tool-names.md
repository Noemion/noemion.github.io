# ADR-0005：项目品牌与工具命令独立命名

- 状态：Superseded by ADR-0008
- 日期：2026-07-12
- 影响范围：工具名称、正式路由、手册、导航、测试与未来 CLI

## 问题

把全部工具命令统一写成 `noem` 前缀能够显示归属，但也会让名称只表达“属于 Noemion”，不能表达工具在对象生命周期中的独立角色。GNU Binutils 同属一个工具集合，却使用 `as`、`ld`、`objdump`、`readelf` 等不同命令；项目品牌、工具集合和单个命令并不需要共享字面前缀。

另一方面，如果为了哲学风格给每个工具强造陌生名称，开发者会失去动作线索，命令检索、脚本阅读和错误定位都会变差。命名需要同时保留工程清晰度与项目思想来源。

## 决定

1. `Noemion` 是项目、对象体系和发布来源的品牌，不是所有命令的强制前缀。
2. 工具可以采用独立名称，但必须同时给出直白的中文职责副标题；未来 CLI 的 `--help` 也必须先说明输入、动作和输出。
3. 哲学名称只提供角色隐喻，工程职责仍由规范、数据结构、不变量、诊断和测试定义。
4. 命令名、路由名和工具显示名可以一致，但不得写入 NOBJ 魔数、Section 类型、格式版本或语义身份；工具改名不得改变对象字节。
5. 只有名称能够提高职责辨识度，并通过大小写无关冲突、常见包管理器、系统命令和商标初筛后才采用。法律与商标可用性仍需发布前独立复核。
6. 没有更好独立名称的工具继续使用现有动作型工作名，不为对称性批量重命名。

## 首批采用名称

| 旧工作名 | 当前名称 | 思想线索 | 工程解释 |
| --- | --- | --- | --- |
| `noemcompile` | `noesis` | 理解与判断活动 | 驱动确定性 Noesis Core，把来源或受控候选形成 NIR/NOBJ。 |
| `noemassemble` | `morphe` | 形式与结构 | 把已经显式化的 Text NIR 赋予确定的对象形式，不新增语义。 |
| `noeminspect` | `theoria` | 观察与考察 | 只读查看对象布局、记录、符号、重定位和来源，不宣告验证通过。 |
| `noemlink` | `synthesis` | 综合与组合 | 把局部对象、符号、约束和依赖组合为全局一致对象。 |

`noesis` 与组件 `Noesis Core` 的关系是“工具驱动组件”，不是同一个接口；标题与手册必须明确这一点。`morphe`、`theoria` 和 `synthesis` 的哲学词源不构成对象格式规则。

## 路由与迁移

正式路由迁移到 `tools/noesis/`、`tools/morphe/`、`tools/theoria/` 和 `tools/synthesis/`。项目处于开发阶段，不保留旧路径、别名、重定向或兼容命令。导航、手册、样式选择器、测试和交叉链接必须在同一变更中更新。

## 依据与限制

- GNU Binutils 使用独立命令名组织同一工具集合：https://www.gnu.org/software/binutils/binutils.html
- Noemion 的哲学采用边界见 `about/intellectual-foundations.html`；哲学术语只能提出角色区分，不能替代工程定义。
- 首批名称仍是开发阶段接口；首个可安装版本发布前需要完成命令冲突、域名、软件包、商标和跨语言可读性复核。
