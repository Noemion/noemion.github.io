# Noemion 语言与命名规范

本规范适用于所有正式页面、手册、组件、制品、命令、目录、文件名和导航标签。它同时解决三件事：让中文技术文档像开发者之间的正常说明；让新领域拥有短、可说、可检索的工程语言；让名称严格服从数据结构、信任边界和测试，而不是反过来决定架构。

## 已冻结的命名立场

1. **Noemion 只作为项目与新领域的品牌。**它不进入格式名、对象类型、命令、ABI、魔数或组件前缀。
2. **Endem 是核心制品名。**它由英语 `end` 与语言学中表示最小区别单位的 `-eme` 融合而成，不是缩写，也不是在既有目标文件术语上添加前缀。
3. **唯一公开应用和 CLI 都叫 `endem`。**正式文件扩展名是 `.endem`；公开动作只有 `poie`、`elenk`、`pleko`、`tasse`、`sphra`、`theor`、`praxe` 与 `peira`。
4. **不保留旧名称。**路由、命令、文件名、标题、导航、测试与示例直接采用当前名称，不提供别名、重定向、兼容命令或迁移垫片。
5. **正式发行前必须通过名称门禁。**当前研究只能支持继续使用候选名，不能替代商标检索和法律判断。

语言学后缀 `-eme` 长期用于表示某一语言结构中的区别单位，例如 phoneme、grapheme 与 morpheme。词源参考：

https://www.etymonline.com/word/-eme

## 写作原则

1. 先回答读者要解决什么问题，再说明制品、动作或规则。不要以项目术语开头要求读者自行推断用途。
2. 使用主动语态和现在时。写“Poiet 解析引用并报告冲突”，不写“引用将会被解析并给出相关报告”。
3. 一个句子只表达一个主要判断。超过三个并列动作时改用步骤或列表。
4. 主语必须明确。避免“进行处理”“完成支持”“实现相关能力”“提供相应功能”等缺少执行者和对象的说法。
5. 直接描述输入、动作、结果和失败。应用页优先使用“读取什么—检查什么—输出什么—何时拒绝”的顺序。
6. 专有词第一次出现时给出直白工程定义。后文只在能够减少歧义时使用英文标识。
7. 规范使用“必须、不得、应当、可以”；说明性文字使用“会、用于、负责”。不要用“赋能、打造、沉淀、闭环、拉通、抓手”等宣传或管理套话。
8. 状态说明只写已经完成、正在设计、尚未提供和下一项工作。不要把内部提示词、制作过程或评审对话写给读者。
9. 示例必须说明前提和可观察结果。未冻结的参数、字节编号和能力不得写成可直接执行的接口。
10. 手册按任务组织，不按实现文件组织；参考章节仍要完整列出格式、动作、选项、错误和索引。

这些规则采用 GNU 文档的长期可读性原则：面向不了解主题的读者、先基础后进阶、首次定义专门术语、按用户问题而不是功能清单组织，并优先使用主动语态和现在时。

## 核心领域语言

| 名称 | 直白工程定义 | 不能被误解为 |
| --- | --- | --- |
| Noemion | 项目、新领域与社区品牌 | 格式前缀、组件层或 CLI 前缀 |
| Endem | 最小、独立有效且可验证的期望终态单元 | Prompt、任务消息、执行脚本或二进制化文本 |
| Synem | 多个 Endem 经符号、依赖、约束和内容身份解析后形成、可以确定性封装的组合闭包 | 文件列表、搜索路径结果、归档目录或运行时临时依赖 |
| Dromen | attested Endem 或 Synem 通过重新验证、策略检查和最小能力授予后形成的一次会话实现态 | 持久格式、签名制品、全局会话或永久权限 |
| Tekmor | 绑定明确主体和声明范围，并封装结构化观察 `phain`、方法、环境、结果与限制的证据 | 无范围的“通过”徽标、最终验收决定或模型自评 |
| Poiet | 决定规范语义、排序、布局与确定性字节的可信核心 | 模型推理服务、Theor 或 Praxor |
| Theor | `theor` 背后的独立只读实现，直接读取不可信字节并形成检查视图 | Poiet 解析器的包装、编辑器或生产写入路径 |
| Praxor | `praxe` 背后的受限装载与实现环境 | Poiet、通用 shell、无限能力会话或验收者 |

### Endem 的最小性

Endem 的“最小”不是字数最少，也不是把所有内容压入一条记录。采用以下可测试边界：

- 如果拆分后任一部分失去独立身份、来源、约束或完成判据，就不应继续拆分。
- 如果两个目标可以独立版本化、引用、验证、替换或失败，就必须拆成两个 Endem。
- 一个 Endem 可以引用其他 Endem，但必须独立说明自己的目标、约束、完成判据和未决信息。
- 自然语言只是来源表达之一；进入规范字节前，确定性 Poiet 必须形成类型化、可定位和可验证的结构。

### 六个语义面

每个 Endem 按 `rhem`、`semion`、`skena`、`telis`、`krin`、`apor` 的固定顺序表达语义。名称保持小写 ASCII，不翻译、不复数化，也不在不同配置中改名。

| 组 | 必须回答 | 规范边界 |
| --- | --- | --- |
| `rhem` | 来源中实际出现了哪些记号？ | 保留原始表达、主体、语言、媒体、范围、版本、摘要与可重定位位置；推断不能伪装成原文。 |
| `semion` | 这些记号怎样投影为符号、指称和关系？ | 保存作用域、极性、量化、时间和来源映射；模型改写不等于已确认意义。 |
| `skena` | 哪一种中性可能事态由对象和关系构成？ | 保存一个根对象—关系图及显式组合与极性；不得混入目标力量或实际观察。 |
| `telis` | 目标要求这个事态发生变化还是持续成立？ | 第一阶段只允许 `kine` 与 `mene`；禁止事项由 `skena` 的否定表达，不设置重复的回避力量。 |
| `krin` | 哪些结构化观察支持满足、反驳、观察不足或故障，谁有权决定？ | 保存 `phain` 类型、比较规则、必需 Tekmor、`agno/fault` 处理和权威；命题不能自称为真。 |
| `apor` | 投影、指称、关系、测量或权威还有什么没有确定？ | 保存候选、冲突、缺口、风险和解决权限；不得把最高概率猜测静默升级。 |

有效 Endem 必须具有可定位 `rhem`、已授权 `semion`、一个根 `skena`、一个适用 `telis` 和覆盖全部必需根路径的 `krin`。没有未决项时，`apor` 显式为空集合。六个名称是顶层类型化语义面，不是六段自由文本；条目类型、标识、关系位置、排序、限制和编码由 Endem 规范定义。

`semion`、`skena` 与适用 Tekmor 中的 `phain` 必须共享稳定符号身份和可比较关系位置。逻辑形式由对象—关系拓扑、角色和组合边显示，规范载荷不得依赖自填的 `logical_form`、`valid` 或 `true` 字段。违反 Profile 或没有任何允许投影的输入以 `aseme` 拒绝；存在候选但无法选择时进入 `apor`；意义已经确定但观察不足时返回 `agno`。

`apor` 是一等信息。Poiet、Theor、Praxor、控制平面和模型都不得用默认值、搜索顺序、概率最高候选或未经记录的人工判断静默消除它。只有具有明确来源和责任的决议才能确认投影，并且必须留下可审计关系。

## 动作与权限边界

唯一 CLI 采用 `endem <action>`。动作名由古希腊语动作词根缩写或再造；每次首次出现都必须同时给出直白职责，不能要求读者用词源猜测行为。

| 动作 | 输入与结果 | 实现与权限边界 |
| --- | --- | --- |
| `poie` | 把受控来源确定性形成 Endem | 由 Poiet 执行；模型只能提供不可信来源候选，不能决定规范字段或字节。 |
| `elenk` | 检查结构、语义、引用前提、策略和资源限制 | 由 Poiet 的生产验证路径执行；失败必须产生稳定错误类别和定位。 |
| `pleko` | 解析多个 Endem 的引用、约束和身份并形成 Synem | 由 Poiet 执行；禁止环境搜索、隐式网络解析和顺序碰运气。 |
| `tasse` | 把已解析闭包确定性封装为可分发产物 | 由 Poiet 执行；相同规范输入必须产生相同字节。 |
| `sphra` | 对确定性产物建立发布者信任包络 | 在独立密钥权限域执行；签名随机性、时间和发布者状态不得改变 Endem/Synem 语义身份。 |
| `theor` | 只读查看字段、引用、闭包、摘要、来源和 Tekmor 范围 | 由独立 Theor 执行；不得复用 Poiet 解析代码，不得拥有写入、封装、密钥或运行权限。 |
| `praxe` | 验证并装载 Synem，建立 Dromen，在最小能力域中实现目标 | 由独立 Praxor 执行；不得修改已封装或已签名产物，也不得自行宣告最终验收。 |
| `peira` | 运行 Golden、Malformed、差分、可复现、安全与互操作测试 | 测试路径必须同时覆盖 Poiet、Theor 和 Praxor 的独立失败面。 |

统一入口不等于统一实现。`endem` 可以作为安装和交互入口分派动作，但 `theor` 与 `praxe` 背后的代码、进程和权限域必须独立；质量测试必须证明 Theor 不能写、Praxor 不能取得密钥、Poiet 不能调用模型。

推荐文案直接写动作，例如：

```text
endem poie goal.txt -o search.endem
endem elenk search.endem
endem pleko search.endem policy.endem
endem theor search.endem
endem praxe search.endem
```

这些示例只冻结名称和动作关系；参数、输出命名和具体行为仍以已发布手册版本为准。

## 模型与验收语言

- 写“模型提出候选”，不写“模型生成 Endem”。
- 写“Poiet 形成规范字节”，不写“模型编译目标”。
- 写“Praxor 获得本次 Dromen 的最小能力”，不写“智能体拥有工具权限”。
- 写“Tekmor 支持声明范围内的判断”，不写“证据证明整个任务成功”。
- 写“验收者根据 `krin`、适用 Tekmor 和决定权威作出结论”，不写“模型或 Praxor 宣告完成”。
- 写“`apor` 保留未决语义”，不写“系统自动选择最合理解释”。

## 名称冲突门禁

`Endem` 进入首次正式发行前，维护者必须保存一份有日期、查询式、结果链接、风险结论和责任人的名称审查记录，至少覆盖：

1. PyPI、npm、crates.io、主要系统包仓库和 GitHub 的精确名、大小写变体与近音名。
2. 人工智能产品、编程语言、文件格式、协议、标准组织和开发者工具的同名或近似名。
3. `.endem` 文件扩展名、CLI 命令、MIME 类型与常见操作系统命令冲突。
4. WIPO Global Brand Database 以及计划发行地区的商标数据库；存在疑问时由合格法律专业人员复核。
5. 主要域名和社区账号。域名可用不等于商标可用，域名不可用也不自动否决工程术语。

截至 2026-07-12 的初步互联网检索没有发现人工智能、编程语言、协议或标准组织中的强冲突，PyPI、npm 与 crates.io 的 `endem` 精确包名当时均未登记；搜索结果主要是人名、地区性企业、外语词形和能源模型变量。这只是时间点证据，不是保留名称、商标权或未来可用性的保证。

## 中文信息架构名称

页面名称必须先说明内容类型，再说明主题。中文站点采用以下固定含义：

| 名称 | 使用场景 | 不使用的含糊写法 |
| --- | --- | --- |
| 项目概览 | 项目目的、范围、现状和主要入口 | 单独使用“了解”作为正式页面标题 |
| 架构设计 | 系统分层、组件关系、制品流和信任边界 | 单独使用“架构”作为页面主标题 |
| 架构决策 | 已采用的职责边界、备选方案和后果 | “架构决定” |
| 入门指南 / 获取与使用指南 / 开发指南 | 帮助读者完成一项任务或形成阅读路径 | 用“文档”概括所有任务型内容 |
| Endem 参考指南 / 规范参考指南 | 按制品、字段、动作、术语或成熟度查找信息 | 单独使用“工具参考”“规范参考”或“参考” |
| Endem 使用手册 | 唯一 CLI 的格式、绑定、安全、运行与参考 | 把动作拆成彼此无关的工具手册 |
| 项目动态 | 已发生且可核对的里程碑、规范变化和发布状态 | “新闻与进展”或暗示固定发布节奏的“新闻” |
| 常见问题 | 对高频问题的直接回答 | 仅使用英文缩写“FAQ” |

导航可以为可用空间使用“项目、规范、Endem、指南、开发”等短标签，但展开后的卡片、页面标题和链接文字必须使用完整名称。字段名、动作、命令和规范标识保持英文；要求是中文分类准确、句意完整，而不是机械消除英文。

## 逐页检查表

每个正式页面都要核对以下项目：

1. 标题和首段是否直接说明读者问题。
2. Endem、Synem、Dromen、Tekmor、Poiet、Theor 与 Praxor 是否在首次出现时给出直白定义。
3. 是否把 Noemion 错当成格式、对象、组件或命令前缀。
4. 是否明确输入、处理、输出、错误和当前状态。
5. 是否分开 `aseme`、`apor` 与 `agno`，并保留模型不可信和最终验收责任。
6. 是否把 `theor`/Theor 与生产写入路径隔离，把 `praxe`/Praxor 与 Poiet 和密钥权限隔离。
7. 是否只使用 `endem` 和八个当前动作，没有旧名称、旧路由、别名或重定向。
8. Tekmor 是否声明主体、范围、方法、环境、结果和限制，而不是泛化为“已证明”。
9. 链接文字是否说明目的，导航是否使用当前名称。
