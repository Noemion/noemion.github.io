# Noemion 语言与命名规范

本规范适用于所有正式页面、手册、组件、制品、命令、目录、文件名和导航标签。它同时解决三件事：让中文技术文档像开发者之间的正常说明；让新领域拥有短、可说、可检索的工程语言；让名称严格服从数据结构、信任边界和测试，而不是反过来决定架构。

## 已冻结的职责与设计阶段名称

1. **Noemion 是项目、新领域与社区的总名和品牌。**为避免品牌侵入工程边界，它不进入格式名、对象类型、命令、ABI、魔数或组件前缀。
2. **Endem 是核心制品名。**它由英语 `end` 与语言学中表示最小区别单位的 `-eme` 融合而成，不是缩写，也不是在既有目标文件术语上添加前缀。
3. **唯一公开应用和 CLI 都叫 `endem`。**正式文件扩展名是 `.endem`；公开动作只有 `ktise`、`elenk`、`pleko`、`theor` 与 `drase`。
4. **不保留旧名称。**路由、命令、文件名、标题、导航、测试与示例直接采用当前名称，不提供别名、重定向、兼容命令或迁移垫片。
5. **正式发行前必须通过完整名称门禁。**当前研究只能支持继续使用设计阶段名称，不能替代商标检索、法律判断、目标语言读音和词表内口头区分证据。

项目级标题、简介、门户主张、入门入口和一级导航必须先以 Noemion 指代整个项目与领域。Endem 不得成为项目名、生态总名或替代品牌；它只在最小制品、对应规范、生命周期、应用和 CLI 语境中使用。Synem、Dromen、Iknem、Ktisor、Theor、Drasor 与动作名同样只承担各自工程职责。

## 无数字词汇原则

Noemion 创造的是可以长期口述、书写和实现的领域语言，不用数字代替尚未完成的命名工作。新造的工具、组件、制品、动作、语义面、结果词和关键概念必须优先采用不含阿拉伯数字的短词：不得添加数字后缀表示“新版”，不得把层级或阶段塞进词中，也不得用 `0`、`1` 等数字模仿字母。

数字只在它确实承担精确引用时保留。ADR 编号、条款 ID、规范和 Profile 标识、协议版本、日期、位宽、字段码、枚举值与测试向量编号属于机器或登记标识，不是领域词汇。例如 `ADR-0027` 与 `END-P1` 用于定位一项决定和一个封闭 Profile；正文仍应先写“精确内容身份决定”与“当前封闭内容 Profile”，不能要求读者把编号当作概念名称。

新增含数字名称只有在数字对互操作性、可验证性或稳定引用不可替代时才可进入提案，并必须同时说明：为什么纯文字名称不足、数字由谁分配、是否随版本变化、怎样避免读者把编号误解为成熟度或代际。没有这组证据时，重新命名而不是追加数字。

语言学后缀 `-eme` 长期用于表示某一语言结构中的区别单位，例如 phoneme、grapheme 与 morpheme。词源参考：

https://www.etymonline.com/word/-eme

## 读音与口头区分门禁

名称必须同时适合阅读、搜索和口头交流。“短”“像一个词”或能够给出某种读法，都不能单独证明名称可传播。每个面向人的品牌、制品名、组件名、动作名、语义面和结果词在正式发行前必须记录：

1. 目标语言与 BCP 47 标签，以及每种语言的 IPA、重音和普通拼读提示。
2. 首次看到拼写的朗读结果；不能先用项目给出的提示训练参与者。
3. 在句子、远程会议和有背景噪声场景中的听写回填结果。
4. 与当前全部关键词的成对混淆矩阵，尤其检查共同词首、共同词尾和动作—组件词族。
5. 参与者语言背景、测试材料、原始结果、失败模式和决定责任。

语音合成、自动转写和语音模型可以用于发现风险，但不能替代人类证据。模型输出仍是不可信候选；未来若支持口述动作，系统必须显示识别出的精确机器请求并按授权边界确认，不能凭转写或置信度直接执行。

当前首轮风险把 `Iknem`、`Ktisor/ktise` 与 `kine/mene` 列为优先复核，把 Endem/Synem 列为成对听辨重点。其余名称没有因此自动通过。发现阶段、独立听者、60 个判断、停止规则和证据记录见[术语与读音验证指南](../docs/terminology-and-pronunciation.html)；风险更新维护在 [`name-audit.md`](name-audit.md)，规范门禁见 [ADR-0034](../architecture/adr-0034-pronunciation-and-oral-distinction.html)。

## 写作原则

1. 先回答读者要解决什么问题，再说明制品、动作或规则。不要以项目术语开头要求读者自行推断用途。
2. 使用主动语态和现在时。写“Ktisor 解析引用并报告冲突”，不写“引用将会被解析并给出相关报告”。
3. 一个句子只表达一个主要判断。超过三个并列动作时改用步骤或列表。
4. 主语必须明确。避免“进行处理”“完成支持”“实现相关能力”“提供相应功能”等缺少执行者和对象的说法。
5. 直接描述输入、动作、结果和失败。应用页优先使用“读取什么—检查什么—输出什么—何时拒绝”的顺序。
6. 专有词第一次出现时给出直白工程定义。后文只在能够减少歧义时使用英文标识。
7. 规范使用“必须、不得、应当、可以”；说明性文字使用“会、用于、负责”。不要用“赋能、打造、沉淀、闭环、拉通、抓手”等宣传或管理套话。
8. 状态说明只写已经完成、正在设计、尚未提供和下一项工作。不要把内部提示词、制作过程或评审对话写给读者。
9. 示例必须说明前提和可观察结果。未冻结的参数、字节编号和能力不得写成可直接执行的接口。
10. 手册按任务组织，不按实现文件组织；参考章节仍要完整列出格式、动作、选项、错误和索引。
11. 哲学引文必须短、可核对并标出作者、书名和命题编号；引文后直接说明工程启发和采用限制，不能让读者从引文本身猜测规范含义。
12. 现行正文优先使用直白中文职责，必要时在括号中给出英文接口名。历史 ADR 可以保留旧词，但必须在首屏标明已被取代；现行页面不得泄漏旧扩展名、旧动作或旧制品名。

这些规则采用 GNU 文档的长期可读性原则：面向不了解主题的读者、先基础后进阶、首次定义专门术语、按用户问题而不是功能清单组织，并优先使用主动语态和现在时。

## 核心领域语言

| 名称 | 直白工程定义 | 不能被误解为 |
| --- | --- | --- |
| Noemion | 项目、新领域与社区品牌 | 格式前缀、组件层或 CLI 前缀 |
| Endem | 最小、独立有效且可验证的期望终态单元 | Prompt、任务消息、执行脚本或二进制化文本 |
| Synem | 多个 Endem 经符号、依赖、约束和内容身份解析后形成、可以确定性封装的组合闭包 | 文件列表、搜索路径结果、归档目录或运行时临时依赖 |
| Dromen | Drasor 为一次 Drase 会话从精确 attested Endem 或 Synem、政策、环境、能力、预算和证据责任封存的只读执行契约 | 持久格式、签名制品、进程映像、凭据包、可恢复会话或永久权限 |
| Iknem | 绑定明确主体和声明范围，并封装结构化观察 `phain`、方法、环境、结果与限制的证据 | 无范围的“通过”徽标、最终验收决定或模型自评 |
| Ktisor | 决定规范语义、排序、布局与确定性字节的可信核心 | 模型推理服务、Theor 或 Drasor |
| Theor | `theor` 背后的独立只读实现，直接读取不可信字节并形成检查视图 | Ktisor 解析器的包装、编辑器或生产写入路径 |
| Drasor | `drase` 背后的受限装载与实现环境 | Ktisor、通用 shell、无限能力会话或验收者 |

### Endem 的最小性

Endem 的“最小”不是字数最少，也不是把所有内容压入一条记录。采用以下可测试边界：

- 如果拆分后任一部分失去独立身份、来源、约束或完成判据，就不应继续拆分。
- 如果两个目标可以独立版本化、引用、验证、替换或失败，就必须拆成两个 Endem。
- 一个 Endem 可以引用其他 Endem，但必须独立说明自己的目标、约束、完成判据和未决信息。
- 自然语言只是来源表达之一；进入规范字节前，确定性 Ktisor 必须形成类型化、可定位和可验证的结构。

### 六个语义面

每个 Endem 按 `rhem`、`semion`、`skena`、`telis`、`krin`、`apor` 的固定顺序表达语义。名称保持小写 ASCII，不翻译、不复数化，也不在不同配置中改名。

| 组 | 必须回答 | 规范边界 |
| --- | --- | --- |
| `rhem` | 来源中实际出现了哪些记号？ | 保存提交后的解码表达及主体、语言、媒体、范围、版本、摘要与可重定位位置；若没有外部原始字节身份，不得声称逐字节保真，推断也不能伪装成来源。 |
| `semion` | 这些记号怎样投影为符号、指称和关系？ | 保存作用域、极性、量化、时间和来源映射；模型改写不等于已确认意义。 |
| `skena` | 哪一种中性可能事态由对象和关系构成？ | 保存一个根对象—关系图及显式组合与极性；不得混入目标力量或实际观察。 |
| `telis` | 目标要求这个事态发生变化还是持续成立？ | 第一阶段只允许 `kine` 与 `mene`；禁止事项由 `skena` 的否定表达，不设置重复的回避力量。 |
| `krin` | 哪些结构化观察支持满足、反驳、观察不足或故障，谁有权决定？ | 保存 `phain` 类型、比较规则、必需 Iknem、`agno/fault` 处理和权威；命题不能自称为真。 |
| `apor` | 投影、指称、关系、测量或权威还有什么没有确定？ | 保存候选、冲突、缺口、风险和解决权限；不得把最高概率猜测静默升级。 |

有效 Endem 必须具有可定位 `rhem`、已授权 `semion`、一个根 `skena`、一个适用 `telis` 和覆盖全部必需根路径的 `krin`。没有未决项时，`apor` 显式为空集合。六个名称是顶层类型化语义面，不是六段自由文本；条目类型、标识、关系位置、排序、限制和编码由 Endem 规范定义。

`semion`、`skena` 与适用 Iknem 中的 `phain` 必须共享稳定符号身份和可比较关系位置。逻辑形式由对象—关系拓扑、角色和组合边显示，规范载荷不得依赖自填的 `logical_form`、`valid` 或 `true` 字段。违反 Profile 或没有任何允许投影的输入以 `aseme` 拒绝；存在候选但无法选择时进入 `apor`；意义已经确定但观察不足时返回 `agno`。

`apor` 是一等信息。Ktisor、Theor、Drasor、控制平面和模型都不得用默认值、搜索顺序、概率最高候选或未经记录的人工判断静默消除它。只有具有明确来源和责任的决议才能确认投影，并且必须留下可审计关系。

### 判断与运行结果

结果使用短而直白的现有词，不再创造新的哲学专名。它们回答不同问题，不能通过名称相似或界面便利互相转换：

| 结果域 | 允许值 | 写作边界 |
| --- | --- | --- |
| 制品生命周期 | `nascent / coherent / attested` | 只描述制品形成与发布见证。 |
| 满足判断 | `met / unmet / agno / fault` | 只描述 `skena` 相对 `krin`、`phain` 与 Iknem 的比较。 |
| 权威决定 | `accepted / rejected / deferred` | 只描述具名权威的最终决定，并保留输入满足结果。 |
| 授权决定 | `grant / deny / defer` | 只回答精确主体在精确语境和范围内是否获准确认或行动；不得写成最终接受。 |
| Drase 会话 | `completed / failed / interrupted` | 只描述一次会话怎样终止。 |
| Iknem 有效性 | `valid / invalid / revoked` | 与证据集合的 `sufficient / insufficient` 覆盖度分开。 |

写“授权决定 grant，会话 completed，满足结果 unmet，最终权威决定 rejected”，不写“运行成功但任务失败”这种把多个结果压成一个成功词的句子。外部 MCP/A2A 状态必须先说明其来源域，不能直接写成 Endem 状态、授权决定或最终决定。

### 权威与授权决定语言

权威不是登录状态、签名密钥、组织名称、模型能力或界面位置的同义词。每次写“已授权”，必须同时能回答：谁请求、谁实际行动、代表谁、依据哪份政策、针对什么对象执行什么动作、出于什么目的、落在哪个范围、何时失效以及由谁承担失败责任。

- 写“具名权威在 AUT-CORE 约束的范围内 grant 当前请求”，不写“用户已登录所以允许”。
- 写“模型提出 semion 候选，授权者核对精确来源与安全显示后确认”，不写“模型理解并批准了意义”。
- 写“委托逐级保留行动者并收窄范围”，不写“子 Agent 继承上游全部权限”。
- 写“重新认证满足认证强度，仍需独立授权决定”，不写“step-up 后权限自动扩大”。
- 写“授权决定允许执行”，不写“授权证明目标为真、证据充分或最终 accepted”。

### mene 的时间与连续性语言

持续目标必须把目标时间、连续要求和证据覆盖分开表达：

| 语义面 | 允许词 | 写作边界 |
| --- | --- | --- |
| 时间范围 | `fixed / elapsed` | `fixed` 是具名权威解析的 UTC 半开区间；`elapsed` 是具名事件后的固定时长，并由单调时钟测量。 |
| 连续政策 | `strict / budgeted` | `strict` 要求全区间成立；`budgeted` 必须同时声明累计、单次和次数上限。 |
| 覆盖分类 | `met / unmet / agno / fault` | 完整覆盖支持判断；覆盖空洞是 `agno`，已证实违约是 `unmet`，时钟或求值契约故障是 `fault`。 |

写“固定区间内采用 strict，观测存在空洞，因此结果为 agno”，不写“采样均正常，所以持续目标通过”。“采样”只描述证据取得方法，不是第三种连续性政策。`now`、`today`、`tomorrow`、默认时区、墙钟经过时长和远端 Agent 完成时间都不能直接成为规范时间。

### 测量与阈值语言

测量页面优先使用已有工程词 `measurement-construct`、`estimand`、`measurement-procedure` 和 `threshold-contract`，不为构念、总体、程序或阈值另造哲学专名。先写清“测量什么、适用于谁”，再写单位、窗口、聚合器和阈值。

### 复合判断语言

复合页面优先使用 `criterion-leaf`、`criterion-composition`、`decisive-basis` 和 `evaluation-coverage`，不另造哲学专名。操作只写 `all_of / any_of`，并先说明为什么这些关系仍属于一个不可分终态。独立目标改用 Endem 与 Synem，不把自然语言连词直接翻译为对象边界。

写“固定测试集上的准确率为 0.82”，不写“模型总体准确率为 0.82”；只有预注册总体、抽样假设、统计模型和不确定区间后，才可以写推广估计。写“区间跨越阈值，因此为 agno”，不写“点估计超过阈值，所以通过”。显示舍入、排名、基准名和模型置信度都不能代替规范判断。

### 否定与缺席证据语言

否定沿用同一关系、角色和顺序，只写显式 `positive / negative` 极性。不得用 `not_*`、`cannot_*` 等新谓词，或用删除关系节点表达否定。

写“查询在声明的数据集内没有匹配，因此当前为 agno”，不写“系统证明事件从未发生”。只有具名权威证明全集、全部角色路径、时间截止、完整枚举和损失边界均已封闭时，才写“完整缺席在该范围支持 met”。观察器未按契约完成时写 `fault`，发现同构正反例时写 `unmet`。

## 动作与权限边界

唯一 CLI 采用 `endem <action>`。动作名由古希腊语动作词根缩写或再造；每次首次出现都必须同时给出直白职责，不能要求读者用词源猜测行为。

| 动作 | 输入与结果 | 实现与权限边界 |
| --- | --- | --- |
| `ktise` | 把受控来源确定性形成 Endem | 由 Ktisor 执行；模型只能提供不可信来源候选，不能决定规范字段或字节。 |
| `elenk` | 检查结构、语义、引用前提、策略和资源限制 | 由 Ktisor 的生产验证路径执行；失败必须产生稳定错误类别和定位。 |
| `pleko` | 解析多个 Endem 的引用、约束和身份并形成 Synem | 由 Ktisor 执行；禁止环境搜索、隐式网络解析和顺序碰运气。 |
| `theor` | 只读查看字段、引用、闭包、摘要、来源和 Iknem 范围 | 由独立 Theor 执行；不得复用 Ktisor 解析代码，不得拥有写入、封装、密钥或运行权限。 |
| `drase` | 验证并装载 Synem，建立 Dromen，在最小能力域中实现目标 | 由独立 Drasor 执行；不得修改已封装或已签名产物，也不得自行宣告最终验收。 |

统一入口不等于统一实现。`endem` 可以作为安装和交互入口分派动作，但 `theor` 与 `drase` 背后的代码、进程和权限域必须独立；质量测试必须证明 Theor 不能写、Drasor 不能取得密钥、Ktisor 不能调用模型。派生封装、外部签名与符合性验证按 ADR-0035 保留责任边界，但不是公开动作。

推荐文案直接写动作，例如：

```text
endem ktise goal.txt -o search.endem
```

这些示例只冻结名称和动作关系；参数、输出命名和具体行为仍以已发布手册版本为准。

## 模型与验收语言

- 写“模型提出候选”，不写“模型生成 Endem”。
- 写“Ktisor 形成规范字节”，不写“模型编译目标”。
- 写“Drasor 获得本次 Dromen 的最小能力”，不写“智能体拥有工具权限”。
- 写“Iknem 支持声明范围内的判断”，不写“证据证明整个任务成功”。
- 写“验收者根据 `krin`、适用 Iknem 和决定权威作出结论”，不写“模型或 Drasor 宣告完成”。
- 写“`apor` 保留未决语义”，不写“系统自动选择最合理解释”。

## 结构化诊断语言

诊断只使用直白工程术语，不为错误处理另造哲学专名。`diagnostic-code` 表示稳定机器身份，`diagnostic-layer` 表示失败发生的边界，`diagnostic-location` 表示类型化有限位置，`primary-diagnostic` 表示唯一主阻断诊断，`recovery-class` 表示不带权限和副作用的恢复分类。

- 写“外部 MCP 工具执行错误映射为本地诊断”，不写“工具失败所以目标 unmet”。
- 写“需要具名权威重新授权”，不写“诊断允许 Agent 自动提权”。
- 写“消息用于解释，机器码用于协议”，不让自动化解析本地化文本取得语义。
- Noemion 是品牌，不进入诊断码；机器码按 `diagnostic.*`、`endem.*`、`synem.*`、`dromen.*` 或 `iknem.*` 的责任域登记。

## 外部协议适配语言

外部协议适配只使用直白工程术语，不为 Agent、工具、任务、重试或交付另造哲学专名。`adapter-binding` 固定协议与对端版本，`peer-binding` 分开声明和认证身份，`invocation-binding` 绑定一次本地调用，`loss-manifest` 记录映射损失，`external-state` 保存外部状态域，`idempotency-class` 约束重试，`delivery-evidence` 约束异步交付。

- 写“外部 A2A Task completed，等待本地 krin 判断”，不写“Agent 已完成目标”。
- 写“调用缺少幂等或去重证据，因此不自动重试”，不写“网络失败所以安全重试”。
- 写“Artifact 是绑定来源和内容身份的候选”，不写“远端产物就是 Endem 或 Iknem”。
- 适配诊断码使用 `adapter.*` 责任域；Noemion 品牌不进入机器码、协议 Profile、schema 或 API 前缀。

## 名称冲突门禁

`Endem` 进入首次正式发行前，维护者必须保存一份有日期、查询式、结果链接、风险结论和责任人的名称审查记录，至少覆盖：

1. PyPI、npm、crates.io、主要系统包仓库和 GitHub 的精确名、大小写变体与近音名。
2. 人工智能产品、编程语言、文件格式、协议、标准组织和开发者工具的同名或近似名。
3. `.endem` 文件扩展名、CLI 命令、MIME 类型与常见操作系统命令冲突。
4. WIPO Global Brand Database 以及计划发行地区的商标数据库；存在疑问时由合格法律专业人员复核。
5. 主要域名和社区账号。域名可用不等于商标可用，域名不可用也不自动否决工程术语。

截至 2026-07-13，PyPI、npm 与 crates.io 的 `endem` 精确包名仍未登记，GNU 包目录与 IANA 媒体类型登记表也没有精确项。但是，GitHub 已有三个第三方精确同名仓库和一个精确同名用户，`endem.com` 也已登记。因此 Endem 只能继续作为中等风险的开发候选名，不能宣称全网唯一；第一次可安装发行仍须通过商标、近音、类别、包名取得、法律复核和 ADR-0034 的口头区分门禁。日期化查询式、候选和停止条件统一维护在 [`name-audit.md`](name-audit.md)。

主要检索入口包括 [PyPI](https://pypi.org/)、[npm](https://www.npmjs.com/)、[crates.io](https://crates.io/)、[GitHub 搜索](https://github.com/search)与 [WIPO Global Brand Database](https://www.wipo.int/en/web/global-brand-database)。入口只提供查询能力，具体查询式、结果时间和风险结论必须进入审计记录。

## 中文信息架构名称

页面名称必须先说明内容类型，再说明主题。中文站点采用以下固定含义：

| 名称 | 使用场景 | 不使用的含糊写法 |
| --- | --- | --- |
| 项目概览 | 项目目的、范围、现状和主要入口 | 单独使用“了解”作为正式页面标题 |
| 架构设计 | 系统分层、组件关系、制品流和信任边界 | 单独使用“架构”作为页面主标题 |
| 架构决策 | 已采用的职责边界、备选方案和后果 | “架构决定” |
| 入门指南 / 获取与使用指南 / 开发指南 | 帮助读者完成一项任务或形成阅读路径 | 用“文档”概括所有任务型内容 |
| Endem 参考指南 / 规范参考指南 | 按制品、字段、动作、术语或成熟度查找信息 | 单独使用“工具参考”“规范参考”或“参考” |
| Endem 使用手册 | 唯一 CLI 的格式、绑定、安全、运行与参考 | 把动作拆成彼此无关的工具手册 |
| 项目动态 | 已发生且可核对的里程碑、规范变化和发布状态 | “新闻与进展”或暗示固定发布节奏的“新闻” |
| 常见问题 | 对高频问题的直接回答 | 仅使用英文缩写“FAQ” |

导航可以为可用空间使用“项目、规范、应用、指南、开发”等短标签；`Endem` 只作为“应用”分组内的应用名、制品名和手册名出现。展开后的卡片、页面标题和链接文字必须使用完整名称。字段名、动作、命令和规范标识保持英文；要求是中文分类准确、句意完整，而不是机械消除英文。

## 逐页检查表

每个正式页面都要核对以下项目：

1. 标题和首段是否直接说明读者问题。
2. Endem、Synem、Dromen、Iknem、Ktisor、Theor 与 Drasor 是否在首次出现时给出直白定义。
3. 是否把 Noemion 错当成格式、对象、组件或命令前缀。
4. 是否明确输入、处理、输出、错误和当前状态。
5. 是否分开 `aseme`、`apor` 与 `agno`，并保留模型不可信和最终验收责任。
6. 是否把 `theor`/Theor 与生产写入路径隔离，把 `drase`/Drasor 与 Ktisor 和密钥权限隔离。
7. 是否只使用 `endem` 和五个当前动作，没有旧名称、旧路由、别名或重定向。
8. Iknem 是否声明主体、范围、方法、环境、结果和限制，而不是泛化为“已证明”。
9. 链接文字是否说明目的，导航是否使用当前名称。
10. 制品、满足、决定、会话和证据结果是否分开，没有把 `completed`、`valid` 或外部 Task 成功写成 `accepted`。
11. `mene` 是否分开时间范围、连续性政策和观测覆盖，没有把相对日期、默认时区、墙钟经过时长或稀疏采样写成确定事实。
12. 诊断是否让机器码与消息分开，固定生产语境和失败层次，并避免把外部错误、恢复建议或重试状态写成本地结果与权限。
13. 新造的工具名、组件名、制品名、动作名和关键术语是否没有数字；含数字内容是否只是必要的精确引用，并同时给出直白名称或职责。
