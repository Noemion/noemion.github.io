背景与边界

从提示文本到持久目标契约，中间缺少的工程原语。

MotivationScopeNon-goals

为什么需要 Endem

提示词适合表达一次请求，却没有稳定回答来源、目标、硬约束、验收和歧义分别是什么。Endem 把这五类信息变成可独立处理的最小契约。

字段回答的问题不能静默做什么
say人或上游系统实际表达了什么，来源在哪里？不能在规范化后丢失来源绑定。
aim唯一根终态是什么？不能把多个相互独立的目标伪装成一个目标。
must什么必须成立，什么绝不能发生？不能由模型把软偏好升级成硬约束，或反向降级。
done由谁、用什么证据判断完成？不能让执行者同时自定标准并自我验收。
open哪些含义、冲突或依赖仍未确定？不能用高置信输出抹去未决状态。

与相邻形式的区别

Prompt

Prompt 是来源表达或运行输入；Endem 是经过来源绑定与规则检查的持久目标单元。

Skill / Agent 包

它们分发说明、脚本与资源；Endem 定义期望终态、约束和验收，不携带任意执行权限。

普通数据容器

CBOR、Protobuf 或 JSON 可以承载编码，但不会自动产生 Endem 的语义与失败规则。

传统目标文件

传统格式绑定符号与地址；Endem 绑定目标、来源、能力上限、验收和仍未解决的语义。

信任边界

  • Core 的形成、规范化、布局、验证、绑定与哈希路径必须无模型、确定且可复现。
  • 所有外部字节、模型候选、工具结果和依赖声明都从不可信状态开始。
  • 解析偏移、长度、计数、索引和对齐只能使用 checked arithmetic，并受深度、内存与时间上限约束。
  • endem see 使用独立只读解析实现；能够显示不代表已经可信。
  • endem run 在隔离进程中重新检查实际 Weave；Harness 独占实时能力句柄，模型只能请求。
  • 外部签名者持有私钥；Core 只产生签名请求并核对返回响应。

当前边界

已接受 Endem 词汇与单一应用拓扑；规范编码、扩展机制、ABI、实现语言和协议适配仍需通过 ADR 与验证条件。第一阶段不引入模型、LLVM、MLIR 或大型框架。
