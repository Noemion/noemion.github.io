背景与边界

Noemion 研究普通人怎样把自然语言目标形成可检查的工程制品，并让它跨人工智能系统保持同一责任边界。

项目目的开发者问题当前策略术语读音待验证

“编译目标”不是生成代码

软件工程长期依靠形式语言，把程序员的代码转成机器能够执行或检查的对象。人工智能系统却常从普通语言出发，并且可以直接调用工具、修改文件或操作外部服务。

Noemion 把“编译”定义为一条可复核的目标形成路径。它保留原始表达，显示意义怎样得到确认，保留尚未解决的歧义，并把期望结果和接受条件写成软件可以检查的关系。

这不是自然语言生成代码。代码、工具调用和远端任务都可能是实现目标的手段，但它们不能替代目标本身，也不能替提出目标的人重新定义成功。

当前这仍是一项研究与规范工作。网站所说的“编译”不表示已经存在可执行编译器、稳定格式或生产兼容性。

从一次依赖升级看清缺口

假设用户要求 Agent“把支付服务依赖升级到经过批准且没有已知高危漏洞的版本”。执行过程中会产生很多成功信号，但每种信号只回答自己的问题。

系统给出的信号它可以证明什么它没有回答什么
用户的 Prompt保存了这次请求的自然语言来源。“经过批准”、漏洞资料截止点、允许修改的仓库和接受者仍可能不明确。
MCP 工具调用成功客户端与服务器完成了一次协议调用，工具返回了结果。调用是否具有业务授权，返回是否覆盖目标条件，以及结果是否应被接受。
A2A TASK_STATE_COMPLETED远端 Agent 报告该 A2A 任务已经成功结束。远端任务的完成定义是否等同于本地目标的满足判据。
包管理器和 CI 成功依赖解析、构建或已选择的测试在对应环境中完成。候选版本是否由正确权威批准，漏洞资料是否足够新，未运行的检查是否仍是缺口。
日志、追踪与扫描结果记录了具名操作、观察和遥测属性。观察覆盖是否足够，由谁按哪项规则把它们合成接受、拒绝或未知结论。

Noemion 不替代这些系统。它把人的原始表达、已确认意义、期望事态、目标方向、满足判据和待确认事项固定为可跨运行引用的目标对象。授权、会话、协议调用和证据继续由各自责任层处理。

Noemion 与相邻层怎样配合

相邻层它主要回答什么Noemion 保留的边界
Model Context Protocol客户端和服务器怎样交换工具、资源、提示、通知及相关授权信息。协议能力和调用结果进入受限会话，但不能改写本地目标意义、权限或满足结果。
Agent2Agent Protocol 1.0独立 Agent 怎样发现能力、交换消息与产物，并管理远端 Task 生命周期。A2A Task 状态保留为外部协议事实；映射为本地会话或目标结果需要显式 Profile 和证据。
工作流、构建与任务运行器动作怎样排序、重试、并行或终止，以及某次工作是否完成。运行状态不替代目标对象、业务判据或具名接受决定。
身份、政策与凭据系统谁可以对哪个资源执行什么动作，凭据在何时有效。动作授权不确认自然语言意义，也不证明目标已经满足。
OpenTelemetry 语义约定怎样用一致属性记录 traces、metrics、logs、events 和相关操作事实。遥测可以成为观察来源；证据范围、满足判据和最终决定仍需单独绑定。

NIST AI Agent Standards Initiative分别推进 Agent 协议互操作、身份与授权研究和安全评价。这一趋势说明责任分层正在变得更重要，但不构成对 Noemion 设计或实现的认可。

为什么需要一个持久目标对象

同一目标经过模型改写、协议转发、会话恢复或供应商切换后，开发者仍需要回答下面六个问题。直白职责是阅读入口；短词只是现行设计字段。

开发者问题现行字段失败时怎样处理
上游实际表达了什么，来源在哪里？rhem · 来源表达来源无法定位时，不声称后续结构忠实于原请求。
哪些意义已经由规则或具名权威确认？semion · 已确认意义投影模型摘要、相似度或高置信输出不能自行取得确认地位。
哪种对象—关系事态构成目标内容？skena · 期望事态多个独立目标不能伪装成一个根事态。
目标要求事态达到成立，还是持续成立？telis · 目标方向否定、询问、许可和动作授权不压进同一个方向值。
哪些观察足以支持判断，由谁决定？krin · 满足判据观察不足保留未知；执行者不能临时改写标准并自我验收。
哪些候选意义或冲突仍未解决？apor · 待确认意义没有解决权限时停止，不用模型选择最高概率解释。

从 GNU 工具链借鉴什么

GNU make把 goal 定义为最终要更新的 target，并通过显式 prerequisite 关系决定还需更新哪些对象。这为 Noemion 提供了重要纪律：依赖必须显式、形成过程应可重放、产出需要独立检查。

但 make 的 goal 不等于自然语言目标契约。文件已经更新只说明构建规则完成，不能回答业务意义、动作授权、观察覆盖和最终接受。

GNU Manuals要求资料按照用户面对的概念和问题组织，而不是照抄实现结构或功能清单。Noemion 因而先解释开发者要作出的判断，再给字段、组件和 ADR。

项目不会复制 GNU Binutils 的工具数量、命令命名或机器指令本体。类比只在能够形成规范条款、失败条件和验证证据时保留。

工程责任主要承担者Noemion 的接口边界

当前开发者可以依赖什么

  • 当前策略：Endem 分开六项语义职责；Synem、Dromen 和 Iknem 分别处理组合闭包、一次会话与有范围证据。
  • 已有成果：CORE 草案、实验 Profile、ADR、威胁模型和规范向量已经定义资料层的规则与失败边界。
  • 正在研究：现行专名的必要性、发行拼写、主要语言读音和口头区分，以及外部协议到本地责任的精确 Profile。
  • 限制条件：当前没有 Ktisor、Theor、Drasor、CLI、运行时或协议适配器；资料构建和向量通过不能证明组件安全或互操作。

当前策略已确定 Endem 所指的最小制品职责；现行拼写和读音仍须通过发行名称审查。职责稳定不等于名称已经适合发布。
