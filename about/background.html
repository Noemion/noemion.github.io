背景与边界

Noemion 为什么存在、解决什么问题，以及不试图成为什么。

MotivationScopeNon-goals

已确认原则：项目目标

Noemion 面向生成式计算建立可执行、可链接的表征对象体系，使目标、约束、偏好、资源引用、歧义和验收规则能够被机器分析、验证、重定位和执行。

对使用者意味着什么：同一个任务即使换一种表达，也应保留相同的目标身份、不可违反条件和验收依据；若系统无法确认语义，就应报告未决项，而不是给出看似确定的结果。

  • GSIR 表达目标、约束、证据、评价与未决语义，而不是只保存一段供模型阅读的文本。
  • 对象格式负责组织、链接、装载和完整性；Runtime 负责生成式求解与 Lowering。
  • 项目借鉴 ELF 的 Section、Segment、符号、重定位、动态链接、调试、strip 和装载工程策略，但不复制传统机器指令语义。

已确认原则：与相邻形式的区别

Prompt

文本提示可以作为输入或来源，但仅将文本二进制化并不能形成可验证、可链接的语义 IR。

Skill 包

SKILL.md、脚本和资源的压缩归档负责分发内容，不等同于具有类型、符号、约束和重定位语义的对象。

传统 IR

传统 IR 通常强调“怎样执行”；GSIR 强调“什么结果满足目标”以及生成式求解不可越过的边界。

普通二进制容器

JSON、CBOR、Protobuf 或 Token ID 编码只能承载数据，不能凭编码方式自动获得 GSIR 语义。

已确认原则：系统与信任边界

  • Strict 模式无模型、确定且可复现；只有确定性的 Compiler Core 能生成 GSIR/GOBJ。
  • 通用模型可作为外部源码生成器，模型输出始终是不可信输入。
  • 模型不得决定二进制布局或直接生成 GOBJ。
  • 歧义是一等信息；证据不足时保留未决语义，不制造错误确定性。
  • 首个实施阶段只做规范与安全二进制核心，不引入模型、LLVM、MLIR 或大型框架。

开放问题：名称与实现选择

总体对象规范的候选名称、类似 .o 的对象最终名称以及首版实现语言尚未冻结。工程讨论可暂用 GOBJ，但候选 AELF 等名称不构成正式规范。完整清单见开放问题。

未来阶段：路线图语境

链接、可信装载、SSO 发布与 NSFE 将在前序规范、安全检查和拒绝路径稳定后推进。开发顺序和完成标准见实施路线图。

研究命题与反证条件

Noemion 的上位命题是：将生成式任务表示为带类型、约束、证据、权限与验收条件的对象，可以比仅依赖文本上下文提供更强的可验证性、组合性和可复现性。该命题必须与结构化 Prompt、工作流 DSL、知识图谱、传统 IR 和普通对象容器进行对照，而不能只用项目内部术语证明自身价值。

  • 如果对象化表示不能在跨模型或跨 Runtime 实验中保持关键约束等价，模型无关性主张应被限制或否定。
  • 如果链接、验证和装载的额外复杂度不能换来可测量的错误定位、安全或复现收益，工程范围必须收缩。
  • 如果歧义集合和证据机制不能降低错误确定率，相关语义结构需要重新设计。
  • 如果对象条款无法形成独立实现和一致性测试，标准化目标尚不成立。

思想资源与工程边界

项目持续阅读分析哲学、语言哲学、逻辑和语用学著作，用来发现表达、指称、满足、行为、语境和推理承诺之间不能混淆的边界。书籍只产生设计问题和反例，不能直接产生规范条款。
