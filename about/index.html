项目背景

为什么自然语言目标需要自己的最小工程制品，以及哪些责任继续由相邻系统承担。

开发者问题工程边界当前策略

问题

生成式系统接收的是文字，却必须兑现目标、遵守硬约束、控制能力并用证据验收。把这些内容留在一次性上下文中，会让每个模型和每次运行重新猜测同一契约。

Noemion 不研究“怎样写出更好的提示词”。它研究一项自然语言目标怎样跨越多次运行，仍保持同一个可检查的工程身份。

  • 形成目标时，保留来源、意义、期望事态和待确认信息。
  • 组合目标时，显式发现引用、约束和权限冲突。
  • 实现目标前，固定能力上限、观察要求和验收权威。
  • 实现目标后，用可定位证据说明发生了什么、结论适用于哪里。

项目定义哪些工程责任

最小目标制品 · Endem

保存一项期望终态的来源、意义、事态、方向、判据和待确认事项。

组合闭包 · Synem

两个或更多 Endem 解析引用和冲突后形成的完整关系图。单个自包含 Endem 不需要组合对象。

一次会话契约 · Dromen

为一次受限运行固定对象、主体、政策、环境、能力、预算和观察责任；它不是持久文件或凭据包。

有范围证据记录 · Iknem

绑定精确主体、对象、方法、观察和可支持结论；模型陈述不能单独证明目标已经完成。

这些专名都是现行设计标识。开发者资料先使用直白职责；发行拼写、读音流畅度及相邻名称的口头区分仍需独立人类验证。

哪些责任由外部系统承担

  • Prompt、Skill 包和 Agent 消息继续承担来源表达与运行输入；它们不会因被保存为 JSON 或 Token 序列而成为目标制品。
  • MCP、A2A、HTTP 与工作流系统继续承担协议交换、任务状态和调度；Noemion 只要求映射不得改写本地意义、权限或结果。
  • 身份、政策、凭据和同意系统继续授予实际动作能力；意义确认不会自动产生运行权限。
  • 成熟外部工具继续承担模型训练、遥测、扫描、构建和发布；项目只在出现不可替代需求时定义受限适配边界。

如何推进
