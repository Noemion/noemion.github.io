项目背景

为什么自然语言目标需要自己的最小工程制品，以及 Noemion 明确不建设什么。

WhyScopeEvidence

问题

生成式系统接收的是文字，却必须兑现目标、遵守硬约束、控制能力并用证据验收。把这些内容留在一次性上下文中，会让每个模型和每次运行重新猜测同一契约。

Noemion 不研究“怎样写出更好的提示词”。它研究一项自然语言目标怎样跨越多次运行，仍保持同一个可检查的工程身份。

  • 形成目标时，保留来源、意义、期望事态和待确认信息。
  • 组合目标时，显式发现引用、约束和权限冲突。
  • 实现目标前，固定能力上限、观察要求和验收权威。
  • 实现目标后，用可定位证据说明发生了什么、结论适用于哪里。

项目创造什么

Endem

最小、独立有效且可验证的期望终态单元。六个语义面围绕一个根 skena 形成完整契约。

Synem

两个或更多 Endem 解析引用和冲突后形成的组合闭包。单个自包含 Endem 不需要 Synem。

Dromen

Drasor 为一次 Drase 会话封存的只读执行契约。它不是文件、凭据包或可恢复权限。

Iknem

绑定主体、范围和结构化观察 phain 的证据记录。模型陈述不能单独证明目标已经完成。

项目不创造什么

  • 不把提示词、Skill 包、JSON 或 Token 序列改名为目标制品。
  • 不把 GNU Binutils 的工具数量、命令命名或机器指令本体照搬到新领域。
  • 不建立另一套通用模型训练平台；只有出现不可替代的研究需求才适配现有生态。
  • 不让模型决定规范字节、扩大权限、修改签名制品或宣告验收通过。

如何推进

第一条纵向切片只包含 endem ktise、elenk、独立实现的 theor 和一致性验证。双读取器、恶意输入、逐字节复现和资源上限验证完成后，才评估组合、封装和运行。
