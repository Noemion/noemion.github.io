Agent Harness

把 Runtime、工具、上下文、反馈和验收策略组织成受约束的执行环境，让每一步都可观察、可拒绝、可复查。

Design StageControl PlaneLeast CapabilityEvidence Loop

它解决什么问题

Fulfillment Runtime 中的模型或求解后端能提出候选和下一步能力请求，但不能自行证明权限合规、工具结果正确或任务已经完成。Agent Harness 位于 Runtime 外侧，负责提供可导航的任务地图、最小能力集合和连续反馈，而不是把整个环境暴露给模型。

这层设计把“提示模型完成任务”改写为可检查的控制循环：系统先固定目标、验收契约和 Acceptance Policy，再逐步披露相关上下文，验证每次 Capability Request，观察环境变化，并在证据闭合后形成明确决定或请求指定权威复核。

执行循环

固定 Profile / Policy
建立 Runtime Request
接收 Capability Request
策略验证并执行
返回 Observation
证据闭合与决定 / 继续

每次循环都产生结构化事件。后续步骤读取工具返回的真实状态，而不是依赖模型记忆或人工复制。失败会保留为诊断、策略拒绝、预算耗尽、环境漂移或验收未满足等明确结果。

上下文与知识组织

智能体先读取短小、稳定的入口，再沿链接取得当前任务需要的规范、架构、计划和测试资料。仓库内版本化文件是可复查的记录系统；没有进入权威资料的对话结论不能成为执行依据。

  • 入口文件只负责路由，不承载整套百科全书。
  • 上下文项携带来源、版本、适用范围和新鲜度信息。
  • 规范、设计、计划、生成资料和运行证据分别存放。
  • 失效链接、孤立规则和与行为不一致的说明必须由自动检查发现。

能力与反馈设计

工具接口应当小、稳定、可组合，并返回机器可解析的结果。错误不仅说明失败，还要给出失败位置、策略依据和允许的下一步。UI、日志、指标、追踪和测试都应向同一会话提供可查询的观察面。

  • 读操作与写操作分离，高风险副作用需要显式授权。
  • 每个能力声明输入结构、输出结构、资源上限和幂等性。
  • 运行环境按任务隔离，结束后按策略清理临时状态。
  • 验证失败回到循环，但重复失败、预算耗尽或需要价值判断时请求人工处理。

与现有组件的关系

  • Horizon Engine 产生带来源的不可信语义候选；Agent Harness 不把候选升级为事实。
  • Noesis Core 是唯一可以形成 NIR/NOBJ 的确定性组件；Harness 只能提交输入并读取诊断。
  • Noema Object System 验证、链接和装载对象并交付 Loaded State；Harness 不能绕过对象与权限检查。
  • Fulfillment Runtime 接收受限语义视图和观察，产生候选、Candidate Assessment 与 Capability Request；Harness 管理外侧会话、实时能力和反馈循环。

存在必要性与最小形态

无外部能力、无多步反馈且验收完全可由确定性规则一次完成的任务，可以把 Harness 缩减为最小适配层，不需要独立服务。只要任务涉及文件、网络、工具、凭据、UI、动态披露、恢复或人工判断，Harness 的会话级权限与观察边界就不可由 Runtime 代替，否则模型会同时成为计划者、执行者和验收者。

验证要求

维度必须验证
任务验收真实环境中的证据闭合并满足预注册 Acceptance Policy；需要外部判断时只能返回 pending-review。
能力安全越权参数、未授权副作用、提示注入和工具返回污染被拒绝或隔离。
反馈闭环智能体能读取 UI、日志、指标、追踪和测试结果，并用它们复现与验证问题。
恢复与停止工具失败、环境漂移、预算耗尽、重复失败和人工升级条件产生确定结果。
可追溯性目标、上下文、行动、观察、对象、环境和验收结论能够通过稳定身份关联。

当前只形成职责和信任边界，尚未发布 Agent Harness。Loaded State Handle、Execution Profile、Context Manifest、Capability Catalog、Runtime Request、Capability Request、Capability Observation、状态快照、证据闭包和 Acceptance Decision 仍需规范和测试后才能冻结。

运行入口见noemexecute，事件规范化见noemobserve，证据闭包见noemcoverage，离线场景评估见noemevaluate。

工程依据与采用边界

这项设计吸收了智能体优先工程中关于版本化知识、可读工具、真实环境观察、机械化架构约束和持续清理的经验。Noemion 不把高吞吐量当作降低确定性、安全或独立验收要求的理由，也不假定一种环境中的自治水平可以直接迁移到不同风险任务。参考原文：https://openai.com/zh-Hans-CN/index/harness-engineering/
