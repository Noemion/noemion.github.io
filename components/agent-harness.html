Agent Harness

把模型、工具、上下文、反馈和验收组织成受约束的执行环境，让每一步都可观察、可拒绝、可复查。

Control PlaneLeast CapabilityEvidence Loop

它解决什么问题

模型能提出下一步，但不能自行证明权限合规、工具结果正确或任务已经完成。Agent Harness 位于 Fulfillment Runtime 外侧，负责给智能体一张可导航的任务地图、最小能力集合和连续反馈，而不是把整个环境暴露给模型。

这层设计把“提示模型完成任务”改写为可检查的控制循环：系统先固定目标和验收契约，再逐步披露相关上下文，验证每次能力调用，观察环境变化，并只在证据满足契约时结束。

执行循环

固定目标与契约
披露最小上下文
提出受限行动
策略验证并执行
观察与重新判断
验收或人工处理

每次循环都产生结构化事件。后续步骤读取工具返回的真实状态，而不是依赖模型记忆或人工复制。失败会保留为诊断、策略拒绝、预算耗尽、环境漂移或验收未满足等明确结果。

职责与非职责

Agent Harness 负责Agent Harness 不负责
从版本化资料构造任务地图并按需披露上下文用临时提示覆盖规范、ADR 或对象声明
暴露类型化能力并检查参数、权限、预算和副作用默认开放 Shell、网络、文件写入或凭据
隔离会话，绑定工具、模型、对象和环境指纹直接生成 NIR/NOBJ 或修改签名对象
收集观察、调用、拒绝、恢复和验收证据让模型自行宣告正确、扩大权限或跳过验证
在预先声明的条件下停止或请求人工判断把无休止重试当作恢复策略

上下文与知识组织

智能体先读取短小、稳定的入口，再沿链接取得当前任务需要的规范、架构、计划和测试资料。仓库内版本化文件是可复查的记录系统；没有进入权威资料的对话结论不能成为执行依据。

  • 入口文件只负责路由，不承载整套百科全书。
  • 上下文项携带来源、版本、适用范围和新鲜度信息。
  • 规范、设计、计划、生成资料和运行证据分别存放。
  • 失效链接、孤立规则和与行为不一致的说明必须由自动检查发现。

能力与反馈设计

工具接口应当小、稳定、可组合，并返回机器可解析的结果。错误不仅说明失败，还要给出失败位置、策略依据和允许的下一步。UI、日志、指标、追踪和测试都应向同一会话提供可查询的观察面。

  • 读操作与写操作分离，高风险副作用需要显式授权。
  • 每个能力声明输入结构、输出结构、资源上限和幂等性。
  • 运行环境按任务隔离，结束后按策略清理临时状态。
  • 验证失败回到循环，但重复失败、预算耗尽或需要价值判断时请求人工处理。

与现有组件的关系

  • Horizon Engine 产生带来源的不可信语义候选；Agent Harness 不把候选升级为事实。
  • Noesis Core 是唯一可以形成 NIR/NOBJ 的确定性组件；Harness 只能提交输入并读取诊断。
  • Noema Object System 验证、链接和装载对象；Harness 不能绕过对象与权限检查。
  • Fulfillment Runtime 在已验证对象和运行配置下求解；Harness 管理外侧会话、能力和反馈循环。

验证要求

维度必须验证
任务成功真实环境中的结果满足验收契约，而不只是生成了看似合理的内容。
能力安全越权参数、未授权副作用、提示注入和工具返回污染被拒绝或隔离。
反馈闭环智能体能读取 UI、日志、指标、追踪和测试结果，并用它们复现与验证问题。
恢复与停止工具失败、环境漂移、预算耗尽、重复失败和人工升级条件产生确定结果。
可追溯性目标、上下文、行动、观察、对象、环境和验收结论能够通过稳定身份关联。

当前只形成职责和信任边界。能力结构定义、执行配置、事件结构定义、状态快照与验收报告仍需规范和测试后才能冻结。

工程依据与采用边界

这项设计吸收了智能体优先工程中关于版本化知识、可读工具、真实环境观察、机械化架构约束和持续清理的经验。Noemion 不把高吞吐量当作降低确定性、安全或独立验收要求的理由，也不假定一种环境中的自治水平可以直接迁移到不同风险任务。参考原文：https://openai.com/zh-Hans-CN/index/harness-engineering/
