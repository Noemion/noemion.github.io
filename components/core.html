Core

确定性地形成、验证、绑定、打包和封装 Endem/Weave，是唯一能够写规范制品的组件。

DeterministicSole WriterFail Closed

为什么 Core 不能省略

如果自然语言前端、模型、绑定器和发布工具各自决定一套对象语义，Endem 将无法复现，也无法判断同一 aim、must、done 和 open 是否在转换中保持。Core 集中执行规范化、状态机和写入不变量，所有合格 Endem/Weave 都必须经过它。

Core 可以是一组库而不是独立服务，但模型、Runner、Reader 和外部签名系统不能绕过它写入规范字节或提升 Endem 状态。

权威输入与生产者

输入生产者Core 检查
Goal Card / 受控来源用户、规范作者或受控生成器编码、来源身份、一个根 aim、五组语义和版本。
Candidate + Binding Decision不可信模型/外部前端，以及有权确认的主体候选范围、替代项、权限、撤销与来源摘要。
Binding Policy + Dependency Lock构建配置和依赖管理者内容身份、冲突、循环、能力收敛和可重放性。
Pack Profile发布配置保留内容、调试外置、覆盖、预算和确定性元数据。
Signature Response外部签名系统原请求、候选载荷、算法、签名者、策略版本和签名有效性。

输出与消费者

输出实际消费者输出成立的前提
open Endem + Form Witnesscheck、see、bind、人工确认结构合法、来源可定位、一个根 aim；open 项仍显式存在。
生产 Check Resultbind、pack、Runner每个请求层明确 pass/fail/not-run，并绑定实际字节与配置。
bound Endem / Weave + Binding Witnesspack、see必需引用与冲突已解析，权限未扩大。
Unsigned Candidate + Signing Request外部签名系统、seal发布载荷冻结，闭包完整，裁剪保持规范属性。
sealed Endem / Weave + Seal Witnesssee、RunnerSignature Response 精确绑定原候选和请求；被签字节未改变。

关键不变量

  • 相同规范化输入、配置、依赖和工具版本产生相同字节与稳定诊断。
  • 所有偏移、长度、计数、索引、对齐、解压和累计预算使用 checked arithmetic。
  • 未知关键记录、状态与内容不一致、悬空引用和权限扩大关闭失败。
  • 模型只能提交候选，不能决定字段、布局、绑定、签名范围或验收结果。
  • open 项只能被授权绑定、保留为明确运行时未知或导致失败，不能静默删除。
  • seal 只在被签载荷之外附加封装；私钥不进入 Core。

失败与拒绝责任

失败类别Core 必须返回不得做
Source invalid来源位置、编码或摘要诊断猜测缺失文本或调用模型修复。
Contract incomplete缺少的 say/aim/must/done/open 字段组和稳定错误码用默认提示补充来源、硬约束、验收或未决项。
Binding conflict参与成员、冲突字段、策略和 Binding Witness按输入顺序、弱绑定或最后写入者静默选择。
Pack changes semantics变化属性、被移除记录和拒绝结论只因文件更小就继续发布。
Seal mismatch请求、载荷、算法、签名者或策略差异修改候选以适配签名响应。

当前成熟度与未冻结接口

Core 的唯一写入责任、确定性、状态机和失败边界已经采用，尚未发布实现。Goal Card 语法、规范记录、物理编码、生产 Verified Handle、Binding Policy、Pack Profile、Signing Request/Response 和错误码仍需规范与测试冻结。
