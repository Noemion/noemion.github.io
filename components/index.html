项目/系统组件

系统组件

未来的软件为什么不能既解释人的目标、又检查自己的输出、还拿着权限直接执行，以及这三项工作怎样分开。

唯一写入者独立读取者受限运行边界均尚未实现

为什么只有三个组件

如果同一段代码既写入目标、又证明自己写得正确，错误可能在两条路径中同时存在；如果写入器还能直接取得文件或网络权限，目标内容又会和运行能力混在一起。Noemion 因此规划三个组件：Ktisor 写入，Theor 独立读取，Drasor 只在一次会话中取得受限能力。

组件入口

确定性制品形成器（Ktisor）把已经确认的目标解释写成 Endem，并在信息缺失、引用不完整或格式错误时停止；它不能替人选择意思，也不能取得运行权限。 独立只读检查器（Theor）使用第二条直接读取实现查看和比较制品，不生成生产可信句柄。 受限运行边界（Drasor）每次运行前重新检查目标、授权和环境，只把本次需要的能力交给会话；它记录观察，但由另一个具名决定者判断结果是否可以接受。

职责与依赖方向

受控来源 + 已确认意义投影
Ktisor 核对语义授权绑定
确定性形成 Endem / Synem
Theor 独立检查
Drasor 另行核对动作授权
Iknem / 满足结果 / 最终决定
组件可以依赖不得依赖
Ktisor规范注册表、确定性算法、外部签名响应验证接口模型开发包、Drasor、实时能力、Theor 输出
Theor公开规范和自己的有界解析实现Ktisor 的形成侧解析器、写入器、组合器、BFD 或错误分类实现
DrasorKtisor 的制品形成侧检查接口、隔离后端适配器、策略和能力提供者Ktisor 写入器、对象改写接口、Theor 诊断作为授权依据

用一次依赖更新走完组件路径

对于“更新服务依赖，并确认当前版本可以发布”，有权主体先确认目标怎样投影到六个语义面；Ktisor 形成精确 Endem，Theor 用独立实现检查相同字节，Drasor 再在一次性 Dromen 中受限调用文件、包管理器和测试能力。工具退出成功只形成带来源的观察；krin 决定目标是否满足，具名权威另行决定是否接受。

先说职责，再说名称：Ktisor、Theor 与 Drasor 都是设计阶段名称，读音流畅度、口头区分度和发布拼写尚未经过使用者验证。当前文档用职责名称帮助开发者建立边界，不以名称悦耳代替可验证定义。

部署不等于逻辑边界

endem 是唯一公开 CLI，三个组件可以随同一个发行包装载；theor 仍必须使用独立代码路径，drase 仍必须进入隔离权限域。简单任务可以不启动 Drasor，多对象不存在时可以不启用 pleko，但所有规范制品都必须通过 Ktisor。

规范与命令入口

制品职责见Endem、Synem 与 Iknem 规范，状态变化见Endem 生命周期，唯一命令的使用说明见Endem 应用页。
