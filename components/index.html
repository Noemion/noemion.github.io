系统组件

Noemion 把确定性编译、对象处理、模型辅助、智能体控制和结果验收分成相互约束的组件。

NoesisNoemaHorizon

为什么按组件分层

Noemion 必须把确定性对象生成、不可信对象处理、生成式求解和模型辅助分隔开，否则模型输出可能绕过格式、权限和完整性规则，系统也无法定位错误属于规范、编译、链接还是运行阶段。

核心组件

Noesis Core把来源表达编译为可检验的 Noema IR，并生成确定性对象。 Noema Object System解析符号和依赖，链接、验证并装载 Noema Object。 Noema Object规定对象记录、引用、布局和完整性边界。 Horizon Engine从长篇来源中提取带证据的候选语义，但不决定对象内容。 Agent Harness装配上下文和受限能力，驱动可观察、可拒绝、可验收的智能体循环。

职责边界

Noema Object 负责组织、链接、装载与完整性。Fulfillment Runtime 在对象通过验证后执行求解和实现化。模型只能提交候选，不能决定对象布局或直接生成 NOBJ。

数据只沿一个方向取得更高信任：候选语义先经过确定性编译，再形成对象和运行绑定。Fulfillment Runtime 可以读取对象声明，但不能回写对象布局；Horizon Engine 可以生成候选和检索信号；Agent Harness 可以组织上下文、能力和反馈，但两者都不能绕过 Noesis Core 或对象验证。

每个组件的权威说明必须定义输入输出、内部状态、关键算法或数据结构、失败条件、资源上限、可观测事件和验证策略。缺少其中任何一项时，只能视为职责级设计，不能推定接口或实现已经稳定。
