GSIR / GOBJ

Compiler Core 是唯一可生成 GSIR/GOBJ 的组件。它负责规范化、符号与概念绑定、类型和约束检查、歧义保留、Source Unit 覆盖、Lowering 以及确定性对象布局。

已确认原则：三种前端边界

模式输入与信任边界
Strict受控 GSL 经无模型路径解析；相同规范化输入和环境产生可复现结果。
AssistedNSFE 只提交带来源、证据、置信度和歧义状态的封闭候选；Core 决定采纳或拒绝。
Generate-Strict外部生成式模型只生成候选 GSL，随后由 Strict 路径检查；模型输出仍是不可信源码。

具体命令行选项属于工具接口，不由组件职责定义冻结；只有 CLI 规范、版本策略和对应一致性测试获批后才形成兼容承诺。

设计提案：编译 Pass

  1. 规范化源文本、上下文、资源、领域命名空间和策略。
  2. 识别目标、结果类型、模态、否定、范围、条件与多个合法解释。
  3. 构造高层 GSIR，执行实体绑定与硬约束、软偏好、推断权限编译。
  4. 分类未指定、可推断、冲突、待确认与 ontology 外语义。
  5. 构造 Artifact Contract 和 Evaluation Graph，执行类型、符号、可满足性和覆盖检查。
  6. Lowering 为可编码 GSIR，再由确定性 Writer 生成对象。

已确认原则：失败边界

  • 模型置信度不能替代类型、覆盖或安全证明。
  • 编译器无法可靠理解的输入是编译失败；源内容显式保留的歧义才是合法运行语义。
  • 精确载荷不能被有损摘要；覆盖缺口、冲突和未分类来源不能静默发布。

未来阶段：长上下文与 Assisted

第一阶段只实现规范与安全二进制核心。Source Graph 长上下文编译和 NSFE Assisted 前端必须在 Strict 路径、覆盖模型与拒绝行为稳定后进入。

详细设计入口与验证证据

尚未冻结的核心契约包括规范化上下文、符号与类型环境、约束求解接口、歧义状态机、Coverage Ledger、Pass 管理、确定性 Writer 和诊断模型。每个 Pass 必须声明输入不变量、输出不变量、是否允许产生未决状态、复杂度或资源上限以及失败后是否可继续。

Compiler Core 的放行证据包括：相同输入的字节级复现、规范化 Round-trip、类型与约束负例、歧义保持、Source Unit 覆盖、精确载荷保护、跨 Pass 属性测试和关闭模型后的完整 Strict 流程。性能数据不能替代语义和安全正确性。
