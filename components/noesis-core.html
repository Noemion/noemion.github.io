NIR / NOBJ

Noesis 在这里表示从表达形成可检验意义的编译活动。Noesis Core 是唯一可以生成 NIR 和 NOBJ 的组件；它规范化输入，绑定符号与概念，检查类型和约束，保留不能消除的歧义，并确定对象布局。

三种输入配置

模式输入与信任边界
Deterministic 配置只读取受控来源，不调用模型；相同的规范化输入和环境必须产生相同结果。
Model-assisted 配置Horizon Engine 提交带来源、证据、置信度和歧义状态的封闭候选；Noesis Core 决定接受或拒绝。
External-generation 配置外部模型可以生成候选来源，但候选仍是不可信输入，必须重新经过完整的确定性检查。

具体命令行选项属于工具接口，不由组件职责定义冻结；只有 CLI 规范、版本策略和对应一致性测试获批后才形成兼容承诺。

当前编译流程

  1. 规范化源文本、上下文、资源、领域命名空间和策略。
  2. 识别目标、结果类型、模态、否定、范围、条件与多个合法解释。
  3. 构造高层 NIR，执行实体绑定与硬约束、软偏好、推断权限编译。
  4. 分类未指定、可推断、冲突、待确认与本体库外语义。
  5. 构造验收契约和证据图，检查类型、符号、可满足性和来源覆盖。
  6. 把高层表示实现化为可编码的 NIR，再由确定性对象写入器生成 NOBJ。

失败边界

  • 模型置信度不能替代类型、覆盖或安全证明。
  • 编译器无法可靠理解的输入是编译失败；源内容显式保留的歧义才是合法运行语义。
  • 精确载荷不能被有损摘要；覆盖缺口、冲突和未分类来源不能静默发布。

后续计划：长上下文与模型辅助

第一阶段只设计规范与安全对象核心。来源图长上下文编译和 Horizon Engine 模型辅助输入，要等确定性路径、覆盖模型和拒绝行为稳定后再开发。

详细设计入口与验证证据

尚未冻结的核心契约包括规范化上下文、符号与类型环境、约束求解接口、歧义状态机、证据账本、编译阶段管理、对象写入器和诊断模型。每个编译阶段必须声明输入和输出不变量、是否允许产生未决状态、资源上限以及失败后能否继续。

验收测试需要覆盖相同输入的字节级复现、规范化往返、类型与约束负例、歧义保持、来源单元覆盖、精确载荷保护、跨阶段属性测试，以及关闭模型后的完整确定性流程。性能数据不能替代语义和安全正确性。
