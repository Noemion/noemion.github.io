NIR / NOBJ

Noesis 在这里表示从表达形成可检验意义的编译活动。Noesis Core 是唯一可以把来源规范化为合格 NIR，并通过规范对象写入路径生成 NOBJ 的组件；外部工具可以编写候选文本或封装输入，但不能绕过 Core 的确定性检查形成合格对象。

模型候选通过类型或约束检查，只能证明候选内部结构成立，不能证明它忠实表达了来源。每项确认语义必须能由受控语法与版本化规则重新推导，或绑定经过授权的 Source Binding Decision；否则必须保留为歧义、待确认或未解决状态，或拒绝编译。

三种输入配置

模式输入与信任边界
Deterministic 配置只读取受控来源，不调用模型；相同的规范化输入和环境必须产生相同结果。
Model-assisted 配置Horizon Engine 提交绑定来源跨度、替代解释、证据、置信信息和未决状态的 Candidate Envelope；Noesis Core 只确认可重推导或已获授权绑定的主张。
External-generation 配置外部模型可以生成候选 Noesis Source 或 Candidate Envelope，但候选仍是不可信输入，必须经过相同的来源绑定、类型、约束、歧义与覆盖检查。

这些配置属于前端与编译编排，不表示 Noesis Core 内部调用模型。具体命令行选项属于工具接口，不由组件职责定义冻结；只有 CLI 规范、版本策略和对应一致性测试获批后才形成兼容承诺。

当前设计的编译流程

  1. 规范化源文本、上下文、资源、领域命名空间和策略。
  2. 验证 Candidate Envelope 的来源身份、跨度、替代项和确认权限；无法建立来源绑定时保留未决或失败。
  3. 识别目标、结果类型、模态、否定、范围、条件与多个合法解释。
  4. 构造高层 NIR，执行实体绑定与硬约束、软偏好、推断权限编译。
  5. 分类未指定、可推断、冲突、待确认与本体库外语义。
  6. 构造验收契约和验证计划，检查类型、符号、可满足性和编译来源覆盖。
  7. 把高层表示规范化到可编码层，再由确定性对象写入器生成 NOBJ。

失败边界

  • 模型置信度不能替代来源绑定、类型、覆盖或安全证明。
  • 结构合法但无法证明忠实于来源的模型主张不得升级为确认语义。
  • 编译器无法可靠理解的输入是编译失败；源内容显式保留的歧义才是合法运行语义。
  • 精确载荷不能被有损摘要；覆盖缺口、冲突和未分类来源不能静默发布。

存在必要性与部署形态

任何合格 NIR/NOBJ 都需要同一套确定性来源绑定、类型、约束、歧义和对象写入规则，因此 Noesis Core 的逻辑边界不能省略。它不必是独立进程：noesis、morphe 或其他前端可以共享同一核心库，但不能各自发展一套不同语义或让模型绕过该路径。

后续计划：长上下文与模型辅助

第一阶段只设计规范与安全对象核心。来源图长上下文编译和 Horizon Engine 模型辅助输入，要等确定性路径、覆盖模型和拒绝行为稳定后再开发。

详细设计入口与验证证据

尚未冻结的核心契约包括 Candidate Envelope、Source Binding Decision、规范化上下文、符号与类型环境、约束求解接口、歧义状态机、证据账本、编译阶段管理、对象写入器和诊断模型。每个编译阶段必须声明输入和输出不变量、是否允许产生未决状态、资源上限以及失败后能否继续。

验收测试需要覆盖相同输入的字节级复现、规范化往返、类型与约束负例、歧义保持、来源单元覆盖、精确载荷保护、跨阶段属性测试，以及关闭模型后的完整确定性流程。性能数据不能替代语义和安全正确性。

语义职责见Noema IR 规范，对象编码见Noema Object 规范，编译与文本对象入口见noesis和morphe。
