Noema Object System

链接、验证和装载 Noema Object，并为结果验收建立可信运行环境。

LinkValidateFulfill

链接器怎样组成完整对象

简要说明：链接器判断多个局部对象能否组成整体；装载器判断整体能否进入当前环境；Fulfillment Runtime 只在前两项检查通过后执行受约束求解。三者分别报告成功或失败，不能合并成含义模糊的“可运行”状态。

  • 合并可兼容的 Section、字符串、符号、语义节点与依赖。
  • 解析导入导出，重新编号对象内部 ID，并应用资源、策略、评价器、模型、结构定义、原型和能力等引用类别的重定位。
  • 按语义类型处理约束、权限、工作流和披露规则冲突，不采用静默的最后写入者覆盖。
  • 删除不可达内容后重建受影响的摘要、完整性结构与签名计划。

R_NIR_* 等具体重定位标识只是候选命名，尚未成为规范。

HOBJ 共享边界

只读语义图、工作流、契约、知识子对象和语义原型可在多个智能体间共享；披露状态、记忆、上下文、调用表、绑定和权限状态属于会话或实例私有。强依赖必须形成可验证闭包。

装载与结果验收

  1. 检查魔数、版本、Header、边界、计数、索引和对齐。
  2. 验证签名范围以及待加载数据的完整性证明。
  3. 按类型和权限映射 Segment，建立符号、模型与调用表并完成允许的重定位。
  4. 检查策略后冻结关键绑定、权限与入口。
  5. Fulfillment Runtime 把可执行 NIR 实现化为模型输入和运行控制结构，并记录生成、评价、选择与终止结果。

当前设计：完整性与装载后保护

当前设计建议采用数据块、Section 与全局根组成的分层 Merkle 验证，并将 Header、表项、偏移长度、入口、符号、重定位、权限、策略和模型要求纳入签名范围。AI-NX/AI-RELRO 是描述段解释权限与重定位后冻结策略的设计名称，具体位标志和 ABI 尚未冻结。

尚待确定：消费端与动态依赖

输入渲染器与直接实现化接口的 ABI、动态依赖声明、撤销、回滚、披露状态同步和预算协议仍需规范与 ADR。Agent Harness 负责外侧会话和能力控制，不能把这些运行状态写回签名对象。

后续计划：可信装载

最小装载器和 Fulfillment Runtime 要等安全对象核心、确定性编译和链接规则稳定后再实现。签名、Merkle 证明、权限映射和状态漂移拒绝都需要畸形输入与对抗测试。

概要设计：事务边界与验证

链接和装载都应采用“先计划、后提交”的事务边界。链接器先完成输入扫描、符号选择、ID 映射、约束与权限冲突分析和布局计划，所有强制检查通过后才产生输出对象；装载器先验证结构、完整性、版本、依赖和权限映射，全部通过后才暴露可执行入口，避免部分可信状态。

组件可观察产物关键失败证据
链接器链接映射、符号选择、ID 映射、依赖闭包、输出摘要未解析、重复定义、类型、约束、权限与配置冲突报告
验证器分层验证结果与规范条款定位字节、记录、节点、策略或证明路径的精确诊断
装载器装载计划、Segment 权限、冻结绑定和运行入口篡改、回滚、越权、状态漂移和资源上限拒绝
Fulfillment Runtime披露决定、实现化计划、评价与终止记录预算不足、能力未授权、契约未满足和外部依赖失败
