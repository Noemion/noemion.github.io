Noema Object System

链接、验证和装载 Noema Object，并向运行层交付不可变的已验证状态。

LinkValidateLoad

链接器怎样组成完整对象

简要说明：链接器判断多个局部对象能否组成整体；验证器判断整体是否满足结构、语义、覆盖和信任要求；装载器判断经过验证的整体能否进入当前环境。三者分别报告成功或失败，不能合并成含义模糊的“可运行”状态。

  • 合并可兼容的 Section、字符串、符号、语义节点与依赖。
  • 解析导入导出，重新编号对象内部 ID，并应用资源、策略、评价器、模型、结构定义、原型和能力等引用类别的重定位。
  • 按语义类型处理约束、权限、工作流和披露规则冲突，不采用静默的最后写入者覆盖。
  • 删除不可达内容后重建受影响的对象摘要与完整性输入材料；任何变换都会使旧签名失效，最终签名范围与回填由发布封装阶段负责。

R_NIR_* 等具体重定位标识只是候选命名，尚未成为规范。

HOBJ 共享边界

只读语义图、工作流、契约、知识子对象和语义原型可在多个智能体间共享；披露状态、记忆、上下文、实时能力句柄、绑定和权限状态属于会话或实例私有。必需依赖必须形成可验证闭包。

验证与装载

  1. 检查魔数、版本、Header、边界、计数、索引和对齐。
  2. 验证签名范围以及待加载数据的完整性证明。
  3. 按类型和权限映射 Segment，建立符号、模型绑定、能力需求描述与外部依赖表，并完成允许的重定位。
  4. 检查策略后冻结关键绑定、权限与入口。
  5. 全部检查通过后产生不可变 Loaded State 并交给 Agent Harness；Runtime 只通过 Harness 获得当前请求所需的只读语义视图，对象系统不执行生成式求解。

设计方案：完整性与装载后保护

当前设计建议采用数据块、Section 与全局根组成的分层 Merkle 验证，并将 Header、表项、偏移长度、入口、符号、重定位、权限、策略和模型要求纳入签名范围。AI-NX/AI-RELRO 是描述段解释权限与重定位后冻结策略的设计名称，具体位标志和 ABI 尚未冻结。

待定事项：消费端与动态依赖

Loaded State Handle、动态依赖声明、撤销、回滚和披露视图仍需规范与 ADR。Loaded State 只携带不可变能力需求和授权上限，不携带实时句柄；Agent Harness 负责会话级能力绑定，Fulfillment Runtime 负责求解；两者都不能把运行状态写回签名对象。

后续计划：可信装载

最小验证器和装载器要等安全对象核心、确定性编译和链接规则稳定后再实现。签名、Merkle 证明、权限映射和状态漂移拒绝都需要畸形输入与对抗测试。运行层的后续设计见Fulfillment Runtime。

存在必要性与部署形态

只做来源编译或静态对象检查时可以不启动装载部分；一旦需要组合多个对象、验证发布包或建立运行状态，就必须有独立于 Runtime 的对象系统边界。它可以作为 synthesis、noemvalidate 与 noemexecute 共享的安全库，而不必单独成为服务，但解析、验证、提交与失败证据必须保持一致。

设计概要：事务边界与验证

链接和装载都应采用“先计划、后提交”的事务边界。链接器先完成输入扫描、符号选择、ID 映射、约束与权限冲突分析和布局计划，所有强制检查通过后才产生输出对象；装载器先验证结构、完整性、版本、依赖和权限映射，全部通过后才暴露可执行入口，避免部分可信状态。

组件可观察产物关键失败证据
链接器链接映射、符号选择、ID 映射、依赖闭包、输出摘要未解析、重复定义、类型、约束、权限与配置冲突报告
验证器分层验证结果与规范条款定位字节、记录、节点、策略或证明路径的精确诊断
装载器装载计划、Segment 权限、冻结绑定和运行入口篡改、回滚、越权、状态漂移和资源上限拒绝
Loaded State 交付不可变对象视图、能力需求描述、授权上限、依赖身份和装载摘要交付前状态漂移、描述不完整、权限不一致或身份失配
