Fulfillment Runtime

在受限语义视图和显式运行配置下求解目标、评价候选并提出能力请求，不改变对象约束，也不宣告任务最终验收。

FulfillmentUntrusted CandidatesCandidate Assessment

它解决什么问题

Noema Object 描述什么结果可以接受以及哪些边界不得违反，但对象本身不执行生成式求解。Fulfillment Runtime 接收 Harness 提供的只读语义视图、模型或其他求解后端和当前观察，把抽象目标实现为候选结果；需要外部能力时只返回类型化 Capability Request。

生成出内容不等于任务完成。每个候选仍是不可信结果，Runtime 只能形成 Candidate Assessment；真实环境证据由 Harness 汇总，最终 Acceptance Decision 由执行前固定的策略或指定外部权威作出。

与 Agent Harness 的边界

Agent Harness 管理任务会话：它选择何时披露上下文、怎样绑定与执行能力、怎样观察环境、何时再次调用 Runtime，以及何时请求人工判断。Fulfillment Runtime 只负责一次受约束求解与候选评价，不拥有会话级权限、实时能力句柄或工具编排权。

  • Harness 从 Loaded State 建立请求所需的只读语义视图和 Runtime Request。
  • Runtime 返回候选、Candidate Assessment、Capability Request、资源消耗和失败分类。
  • Harness 校验能力请求，执行或拒绝后返回结构化 Capability Observation。
  • Harness 根据预先声明的停止和升级条件决定下一步。
  • Runtime 内部可以执行受预算约束的候选采样，但会话级重试和权限变化只能由 Harness 决定。

候选评价与会话停止

Runtime 评价器可以是确定性规则、模型或外部检查器，但它们必须声明输入、权限、失败语义和证据强度。模型评价不能替代确定性安全检查，Candidate Assessment 也不能替代 Harness 的证据闭包与最终验收决定。

  • 硬约束失败时拒绝候选，不通过软偏好补偿。
  • 预算、迭代次数和截止条件由运行配置提供。
  • 重复失败、评价器分歧或需要价值判断时返回 Harness 处理。
  • Runtime 只记录候选通过、候选未满足或无法评价；最终 accepted 必须绑定对象、配置、后端、证据、环境和决定权威。

当前成熟度

当前只形成组件职责、信任边界和验证路线，尚未发布 Fulfillment Runtime。Runtime Request、Capability Request、Capability Observation、Candidate Assessment、后端适配器、评价器 ABI、预算单位和跨 Runtime 等价要求仍需规范、ADR、原型和对抗测试。

存在必要性与替换边界

编译、查看、链接、验证和发布对象不需要 Fulfillment Runtime；只有把对象目标转化为具体候选时才启用。Runtime 的价值是用统一请求和结果协议隔离模型、规则求解器或其他后端差异，因此具体后端可替换，受限语义视图、Candidate Assessment、Capability Request 和失败分类的逻辑边界不能消失。

验证要求

  • 同一 Loaded State 在相同后端与锁定环境下产生可比较的运行记录，随机性必须显式受控。
  • 硬约束、权限和验收契约通过变异测试证明不会在实现化中丢失。
  • 无效模型输出、工具污染、评价器失败、能力拒绝、预算耗尽和状态漂移都有稳定返回路径。
  • 跨后端结果比较区分“同一结果”“都满足契约”和“无法比较”，不能把文本相似度当作语义等价。

装载边界见Noema Object System，共享披露对象见Horizon Object，运行驱动与证据入口见noemexecute、noemobserve与noemcoverage。
