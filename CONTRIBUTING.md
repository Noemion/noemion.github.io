# 为 Noemion 贡献

Noemion 当前公开的是研究、术语、架构、规范、威胁模型、验证方案和网站资料，还没有组件实现。贡献的第一目标是让一个可检验的主张更准确、更清楚或更容易验证；增加页面、术语、对象或动作数量本身不算进展。

## 先选择正确入口

- 发现事实错误、失效链接、难懂段落、无障碍问题或响应式问题，请提交[资料与网站问题](https://github.com/Noemion/noemion.github.io/issues/new?template=01-documentation.yml)。
- 提出术语、对象边界、规范义务、威胁或验证方法，请提交[研究与规范问题](https://github.com/Noemion/noemion.github.io/issues/new?template=02-research.yml)。
- 开始工作前先搜索[现有问题](https://github.com/Noemion/noemion.github.io/issues)，避免重复讨论同一边界。

当前没有公开的私密安全报告渠道。不要在公开 Issue 中提交未公开漏洞、凭据、密钥、个人数据或可被滥用的样本。公开渠道可以报告不含敏感细节的文档错误和安全边界缺口；需要私密材料的问题应等候项目公布专用渠道。

## 把意见写成可复核主张

一项可处理的贡献至少回答以下问题：

1. 哪个读者、对象、条款或运行边界受到影响？
2. 当前资料具体说了什么，问题可在什么条件下观察？
3. 你建议改成什么，什么反例会推翻这个建议？
4. 哪份规范、论文、标准、实现资料或真实案例支持判断？
5. 修改最多能证明什么，还有什么不能据此声称？

术语提案还要说明：普通工程词为什么不足、名称指向哪个独立对象、谁是实际消费者、与相邻词怎样区分，以及中文和英语口头交流中可能出现的读音或听写混淆。词源、检索结果少或个人觉得顺口，都不能单独证明名称适合发行。

## 权威源与修改位置

| 修改内容 | 先检查 | 修改后应保持 |
| --- | --- | --- |
| 公开解释与学习路径 | 对应 HTML/Markdown 页面、现行 ADR 与规范源 | 首段给出直接结论；链接指向唯一权威源 |
| 规范义务 | `spec/*.md` 与 `spec/registry.json` | 条款、成熟度、威胁、验证方式和向量引用一致 |
| 术语或对象边界 | 命名规范、名称审计、相关 ADR 与研究提案 | 职责、机器身份、权限、结果域和读音证据不混淆 |
| 场景与反例 | 对应 `spec/*-scenarios.md` | 案例明确标为非规范；只有转成条款和向量后才支持符合性主张 |
| 路由、导航或版式 | `sitemap.md`、页面角色与设计系统 | 正式路由唯一；桌面和窄屏均保持可读与可操作 |

不要在解释页复制第二套规范，也不要把研究提案、资料检查或页面构建写成组件已经实现。

## 写作要求

- 先写现实问题和直接结论，再引入 Noemion 专名。
- 使用主动语态，明确输入、执行者、动作、结果和失败责任。
- 一句话只推进一个主要判断；三项以上并列关系改用列表、步骤或表格。
- 区分思想来源、工程类比、规范要求、外部协议事实和验证证据。
- 模型输出、协议状态、签名、授权、满足判断和最终决定分别表达。
- 使用“当前策略、已有成果、正在研究、待定内容、限制条件”等读者可理解的状态，不把内部工作过程写进公开内容。

## 提交前验证

资料修改至少运行：

```bash
python3 tests/site_quality_test.py --source-only
```

规范、登记或向量修改还要运行与修改主题对应的 `tests/*_test.py`。完整网站修改需要用仓库锁定的 Jekyll 环境构建 `_site`，复制 `sitemap.md` 后执行成品检查。请在 Pull Request 中列出实际运行的命令、结果和未验证范围；不要把“测试通过”扩大成测试没有覆盖的结论。

## Pull Request 应说明什么

- 解决的问题和可证伪主张。
- 修改过的权威源与读者页面。
- 支持案例、拒绝案例或威胁。
- 实际验证结果和声明上限。
- 新增术语的必要性、冲突与读音风险；没有新增术语时直接说明。
