项目/规范参考指南/Iknem

有范围证据记录
Iknem

记录谁在何时、用什么方法观察了哪个精确对象，并限定这项观察最多支持什么结论。

IKN-CORE 0.1.0-draft主张必须有范围尚无物理格式尚未实现

先用一次发布判断理解它

有范围证据记录（设计阶段名称 Iknem）只回答一件事：某个生产者通过已说明的方法，对一个精确对象观察到了什么，这项观察最多支持哪一句结论。

假设目标是“更新服务依赖，并确认当前版本可以发布”。锁文件、部署接口、健康探针和模型说明都能提供信息，但它们回答的问题不同。开发者先保留各自的对象、方法和时间，再判断这些记录是否适用于当前目标。

取得的信息这项记录可以声称什么仍需补充什么
锁文件中的版本与摘要指定解析器从该文件中读到了这组依赖版本和摘要实际部署对象和运行状态
部署接口返回成功指定接口在该时刻接受了请求变更是否生效、目标实例是否正确
精确实例的健康探针响应该实例在声明方法和时刻满足了探针条件判据要求的完整观察窗口
模型对日志的解释保存为待核查的 model-candidate原始观察、变换过程和独立评估

这四项信息不能压成一个“发布成功”。Iknem 保存可以复核的有限主张；krin 说明当前目标需要哪些观察，具名权威最后决定是否接受结果。

当前状态：IKN-CORE 0.1.0-draft 已定义抽象职责。物理格式和组件接口尚未形成，也没有相应的采集、验证、归并、撤销或决定实现。这份说明只提供规范阅读路径，不能作为可用组件或稳定互操作格式。

按九项责任建立一项记录

处理顺序仍是固定对象、保留观察、评估有效性、计算覆盖度、判断满足，最后由权威作出决定。九项条款分别固定每一步必须保存的事实和失败责任。

固定对象与问题
保留观察与变换
评估记录有效性
计算判据覆盖度
判断目标是否满足
由权威作出决定
规范责任开发者必须固定失败时怎样处理
IKN-SCP-001
主体与范围		精确对象、有限主张、生产者、方法、环境、适用时间和截止点主体或方法含糊时缩小主张，越界部分不能使用
IKN-PRV-001
有限溯源		活动、精确输入、父记录、派生关系和责任主体；关系必须有限且无环未知输入进入限制，循环自证或悬空依赖使溯源无效
IKN-OBS-001
观察与变换		原始观察、解析、过滤、聚合、脱敏、模型解释、信息损失和结构化 phain推断冒充观察或变换无法定位时，相关主张无效
IKN-CLS-001
记录种类与来源类别		observation / derivation / attestation / appraisal / decision-record 与 direct-observation / deterministic-derivation / external-assertion / human-judgment / model-candidate签名、分数、重复次数或生产者自述不能提升类别
IKN-INT-001
完整性分离		内容身份、顺序、生产者认证、签名、时间戳和透明日志关系完整性只能说明来源与未检测到的修改，不能代替事实正确和授权
IKN-VAL-001
外部有效性评估		验证者、政策、参考值、信任根、环境、截止点和撤销状态评估依据变化后重新求值；记录不能给自己填写 valid
IKN-COV-001
判据覆盖度		精确 krin、关系位置、主体集合、时间范围和不同观察责任存在缺失路径、未知损失、开放集合或时间空洞时输出 insufficient
IKN-DEC-001
决定分离		输入记录、有效性、覆盖度、满足结果、决定规则和具名权威decision-record 只记录已发生的决定，不创造决定权限
IKN-PRI-001
最小披露		外置对象、摘要、脱敏规则、可见范围、不可见范围及其对主张的影响Bearer token、私钥、会话 cookie、文件描述符和实时能力句柄不得进入记录；关键材料不可得时相关依据不可验证

四个结果必须分别保存

结果域允许值回答的问题
Iknem 有效性valid / invalid / revoked验证者是否在指定政策、环境和截止点下接受这项记录
证据覆盖度sufficient / insufficient适用记录是否覆盖精确 krin 的全部观察责任
满足判断met / unmet / agno / fault目标要求与结构化观察比较后得到什么结果
权威决定accepted / rejected / deferred具名权威依据结果和决定规则作出什么决定

valid 是外部评估，不是记录给自己的标签。覆盖度按不同观察责任计算，不按日志、签名或重试次数计算。政策、密钥、方法、对象或截止点变化后，验证者必须重新评估。

外部输入只按原身份进入

模型、Agent 协议、遥测和签名材料都可以进入证据路径，但它们只能支持与自身来源、方法和范围一致的主张。

复核日期：2026-07-16。外部标准只说明为什么要分开证据、评估和使用方决定；它们不定义 Iknem 的物理编码、身份或最终权威。

相邻问题按需要继续展开

IKN-CORE 只定义证据记录的共同边界。开发者提出更强主张时，再进入对应研究资料；这些提案不会在这里复制第二套规则。

开发者问题继续阅读当前上限
模型评分能否参与评测，训练反馈能否成为发布依据？模型参与评测研究与模型训练与更新研究正在研究；模型输出保持有范围候选
一条轨迹能否证明动作、状态转变或因果？状态变化与因果归因研究正在研究；现行记录只支持已声明的方法和范围
历史与检查点何时能支持证据？记忆、检查点与恢复研究保存状态只说明生产者保留了什么，不恢复旧权限或补齐未知副作用
删除收据能否证明全部副本已经清除？数据使用、保留与删除研究正在研究；请求、操作、验证、确认、备份和派生物分别记录

规范来源与当前上限

资料层级权威入口能够证明什么
现行规范IKN-CORE 规范源主体、溯源、观察、类别、完整性、有效性、覆盖、决定与披露的唯一条款源
设计与一致性材料当前草案威胁模型、非规范场景与提案向量只支持设计复核和资料关系；向量仅检查资料一致性，不能证明组件实现或稳定格式
背景与边界ADR-0022解释采用理由、外部先例、限制和待定内容，不建立第二套条款

当前策略：IKN-CORE 已确定抽象证据与评估边界，但不锁定物理容器、扩展名、字段编号、摘要与签名算法、透明日志、撤销分发、远程查询、跨生产者时钟归并、隐私策略语言和长期归档。未来实现必须以新 Profile 和规范字节证明互操作。

名称仍在研究：Iknem 是设计阶段名称，发行拼写、读音流畅度和与 Endem、Synem、Dromen 的口头区分度尚未通过真实使用者验证。开发者文档必须先写“有范围证据记录”，不能让名称替代可验证定义。

spec/iknem-core.md 是唯一现行规范条款源；这份说明只提供阅读路径，不建立第二套字段或符合性要求。
