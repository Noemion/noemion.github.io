SSO 与渐进式披露

Shared Skill Object 的语义完整性、依赖闭包、链接、strip 与受控披露。

Semantic CompleteProgressive DisclosureStrippable

已确认原则：完整性边界

SSO 不是原始 Prompt 或 Skill 的压缩包，而是经过语义覆盖验证、可链接且支持渐进式披露的对象。说明性原文可被移除，但所有影响行为的目标、约束、条件、流程、例外、知识与依赖必须保留。

  • 规范化 Skill 与任意 Prompt 可使用不同前端，但汇入同一确定性 Compiler Core。
  • 大型知识的完整性由主 SSO 与经过验证的内容寻址依赖闭包共同承担。
  • 任意自然语言的语义等价不能伪装成数学证明；覆盖证书必须可审计并允许保守拒绝。

已确认原则：链接与 strip

  • 硬约束和接口冲突导致链接失败；权限默认取更严格交集；低信任对象不能覆盖高信任策略。
  • 工作流默认保留命名空间，软偏好按显式优先级合并，链接器输出合并与冲突报告。
  • 开发对象与发布对象具有同一运行语义；发布形态通过删除非运行时调试信息获得。
  • strip 前后目标、约束、Artifact Contract、依赖、披露行为和运行语义哈希保持一致。

设计提案：披露图与覆盖状态

当前提案使用 Discovery、Contract、Execution、Knowledge 和 Debug 五层披露视图；固定层级保证审计，查询驱动选择减少上下文。Source Unit 的覆盖状态拟区分已映射、重复、装饰、精确 Artifact、未解决和冲突。

代码、数值表、公式、Schema、逐字模板、法律条款和二进制资源等不可安全摘要的内容必须精确保留或由可信外部资源承载。

已确认原则：受控披露

  1. 调用方请求能力、主题或符号。
  2. Runtime 检查 Trigger、权限与信任策略，并计算依赖闭包。
  3. 相关 Contract 与硬约束强制加入，再在预算内加入 Knowledge 与 Example。
  4. 必需内容超过消费端限制时拆分任务或拒绝，不能裁掉安全边界。

开放问题：披露 ABI

节点类型、Section 名称、状态哈希、预算单位、继续句柄、检查点与消费端确认协议仍是设计提案。跨轮 Bundle 的字段示意不能视为已冻结 ABI。

未来阶段：长上下文编译

Source Graph、局部抽取、全局链接、定向回读、Coverage Audit 与 Disclosure Planning 将分阶段实现；无法安全拆分或可靠理解的输入必须报错，不得静默截断。

概要设计：发布闭包与研究验证

SSO 发布闭包由主对象、强依赖、版本与内容摘要、权限和信任策略、披露图、Artifact Contract、评价器以及必需精确载荷组成。弱依赖必须声明缺失行为；动态依赖必须受 Profile、锁定和运行授权约束。

验证工作需要回答三类问题：语义完整性是否优于普通压缩或摘要；渐进式披露能否在不丢失安全约束的前提下降低上下文成本；不同编译器或 Runtime 是否能对同一 SSO 得出一致的必需闭包与拒绝结论。每项实验都应包含长上下文基线、对抗遗漏、权限冲突、预算不足和不可安全摘要材料。
