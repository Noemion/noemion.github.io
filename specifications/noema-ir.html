Noema IR

把目标、约束、歧义和验收条件表示为可以检查与链接的中间对象。

Noema IR 表示什么

Noema 在这里指不同表达可以共同指向的意向相关项。Noema IR（NIR）描述什么结果可以满足任务，以及求解过程不得越过哪些边界；它不是二进制化的提示词、文本包或普通序列化容器。

直白解释：提示词说明“怎样表达请求”，NIR 则保存“最终必须实现什么、哪些条件绝不能违反、哪些信息仍不确定、用什么证据验收”。措辞可以变化，但经过规范化的目标身份和硬约束不能因此漂移。

CandidateAssessment(y, G, H, C) ∈ {satisfied, unsatisfied, unknown}
AcceptanceDecision = Decide(Policy, CandidateAssessment, EvidenceClosure, Authority)

非规范性直觉：G 表示目标，H 表示硬约束，C 表示登记上下文。候选评价和真实环境证据闭包是两个不同输入，决定还必须绑定执行前固定的策略与有权作出判断的主体。有效结果可以有多个或形成偏序；任何必需条件为 unknown 时只能拒绝或转入 pending-review，不能假设存在唯一可计算最优解。

核心语义

下列结构名称描述语义职责，不等同于已冻结的二进制 Section 或字段。实现必须先满足职责与不变量，再由 NOBJ 规范决定具体编码。

结构职责
Goal Graph目标、目标状态、优先级和完成条件。
Semantic Context Graph实体、概念、资源、作用域与来源。
Constraint Lattice硬约束、软偏好、优先级、覆盖与冲突。
Ambiguity Set候选解释、证据、风险和处理策略。
验收契约结果类型、结构、语义属性与输出编码要求。
Validation Plan规则验证器、评价器、阈值、所需证据与无法自动判断时的处理方式；真实运行证据不写回该计划。

歧义、模态与推断权限

实现必须能表达已知、未指定、可推断、歧义、冲突、未解决和本体库外等状态。硬约束与软偏好分离，推断权限至少区分允许推断、禁止推断和需要确认。证据不足不能被模型置信度替换为唯一结论。

设计方案：语义关系、来源与满足环境

根据思想与方法基础提出的分析框架，当前研究提议把经常混用的四类记录关系分开。它们并非同一种“身份”；此处只登记待验证的结构，不构成已冻结字段或编码。

关系回答的问题需要保留的证据
Source Expression用户、规范或工具实际给出了什么表达？来源、跨度、版本、许可和原始上下文。
Compiled MeaningNoesis Core 依据什么确认或保留了哪些规范化语义？类型、确定性规则、Source Binding Decision、歧义分支和编译诊断。
Reference Binding符号此刻绑定到哪个对象、版本或能力？作用域、解析路径、重定位和稳定标识。
Acceptance Contract什么可观察证据使目标或约束被判为满足？评价环境、变量绑定、阈值、验证器和失败语义。

语义来源还应区分明示内容、结构蕴涵、可撤销的语境推论、工具推导和模型候选。它们不能共享同一确认权限；Candidate Envelope 通过类型检查也不等于来源忠实。满足判断只能读取配置明确登记的证据、版本、能力和背景假设。

设计方案：混合语义与分层 IR

当前提案以符号语义图和带权约束承担边界与验证，以语义原型或连续残差表达难以完全离散化的要求。高层语义、规范化编码、后端适配和行为请求等分层视图仍只是研究候选；各层是否需要独立形式、怎样命名以及如何保持等价都须规范化。

待定事项：最小类型与连续表示

核心类型集、连续残差互操作、评价器接口和模型原生载荷边界尚未冻结，见开放问题。

设计概要：规范不变量与验证路线

  • 每个硬约束必须具有稳定标识、作用域、来源和可判定或显式不可判定状态；不得在实现化中降级为软偏好。
  • 每个推断结果必须能追溯允许它的权限、规则和证据；禁止推断与需要确认状态不能被模型置信度覆盖。
  • 验收契约必须定义结果结构、必需字段、编码和验收条件，使评价失败能够定位到契约项。
  • 只有在规范版本、规范化表示和等价规则冻结后，才能定义可互操作的 NIR 语义哈希；在此之前只能比较明确登记的结构与契约属性。

研究验证需要覆盖表达能力、来源忠实性、歧义保留、错误确定率、约束冲突定位、跨模型/Fulfillment Runtime 契约等价和表示成本。最小类型集应由代表性语料、反例和实现复杂度共同决定，而不是按预设数量凑齐。
