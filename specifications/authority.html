权威与授权决定边界

定义谁可以在什么范围内确认自然语言意义、解决待确认项、授予运行能力或作出决定，并让委托、同意、撤销和多人冲突都可核对。

AUT-CORE 0.1.0-draft12 ClausesNo Wire FormatNo Policy Engine

它解决什么问题

模型可以提出候选，界面可以收集点击，签名可以认证陈述，但只有落在精确政策与范围内的具名权威才能形成授权。

当前策略已确定这项跨制品边界：认证、授权、满足和最终决定必须分别判断，任何一层不得替另一层作结论。

此前 Endem、Dromen、Iknem、身份和协议适配规范都要求“具名权威”或“当前政策”，却没有共同定义主体、作用范围、委托、多人冲突、同意、撤销和重放。AUT-CORE 收敛这些跨对象不变量，不创建新的制品、文件格式、令牌、角色系统或组件。

当前仍在研究预览、dry-run 与批准之间的横切边界。非规范研究提案要求把显示、模拟、授权、执行尝试、事后观察、满足和最终决定分开；该提案仍处于研究阶段，不构成新的 AUT-CORE 条款。

权威边界：实现义务以 AUT-CORE 条款源为准。当前只有规范、威胁、场景和提案向量，没有身份提供方、政策求值器、同意界面、能力代理或决定服务。

一个候选即使结构合法、来自强模型并具有很高置信度，也只能停留在候选层。授权者必须看到候选相对来源改变了什么、影响哪些关系和角色、由谁产生、哪些待确认项仍然存在。权限只覆盖已展示且落在主体职责内的部分。

确定性规则也不能获得无限权威。规则必须固定版本、输入域、输出位置和失败责任；超出规则声明域的输入仍进入 apor 或拒绝，不能用“自动规则”掩盖新的自然语言判断。

Agent 委托为什么必须保留行动者

RFC 8693 区分 delegation 与 impersonation：委托保留实际行动者代表谁，冒充则可能让接收方只看到被代表主体。Noemion 默认只接受可追踪委托；即使外部协议允许冒充，本地记录仍保留真实行动者，并要求独立政策明确允许。

A2A 允许 AUTH_REQUIRED 沿 Agent 链继续委托，MCP 允许 scope 不足后重新授权。这些机制只传递请求或凭据，不证明某个 Agent 有权批准，也不能修改已有 Dromen。扩大能力必须产生新的授权决定和新的 Drase 会话。

Agent 名称为什么不能代替实际行动者

“用了哪个模型”“部署了哪套 Agent 定义”“哪个工作负载通过认证”“哪个运行实例调用工具”和“代表谁行动”是不同事实。AUT-CORE 已经分开请求主体、实际行动者与被代表主体；当前研究进一步要求实际行动者能够关联精确工作负载、运行实例、会话和一次调用，而不是只保存产品名或服务账户。

当前非规范软件 Agent 身份、委托与责任链研究提案用九个层次检查这组边界。它不修改 AUT-CORE，不创建 Agent 身份制品、目录、服务、组件或新专名；现阶段只把问题归还 ID、AUT、DRO、ADP、IKN、DIA 与 TEXT-IDENTIFIER。

访问、使用、保留与删除为什么不是同一授权

允许 Agent 读取精确对象，只允许一次具名访问尝试。把内容发送给模型或工具、写入日志或记忆、建立索引或嵌入、用于训练、披露给下游、延长保留期限以及执行删除，都需要分别绑定对象、动作、目的、接收方、条件和截止点。

删除请求也不等于删除授权或清除完成。审计保留、备份、派生物和下游对象可能具有不同控制者与约束；AUT 只记录谁允许哪个精确动作，实际执行、残留和清除效果继续由 ADP、ID、IKN 与 DIA 观察。

当前非规范软件 Agent 数据使用、保留与删除边界研究提案把十八种生命周期事实归还现有责任。它不修改 AUT-CORE，不创建数据、隐私或删除 CORE，也不形成法律结论。

同意和多人批准怎样避免形式化陷阱

人类点击只有在安全视图与精确机器请求一致、目的和接收方清楚、拒绝路径可达、期限和撤销后果明确时，才可能成为同意依据。点击者仍需通过当前角色和范围核对；同意不是身份认证或权威证明的替代品。

多人决定必须在看到结果前固定合格主体、不同身份规则、所需角色或数量、否决和冲突处理。同一控制主体的多个账户、重复签名和重放事件只能计一次；未解决冲突保持 defer，不能由最后到达的事件决定。

权威资料与采用限制

  • RFC 9396 精细授权请求提供动作、资源和部分授予边界；项目不采用 OAuth JSON 作为本地格式。
  • RFC 8693 Token Exchange区分委托与冒充；项目要求委托链始终保留实际行动者。
  • RFC 9470 Step-up Authentication只解决认证强度不足，不定义资源授权政策；重新认证成功不会自动授权动作。
  • RFC 9700 OAuth 安全实践要求令牌限制发送者、受众、资源与动作；项目把令牌视为能力上限而非权威本身。
  • MCP 2025-11-25 授权规范与A2A 1.0 规范提供外部授权和委托机制；本地语义与最终决定仍由 AUT-CORE 管理。
  • NIST 软件与 AI Agent 身份授权概念论文提出人类与 Agent 区分、委托、日志透明和身份元数据问题；它仍是概念工作，不是 Agent 身份标准。
  • SPIFFE 工作负载身份让软件主体在信任域中认证，但一个工作负载可以覆盖多个副本，SVID 也不授予应用动作权限。
  • MCP 企业管理授权集中处理组织级服务器接入；每次工具动作仍须保留实际行动者、被代表主体、资源、受众和本地授权决定。
  • GNU Guix 授权实践要求本地显式授权替代服务器的签名密钥；GnuPG 信任模型也把密码验证与本地信任判断分开。
  • GNU Coreutils id分别显示真实身份、有效身份、组和安全上下文；项目只采用这种分开表达的纪律，不把 Unix 进程模型直接当作 Agent 身份标准。

语言哲学边界

维特根斯坦在《逻辑哲学论》4.022 写道：“命题显示其意义。命题显示当它为真时事情是怎样的，而且宣称事情就是这样的。”Noemion 由此坚持：候选命题的结构与真假判断必须分开。

这段话不能决定组织中谁有权批准、同意怎样有效、委托能否成立或冲突怎样解决。授权是额外的制度与安全关系，必须由精确政策、主体、范围、决定和证据定义，不能从命题的逻辑形式或措辞本身推出。

待定内容

权威目录、角色和政策语言、授权事件物理编码、同意 UI Profile、法定人数算法、凭据格式、外部协议映射、撤销分发、能力代理、审计存储和法律效力仍保持开放。Agent 定义、工作负载、运行实例、会话、凭据与一次调用怎样形成可互操作绑定，也必须等待真实消费者、协议 Profile、威胁模型和规范字节，不能由当前研究提案提前固定。
