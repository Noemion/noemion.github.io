## 先看一次发布怎样确认“还是同一份”

团队先形成一份保留原始自然语言的目标制品，再为最终发布移除原文和可逆重建材料。开发者不能沿用文件名、旧摘要或形成版签名；每个对象都要先取得自己的精确身份。

1. 定位候选
2. 固定精确字节
3. 建立完整身份
4. 验证有类型陈述
5. 按截止点评估
6. 另行决定授权

| 发布材料 | 开发者可以确认什么 | 仍然不能确认什么 |
| --- | --- | --- |
| `service-release.endem`、仓库路径、URL 或版本标签 | 找到一个等待核对的候选对象 | 当前字节就是批准过的发布对象 |
| 身份域、对象种类、规范、表示、Profile、算法、长度和完整摘要 | 在完全相同的身份语境内比较精确字节 | 内容为真、语义等价、目标满足或制品已接受 |
| 覆盖对象、用途、受众和主体的签名陈述，以及证书、撤销与时间材料 | 按固定信任根、政策和截止点验证谁签署了哪一份受保护陈述 | 签名主体当前拥有发布权限，或陈述内容必然正确 |
| 裁剪发布版与受控来源伴随资料 | 分别建立新身份，并用显式变换关系连接含来源形成版 | 继承形成版身份、签名、证据、权限或接受状态 |

**当前策略：**身份相同只说明同一身份域与版本语境中的精确字节相同。签名验证、主体授权、目标满足、证据覆盖和最终接受仍由各自规则判断。

## 按十二项责任检查身份与签名

| 规则 | 开发者应做什么 | 必须拒绝什么 |
| --- | --- | --- |
| `ID-DOM-001` | 把身份绑定到对象种类、规范、表示、Profile 和版本化身份域 | 跨语境只比较摘要文本 |
| `ID-BYT-001` | 明确摘要覆盖的字节、顺序、长度和排除项 | 隐式规范化、可变输入、验证时猜测和自包含摘要 |
| `ID-REF-001` | 让安全引用携带完整身份，并唯一闭合到一个允许对象 | 名称、路径、URL、`latest`、build ID 或外部任务 ID 选择目标 |
| `ID-ALG-001` | 用版本化政策固定算法、参数、用途、长度和允许状态 | 未知算法、参数歧义、静默替换、试到成功或降级 |
| `ID-DSP-001` | 短摘要只用于显示，安全比较始终使用登记长度的完整值 | 前缀碰撞、首个匹配、缓存顺序或 build ID 校验 |
| `ID-EQV-001` | 把精确字节身份与语义等价分开 | 摘要相同推出真假，或不同字节自动继承 Semantic Key |
| `ID-STM-001` | 签名保护对象、类型、用途、受众、主体、政策和解释所需上下文 | 裸摘要签名跨对象、跨类型或跨用途重放 |
| `ID-ENV-001` | 把签名、证书、时间、日志和状态材料放在外置验证包络中 | 包络改变对象身份，或只保存密钥提示 |
| `ID-AUT-001` | 分别计算密码验证、主体认证、授权、适用性和最终决定 | 签名或日志包含直接证明真实、获权、满足或接受 |
| `ID-VAL-001` | 用具名截止点、信任根、政策、撤销和新鲜状态评估有效性 | 无限缓存成功，或在撤销、过期和政策变化后沿用旧结论 |
| `ID-REP-001` | 固定输入闭包、工具和参数，并比较独立路径产出的完整字节 | 相同日志、build ID、缓存副本或单次重跑冒充可复现性 |
| `ID-REL-001` | 让形成版、裁剪发布版和伴随资料各有身份、损失清单与显式关系 | 派生物继承来源身份、签名、证据、接受状态或能力 |

> **权威边界：**[ID-CORE 条款源](https://noemion.github.io/spec/identity-core.html)定义实现义务。页面只提供检查顺序，不建立第二套身份格式、算法登记或签名 Profile。

## 形成版、裁剪发布版与验证材料怎样连接

| 对象 | 应当保留什么 | 验证结论上限 |
| --- | --- | --- |
| 含来源形成版 | 原始自然语言、经过确认的目标结构、所用 Profile 和自己的完整身份 | 有权审核者可以核对来源到目标结构的形成关系 |
| 裁剪发布版 | 经过确认的目标结构、发布 Profile、重写后的内部引用和新的完整身份 | 所有验证者可以核对发布字节和适用外部陈述；发布版不重新包含原文 |
| 受控来源伴随资料 | 精确来源、变换、保留与删除清单、双方身份和访问边界 | 没有伴随资料的验证者不得声称已经独立复核原始表达的忠实转换 |
| 外置验证包络 | 有类型签名陈述、签名、证书或公钥材料、可信时间、日志与撤销状态 | 只支持具名截止点下的有限验证，不改变被指对象身份，也不授予动作权限 |

可复现性还需要精确输入闭包、工具版本、参数、环境约束和彼此独立的产出路径。输出相同支持声明范围内的逐字节复现；输出不同只证明存在字节差异，不能仅凭摘要定位原因。

## 规范来源与当前上限

[RFC 6920](https://www.rfc-editor.org/rfc/rfc6920.html)用算法、长度和完整摘要建立名称与数据的完整性绑定，但具体摘要输入仍由使用它的规范定义。[RFC 9052 COSE](https://www.rfc-editor.org/rfc/rfc9052.html)明确把 `kid` 作为不保证唯一的密钥查找提示；[DSSE 1.0.2](https://github.com/secure-systems-lab/dsse/blob/master/envelope.md)则把载荷类型与载荷共同纳入签名协议。它们支持身份语境和类型分离，不替 Noemion 选择物理包络。

截至 2026 年 7 月 16 日，[Sigstore Bundle 0.3.2](https://docs.sigstore.dev/about/bundle/)仍把签名内容与证书、时间戳和透明日志等验证材料组合保存；[SLSA 1.2 制品验证](https://slsa.dev/spec/v1.2/verifying-artifacts)要求先匹配陈述主体摘要，再按验证者配置的信任根、构建者和参数判断来源。[GNU `ld` build ID](https://sourceware.org/binutils/docs/ld/Options.html#index-_002d_002dbuild_002did_003dstyle)只标识原始链接输出，不是当前文件校验和；[GNU Guix challenge](https://guix.gnu.org/manual/en/guix.pdf)把可复现性落到独立产出的逐字节比较。

- [身份与签名威胁模型](https://noemion.github.io/spec/identity-threat-model.html) — 覆盖身份混淆、降级、重放、撤销和信任继承威胁。
- [设计场景](https://noemion.github.io/spec/identity-scenarios.html) — 用支持案例、反例与边界场景检查身份责任；场景不是签名格式或组件证据。
- [精确身份提案向量](https://github.com/Noemion/noemion.github.io/tree/main/vectors/identity) — 覆盖允许包络与确定拒绝，并绑定对应条款；向量不证明摘要器、签名器或验证器已经实现。
- [ADR-0027 · 签名有效，不等于制品获准](../architecture/adr-0027-exact-identity-and-attestation.html) — 说明共同身份标准、有类型签名和验证材料边界。
- [ADR-0036](../architecture/adr-0036-source-bearing-and-stripped-release.html) — 说明含来源形成版、裁剪发布版与受控伴随资料的关系。
- [模型训练与更新边界研究](https://noemion.github.io/spec/model-training-and-update-boundaries-proposal.html) — 把训练数据、基础模型、适配器和派生权重的身份问题留在非规范研究中。

**待定内容：**发行摘要与签名算法、算法登记表、摘要文本语法、COSE、DSSE 或 Sigstore Profile、证书体系、透明日志、时间戳、撤销分发、Semantic Key、跨版本等价和来源伴随物理格式均保持开放。

**限制条件：**ID-CORE 不是新制品、命令、密码算法或 Semantic Key。当前没有摘要器、签名器、验证器、透明日志客户端、可复现构建系统或发布组件。
