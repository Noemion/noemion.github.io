结构化诊断
让失败可以被人和机器共同理解

统一 Endem、Synem、Dromen、Iknem 与外部协议错误的机器身份、位置、恢复和安全边界。

DIA-CORE 0.1.0-draftDIA-CATNo Wire FormatNo Runtime

先给直白定义

结构化诊断是一份有限、可定位、机器可读的失败说明。它告诉调用者哪个操作依据哪条规则，在检查哪个精确输入时，因什么原因停止或需要注意。

诊断不是目标结果、证据、最终决定、权限或自动修复命令。一个诊断可以帮助人和 Agent 选择下一步，但必须经过原有政策、预算和授权边界。

用同一次发布理解一次失败

继续使用服务发布案例：外部调用已经发出，但连接在响应前中断，当前没有证据证明副作用是否发生。此时诊断负责说明“哪里失败、依据什么停止、下一步需要什么”，不能替系统猜测发布成功或自动再执行一次。

诊断字段本例记录开发者据此怎样处理
机器身份adapter.retry.not_authorized，主条款 ADP-RTY-001自动化依赖登记码，不解析“超时”“请重试”等消息
生产语境适配器版本、本地调用身份、精确发布输入、协议/Profile 与 DIA-CAT 版本防止把本次诊断缓存到另一个发布对象或调用
层次与位置policy 层、外部请求关联；重试检查之后的步骤标为 not-run保留传输事实，同时说明本地为何没有继续尝试
恢复分类operator-review，复核前置条件是取得去重或事后核对证据当前不自动重试；下一步仍受 Dromen 预算和具名授权约束
结果边界公开消息脱敏，保留主诊断和有限相关项不产生 unmet、accepted、回滚完成或新的权限

如果协议返回的是未知结构化错误，则保留协议版本、远端主体和原始码，再映射到受限通用诊断。无论消息怎样措辞，未知状态都不能降级为成功。

十条核心规则

规则直接含义
DIA-IDN-001稳定机器码与人类消息分开；程序不得解析措辞取得语义。
DIA-PIN-001固定生产者、操作、实际输入、规范/Profile/目录版本和本次发生。
DIA-LAY-001来源、结构、语义、会话、证据、协议和后端错误不得互相洗白，也不得替代结果域。
DIA-LOC-001来源范围、字节范围、语义路径和会话绑定必须类型化、有界并安全显示。
DIA-PRI-001主阻断诊断按规则和规范位置确定，不能取决于并发返回顺序。
DIA-REC-001恢复建议必须声明前置条件；不得授予权限、无限重试或直接执行副作用。
DIA-EXT-001外部错误保留协议版本、远端主体与原始结构化状态，再映射到本地码。
DIA-SEC-001默认最小披露，不保存令牌、私钥、cookie、能力句柄或未授权正文。
DIA-BND-001数量、深度、位置、消息和总输出都使用先检查的有限预算。
DIA-ATM-001阻断错误使操作原子失败，不能同时返回部分可信对象或成功状态。

一项完整诊断包含什么

内容例子为何需要
机器码与标题dromen.environment.drift让自动化依赖稳定身份，而不是消息文本。
生产语境生产者、drase、精确制品身份、DRO-CORE 与 DIA-CAT 版本阻止诊断被错误复用到另一个输入或规则版本。
层次与执行范围session，执行到 session，后续为 not-run避免“没有检查”被写成“已经通过”。
类型化位置session_binding: environment/adapter让人定位问题，同时限制攻击者控制的字符串。
恢复分类refresh-binding，需要重新观察环境给出下一步，但不自行修改 Dromen 或扩大权限。
披露与预算公共脱敏政策、最多 32 项、总输出 64 KiB防止秘密泄露和诊断风暴。

为什么不能只返回一段错误文字

文字适合解释，不适合承担机器协议。语言、本地化和措辞会变化；错误正文还可能来自攻击者、模型或远端服务。若 Agent 通过字符串匹配决定“重试”“提权”或“任务失败”，消息就变成了未授权控制接口。

DIA-CORE 因此要求机器码、条款、层次、位置和恢复分类独立存在。人类消息可以更清楚，但自动化只能消费登记字段，并继续服从 Dromen 的能力与预算。

MCP 的 JSON-RPC 协议错误、工具执行错误，A2A 状态，HTTP Problem Details，操作系统错误和模型 SDK 异常必须保留各自来源。适配器不能因为消息中出现“success”“failed”或“permission”就改写目标满足、最终决定或当前会话权限。

诊断目录怎样组织

DIA-CAT 使用对象或诊断系统本身作为首段，例如 endem.*、synem.*、dromen.*、iknem.* 与 diagnostic.*。Noemion 是项目品牌，不进入机器码。

目录保留已有 Endem 结构与内容草案码，并为 SYN-CORE、DRO-CORE、IKN-CORE 和 DIA-CORE 的每条核心拒绝责任登记初始码。目录版本必须与诊断一起固定；当前机器码仍是草案接口，不是稳定 ABI。

权威规范和验证资料

DIA-CORE 诊断内容条款十条关于身份、语境、层次、位置、主错误、恢复、外部来源、安全、预算和原子性的规范要求。 DIA-CAT 诊断目录登记跨 Endem、Synem、Dromen、Iknem 与诊断系统自身的草案机器码。 诊断威胁模型检查消息解析、状态洗白、主错误竞态、自动提权、泄密、风暴和部分成功。 诊断自然语言场景十五个支持案例、反例和边界场景；它们不是实现或符合性声明。 诊断提案向量二十个资料一致性提案，覆盖十项允许包络与十项确定拒绝。

当前状态

当前策略：DIA-CORE 用十条核心规则定义诊断内容。未来实现必须遵守这些规则，不得把诊断当作证据、结果、权限或自动修复命令。

物理编码、CLI 退出状态、SARIF/HTTP/MCP 映射、持久日志和遥测仍待确定，需要分别设计与验证。当前没有诊断生产器、渲染器、日志服务、重试引擎、协议适配器或 CLI；提案向量只描述规范预期。
