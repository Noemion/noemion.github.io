文本与标识符边界

定义系统怎样接收、比较、定位、显示和提交 Unicode 文本，并保证人类复核的内容与模型实际取得的内容可以核对。

TEXT-IDENTIFIER-CORE 0.1.0-draft12 Clauses24 VectorsNo Unicode Code

它解决什么问题

看起来相同的文字不一定是相同字节、相同码点或相同意义。机器实际读取的内容也可能与审核者在屏幕上看到的内容不同。

TEXT-IDENTIFIER-CORE 给现有字符串字段建立共同边界。它分开来源字节、解码文本、结构标识符、规范化结果、比较规则、文本范围、显示视图和模型输入。

当前已接受十二条跨制品文本边界。未来生产者和使用者必须明确自己处理哪一层，不得让显示、搜索、同形提示、语言标签或模型相似度替代精确机器内容与授权意义。

权威边界：实现义务以 TEXT-IDENTIFIER-CORE 条款源为准。当前没有 Unicode 处理器、规范化器、同形检测器、模型输入网关或新的文本格式。

先分清五类文本职责

文本槽承担的职责当前关键边界
source-content保存提交后解码得到的来源表达允许 Unicode；不静默规范化；不冒充原始来源字节
structural-identifier承担机器引用、闭包和结构比较END-P1 继续只允许 ASCII；大小写敏感；与 locale 无关
registered-token保存规范中登记的有限枚举和值域只接受登记值；不能由本地化文本替代
descriptive-text供人阅读的说明、限制和背景可以本地化；不能进入身份或授权判断
display-message呈现诊断、日志摘要和界面提示可以转义或截断；必须披露变换；不能成为安全引用

十二条文本规则

规则直白要求必须拒绝
TEXT-SLT-001每个字符串字段先声明文本槽和失败责任把来源正文、本地化消息和机器标识符混用
TEXT-ENC-001输入先按 RFC 3629 严格解码 UTF-8过长编码、代理项、非法标量和本地编码猜测
TEXT-SRC-001分开来源字节、解码文本、变换和损失把 END-SRCM 解码内容称为原文件逐字节副本
TEXT-IDN-001当前结构标识符保持封闭 ASCII ProfileUnicode 同形、默认 locale 和静默大小写折叠
TEXT-NRM-001规范化必须绑定文本槽、Unicode 版本和身份影响盲目 NFC、NFKC 或覆盖原文
TEXT-CMP-001每次比较明确字节、标量、规范化或排序域搜索命中、同形骨架或模型相似度替代精确相等
TEXT-RNG-001范围绑定精确主体、表示和 Unicode 标量单位混用字节、UTF-16、字素、屏幕列和模型 token
TEXT-BID-001逻辑存储顺序与双向显示顺序分开视觉顺序参与身份或安全相关方向控制被隐藏
TEXT-HID-001不可见字符与同形风险可被盘点和显式审查静默删除、自动合并身份或仅凭警告判断恶意
TEXT-MET-001语言、媒体、方向和 locale 元数据只承担声明自报语言触发未登记变换或取得语义权威
TEXT-AIM-001绑定模型实际输入、预处理、tokenizer 和复核视图人看清洗文本而模型接收另一组码点
TEXT-OUT-001显示、复制和导出披露转义、截断、脱敏与方向处理短显示或经过清洗的界面文本充当安全引用

当前怎样处理来源表达

END-SRCM 接受 UTF-8 来源清单，并把转义与换行规则应用后得到的文本写入 rhem.content。因此当前 Endem 保存的是提交后的解码表达，不是原始 .ends 文件的逐字节副本。

如果未来需要证明某份原始文件未经改变，必须另外绑定原始来源字节身份、解码 Profile、变换链和损失清单。不能用 rhem.content 反推原文件的字节序列。

为什么模型输入需要单独约束

不可见字符、同形替换和变体选择符可能改变 tokenizer 与模型行为，却不一定在普通界面中显现。未来任何模型或检索调用都必须绑定实际输入身份、全部预处理、隐藏字符清单、模型与 tokenizer 身份，以及供人核对的显示视图。

这项绑定只让候选过程可审查。模型输出仍然是 model-candidate，不能直接写入规范字节、删除 apor、扩大权限或宣告目标已经满足。

当前仍未冻结

Unicode 国际化结构标识符、来源文本重写、语言检测、分词、字素级编辑、搜索排序、字体信任、tokenizer 协议、同形自动处置、双向编辑器和原始来源字节字段都保持开放。任何新增字段或 Profile 都必须先证明真实生产者、消费者、身份影响和正反向量。
