项目/规范/HOBJ

Horizon Object

保存可以按权限和任务需要逐步披露的共享知识、依赖与行为边界。

Proposed ObjectUnfrozen ABIHOBJProgressive Disclosure

完整性边界

Horizon 表示当前尚未展开、但与对象理解有关的背景和可能状态。Horizon Object（HOBJ）不是提示词或 Skill 的压缩包；它保存经过覆盖检查、可以链接并按策略披露的共享内容。系统可以移除说明性原文，但必须保留所有会影响行为的目标、约束、条件、例外、知识和依赖。

直白解释：HOBJ 让多个任务复用同一份经过验证的能力，同时只向当前运行披露必要部分。节省上下文不能以遗漏安全条件、隐藏必需依赖或扩大访问权限为代价。

  • 结构化文档、受控源语言和其他来源可以使用不同前端，但必须汇入同一确定性 Noesis Core；模型候选仍须经过来源绑定。
  • 大型知识的完整性由主 HOBJ 与经过验证的内容寻址依赖闭包共同承担。
  • 任意自然语言的语义等价不能伪装成数学证明；Release Coverage Proof 必须可审计并允许保守拒绝。

产生与再次链接

Horizon Engine 只产生 Candidate Envelope，不产生 HOBJ。共享来源先由 Noesis Core 编译为可重定位 NOBJ；synthesis 再把一个或多个 NOBJ 与已有 HOBJ 合并为新的 HOBJ。HOBJ 可以继续作为链接输入，但内容身份、必需依赖闭包、披露规则和权限收敛必须保持可验证。

链接与 strip

  • 硬约束和接口冲突导致链接失败；权限默认取更严格交集；低信任对象不能覆盖高信任策略。
  • 工作流默认保留命名空间，软偏好按显式优先级合并，链接器输出合并与冲突报告。
  • 开发对象与发布对象具有同一运行语义；发布形态通过删除非运行时调试信息获得。
  • strip 前后目标、约束、验收契约、依赖和披露行为必须满足规范化等价检查；只有等价规则冻结后，才能把运行语义哈希作为互操作要求。

设计方案：披露图与覆盖状态

当前提案使用五层披露视图：Discovery（可发现的名称与摘要）、Contract（接口、约束与验收契约）、Execution（执行所需结构）、Knowledge（按请求选择的知识）和 Debug（开发诊断与来源细节）。固定层级保证审计，查询驱动选择减少上下文。来源单元的覆盖状态拟区分已映射、重复、装饰、精确产物、未解决和冲突。

代码、数值表、公式、结构定义、逐字模板、法律条款和二进制资源等不可安全摘要的内容必须精确保留或由可信外部资源承载。

受控披露

  1. 用户请求、Runtime 的 Capability Request 或预定义工作流触发披露需求，但都不能直接读取共享内容。
  2. Agent Harness 生成带会话、对象、请求目标和预算身份的 Disclosure Request；Noema Object System 检查 Trigger（对象中预定义的披露条件）、权限与信任策略，并计算依赖闭包。
  3. 对象系统把相关契约与硬约束强制加入，再在预算内建立 Knowledge 与 Example 的只读披露视图。
  4. Fulfillment Runtime 只消费已经授权的披露视图，不自行扩展依赖或上下文。
  5. 必需内容超过运行端限制时，Agent Harness 只能按预定义拆分规则保持原目标、硬约束、验收契约和覆盖闭包，无法保持时必须拒绝。

待定事项：披露 ABI

节点类型、Section 名称、状态哈希、预算单位、继续句柄、检查点与消费端确认协议仍是待验证设计。跨轮 Bundle 的字段示意不能视为已冻结 ABI。

后续计划：长上下文编译

来源图、局部抽取、全局链接、定向回读、Coverage Audit 与 Disclosure Planning 将分阶段实现；无法安全拆分或可靠理解的输入必须报错，不得静默截断。

设计概要：发布闭包与研究验证

HOBJ 发布闭包由主对象、必需依赖、版本与内容摘要、权限和信任策略、披露图、验收契约、评价器以及必需精确载荷组成。可选依赖必须声明缺失行为；动态依赖必须受配置、锁定和运行授权约束。

验证工作需要回答三类问题：语义完整性是否优于普通压缩或摘要；渐进式披露能否在不丢失安全约束的前提下降低上下文成本；不同 Noema Object System 实现是否能对同一 HOBJ 得出一致的必需闭包、披露视图与拒绝结论。每项实验都应包含长上下文基线、对抗遗漏、权限冲突、预算不足和不可安全摘要材料。
