# Noemion Pages Repository Rules

- 当前尚未进入代码开发阶段。除非用户以后明确宣布进入代码开发阶段并指定范围，本仓库只维护哲学与语言研究、术语、架构、规范、威胁模型、验证方案、公开文档和网站；不得新增或迁入任何实际项目组件实现。
- 门户主标题与承担核心主张的大型标题不得使用逗号、句号等标点制造停顿；应按完整语义分行，以排版层级形成节奏。标题必须先说明 Noemion 定义的对象和基础作用，宏大定位只能写成项目使命，不能暗示尚未完成的规范、组件或生态已经建成。
- 当前禁止 `implementation/`、生产 crate、可执行 CLI、Poiet/Theor/Praxor 源码、运行时、实现型 GitHub Actions、fuzz target、安装包和二进制发行物。不得把语言原型、规范向量检查器、Python 契约测试或页面构建写成“安全核心已经实现”。
- 现有 `experiments/` 与规范向量只能作为待重新审查的历史研究材料；不得扩展、运行后形成新的实现结论或作为进入组件开发的默认授权。是否保留这些材料由本轮阶段回退审计决定。
- 网站与规范的自动检查可以继续运行，但只证明页面、登记、条款和向量之间的一致性。组件开发阶段必须由用户明确开启，不能由 ADR、路线图、测试计划或“第一纵向切片”等文字自动触发。
- 本目录对应 `https://github.com/Noemion/noemion.github.io`，发布地址为 `https://noemion.github.io/`；仓库名和站点域名使用全小写。
- 本仓库是 GitHub Pages 的 Jekyll 源目录。保留 `_config.yml` 与 `.github/workflows/pages.yml`，不得添加 `.nojekyll`。
- 站点必须由 Jekyll 生成：公共外壳只维护在 `_layouts/` 与 `_includes/`；普通 HTML 源只保留 Front Matter 和职责对应的 `<main>` 正文，手册 Markdown 只保留 Front Matter 与正文，不复制 `<head>`、站点头部、面包屑、分页、侧栏或页脚。
- 修改前先读 `README.md` 与 `sitemap.md`；`sitemap.md` 是唯一正式 HTML 路由注册表，README 维护页面角色、模块化内容规范和维护规则。
- 当前工作环境资源有限；本地预览服务、浏览器标签、临时进程和其他验收资源只在实际需要时创建，使用完成后立即关闭或停止，不得为了可能的后续复用长期占用。后续确有需要时重新创建。
- 当前处于开发阶段，不保留旧 HTML 路径、别名、重定向或兼容垫片；路由变化同步 README、共享目录配置、入口和交叉链接。
- 全站统一采用 Noemion 设计系统：共享顶部悬停展开导航、四列发现页脚、可持久化的 `Light / Dark / System` 主题选择、约 1200px 居中画布、编辑式排版、横向内容条目与快速微交互；不同页面角色分别使用列表、文章、Endem 应用和文档范式，不套用同一内容模板。视觉识别使用自有语义频谱、制品记录、证据状态、内容和产品状态语言。
- 全部正式页面必须按 `design-system/geometric-layouts.md` 继承所属模块的几何母题；同一模块保持一致，目录、专题、Endem 应用和手册只按页面角色调整面积与密度。三角形、梯形、平行四边形、折页和缺口形状必须对应真实内容关系，不能为了新奇堆叠装饰，也不能裁切正文、交互命中区或规范性信息。
- 分析哲学相关视觉统一遵循 `design-system/philosophical-visual-language.md`：只把表达与意义、命题与条件、部分与整体、主张与证据等项目相关区分转成克制的线条、节点和关系图；不得使用哲学家肖像、古典装饰或名言替代工程解释，也不得把哲学启发写成规范依据。
- 页面布局不得通过固定偏移制造没有标题、导航、状态或视觉内容的空白栏。普通专题应根据真实内容在交错论述、标题侧栏、全宽表格/流程、并排卡片、折叠问题、时间线和必要图片之间选择；同页可以组合多种版式，但移动端必须按语义顺序回到单列。手册目录栏、应用状态栏等有明确职责的侧栏不属于无意义留白。
- 可读性优先于填满画布：1200px 是页面画布而不是段落宽度，技术长文和 Endem 应用说明的普通正文保持约 700–760px 阅读列与 17–18px/约 1.75–1.8 行高；桌面可使用约 800px 主阅读列和约 380px 粘性信息栏，但信息栏必须从所属页面角色的引言区起承担目录、状态、关键制品、输入输出或后续入口等真实职责，内容不足时回到单列。具体尺度与验收读取 `design-system/readability.md`。
- 页面结构不得使用通用营销式首屏组件。门户、目录、专题、Endem 应用和手册分别维护门户引言、目录引言、专题引言、应用引言和手册引言；标题、摘要、状态标记与模块图形归属对应页面角色，不建立跨角色的全局首屏接口。
- 页面可以使用真实图片或生成式视觉，但必须先判断图片是否承担解释、辨识或氛围职责；优先检索许可清晰的素材库并记录来源与许可，现成素材无法准确表达项目语义时再生成项目定制图片。公开仓库不得记录模仿或借鉴目标，图片中的文字优先使用英文或由 HTML 覆盖；所有图片必须本地化、压缩、提供尺寸和合适的替代文本，并按 `design-system/images.md` 处理裁切、明暗主题描边、动效与减少动态效果。
- 全站正文必须先回答读者问题再引入项目术语：先交代现实场景、核心缺口和直白结论，再说明哲学启发、工程映射与技术名称；思想来源、工程类比、规范要求和已验证事实必须明确区分。
- 哲学一致性是公开内容的首要标准，可读性紧随其后。每个正式页面必须让第一次接触 Noemion 的读者仅凭当前页面理解标题、首段、专名首次定义和下一步；不能要求读者先掌握整套词汇。长句只保留一个主要判断，连续三个以上动作、条件或例外改用步骤、列表或表格。
- 《逻辑哲学论》等著作只允许使用能够直接澄清当前问题的短引文；引文必须标出作者、书名和命题编号，随后用直白中文说明工程启发与不采用的部分。不得用引文替代规范、不变量、错误语义或证据，也不得把一种哲学解释写成软件事实。
- `spec/endem-scenarios.md` 是六语义面、未知分类、根拆分和满足判断的非规范自然语言设计审查语料。相关语义变化必须同步至少一个支持案例、一个反例或一个明确未决场景；案例只有转化为唯一条款、登记验证方式并形成正反向量后才可能成为符合性要求，不能被描述为解析器、可执行测试或组件证据。
- 所有正式文案、组件、制品、动作、目录和文件命名统一遵循 `design-system/language-and-naming.md` 与命名 ADR；中文使用主动语态和现在时，明确输入、动作、结果与失败，禁止无主语被动句、名词堆叠和管理套话。旧名称与旧路由不保留兼容入口。
- Noemion 是项目、新领域与社区的总名和品牌；为避免品牌侵入工程边界，它不进入格式名、对象类型、命令、ABI、魔数或组件前缀。Endem 是最小、独立有效且可验证的期望终态单元，正式扩展名为 `.endem`；每个 Endem 使用 `rhem/semion/skena/telis/krin/apor` 六个语义面并只有一个根 `skena`。唯一公开命令行为 `endem`，动作固定为 `poie`、`elenk`、`pleko`、`tasse`、`sphra`、`theor`、`praxe` 与 `peira`。不得建立旧字段或旧命令兼容入口。
- 项目级标题、门户主张、简介、入门入口和一级导航必须先以 Noemion 指代整个项目与领域，再按需要引出 Endem 等局部工程名；不得把 Endem 写成项目名、生态总名或 Noemion 的替代品牌。Endem 只属于最小制品、对应规范、生命周期、应用与 CLI 语境，其余专名也只承担各自已定义职责。
- 语言哲学相关设计严格遵守 ADR-0010 的工程边界：来源记号、意义投影、中性可能事态、目标方向、结构化观察与满足判断必须分层；`skena` 不携带目标力量，`telis` 第一阶段只使用 `kine/mene`，禁止事项由事态中的显式否定表达；逻辑形式由结构显示，不能由载荷自我宣称；`aseme`、语义 `apor` 与观察不足 `agno` 不得合并。公开页面必须同时说明《逻辑哲学论》不能直接推出软件规范，也不得声称格式编码了不可说之物。
- 正式术语必须同时保持制品与权限边界：Synem 是已解析并可封装的组合闭包，Dromen 是一次会话的实现态，Tekmor 是封装结构化观察 `phain` 与声明范围的证据。`theor` 背后使用与生产写入路径独立实现、只读且无写入/密钥/运行权限的 Theor；`praxe` 背后使用独立 Praxor 与最小能力域。统一 CLI 不得被描述成共享实现或共享权限。
- 面向读者的信息架构必须使用中文技术网站常见且职责明确的栏目名称：整体结构与关键决策写“架构设计”或“架构决策”，任务型内容写“指南”，按术语、制品或动作查找的内容写“参考指南”或“参考资料”，Endem 单一应用的连续专题写“使用手册”。不得用孤立的“架构”“文档”“参考”作为无法判断内容类型的页面主标题或导航名称；项目专有英文名和规范标识不受此限制。
- 正式 HTML 只面向开发者、普通用户和潜在使用者，不得出现提示词、生成过程、用户指令、内部 review、补写过程、页面制作或为了验收而编写等幕后表述；状态信息只说明产品、规范、实现和证据本身。公开产品文案不得使用“本页”“阶段门”“证据门”“放行”“退出证据”或未解释的 IPD 等内部制作与管理术语，必须改写为实际状态、开发计划、完成标准或验证结果；规范性条款、安全约束和学术论证中的专业术语不受此限制，但首次出现仍需直白解释。
- 网站是 Noemion 后续设计、实现与互操作工作的标准和规范入口；采用“直白解释 + 精确定义”双层表达，不能为了可读性删减术语边界、规范性强度、不变量、失败语义、成熟度或权威来源。
- 实现义务的唯一条款源维护在 `spec/*.md`，机器可读的规范、Profile、术语、条款、成熟度和验证映射维护在 `spec/registry.json` 与 `spec/profiles/`，正反向量维护在 `vectors/`。公开 HTML 只解释并链接精确源文件，不复制第二套条款；`vectors/semantic/` 的 JSON 是语义测试外壳，`vectors/wire/` 的十六进制表达 END-FMT 实验字节，`vectors/source/` 表达实验性 Poiet 来源清单，三者都不得冒充稳定 ABI。新增或修改条款时必须同步登记、向量或明确的计划验证，并运行 `tests/spec_contract_test.py`、`tests/semantic_vector_test.py`、`tests/wire_vector_test.py`、`tests/p1_payload_test.py` 与 `tests/source_manifest_test.py`；向量只有被执行并比较预期后才能称为验证证据。
- `experiments/` 只保存当前冻结、可删除的历史非生产研究材料。代码阶段开启前不得自动运行、扩展或据此形成新的实现结论；历史结果只能说明当时有限输入下的观察，不能用源码行数、单次速度或“内存安全”标签替代未来边界验证。
- ADR-0012 记录未来 Poiet 与生产读取核心的 Rust 1.97.0 设计基线。用户开启代码阶段后，仍须重新评审工具链版本；`forbid(unsafe_code)`、零第三方 crate、显式 checked arithmetic、release 溢出检查、工具链锁和 Theor 独立解析路径都是未来实现约束，不表示组件已经存在。
- ADR-0013 规定 `profile_id=2` 的 END-P1 是未来首条实现路径的设计 Profile；END-P0 只保留为结构实验。用户开启代码阶段后，未来实现不得增加未登记字段、复用 JSON 键名作为线格式、支持未定义的 mene 区间，或把 P0 结构接受升级为语义接受。
- ADR-0014 与 `spec/endem-source-manifest.md` 规定首个 Poiet 来源清单；它只是一项可替换的实验输入，不是新的哲学专名、稳定源语言或 Endem 身份。任何语法变更必须同步 `vectors/source/` 并证明仍确定性映射到同一 END-P1；正式来源语言出现后直接删除该入口。
- 架构组件、动作职责和页面组件都必须说明存在必要性、自身价值、上游输入或触发者、下游消费者、与相邻组件的边界，以及可以省略、合并或停止建设的条件；不得仅为架构图对称、目录完整或视觉填充保留无人消费的产物、重复动作或无信息价值的界面元素。
- Praxor 外侧的控制平面负责上下文装配、类型化能力、策略检查、观察反馈、预算与人工升级，不属于确定性 Poiet。模型只能提出不可信来源候选、计划和能力参数；不得决定 Endem/Synem 规范字节、修改已封装或已签名制品、自行扩大权限、删除 `apor` 项或宣告验收通过。涉及智能体工程的页面必须同时说明 Praxor、控制平面、Tekmor 与人工验收的责任边界。
- 外部书目、论文、规范、下载和其他资源链接必须使用能说明内容或来源的可读标签，不得把完整 URL 当作正文链接文字。链接目标仍保留原始地址，读者可以通过浏览器检查；内部路由同样使用能说明目标内容的标签。
- Endem 应用入口与使用手册分离：`endem/index.html` 是唯一应用项目页，`endem/docs/*.md` 是唯一 CLI 手册源。不得再创建并列工具树、旧工具路由或兼容入口。
- `endem/index.html` 必须保持 HTML 源，并按 `design-system/internal-tools.md` 使用统一应用视觉；`endem/docs/*.md` 使用纯通用 Markdown 正文。八个动作可以拥有有限的状态色和图形强调，但不能被包装为八个独立应用、命令品牌或权限域。
- Endem 应用页“继续阅读”章节只放置可点击资源入口，不得在卡片之后追加孤立的候选接口。接口状态、开发顺序或候选契约必须放在“当前状态”或“它不会做什么”的明确上下文中，且不能与全站未发布声明重复。
- 手册与指南正文的权威源文件必须使用 Markdown；Jekyll 负责把 Markdown 转换为正式 `.html` 路由。正文只允许通用 Markdown 语法，不得包含原始 HTML 或 Kramdown 专有属性。不得把构建生成的 HTML 当作正文维护入口，也不得同时保留内容重复的 `.md` 与手写 `.html` 两套权威源。新增页面只通过 Markdown Front Matter 加入手册，目录、索引和分页必须根据 `manual_id`、分组与顺序动态生成。
- 全站样式只在 `assets/style.css` 和 `assets/directory.css` 维护；正式导航数据只在 `_data/navigation.yml` 与 `_data/manuals.yml` 维护，服务端导航由 `_includes/` 生成，按需交互由 `assets/site.mjs` 与 `assets/modules/*.mjs` 维护。禁止页面级 CSS、页面内交互脚本和复制目录链接。
- 全站顶部一级入口在悬停与键盘聚焦时展开分组面板；卡片标题、箭头和表面必须提供一致悬停反馈，并支持 `prefers-reduced-motion`。页面类型映射、视觉元素和验收细则维护在 `sitewide-design-system.md`，README 只保留公开层面的概要。
- 修改任何页面、布局、目录、CSS 或交互动效前，必须先读取 `design-system/README.md`，再按其中的路由表读取目标 `page_role` 或组件对应的设计文档；改动跨越多个页面角色时必须同时遵守所有相关文档。新增页面范式时先新增设计文档和路由，不得只在实现中形成隐式规则。
- 公开页面不得把设计提案、候选名称、未来能力、论文、专利、软著或标准化意图表述为已实现或已获认可的成果。
- 规范、登记或向量变化后先运行 `python3 tests/spec_contract_test.py`、`python3 tests/semantic_vector_test.py`、`python3 tests/wire_vector_test.py`、`python3 tests/p1_payload_test.py` 与 `python3 tests/source_manifest_test.py`；路由或页面角色变化后运行 `python3 tests/site_quality_test.py`。发布前构建 `_site`，再运行 `python3 tests/site_quality_test.py _site` 并检查浏览器实际渲染。
- 网页改动完成且源码检查、Jekyll 构建、构建产物检查与浏览器验收均通过后，必须主动提交并推送到 GitHub，确认 GitHub Pages 构建状态，并向用户提供可直接查看的在线地址；除非用户明确要求只保留本地改动。发布所需的认证、权限或工具缺失时必须立即说明阻塞，不能把“本地完成”当作最终完成。
- 内容审查状态记录在 `content-quality-audit.md`；项目级要求变化时同步本文件或 README，避免规则散落。
