项目/Endem

endem 目标制品应用

设计中的单一命令入口将贯穿目标制品的形成、检查、组合、发布和受控实现。

Application DesignActions Specified尚未实现尚未发布

它解决什么问题

Endem 处理目标契约，而不是机器指令。用户需要沿同一制品生命周期完成工作，并在每次状态变化时得到明确结果或失败原因。

传统对象工具把汇编、查看、链接和打包拆成许多程序。Noemion 借鉴这些职责边界，但不照搬工具数量。

endem 是唯一公开 CLI。它的五个动作共享术语、诊断和发行入口，但 Ktisor、独立 Theor、隔离 Drasor 和外部签名系统继续保持不同实现与权限边界。

当前状态

当前没有可执行组件。现有材料定义未来 endem 应怎样读取来源、形成 END-P1、分离生产检查与独立 Theor。参数、退出状态、安装包和稳定 ABI 尚待确定。

  • ADR-0012 只把 Rust 1.97.0、禁止 unsafe 和依赖边界记为未来实现约束。
  • END-P1 字节只作为规范向量，不是 Ktisor 的产出证据。
  • elenk 与独立 theor 目前只有职责、输入输出和验证要求。
  • 没有实现仓库、安装包、签名或跨平台构建，因此当前不是可下载版本。
  • pleko 等真实多 Endem 案例与最小格式通过双实现验证后开发。
  • drase 要等待运行消费者和隔离边界；派生封装与外部签名没有公开动作承诺。

它怎样工作

ktise · 形成 nascent Endem
elenk · 生产验证
pleko · 形成 coherent / Synem
theor · 独立只读观察
外部签名 · 独立集成
drase · 隔离实现
Iknem · 有界证据
权威 · 最终决定
  • ktise 是唯一规范写入入口；模型或外部前端只能提供候选。
  • elenk 使用 Ktisor 的生产读取路径，并按结构、语义、状态、完整性和策略分层报告。
  • pleko 使用内容身份和显式策略解析引用，不能读取环境隐式路径。
  • 物理封装由对应生产动作在 Profile 锁定后承担；外部签名集成不持有 Ktisor 写入权。
  • theor 使用不共享 Ktisor 解析代码的 Theor；drase 启动不拥有写入器的隔离 Drasor。

它不会做什么

  • 不把任意自然语言、模型 JSON 或结构化输出直接视为合格 Endem。
  • 不让 theor 的可读视图替代生产 elenk 或生成可信句柄。
  • 不提供可以任意删除、改名或重写 semion、skena、telis、krin、apor 的通用变换命令。
  • 不通过环境变量、当前目录或网络“最新版本”解析正式依赖。
  • 不持有发布私钥；没有外部签名响应时只能保留未签候选。
  • 不让 drase 后端持有实时能力句柄、改写 attested 制品或自行宣布 accepted。
  • 不把 Iknem、签名、模型评分或工具数量描述为数学证明或成熟实现证据。
