项目/Endem

endem · 目标制品工具

用一个命令入口形成、检查、绑定、发布、独立查看和隔离运行 Endem/Weave。

Single CLI8 CommandsDesign StageNo Executable Yet

它解决什么问题

传统对象工具常把汇编、查看、符号、链接、裁剪、打包和执行拆成许多命令。Endem 处理的是目标契约而不是机器指令；用户真正需要的是沿同一制品生命周期完成工作，并在每次状态变化时得到明确失败。

endem 是唯一公开 CLI。它的八个子命令共享术语、诊断和发行入口，但 Core、独立 Reader、隔离 Runner 和外部签名系统继续保持不同实现与权限边界。

当前状态

设计阶段，尚未提供可执行程序。Endem、Weave、Frame、Witness、五组投影语义、三个状态、.endem 扩展名、命令名和职责已经采用；参数、退出状态、安装包和稳定 ABI 尚未冻结。

  • 首个实现只覆盖 form、check、see 与 test。
  • bind 等真实多 Endem 案例与最小格式稳定后开发。
  • pack、seal 和 run 等发布与安全边界冻结后开发。

它怎样工作

form · 形成 open Endem
check · 生产验证
bind · 形成 bound / Weave
pack · 冻结候选
seal · 核对外部签名
see · 独立只读观察
run · 隔离实现
test · 一致性验证
  • form 是唯一规范写入入口；模型或外部前端只能提供候选。
  • check 使用 Core 的生产读取路径，并按结构、语义、状态、完整性和策略分层报告。
  • bind 使用内容身份和显式策略解析引用，不能读取环境隐式路径。
  • pack 产生不可变候选和 Signing Request；seal 只核对并附加外部响应。
  • see 使用不共享 Core 解析代码的 Reader；run 启动不拥有 Writer 的隔离 Runner。

它读取什么，产生什么

入口主要输入主要输出
formSource Card、投影候选、Binding Decision、规范版本open Endem、Form Witness、诊断
checkEndem/Weave、验证层、限制和信任材料分层结论、Check Witness；内部 Verified Handle
bindEndem 集合、Binding Policy、Dependency Lockbound Endem 或 Weave、Binding Witness
packbound 制品、发布 Profile、覆盖与调试策略不可变候选、manifest、Signing Request
seal原候选、原请求、Signature Responsesealed Endem/Weave、Seal Witness
see任意原始制品字节、视图和预算只读文本/结构视图、差分和诊断
runsealed 制品、Execution Policy、后端与能力目录Frame、Run Record、Run/Decision Witness
test规范向量、实现构建、恶意语料和复现配置一致性、差分、模糊测试与复现报告

它不会做什么

  • 不把任意自然语言、模型 JSON 或结构化输出直接视为合格 Endem。
  • 不让 see 的可读视图替代生产 check 或生成可信句柄。
  • 不提供可以任意删除、改名或重写 mean、case、when、open 的通用变换命令。
  • 不通过环境变量、当前目录或网络“最新版本”解析正式依赖。
  • 不持有发布私钥；没有 Signature Response 时只能保留未签候选。
  • 不让 run 后端持有实时能力句柄、改写 sealed 制品或自行宣布 accepted。
  • 不把 Witness、签名、模型评分或工具数量描述为数学证明或成熟实现证据。
