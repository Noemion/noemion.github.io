项目/Endem

endem · 目标制品应用

一个命令入口贯穿目标制品的形成、检查、组合、发布和受控实现。

Single CLI8 ActionsDesign StageNo Executable Yet

它解决什么问题

Endem 处理目标契约，而不是机器指令。用户需要沿同一制品生命周期完成工作，并在每次状态变化时得到明确结果或失败原因。

传统对象工具把汇编、查看、链接和打包拆成许多程序。Noemion 借鉴这些职责边界，但不照搬工具数量。

endem 是唯一公开 CLI。它的八个子命令共享术语、诊断和发行入口，但 Poiet、独立 Theor、隔离 Praxor 和外部签名系统继续保持不同实现与权限边界。

当前状态

设计阶段，尚未提供可执行程序。Endem、Synem、Dromen、Tekmor、六个语义面、三个状态、.endem 扩展名、命令名和职责已经采用；参数、退出状态、安装包和稳定 ABI 尚未冻结。

  • 首个实现只覆盖 poie、elenk、theor 与 peira。
  • pleko 等真实多 Endem 案例与最小格式稳定后开发。
  • tasse、sphra 和 praxe 等发布与安全边界冻结后开发。

它怎样工作

poie · 形成 nascent Endem
elenk · 生产验证
pleko · 形成 coherent / Synem
tasse · 冻结候选
sphra · 核对外部签名
theor · 独立只读观察
praxe · 隔离实现
peira · 一致性验证
  • poie 是唯一规范写入入口；模型或外部前端只能提供候选。
  • elenk 使用 Poiet 的生产读取路径，并按结构、语义、状态、完整性和策略分层报告。
  • pleko 使用内容身份和显式策略解析引用，不能读取环境隐式路径。
  • tasse 产生不可变候选和 Signing Request；sphra 只核对并附加外部响应。
  • theor 使用不共享 Poiet 解析代码的 Theor；praxe 启动不拥有写入器的隔离 Praxor。

它读取什么，产生什么

入口主要输入主要输出
poie受控来源、投影候选、语义决定和规范版本nascent Endem、Poie Tekmor、诊断
elenkEndem/Synem、验证层、限制和信任材料分层结论、Elenk Tekmor 和内部验证句柄
plekoEndem 集合、组合策略和依赖锁coherent Endem 或 Synem、Pleko Tekmor
tassecoherent 制品、发布配置、覆盖与调试策略不可变候选、清单和签名请求
sphra原候选、原请求和外部签名响应attested Endem/Synem、Sphra Tekmor
theor任意原始制品字节、视图和预算只读文本/结构视图、差分和诊断
praxeattested 制品、运行策略、后端与能力目录Dromen、运行记录、运行与决定类 Tekmor
peira规范向量、实现构建、恶意语料和复现配置一致性、差分、模糊测试与复现报告

它不会做什么

  • 不把任意自然语言、模型 JSON 或结构化输出直接视为合格 Endem。
  • 不让 theor 的可读视图替代生产 elenk 或生成可信句柄。
  • 不提供可以任意删除、改名或重写 semion、skena、telis、krin、apor 的通用变换命令。
  • 不通过环境变量、当前目录或网络“最新版本”解析正式依赖。
  • 不持有发布私钥；没有外部签名响应时只能保留未签候选。
  • 不让 praxe 后端持有实时能力句柄、改写 attested 制品或自行宣布 accepted。
  • 不把 Tekmor、签名、模型评分或工具数量描述为数学证明或成熟实现证据。
